# CLAUDE.md

行为准则，减少常见的 LLM 编码错误。

**权衡：** 这些准则偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 先思考，再编码

**不要假设。不要掩饰困惑。呈现权衡。**

实现之前：
- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种理解，全部呈现出来——不要默默选择。
- 如果有更简单的方案，说出来。必要时提出反对意见。
- 如果有不清楚的地方，停下来。指出困惑之处。然后问。一次问不清就再问，不要自己猜。

## 2. 上下文先行

**先拿到刚好够用、刚好相关、可解释的上下文，再动手。**

实现之前先确认：
- 项目入口、模块边界、相关配置和可运行/测试命令。
- 直接相关文件、影响判断文件、背景参考文件分别是什么。
- 看过但排除的文件/路径，并说明为什么暂不纳入。
- 如果涉及 DB/API/权限/状态机/跨模块变更，未完成上下文收敛前不要开始设计或编码。

跨后端接口、持久化、前端展示、导出、测试的复杂链路变更，先输出覆盖表再实现。覆盖表至少包括：数据产生点、状态/字段定义、持久化位置、对外接口、前端展示、导出/报表、测试入口、明确排除项。每项都要写出具体文件/函数；找不到就标注“不确定/未找到”。

如果你声称解决了性能风险（例如 N+1 查询、重复扫描、阻塞 I/O），必须指出对应代码路径和验证方式；不要只在注释或汇报里宣称。

只读分析阶段不得写文件；如果需要保存报告，必须先确认。

上下文不要无限扩大。先按“项目地图 → 变更邻域 → 精准证据”收敛；如果范围继续变大，最多问 3 个问题让用户帮助缩小。

## 3. 简单优先

**用最少的代码解决问题。不做任何推测性设计。**

- 不添加未被要求的功能。
- 不为一次性代码创建抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景编写错误处理。
- 如果你写了 200 行代码，而 50 行就能搞定，重写它。

问问自己："资深工程师会认为这太复杂了吗？" 如果是，简化它。

## 4. 精准修改

**只改必须改的。只清理自己制造的混乱。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你不会那样写。
- 如果你注意到无关的死代码，提一下——不要删除它。

当你的修改产生了孤立代码：
- 删除因你的修改而变得未使用的导入/变量/函数。
- 不要删除原本就存在的死代码，除非被要求。

检验标准：每一处修改都应该能直接追溯到用户的需求。

## 5. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让测试通过"
- "修复 bug" → "编写能复现问题的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

明确的成功标准让你能够独立循环。模糊的标准（"让它能用"）则需要不断确认。

验证失败时，先读首个编译/测试错误并归因，不要把未确认的失败归因为权限、环境或工具问题。汇报失败必须包含：实际命令、退出码、首个错误、下一步修复判断。

## 6. 改前确认语义约定

**修改现有字段/类型时，先回看原定义中所有合法值。不要凭直觉赋值。**

修改现有代码时：
- 改 status/enum/常量前，先回看原始定义的所有合法值
- 如果原定义中没有你需要的值，先修改定义，再使用
- 不要假设「退款了所以 status 该是 X」——去确认原代码里 X 是否合法
- 新增 public class/record/interface 前，确认语言和框架的文件约束。例如 Java 的 public 顶级类型必须与文件名一致；仅在当前类内部使用时，优先使用 private static 嵌套类型。

检验标准：你赋的每一个值，都应该能在原定义中找到对应。

## 7. 测试策略

**知道什么该测，什么可以不测。**

必须写测试：
- 业务逻辑、算法、状态转换
- 修复 bug 时（先写复现测试）
- 公共 API 或库代码
- 任务明确要求补测试时，不能只跑现有测试；必须新增或修改至少一个直接覆盖变更的测试，除非说明为什么不需要。

可以不写测试：
- 一次性脚本、原型验证
- 纯 UI 展示层
- 配置文件、声明式代码

不确定时，询问用户。

## 8. 规则适用性

**不同任务规模，检查不同规则：**

小任务（≤20 行改动）：
- 必检：[先思考再编码]、[上下文先行（仅确认目标文件和直接引用）]、[精准修改]、[改前确认语义约定]
- 可跳过：其他规则

中等任务（介于小和大之间）：
- 必检：前 7 条规则
- 按需检查：本条

大任务（>5 个文件或 >50 行改动）：
- 必检：所有规则
- 若涉及后端接口、DB、前端、导出、测试中的 3 类及以上，按复杂链路处理：先覆盖表，后实现。

任务规模判断在开始时一次性完成，后续按该规模执行。

---

**这些准则有效的标志：** diff 中不必要修改更少，因过度复杂导致的重写更少，澄清问题在实现之前提出而非犯错之后，关键上下文在动手前收敛，危险任务被及时拦截。
