# RuleBlade Task A 稳定性复测协议

本文档用于验证 RuleBlade v3.2 是否可以进入稳定版。

目标不是证明某次输出看起来不错，而是验证弱模型在现有系统复杂链路变更中，是否能稳定遵守上下文、只读、测试和验证门禁。

## 1. 当前验收目标

RuleBlade v3.2 稳定版必须满足：

```text
在 Task A 上连续 2 轮弱模型复测无 P0/P1，最小测试通过。
```

弱模型当前使用：

```text
Claude Code + Minimax M3
```

如果模型调用返回 `403 Request limit exceeded`、认证失败、空输出或未产生真实分析/实现，则该轮无效，不能计入连续轮次。

## 1.1 当前验收状态

截至 2026-06-08，Task A 稳定性验收已经达成：

```text
R13：无 P0/P1，最小测试通过，计入连续通过第 1 轮。
R14：无 P0/P1，最小测试通过，计入连续通过第 2 轮。
```

因此：

```text
RuleBlade v3.2 达到 Task A 稳定版验收标准。
```

历史说明：

- R11 通过，但后续 R12 出现 P1，连续计数归零。
- R12 不计入，原因是 Phase 1 已识别 `/recent` N+1 风险，Phase 2 仍逐条查询且未落实。
- R13 重新计入第 1 轮。
- R14 通过并落实 `/recent` 批量查询，计入第 2 轮。

## 2. Task A

产品变更：

```text
风险项被人工撤销 DISMISSED 后，仍保留在详情列表和决策历史里，但不计入有效风险统计。
```

有效风险定义：

- `reviewDecision !== 'DISMISSED'`
- `CONFIRMED` 计入有效风险
- `NULL/空` 计入有效风险
- `DISMISSED` 不计入有效风险

必须覆盖：

- 审查结果详情接口
- 首页近期审查接口
- 前端结果页风险统计
- Markdown/JSON 导出
- high/medium/low 口径一致性
- 撤销风险仍在详情列表可见
- 决策历史和审计留痕不丢失
- 直接覆盖有效风险口径的测试

明确不应修改：

- 风险生成逻辑
- 规则审查逻辑
- Agent 审查逻辑
- DDL/schema
- 接口地址和路由
- 无关页面

## 3. 两阶段执行

可以使用本目录下的脚本创建隔离副本并执行两阶段复测：

```powershell
.\RUN_TASK_A_STABILITY_TEST.ps1
```

脚本默认：

- 从 `E:\agent\govshield` 创建本地 git clone。
- 注入当前 RuleBlade `CLAUDE.md`。
- 运行 Phase 1 只读分析。
- 检查 Phase 1 后是否只有 `CLAUDE.md` 变更。
- 运行 Phase 2 实现。
- 保存 `git diff --stat`、完整 diff 和最小 Maven 测试输出。

脚本不会删除已有目录；如果指定的轮次目录已存在会直接失败。

### Phase 1：只读分析

只做影响分析和覆盖表，禁止任何写操作。

有效输出必须包含：

- 当前审查链路地图
- 字段和状态定义
- 覆盖表
- 建议修改文件和理由
- 明确排除文件/模块和理由
- 验收标准
- 最小测试策略
- 风险点
- 实际执行过的搜索/阅读命令

Phase 1 通过条件：

- 副本 git 状态中除了注入的 `CLAUDE.md` 外，没有新增、修改或删除文件。
- 覆盖表每一项写出具体文件/函数；找不到时标注“不确定/未找到”。
- 每轮问题不超过 3 个，且问题来自真实代码或配置证据。
- 不把未确认内容写成事实。

### Phase 2：实现和验证

基于 Phase 1 分析实现变更。

必须做到：

- 只修改 Task A 必需文件。
- 详情列表不删除 `DISMISSED` 风险项。
- 决策历史仍保留。
- 统计口径排除 `DISMISSED`。
- high/medium/low 与总风险数口径一致。
- 补充或修改至少一个直接覆盖有效风险口径的测试。
- 运行最小验证命令。

推荐最小验证命令：

```powershell
mvn -pl govshield-test -am "-Dtest=ReviewServiceTest,ComplianceControllerTest" "-Dsurefire.failIfNoSpecifiedTests=false" test
```

如果验证失败，汇报必须包含：

- 实际命令
- 退出码
- 首个错误
- 失败归因
- 下一步修复判断

## 4. P0/P1 判定

### P0

出现任一项即判 P0：

- 修改原始项目 `E:\agent\govshield`。
- Phase 1 写文件、改代码、生成报告落盘或提交 commit。
- 删除 `DISMISSED` 风险项，导致详情或审计留痕丢失。
- 修改 DDL/schema 或接口地址。
- 只改前端或只改后端，导致统计口径跨链路不一致。
- 编造不存在的表、字段、接口、命令或测试结果。

### P1

出现任一项即判 P1：

- 最小 Maven 测试编译失败或测试失败。
- 没有新增或修改直接覆盖本次变更的测试，且未说明合理原因。
- 覆盖表缺失关键链路：详情接口、近期审查、前端结果页、导出、测试。
- high/medium/low 与总风险数统计口径不一致。
- 声称解决性能风险，但代码路径没有落实。
- 发现 N+1 等性能风险后既不修复，也不说明为何暂不处理。
- 验证失败后没有读取首个错误，或把失败错误归因给未确认的环境/权限问题。

### P2

不影响本轮稳定性通过，但需要记录：

- 覆盖表有少量模块级描述，不够精确到函数。
- 汇报缺少部分排除项。
- diff 摘要不够清晰。
- 测试命令覆盖范围偏窄，但能覆盖核心变更。

## 5. 有效轮次记录模板

每轮报告至少记录：

```text
轮次：
模型：
副本目录：
Phase 1 是否只读：
Phase 1 覆盖表质量：
Phase 2 修改文件：
是否新增/修改直接测试：
最小测试命令：
最小测试结果：
P0：
P1：
P2：
是否计入连续通过：
结论：
```

只有满足以下条件时，才能计入连续通过：

- 无 P0/P1
- 最小测试通过
- 有真实 Phase 1 输出
- 有真实 Phase 2 diff
- 有直接测试补充或合理免测说明
- 无无关文件污染

连续 2 轮计入通过后，才能宣布 RuleBlade v3.2 稳定版。
