# 律刃 RuleBlade

律刃是一组写给 AI 编码助手的通用行为规则。当前版本为 v3.3，包含 8 条编码规则和 1 条中文表达要求。

它解决的不是“模型会不会写代码”，而是更常见的执行问题：没有弄清需求就动手、漏查调用方、顺手修改无关代码、把测试失败归咎于环境，或者用生硬的内部术语向用户解释结果。

## 适用范围

律刃适用于 0→1 新系统、已有系统变更、重构、Bug 修复和测试补充。复杂变更的验证主要在已有系统中完成，但这只是测试场景，不代表规则只能用于旧项目。

它可以单独使用，也可以和 IntentAnchor、Pathfinder、ImpactRadar 等 Skill 配合：

- 律刃规定通用的编码行为。
- 专项 Skill 负责各自的需求澄清、项目摸底或影响分析流程。

## 使用方法

Claude Code 用户可以把 [CLAUDE.md](CLAUDE.md) 放到目标项目根目录。Codex 用户可以使用同一份内容作为 `AGENTS.md`。

这套规则偏向谨慎。小改动会按第 8 条缩短检查范围；涉及数据库、接口、权限、状态机或多个模块时，则要求先把影响范围查清楚。

## 当前 8 条规则

| # | 规则 | 主要要求 |
|---|---|---|
| 1 | 先思考，再编码 | 说明假设和歧义；重要问题不清楚时先问 |
| 2 | 上下文先行 | 先找入口、定义、调用方、测试和明确排除项 |
| 3 | 简单优先 | 不增加需求以外的功能、抽象或配置 |
| 4 | 精准修改 | 只改与需求直接相关的内容，不顺手重构 |
| 5 | 目标驱动执行 | 先定义成功标准；验证失败时读取首个真实错误 |
| 6 | 改前确认语义约定 | 修改状态、枚举、常量和公开类型前先读原定义 |
| 7 | 测试策略 | Bug 先写复现测试；测试粒度要与风险匹配 |
| 8 | 规则适用性 | 根据改动规模决定检查范围 |

此外，[CLAUDE.md](CLAUDE.md) 还要求面向用户的中文表达自然、直接。技术名词可以保留，但不能用“闭环、赋能、链路收口”之类词语代替具体动作和证据。

## 规则是怎样形成的

### 早期筛选

最初的实验比较了没有规则、4 条英文规则、21 条中文规则和 7 条精选规则。

| 版本 | 规则数 | 平均分 | 主要发现 |
|---|---:|---:|---|
| 无规则组 | 0 | 3.92 | 容易自行扩展需求 |
| v1 | 4 | 4.20 | 规则过于描述性，模型仍会自行选择需求解释 |
| v2 | 21 | 4.60 | 分数最高，但 14 条规则没有可归因的作用，内容过多 |
| v2.5 | 7 | 4.47 | 分数接近 21 条版本，并且是唯一主动向用户提问的组 |

早期结果说明，规则数量并不是越多越好。7 条版本已经保留了大部分收益，也更容易被模型完整执行。

### v3：修正状态值误判

v2.5 的测试中，模型在退款逻辑里直接把 `payment.status` 设成 `pending`，没有先确认原定义是否支持 `refunded`。v3 因此加入“改前确认语义约定”，要求修改状态、枚举和常量前先查看所有合法值。

同一版本还加强了第 1 条：一次提问没有消除歧义时，必须继续追问，不能问完后仍靠猜测继续。

### v3.1：增加上下文检查

后续测试发现，已有系统中的错误往往来自上下文不足，而不是代码生成能力不足。v3.1 增加“上下文先行”，要求在实现前确认：

- 项目入口、模块边界和运行方式。
- 目标定义、调用方、引用方、注册位置、生成文件和测试。
- 看过但暂不纳入的文件，以及排除原因。

这条规则只保留通用要求，不复制 ImpactRadar 的完整分析流程。简单任务仍只需确认目标文件和直接引用。

### v3.2：覆盖跨模块变更

v3.2 在 GovShield 上测试了一项跨后端接口、持久化、前端、导出和测试的变更：人工撤销风险后，详情和审计记录仍保留，但该风险不再计入有效统计。

最初测试暴露了四类问题：

- 影响分析完整，但新增 Java `public record` 的文件名不符合语言规则，编译失败。
- 模型把 Maven 的真实编译错误误报成权限问题。
- 声称解决 N+1 查询，但代码仍在循环中逐条查询。
- 用户要求补测试时，只运行了原有测试，没有新增直接覆盖本次变更的测试。

v3.2 没有增加第 9 条，而是把这些要求放进现有规则：复杂变更先列覆盖表；新增公开类型前确认语言约束；验证失败时读取首个错误；性能结论必须指出代码路径和验证方式；要求补测试时必须增加直接测试。

稳定性复测结果：

| 轮次 | 是否计入 | 结果 |
|---|---|---|
| R11 | 是 | 通过；新增直接测试，最小与完整测试通过 |
| R12 | 否 | 发现 N+1 风险但没有落实修复，记为 P1 |
| R13 | 是 | 通过；无 P0/P1，N+1 取舍有明确说明 |
| R14 | 是 | 通过；使用批量查询解决 N+1，前后端测试通过 |

R13 和 R14 连续两轮无 P0/P1，达到 [Task A 稳定性测试](TASK_A_STABILITY_TEST.md) 的验收条件。

### v3.3：把抽象要求改成可检查条件

v3.3 没有增加规则数量，只调整了 5 处表达：

| 位置 | 调整 |
|---|---|
| 第 1 条 | 区分“影响大且难以撤销，必须提问”和“影响小且容易恢复，说明理解后直接做” |
| 第 2 条 | 用普通中文描述上下文收集过程；范围继续扩大时，只问当前最影响推进的问题 |
| 第 5 条 | 没有运行环境时，明确降级为读代码、查引用和推演关键路径，不能虚构测试结果 |
| 第 7 条 | 纯函数用单元测试，跨模块状态变化用集成测试，对外接口使用契约测试或端到端测试 |
| 第 8 条 | 用改动行数、风险类型、删除或重命名、跨文件引用四项条件决定能否走小任务流程 |

Step 3.7 Flash 按 [v3.3 验证方案](TASK_V33_VERIFY.md) 运行了 6 个针对性场景，结果为 6/6 通过。测试覆盖提问阈值、上下文收集、没有运行环境时的验证说明、测试粒度，以及小任务流程的正确进入和退出。

## 为什么保持 8 条

早期实验中，规则从 7 条增加到 21 条后，平均分只从 4.47 增加到 4.60。新增内容越多，模型越难完整执行，单条规则的作用也越难判断。

因此后续版本遵循两个原则：

1. 新问题优先补进现有规则，不轻易增加规则数量。
2. 把要求写成可以观察和验证的动作，而不是抽象口号。

第 8 条会根据任务规模调整成本：

- 小任务只检查目标、直接引用、修改范围和原有语义。
- 中等任务执行前 7 条规则。
- 大任务执行全部规则。
- 同时涉及后端接口、数据库、前端、导出和测试中的 3 类及以上时，先列出覆盖表，再开始实现。

## 能力边界

律刃可以减少不必要修改、漏查引用和虚假验证，但不能代替业务知识。比如“退款后状态应该是什么”，最终答案仍要来自原始定义、业务规则和测试。

它也不能保证模型一定遵守文字指令。涉及数据库写入、生产环境或不可逆操作时，还应使用只读账号、工具权限和运行前 Hook 等系统级保护。

## 来源与测试范围

最初的 4 条英文规则参考了 multica-ai/andrej-karpathy-skills 的 [CLAUDE.md](https://github.com/multica-ai/andrej-karpathy-skills/blob/main/CLAUDE.md)。后续中文表达、规则取舍和测试方案由本项目根据真实编码任务调整。

“修改前检查调用方和引用方，再判断哪些必须同步修改”来自 [hxd-ggsddu](https://github.com/hxd-ggsddu) 提交的 issue。

主要测试记录：

| 范围 | 模型与日期 | 结论 |
|---|---|---|
| 早期规则筛选 | Claude Code v2.1.139（claude-opus-4-8），2026-06-01 | 7 条版本在规则数量和效果之间更合适 |
| v3.2 Task A | Claude Code + MiniMax M3，2026-06-08 | R13、R14 连续两轮无 P0/P1 |

这些结果只对应当时的模型版本、项目和任务，不能证明所有模型在所有代码库中都会得到相同表现。
