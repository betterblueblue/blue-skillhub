# 律刃 RuleBlade

> 8 条给 AI 编码助手用的核心行为规则，外加一条面向用户的中文表达要求。目标很朴素：少猜、少乱改、先拿对上下文、先验证，也把话说得像自然中文。

## 当前版本：v3.3 稳定版（五处补强 + 弱模型验证）

## 定位

律刃是通用 AI 编码行为规则，不绑定具体任务类型。它可以用于 0→1 新系统开发、已有系统变更、重构、修 bug、测试补充等场景，也可以搭配 `impact` 这类影响分析 Skill 使用。

v3.2 的稳定性复测是在 GovShield 已有系统复杂链路变更中完成的，因此它对已有系统里的多模块改动尤其有帮助；但这只是验证场景，不是适用范围上限。v3.3 的补强验证是在低成本弱模型（Step 3.7 Flash）上完成的，确认五处改动对弱模型也有效。

## 迭代历程

### v1 — 对照组（4 条英文规则）

来源：最初版参考了 multica-ai/andrej-karpathy-skills 中的 [`CLAUDE.md`](https://github.com/multica-ai/andrej-karpathy-skills/blob/main/CLAUDE.md)。律刃是在这份大佬版 4 条英文行为准则的基础上，做中文化、项目化实验和后续复杂链路复测后逐步迭代出来的。

| # | 规则 | 写法 |
|---|------|------|
| 1 | Think Before Coding | 英文，描述性 |
| 2 | Simplicity First | 英文，描述性 |
| 3 | Surgical Changes | 英文，描述性 |
| 4 | Goal-Driven Execution | 英文，描述性 |

**暴露的问题：**
- 任务 B 中删了原有代码又恢复，说明“精准修改”写得太抽象，没拦住多余操作
- 全程自问自答，从未向用户提问（"present multiple interpretations" 被理解为「列出理解然后自己选」）
- 平均分 4.2

### v2 — 实验组（21 条中文规则）

来源：在 v1 基础上扩展，覆盖开发阶段、测试策略、中断恢复、并行任务等。

**暴露的问题：**
- 21 条中 14 条没有带来可归因差异，更像噪音
- 查证习惯仍未被激活——规则被稀释了注意力
- 规则过多导致模型无法全部记住和遵守
- 平均分 4.6，但收益很低（每条新增规则仅贡献 0.009）

### v2.5 — 精选组（7 条中文规则，实验前版）

从 21 条中挑出 7 条核心规则，改成更直接的中文指令。

| # | 规则 | 改动 |
|---|------|------|
| 1 | 先思考再编码 | "present multiple interpretations" → 「如果不确定，就问」 |
| 2 | 简单优先 | 保留核心 |
| 3 | 精准修改 | 保留核心 |
| 4 | 目标驱动执行 | 保留核心 |
| 5 | 知道何时说不 | 新增 |
| 6 | 测试策略 | 新增 |
| 7 | 规则适用性 | 新增（元规则） |

**实验结果：**
- ✅ 查证习惯被激活！唯一在任务 B 主动提问的组
- ✅ 平均分 4.47，接近 21 条的 4.6，投入产出最好
- ❌ payment.status 隐性问题：退款后设为 'pending' 而非 'refunded'（业务理解错误，单靠行为规则管不住）
- ❌ 第 5 条「知道何时说不」全程未触发

### v3 — 律刃（7 条中文规则，实验后定稿）

基于 v2.5 实验结果做了两处调整：

**优化 1：第 5 条替换**

| 旧 | 新 |
|---|---|
| 知道何时说不（未触发） | 改前确认语义约定（针对隐性问题） |

理由：实验暴露的主要问题是 payment.status 被模型凭直觉赋值，没有回看原定义。新规则直接要求「赋值前必须回看原定义的合法值」。

**优化 2：第 1 条加强追问力度**

| 旧 | 新 |
|---|---|
| 如果有不清楚的地方，停下来。指出困惑之处。然后问。 | 如果有不清楚的地方，停下来。指出困惑之处。然后问。**一次问不清就再问，不要自己猜。** |

理由：追问不能只问一次就结束。实验中 7 条组问了问题，但没有继续追 status 歧义，加强后更容易覆盖这类场景。

### v3.1 — 上下文先行（8 条轻量补强）

来源：后续在 `impact` 影响分析 Skill 中反复验证上下文包能力时发现，AI 编码助手在已有系统中犯错，很多时候不是因为不会写代码，而是因为动手前没有拿到足够准确的上下文。

新增第 2 条：

| 新增规则 | 目的 |
|----------|------|
| 上下文先行 | 在实现前先确认项目入口、模块边界、相关配置、运行/测试命令、直接相关文件、调用方/引用方、影响判断文件、背景参考文件和排除项 |

这条规则不是把完整的 `Context Pack` 流程搬进通用行为准则。完整上下文包仍属于 `impact` 的专业流程；律刃这里只保留轻量原则：

```text
了解项目整体结构 → 定位变更涉及的模块 → 收集精准的文件级证据
```

这样做的原因：

- 通用行为准则不能太重，否则简单任务会被拖慢。
- 但中大型任务如果不先收敛上下文，很容易误改、漏改或过早设计。
- 改代码前不反查调用方和引用方，容易只改当前文件，留下接口、生成物、测试或注册点断裂。
- “看过但排除”的文件也要说明原因，避免后续反复搜索同一批无关内容。
- 涉及 DB/API/权限/状态机/跨模块时，未完成上下文收敛前不应开始设计或编码。

这和 Augment Code 在《[How to write good AGENTS.md files](https://www.augmentcode.com/blog/how-to-write-good-agents-dot-md-files)》里的观点一致：主指令文件不是越长越好，应保持短、高信号、可执行；复杂细节通过按需加载的参考文档或 Skill 承载。

v3.1 的定位：保留 v3 的实验结论，把上下文能力作为后续实战暴露出的关键补强项。

### v3.2 — 复杂链路门禁（8 条不变，硬闸补强）

来源：在 GovShield 审查链路上用 Claude Code + Minimax M3 做 Round 6 / Round 7 压测。测试任务是修改“人工撤销风险后不计入有效风险统计，但仍保留详情和审计留痕”的跨链路变更，涉及后端接口、持久化字段、前端结果页、首页近期审查、导出和测试。

Round 6 直接实现，no-context / with-context 都超时。Round 7 改成两段式：

1. Phase 1：只做影响分析和覆盖表，不许改代码。
2. Phase 2：基于各自分析结果实现。

**实验暴露的问题：**

- “上下文先行”能显著提升复杂链路分析质量，但不能保证最终代码一定可编译。
- with-context 组分析阶段明显更完整，能识别 medium/low 统计、导出、N+1、缓存等边界。
- with-context 组实现阶段设计更好，但把 Java `public record` 放进了不匹配的文件，导致编译失败。
- 模型把 Maven 失败误归因为“权限门控”，而实际首个错误是 Java public 顶级类型文件名不匹配。

**v3.2 的补强：**

| 位置 | 补强 |
|------|------|
| 第 2 条 上下文先行 | 复杂链路变更必须先输出覆盖表，再实现 |
| 第 5 条 目标驱动执行 | 验证失败必须读取首个错误并归因，不能随便甩锅环境 |
| 第 6 条 改前确认语义约定 | 新增类型、模块、导出项或公开 API 前，确认当前语言和框架的约束 |
| 第 8 条 规则适用性 | 涉及后端接口、DB、前端、导出、测试中 3 类及以上时，按复杂链路处理 |

v3.2 没有新增第 9 条。原因是规则数量继续增加会稀释注意力；这次是把真实压测暴露出的硬失败点补进现有规则，而不是回到 v2 的规则堆叠模式。

Round 8 用 v3.1 / v3.2 做同题回归后，v3.2 成功避免了 Java `public record` 文件名错误，也减少了临时脚本污染；但仍暴露出两个问题：

- 覆盖表有时会变短，只停在“需要确认”，没有稳定落到具体文件/函数。
- 模型可能声称“避免 N+1”，但实际代码仍是循环查询，只是注释写得像解决了。

因此 v3.2 继续做了两处文字收紧：

```text
覆盖表每项必须写出具体文件/函数；找不到就标注“不确定/未找到”。
如果声称解决了性能风险（如 N+1），必须指出对应代码路径和验证方式，不能只在注释或汇报里宣称。
```

Round 9 用 v3.2 原版 / v3.2 收紧版继续做同题回归后，收紧版的覆盖表具体度明显提升，最小 Maven 测试通过；原版则因为 `Map.of(...)` 参数过多编译失败。收紧版仍暴露出两个缺口：

- Phase 1 只读分析阶段写入了 `.scratch/round9-analysis.md`，虽然内容是分析报告，但仍属于写操作越界。
- 任务要求补测试时，模型只跑了现有测试，没有新增或修改直接覆盖本次变更的测试。

因此 v3.2 再补两条门禁：

```text
只读分析阶段不得写文件；如果需要保存报告，必须先确认。
任务要求补测试时，不能只跑现有测试；必须新增或修改至少一个直接覆盖变更的测试，除非说明为什么不需要。
```

这些仍属于 v3.2，不新增版本号。它们都在同一条“复杂链路门禁”上把验收条件写得更可检查，而不是增加新的规则类别。

### v3.2 稳定版验收

2026-06-08 继续用 Claude Code + Minimax M3 对 Task A 做稳定性复测。Task A 是 GovShield 审查链路上的复杂变更：

```text
风险项被人工撤销 DISMISSED 后，仍保留在详情列表和决策历史里，但不计入有效风险统计。
```

验收规则见 `TASK_A_STABILITY_TEST.md`：

- Phase 1 只读分析不得写文件。
- Phase 2 必须补直接测试。
- 最小 Maven 测试必须通过。
- 不能有 P0/P1。
- 需要连续 2 轮弱模型复测通过。

复测结果：

| 轮次 | 结果 | 说明 |
|------|------|------|
| R11 | 通过 | Phase 1 只读，补直接测试，最小与全量测试通过 |
| R12 | 不计入 | 已识别 `/recent` N+1 风险但实现仍逐条查询，判 P1 |
| R13 | 通过 | 无 P0/P1，最小/全量后端/前端 node 测试通过；`/recent` N+1 有明确取舍说明，记 P2 |
| R14 | 通过 | 使用 `findByReviewIdIn` 批量查询落实 `/recent` 性能风险，最小/全量后端/前端 node 测试通过 |

最终结论：

```text
R13 + R14 连续 2 轮无 P0/P1，最小测试通过。
RuleBlade v3.2 达到 Task A 稳定版验收标准。
```

### v3.3 — 五处补强（8 条不变，规则文字收紧）

来源：v3.2 稳定后，在低成本弱模型（Step 3.7 Flash）上做针对性场景测试时发现，几处规则的文字写法仍偏抽象，弱模型容易误解或无法触发正确行为。v3.3 不新增规则条目，只对现有文字做五处收紧。

| 位置 | 旧写法 | 新写法 | 针对的问题 |
|------|--------|--------|------------|
| 第 1 条 先思考再编码 | 只有「如果有不清楚的地方，停下来……然后问」 | 补「不确定且不可逆→必须停下来问；不确定但影响范围小、改了容易退回来→用一句话说明理解然后直接做」 | 「自行判断」太主观，弱模型要么什么都问、要么什么都不问 |
| 第 2 条 上下文先行 | 「项目地图 → 变更邻域 → 精准证据」；「按优先级提问」 | 改为「了解项目整体结构 → 定位变更涉及的模块 → 收集精准的文件级证据」；补「每次只问当前最阻塞的问题」 | 概念泄漏（skill 内部术语出现在通用规则里）；提问次数不灵活 |
| 第 5 条 目标驱动执行 | 只有「验证失败时先读首个错误并归因」 | 补「没有测试或运行环境时，验证降级到静态确认——读代码 + grep 引用检查 + 人工推演关键路径，并说明验证到哪个层级」 | 无测试环境时模型编造验证结果或假装有测试环境 |
| 第 7 条 测试策略 | 只有「必须写测试 / 可以不写测试」的分类 | 补「匹配测试粒度到风险面：纯函数和工具方法用单测；跨模块逻辑和状态转换用集成测试；对外接口契约用契约测试或 e2e」 | 模型对工具函数和接口用同一种测试方式 |
| 第 8 条 规则适用性 | 小任务可跳过其余规则，「自行判断」是否满足条件 | 换成四个硬性条件清单：① 改动 ≤20 行 ② 不涉及 DB/API/权限/状态机/enum/配置键 ③ 不涉及删除或重命名 ④ 影响范围已在目标文件内完全确认 | 「自行判断」不可检查，弱模型无法触发快速通道 |

v3.3 不新增版本号内的规则条目，仍然保持 8 条。五处改动都是把抽象描述换成可检查的硬条件，对弱模型的改善尤其明显。

## 当前版本为什么这样写

### 1. 有实验依据，不是拍脑袋

每条规则都能对应到实验里的行为变化：
- 精准修改：4 条组删代码再恢复 vs 其他组干净追加
- 不确定就问：7 条组唯一主动提问
- 简单优先：无规则组擅自加扣款 vs 有规则组不加
- 改前确认语义约定：针对 payment.status 隐性问题
- 上下文先行：来自后续已有系统变更实践，针对“上下文不足/过载/不可解释”导致的误判和漏改

### 2. 写法比数量重要

核心发现：4 条组的 "present multiple interpretations" 和 7 条组的「如果不确定，就问」**内容相同**，但写法不同导致行为质变。

律刃的写法原则：
- **直接指令 > 描述句**：「不要假设」优于 "consider whether"
- **短指令 > 长解释**：每条规则一句话标题 + 3-5 条要点
- **中文对中文任务**：语言匹配减少翻译损耗

### 3. 7 条是实验甜点位，当前保持轻量 8 条

```
0条 3.92 → 4条 4.2 (+0.28) → 7条 4.47 (+0.27) → 21条 4.6 (+0.13)
```

原始实验说明：7 条之后每条新增规则贡献仅 0.009 分，边际收益下降很快。

所以 v3.1 没有继续堆很多流程规则，只新增 1 条"上下文先行"。v3.2 也保持 8 条，只把复杂链路覆盖表、编译门禁、验证失败归因写进现有规则，不回到 v2 的 21 条规则模式。v3.3 继续保持 8 条，只把五处抽象描述换成可检查的硬条件。

### 4. 分级适用

第 8 条（规则适用性）按任务规模分级：
- 小任务只检关键规则；上下文先行只做极简版，确认目标文件、直接引用和调用方，不跑完整上下文包
- 中任务检 7 条
- 大任务全检
- 复杂链路变更先覆盖表后实现，避免一边搜索一边乱改

### 5. 管行为，不替代业务理解

规则能约束「改不改、问不问、改多少、先看哪些上下文」，但不能保证模型天然懂业务含义，比如「退款后 payment 该是什么状态」。这类正确性要靠查原定义和测试覆盖，所以第 6 条要求先确认语义，第 7 条要求业务逻辑和状态转换必须写测试。

同理，规则能提醒模型先做覆盖表，但不能替代语言和框架基本功。Round 7 里 with-context 组分析很好，仍然因为 Java public record 文件约束编译失败。这个 Java 错误只是当轮样本暴露出的具体例子；通用规则真正要求的是：新增类型、模块、导出项或公开 API 前，先确认当前技术栈自己的文件命名、可见性、导出方式、模块边界和测试发现规则。v3.2 把这类“新增前确认约束”和“验证失败读首个错误”写成硬要求。
