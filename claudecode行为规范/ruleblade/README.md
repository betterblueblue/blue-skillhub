# 律刃 RuleBlade

> 8 条给 AI 编码助手用的行为规则。目标很朴素：少猜、少乱改、先拿对上下文、先验证。

## 当前版本：v3.1（上下文先行补强）

---

## 迭代历程

### v1 — 对照组（4 条英文规则）

来源：Claude Code 官方推荐规则的精简版。

| # | 规则 | 写法 |
|---|------|------|
| 1 | Think Before Coding | 英文，描述性 |
| 2 | Simplicity First | 英文，描述性 |
| 3 | Surgical Changes | 英文，描述性 |
| 4 | Goal-Driven Execution | 英文，描述性 |

**暴露的问题：**
- 任务 B 中删了原有代码又恢复，说明“精准修改”写得太抽象，没拦住多余操作
- 全程自问自答，从未向用户提问（"present multiple interpretations" 被理解为「列出理解然后自己选」）
- 平均分 4.2

### v2 — 实验组（21 条中文规则）

来源：在 v1 基础上扩展，覆盖开发阶段、测试策略、中断恢复、并行任务等。

**暴露的问题：**
- 21 条中 14 条没有带来可归因差异，更像噪音
- 查证习惯仍未被激活——规则被稀释了注意力
- 规则过多导致模型无法全部记住和遵守
- 平均分 4.6，但收益很低（每条新增规则仅贡献 0.009）

### v2.5 — 精选组（7 条中文规则，实验前版）

从 21 条中挑出 7 条核心规则，改成更直接的中文指令。

| # | 规则 | 改动 |
|---|------|------|
| 1 | 先思考再编码 | "present multiple interpretations" → 「如果不确定，就问」 |
| 2 | 简单优先 | 保留核心 |
| 3 | 精准修改 | 保留核心 |
| 4 | 目标驱动执行 | 保留核心 |
| 5 | 知道何时说不 | 新增 |
| 6 | 测试策略 | 新增 |
| 7 | 规则适用性 | 新增（元规则） |

**实验结果：**
- ✅ 查证习惯被激活！唯一在任务 B 主动提问的组
- ✅ 平均分 4.47，接近 21 条的 4.6，投入产出最好
- ❌ payment.status 隐性问题：退款后设为 'pending' 而非 'refunded'（业务理解错误，单靠行为规则管不住）
- ❌ 第 5 条「知道何时说不」全程未触发

### v3 — 律刃（7 条中文规则，实验后定稿）

基于 v2.5 实验结果做了两处调整：

**优化 1：第 5 条替换**

| 旧 | 新 |
|---|---|
| 知道何时说不（未触发） | 改前确认语义约定（针对隐性问题） |

理由：实验暴露的主要问题是 payment.status 被模型凭直觉赋值，没有回看原定义。新规则直接要求「赋值前必须回看原定义的合法值」。

**优化 2：第 1 条加强追问力度**

| 旧 | 新 |
|---|---|
| 如果有不清楚的地方，停下来。指出困惑之处。然后问。 | 如果有不清楚的地方，停下来。指出困惑之处。然后问。**一次问不清就再问，不要自己猜。** |

理由：追问不能只问一次就结束。实验中 7 条组问了问题，但没有继续追 status 歧义，加强后更容易覆盖这类场景。

### v3.1 — 上下文先行（8 条轻量补强）

来源：后续在 `impact` / `impact-pro` 影响分析 Skill 中反复验证上下文包能力时发现，AI 编码助手在已有系统中犯错，很多时候不是因为不会写代码，而是因为动手前没有拿到足够准确的上下文。

新增第 2 条：

| 新增规则 | 目的 |
|----------|------|
| 上下文先行 | 在实现前先确认项目入口、模块边界、相关配置、运行/测试命令、直接相关文件、影响判断文件、背景参考文件和排除项 |

这条规则不是把完整的 `Context Pack` 流程搬进通用行为准则。完整上下文包仍属于 `impact` / `impact-pro` 的专业流程；律刃这里只保留轻量原则：

```text
项目地图 → 变更邻域 → 精准证据
```

这样做的原因：

- 通用行为准则不能太重，否则简单任务会被拖慢。
- 但中大型任务如果不先收敛上下文，很容易误改、漏改或过早设计。
- “看过但排除”的文件也要说明原因，避免后续反复搜索同一批无关内容。
- 涉及 DB/API/权限/状态机/跨模块时，未完成上下文收敛前不应开始设计或编码。

这和 Augment Code 在《[How to write good AGENTS.md files](https://www.augmentcode.com/blog/how-to-write-good-agents-dot-md-files)》里的观点一致：主指令文件不是越长越好，应保持短、高信号、可执行；复杂细节通过按需加载的参考文档或 Skill 承载。

v3.1 的定位：保留 v3 的实验结论，把上下文能力作为后续实战暴露出的关键补强项。

## 当前版本为什么这样写

### 1. 有实验依据，不是拍脑袋

每条规则都能对应到实验里的行为变化：
- 精准修改：4 条组删代码再恢复 vs 其他组干净追加
- 不确定就问：7 条组唯一主动提问
- 简单优先：无规则组擅自加扣款 vs 有规则组不加
- 改前确认语义约定：针对 payment.status 隐性问题
- 上下文先行：来自后续已有系统变更实践，针对“上下文不足/过载/不可解释”导致的误判和漏改

### 2. 写法比数量重要

核心发现：4 条组的 "present multiple interpretations" 和 7 条组的「如果不确定，就问」**内容相同**，但写法不同导致行为质变。

律刃的写法原则：
- **直接指令 > 描述句**：「不要假设」优于 "consider whether"
- **短指令 > 长解释**：每条规则一句话标题 + 3-5 条要点
- **中文对中文任务**：语言匹配减少翻译损耗

### 3. 7 条是实验甜点位，当前保持轻量 8 条

```
0条 3.92 → 4条 4.2 (+0.28) → 7条 4.47 (+0.27) → 21条 4.6 (+0.13)
```

原始实验说明：7 条之后每条新增规则贡献仅 0.009 分，边际收益下降很快。

所以 v3.1 没有继续堆很多流程规则，只新增 1 条“上下文先行”。它来自后续真实 Skill 设计和多栈验证，不是回到 v2 的 21 条规则模式。

### 4. 分级适用

第 8 条（规则适用性）按任务规模分级：
- 小任务只检关键规则；上下文先行只做极简版，确认目标文件和直接引用，不跑完整上下文包
- 中任务检 7 条
- 大任务全检

### 5. 管行为，不替代业务理解

规则能约束「改不改、问不问、改多少、先看哪些上下文」，但不能保证模型天然懂业务含义，比如「退款后 payment 该是什么状态」。这类正确性要靠查原定义和测试覆盖，所以第 6 条要求先确认语义，第 7 条要求业务逻辑和状态转换必须写测试。
