## 迭代历程

### v1 — 对照组（4 条英文规则）

来源：Claude Code 官方推荐规则的精简版。

| # | 规则 | 写法 |
|---|------|------|
| 1 | Think Before Coding | 英文，描述性 |
| 2 | Simplicity First | 英文，描述性 |
| 3 | Surgical Changes | 英文，描述性 |
| 4 | Goal-Driven Execution | 英文，描述性 |

**暴露的问题：**
- 任务 B 中删了原有代码再恢复（违反精准修改，但规则写得太抽象没拦住）
- 全程自问自答，从未向用户提问（"present multiple interpretations" 被理解为「列出理解然后自己选」）
- 平均分 4.2

### v2 — 实验组（21 条中文规则）

来源：在 v1 基础上扩展，覆盖开发阶段、测试策略、中断恢复、并行任务等。

**暴露的问题：**
- 21 条中 14 条无可归因差异，是噪音
- 查证习惯仍未被激活——规则被稀释了注意力
- 规则过多导致模型无法全部记住和遵守
- 平均分 4.6，但 ROI 极低（每条新增规则仅贡献 0.009）

### v2.5 — 精选组（7 条中文规则，实验前版）

从 21 条中精选 7 条核心规则，改写为中文祈使句。

| # | 规则 | 改动 |
|---|------|------|
| 1 | 先思考再编码 | "present multiple interpretations" → 「如果不确定，就问」 |
| 2 | 简单优先 | 保留核心 |
| 3 | 精准修改 | 保留核心 |
| 4 | 目标驱动执行 | 保留核心 |
| 5 | 知道何时说不 | 新增 |
| 6 | 测试策略 | 新增 |
| 7 | 规则适用性 | 新增（元规则） |

**实验结果：**
- ✅ 查证习惯被激活！唯一在任务 B 主动提问的组
- ✅ 平均分 4.47，接近 21 条的 4.6，ROI 最高
- ❌ payment.status 暗病：退款后设为 'pending' 而非 'refunded'（认知错误，规则管不住）
- ❌ 第 5 条「知道何时说不」全程未触发

### v3 — 律刃（7 条中文规则，实验后定稿）

基于 v2.5 实验结果的两处优化：

**优化 1：第 5 条替换**

| 旧 | 新 |
|---|---|
| 知道何时说不（未触发） | 改前确认语义约定（针对暗病） |

理由：实验暴露的唯一可改进点是 payment.status 暗病——模型凭直觉赋值，不回看原定义。新规则直接要求「赋值前必须回看原定义的合法值」。

**优化 2：第 1 条加强追问力度**

| 旧 | 新 |
|---|---|
| 如果有不清楚的地方，停下来。指出困惑之处。然后问。 | 如果有不清楚的地方，停下来。指出困惑之处。然后问。**一次问不清就再问，不要自己猜。** |

理由：吸收 grill-me skill 的精神——逐分支追问，不要问一次就放弃。实验中 7 条组问了但没追问 status 歧义，加强后有望覆盖此类场景。

## 当前版本的优点

### 1. 实验验证，不是拍脑袋

每条规则都有可归因的实验证据：
- 精准修改：4 条组删代码再恢复 vs 其他组干净追加
- 不确定就问：7 条组唯一主动提问
- 简单优先：无规则组擅自加扣款 vs 有规则组不加
- 改前确认语义约定：针对 payment.status 暗病（认知错误）

### 2. 写法比数量重要

核心发现：4 条组的 "present multiple interpretations" 和 7 条组的「如果不确定，就问」**内容相同**，但写法不同导致行为质变。

律刃的写法原则：
- **祈使句 > 描述句**：「不要假设」优于 "consider whether"
- **短指令 > 长解释**：每条规则一句话标题 + 3-5 条要点
- **中文对中文任务**：语言匹配减少翻译损耗

### 3. 7 条是 ROI 甜点

```
0条 3.92 → 4条 4.2 (+0.28) → 7条 4.47 (+0.27) → 21条 4.6 (+0.13)
```

7 条之后每条新增规则贡献仅 0.009 分，边际递减极快。

### 4. 分级适用

第 7 条（规则适用性）按任务规模分级：
- 小任务只检 3 条，不搞形式主义
- 中任务检 6 条
- 大任务全检

### 5. 管行为，不管认知

规则能约束「改不改、问不问、改多少」，但约束不了「退款后 payment 该是什么状态」。认知级正确性靠测试覆盖，律刃对此有清醒认知——第 6 条强调「业务逻辑、状态转换必须写测试」。
