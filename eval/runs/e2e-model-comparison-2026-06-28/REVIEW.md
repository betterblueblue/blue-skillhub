## 三、项目 Ground Truth（评审基准）

> 以下是从源码直接读取的确定性事实，五个副本源码完全一致，评审时统一对照。

### 3.1 技术栈

| 事实 | 证据 |
|------|------|
| Node.js + Express 4.18 + TypeScript 4.9 | `package.json` dependencies |
| Prisma 4.5 + PostgreSQL | `prisma/schema.prisma` datasource |
| JWT 认证（`express-jwt` 8.3 + `jsonwebtoken` 9.0） | `package.json` + `src/middleware/auth/authenticator.ts` |
| bcrypt 密码哈希 | `package.json` + `src/utils/hashPasswords.ts` |
| Winston 日志 | `package.json` dependencies |
| Jest 测试框架 | `package.json` devDependencies |

### 3.2 数据模型（4 个模型）

| 模型 | 主键 | 关键字段 | 关系 |
|------|------|---------|------|
| User | `username`（String @id） | email(@unique)、password、bio?、image? | follows/followedBy（自引用 M:N）、authored（1:M Article）、favorites（M:N Article）、comment（1:M Comment） |
| Article | `slug`（String @id） | title(@unique)、description、body、createdAt、updatedAt | author（M:1 User）、favoritedBy（M:N User）、comments（1:M Comment）、tagList（M:N Tag） |
| Comment | `id`（Int @id autoincrement） | body、createdAt、updatedAt | author（M:1 User）、Article（M:1 Article） |
| Tag | `tagName`（String @id） | — | article（M:N Article） |

### 3.3 认证链路

| 事实 | 证据 |
|------|------|
| JWT 签名算法 HS256 | `authenticator.ts:27` |
| 支持 `Token` 和 `Bearer` 两种 header 前缀 | `authenticator.ts:21` |
| JWT payload 含 `{ username, email }` | `createUserToken.ts:15` |
| 有两个 auth 中间件：`authenticate`（强制）和 `optionalAuthenticate`（可选） | `authenticator.ts:26-38` |
| JWT_SECRET 从环境变量读取，缺失时启动报错 | `authenticator.ts:7-9` |
| **无 role/权限字段**——纯身份认证，无 RBAC | User 模型无 role 字段，无权限中间件 |

### 3.4 注册流程（C2 变更基准）

| 事实 | 证据 |
|------|------|
| 注册路由 `POST /api/users` | `routes/api/users.ts:9` |
| 注册只接收 `{ email, password, username }` | `usersRegister.ts:19` |
| 密码哈希后存入 DB | `usersRegister.ts:22` |
| 注册成功后立即签发 JWT token | `usersRegister.ts:28` |
| **无邮箱验证步骤**——注册即用 | 全流程无 verify/email/confirmation 相关代码 |
| **User 模型无 emailVerified / verified 字段** | `schema.prisma` User 模型 |

### 3.5 请求体解析（C1 变更基准）

| 事实 | 证据 |
|------|------|
| 请求体解析在 `app.ts:18`：`app.use(express.json())` | `app.ts:18` |
| **未设 `limit` 参数**——默认 100kb（Express 4 内置 body-parser 默认值） | 无第二参数 |
| 全局生效，所有路由共用这一个 `express.json()` | `app.ts` 只调用一次 |

### 3.6 负向事实表（当前项目中不存在的依赖/能力）

| 事实 | 证据 |
|------|------|
| **无邮件发送依赖**（无 nodemailer/sendgrid 等） | `package.json` dependencies 中无邮件库 |
| **无消息队列**（无 bull/rabbitmq/amqp 等） | `package.json` dependencies 中无队列库 |
| **无缓存层**（无 redis/ioredis 等） | `package.json` dependencies 中无缓存库 |
| **无 RBAC / 权限中间件** | User 模型无 role 字段，无权限中间件 |
| **无邮箱验证字段** | User 模型无 emailVerified / verified 字段 |

### 3.7 分层架构

```
src/
├── app.ts                    # Express 应用配置 + 全局中间件
├── server.ts                 # 启动入口
├── routes/api/               # 路由层（5 个路由文件）
├── controllers/              # 控制器层（按资源拆分目录）
├── middleware/               # 中间件层（auth / validator / errorHandling）
├── utils/
│   ├── auth/                 # JWT 工具
│   ├── db/                   # Prisma 数据访问层（按资源拆分）
│   ├── hashPasswords.ts      # 密码工具
│   ├── logger.ts             # Winston 日志
│   └── slugfy.ts             # Slug 工具
└── view/                     # 响应格式化层（5 个 viewer）
```

## 九、评分体系（三层固化标准）

> **本章是 Skill 质量评审的固化标准，后续所有模型评测均按此体系执行。**
>
> 评分分为三层：硬性门禁（Gate）→ 流程合规（Compliance）→ 产出质量（Quality）。
> 第一层不通过直接降级，不参与计分；第二层和第三层参与计分，合计为模型总分。

### 9.1 三层体系总览

```
第一层：硬性门禁 (G1-G4)   → MUST 项，任一 FAIL 直接降到 B/C 级，不参与计分
第二层：流程合规 (G5-G9)   → SHOULD 项，每项 PASS 得分，FAIL 扣分，满分 40
第三层：产出质量 (Q1-Q6)   → 梯度评分，满分 60
─────────────────────────────────────────────────────────────
模型总分 = 合规得分（≤40）+ 质量得分（≤60）= 满分 100
```

| 层级 | 性质 | 失败后果 | 计分方式 |
|------|------|---------|---------|
| 硬性门禁 G1-G4 | MUST | 一票否决，直接降级 | 不计分，只判 PASS/FAIL |
| 流程合规 G5-G9 | SHOULD | 扣分但不降级 | 每项 PASS 得 8 分，满分 40 |
| 产出质量 Q1-Q6 | 梯度 | 不影响等级 | 每项 1-10 分，满分 60 |

### 9.2 第一层：硬性门禁（G1-G4）

> 任一 FAIL → 直接降到 B 级或更低， regardless of 合规/质量得分。

| # | 维度 | 评估什么 | 对应检查项 | 判定方式 |
|---|------|---------|-----------|---------|
| **G1** | 文件产出完整性 | Skill 要求的产出文件是否全部生成且非空 | P-F1~F4, C1-F1~F3, C2-F1~F5 | 全部存在且非空 = PASS；缺任一 = FAIL |
| **G2** | 章节完整性 | Pathfinder 15 核心节齐全且非空壳；Impact 文档结构完整 | P-S1~S4 | 15 节全在 + 【14】有实质内容 = PASS |
| **G3** | 事实准确性 | 产出事实与 Ground Truth 一致，无编造 | P-A1~A7 | 零事实错误 = PASS；≤1 处 = PASS(记注)；≥2 处 = FAIL |
| **G4** | Script Gate 通过 | `pf_validate.py` 执行且 SUMMARY `0 failed` | P-V1~V3 | 执行 + 无 FAIL = PASS |

**G3 特别说明**：事实编造（如将 PostgreSQL 说成 SQLite）直接判 FAIL，不论数量。事实遗漏（如少列一个数据模型）≤1 处记注但不 FAIL，≥2 处 FAIL。

### 9.3 第二层：流程合规（G5-G9）

> 每项 PASS 得 8 分，FAIL 得 0 分，满分 40 分。

| # | 维度 | 评估什么 | 对应检查项 | 判定标准 |
|---|------|---------|-----------|---------|
| **G5** | 不确定项分类 | Step 3.0 执行：代码可推断 vs 业务需决策正确区分 | C1-U1~U3, C2-U1~U5 | 有分类 + 标注规范（`[假设]` / `【代码推断】`）= 8 分；有分类但标注不全 = 4 分；无分类 = 0 分 |
| **G6** | L1 上下文交接 | Impact 实际读取并复用了 Pathfinder 地图 | C2-L1~L4, E2E-1~E4 | 读取 + 复用 + 重新取证推断项 + HEAD 一致 = 8 分；缺一项 = 4 分；未读取 = 0 分 |
| **G7** | 影响覆盖完整性 | C2 七层影响全覆盖（schema/注册/登录/端点/邮件/auth/JWT） | C2-A1~A7 | 7/7 = 8 分；6/7 = 4 分；≤5/7 = 0 分 |
| **G8** | 定级与流程完整性 | light/full 判档正确 + 判档决策表存在 + 高风险标注 + `_active-state.md` 产出 | C1-L1~L2, C2-L5~L7, C1-F3, C2-F5 | 全部满足 = 8 分；缺 1 项 = 6 分；缺 2 项 = 4 分；缺 3 项 = 2 分；判档错误 = 0 分 |
| **G9** | 文档间一致性 | 010/020/030 三文档间无自相矛盾 | C2-X1~X3 | 零矛盾 = 8 分；1 处矛盾 = 4 分；≥2 处 = 0 分 |

**合规附加分（最多 +0 分，仅记注）**：
- 认证-鉴权自检完整执行（P-Auth1~3 全通过）→ 记注，不加分
- 三张 Mermaid 图齐全且箭头语义正确（P-G1~G4 全通过）→ 记注，不加分
- 可选节产出（P-S5）→ 记注，不加分

> 以上附加项已体现在质量维度 Q1-Q6 中，合规层不再重复加分。

### 9.4 第三层：产出质量（Q1-Q6）

> 每项 1-10 分梯度评分，满分 60 分。以 C2（复杂变更）的 010/020/030 三件套为主要评分对象。

| # | 维度 | 评估什么 | 评分标准 |
|---|------|---------|---------|
| **Q1** | 需求文档完整性 | 验收标准覆盖度、假设标注质量、模糊点处理 | 10=验收标准全覆盖+假设每条有依据/影响+模糊点逐条处理；7=有验收标准+有假设标注+有模糊点处理；4=有但不全；1=严重缺失 |
| **Q2** | 设计方案深度 | 替代方案对比、横切关注点、向后兼容评估 | 10=多组方案对比+19维横切+兼容性独立章节；7=有方案对比+有横切表+有兼容评估；4=有但简略；1=无 |
| **Q3** | 实施可执行性 | 步骤清晰度、代码示例质量、前置检查、回滚方案 | 10=步骤完整+每步有代码示例+前置检查+逐步骤回滚；7=步骤完整+部分代码+有回滚；4=步骤但缺代码/回滚；1=步骤混乱 |
| **Q4** | 代码风格适配 | 风格约束提取精度、是否带证据引用 | 10=8+条约束+每条带行号引用；7=有约束+有证据引用；4=有约束但无引用；1=无风格约束 |
| **Q5** | 验证严谨性 | 验证用例覆盖、API 方法存在性检查、异常行为确认 | 10=用例全覆盖+方法存在性验证+异常行为确认；7=用例充足+有方法验证或异常确认之一；4=有用例但无方法验证；1=用例严重不足 |
| **Q6** | 设计决策质量 | 方案选择理由、被否方案论证、语义约定确认 | 10=选择理由清晰+被否方案充分论证+语义约定确认+独立推理链；7=有理由+有被否方案+有语义确认；4=有但简略；1=无论证 |

### 9.5 等级定义（基于三层体系）

| 等级 | 含义 | 判定条件 |
|------|------|---------|
| **S** | 生产可用 | G1-G4 全 PASS + 合规 ≥36/40 + 质量 ≥54/60 |
| **A** | 基本可用 | G1-G4 全 PASS + 合规 ≥28/40 + 质量 ≥36/60 |
| **B** | 有缺陷 | G1-G4 有 1 项 FAIL，或合规 <28，或质量 <36 |
| **C** | 不可用 | G1-G4 有 ≥2 项 FAIL，或事实编造 ≥2 处 |

### 9.6 跨模型对比定级规则

| 对比维度 | 判定方法 |
|---------|---------|
| Pathfinder 最强 | 任务 1 合规+质量合计得分最高且评级 ≥ A |
| Impact 最强 | 任务 2+3 合规+质量合计得分最高且评级 ≥ A |
| E2E 协作最强 | G6（L1 交接）满分且质量维度 Q1-Q6 合计最高 |
| 综合最佳 | 总分（合规+质量）最高且无 C 级任务 |
| 存在分化 | 某模型 Pathfinder 评级比 Impact 高 ≥2 级（或反之） |
| **弱模型能力下限** | **最弱模型与最强模型的总分差，以及 G1-G4 闸门能否拦截弱产出** |
| **质量断档判定** | **某模型质量得分比第二名高 ≥8 分（60 分制），且在 Q5（验证严谨性）上独有其他模型不具备的检查能力。注：本次评审中 G52 质量分修正后为 56 分，与第二名 S37 的 50 分差距 6 分 < 8 分阈值，不构成断档** |

### 9.7 评分执行流程

```
1. 硬性门禁检查（G1-G4）
   a. G1：检查所有产出文件存在且非空
   b. G2：检查 Pathfinder 15 节齐全 + Impact 文档结构
   c. G3：逐项对照 Ground Truth 检查事实准确性
   d. G4：运行 pf_validate.py 确认 Script Gate
   → 任一 FAIL：标记降级，记录原因，继续后续评分（但不影响降级判定）

2. 流程合规检查（G5-G9）
   a. G5：检查不确定项分类（[假设] / 【代码推断】标注）
   b. G6：检查 L1 上下文交接痕迹
   c. G7：检查 C2 七层影响覆盖
   d. G8：检查定级判档 + 决策表 + 高风险标注

   → 每项 PASS 得 8 分，部分通过得 4 分，FAIL 得 0 分

3. 产出质量评分（Q1-Q6）
   a. 读取 010/020/030 三件套
   b. 按 9.4 的评分标准逐项打分（1-10 分）

4. 计算总分
   合规得分 = G5~G9 得分之和（≤40）
   质量得分 = Q1~Q6 得分之和（≤60）
   总分 = 合规得分 + 质量得分（≤100）

5. 判定等级
   按 9.5 等级定义判定 S/A/B/C

6. 跨模型对比
   按 9.6 规则判定各维度最佳模型
```

**G52 在产出质量上明显领先**，但不是"断档领先"。差距 6 分（56 vs 50），未达 8 分阈值。G52 的领先集中在两个其他模型完全不具备的检查能力：

1. **方法存在性验证**——写实施文档时，回头验证"我写的这些 API 调用，方法真的存在吗"
2. **异常行为确认**——验证"这些方法在边界情况下会怎样"

这两个能力使 G52 的实施文档从"设计方案"升级为"经过验证的可执行方案"。其他四个模型——包括总分第二的 S37——都停留在"设计了但没验证"的阶段。

但同时，G52 自身有 4 处质量问题：verificationToken 缺 @unique（设计疏漏）、token 过期方案繁琐（违反简单优先）、Step 8 缩进错误（格式校验缺失）、050-validation/ 空目录（交付物不完整）。这些问题在原评审中被列出但未扣分，现修正后 G52 质量从 60 降至 56。

如果把五个模型的实施文档直接交给一个不熟悉项目的开发者去执行，G52 的文档踩坑概率最低，因为它已经排除了"方法不存在""方法行为与预期不符"这两类最常见的实施失败原因。但它不是完美的——开发者可能还会遇到 token 过期逻辑理解成本高、findFirst 性能不如 findUnique 等问题。

---

## 十二、Skill 优化方案：缩小 C25/S37 与 G52 的质量差距

> **目标**：G52 的测试结果不动，作为标杆。通过优化 Skill 协议和模板，让 C25 和 S37 在下次执行时能自发产出 G52 级别的验证深度和代码示例，减少质量分差距。

### 12.1 根因分析：为什么弱模型做不到？

G52 独有的两个领先能力——方法存在性验证和异常行为确认——其实**在优化前的 Skill 中已经定义了**（`phase-4-output.md` 检查 1 和检查 2）。但弱模型没有执行，原因有三：

| 根因 | 具体表现 | 影响 |
|------|---------|------|
| **检查要求埋在 reference 文件中，模板不可见** | 检查 1/2 写在 `references/phase-4-output.md` 里，弱模型读到后"忘了"执行；而 `030-implementation.md` 模板里没有对应的填写字段 | Q5（验证严谨性）：C25=6, S37=7, G52=9 |
| **Step 格式中"操作"字段太模糊** | 模板写 `[具体命令或代码]`，弱模型只写一行文字描述；G52 自觉写了 5 个完整 TypeScript 代码块 | Q3（实施可执行性）：C25=8, S37=8, G52=9 |
| **风格约束不要求行号引用** | 模板写"参考设计文档 §4"，弱模型只写约束内容不标行号；G52 每条都标 `参考 xxx.ts:行号` | Q4（代码风格适配）：S37=9(无行号), G52=10(有行号) |
| **横切关注点表可被跳过** | `020-design.md` 模板有 19 行表但无"不得跳过"提示，C25 直接删表 | Q2（设计深度）：C25=7(无横切表), G52=9(有横切但有@unique疏漏) |

**核心发现：问题不是 Skill 缺少规则，而是规则被埋在了 reference 文件里，弱模型"看不到所以不做"。G52 因为模型能力强，主动执行了这些检查。**

### 12.2 优化方案：把隐藏规则提升到模板可见层

#### 改动 1：`030-implementation.md` 模板 — 新增 §3.2 API 方法验证表

**目标维度**：Q5（验证严谨性）

在 §3 执行步骤和 §4 回滚方案之间，新增 §3.2「API 方法验证」表：

```markdown
## 3.2 API 方法验证（提交确认前必做）

| 方法名 | 来源文件 | grep 验证 | 异常行为 | 验证标注 |
|--------|---------|----------|---------|---------|
| `existingMethod(` | `src/path/to/file.ts` | ✅ 存在 `【已核实: path:line】` | 返回 null / 抛异常 | 已确认 |
| `newMethod(` | — | 新增，本次定义 | — | 新增 |
```

**原理**：弱模型看到模板里有空表就要填，比让它们"自觉"去执行 reference 里的检查可靠得多。G52 是自觉做的，现在把 G52 的行为模式固化到模板里。

**预期效果**：C25 的 Q5 从 6→8+，S37 的 Q5 从 7→8+。

#### 改动 2：`030-implementation.md` 模板 — Step 格式要求代码块

**目标维度**：Q3（实施可执行性）

Step 的"操作"字段从：
```
- **操作**：`[具体命令或代码]`
```
改为：
```
- **操作**：
  ```typescript
  // 涉及代码变更的 Step 必须给出实际代码片段，不得只用文字描述
  // 仅 schema/配置类变更可用片段或命令
  ```
```

**原理**：弱模型看到 `[具体命令或代码]` 会写一行文字描述。看到代码块模板就会往里填代码。这是"模板引导产出格式"的经典手法。

**预期效果**：C25 的 Q3 从 8→9，S37 的 Q3 从 8→9。

#### 改动 3：`030-implementation.md` 模板 — 风格约束要求行号引用

**目标维度**：Q4（代码风格适配）

风格约束字段从：
```
- [轴名] [约束内容]（参考设计文档 §4 和 profile `style_axes` 表）
```
改为：
```
- [轴名] [约束内容]（`参考 path/to/file.ext:行号`）
```

**原理**：不给行号格式，弱模型只写约束内容。给了格式模板，弱模型会照着填行号。

**预期效果**：S37 的 Q4 从 9→10，C25 的 Q4 从 8→9。

#### 改动 4：`020-design.md` 模板 — 横切关注点表不得跳过

**目标维度**：Q2（设计深度）

在横切表的 HTML 注释中追加：
```
提交前自检：19 行全部检查完毕，☐ 行的"本变更的处理"列应填"不涉及"或留空，
☑ 行必须有具体措施。跳过此表（删表或全留空）= 提交不完整，不得提交确认。
```

**原理**：C25 直接删了整个横切表。加上"不得跳过"的显式提示后，弱模型会保留表格并逐行填写。

**预期效果**：C25 的 Q2 从 7→8+。

#### 改动 5：`phase-4-output.md` — 强化检查语言

**目标维度**：Q5（验证严谨性），配合改动 1

三处修改：
- 检查 1：从"静默执行"改为"**验证结果必须写入实施文档 §3.2 表**"
- 检查 2：从"保持不变"改为"在 §3.2 表中记录'返回 null 不抛异常'"
- 汇总：从"静默执行"改为"跳过 §3.2 表 = 提交不完整，不得提交确认"

**原理**：把"静默执行"改为"必须在文档中留下痕迹"，消除弱模型"做了但不记录"的灰色地带。

#### 改动 6：`impact_validate.py` — V3 检查从 WARN 升级为 FAIL/WARN

**目标维度**：Q5（验证严谨性），脚本层强制

V3 检查逻辑从：
- 有方法调用但无验证标记 → WARN

改为两级：
- 有方法调用但无 §3.2 表 → **FAIL**（阻止提交）
- 有 §3.2 表但无验证标记 → WARN

**原理**：WARN 不阻止提交，弱模型会忽略。FAIL 阻止提交，弱模型必须补上 §3.2 表才能提交。这是最后一道防线。

### 12.5 优化策略总结

**核心思路：把 G52 的自发行为固化为模板强制字段。**

G52 之所以明显领先，不是因为 Skill 没教，而是因为它更善于"自觉执行"reference 里的检查。弱模型需要模板层面的"可见引导"才能做到同样的事。

这次优化遵循三个原则：
1. **不改 Skill 核心流程** — Phase 1-5 的流程不变，只在 Phase 4 的模板层做增强
2. **不加新检查维度** — 检查 1/2 已经存在，只是从"隐藏"变为"可见"
3. **脚本兜底** — V3 从 WARN 升级为 FAIL，确保即使弱模型跳过模板，脚本闸门也能拦截

**什么改不了**：Q6（设计决策质量）和 Q1（需求完整性）更多依赖模型自身的推理能力，模板引导的效果有限。G52 的 4 个被否方案和独立推理链（"不用 JWT 做验证 token"）是模型能力的体现，不是模板能补的。剩余的 3-5 分差距主要在这里。
