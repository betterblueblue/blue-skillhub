## 六、关键发现

### 发现 1：v3.9 关键链路追踪是最有效的改进

F3（关键链路追踪）直接捕获了 3 个 V7 遗漏项：

| 链路类型 | 捕获的遗漏项 | 如何发现 |
|---------|------------|---------|
| 数据流路径 | refreshToken TTL 同步 | 追踪 createToken/refreshToken/delLoginUser 三个操作对 userId→token 映射的影响 |
| 错误处理链 | passport.ts select | 追踪未验证用户访问受保护接口时 passport 需要加载 isEmailVerified |
| 数据流路径 | refreshAuth 检查 | 追踪 token 刷新链路发现不检查验证状态 |

**归因**：V7 的 skill 流程没有要求追踪链路二级影响，模型只做了"表面引用反查"。v3.9 的关键链路追踪表强制模型对错误处理链、中间件管线、数据流路径做追踪式分析，从而发现二级影响。

### 发现 2：light 深度检查让 light 模式不再"跳过"关键分析

V7 中 C2 GLM skill 选 light 模式后遗漏了 errorConverter 问题。v3.9 在 light 模板中增加了 3 项强制检查，V8 C2 B2 的 light 产出全部完成：

- 错误处理链：✅ 识别 413→500 并修复
- 中间件管线：✅ 识别 limit 在 xss 前阻断放大
- 数据流路径：✅ 确认仅影响 body 解析

**结果**：V8 C2 B2（light）的 D3=19，与 V8 C1 B2（noskill）的 D3=19 持平。而 V7 C4 B2（full）的 D3=17，低于两者。**v3.9 的 light 深度检查使 light 模式达到了甚至超过了 V7 full 模式的分析深度。**

### 发现 3：改动完整性自检防止"过早收敛"

V7 中 C6 Step skill 在 B2'/B3' 只做了 1 个 Step，遗漏大量关键改动。v3.9 的 F2（改动完整性自检）要求验收标准→Step 映射：

- B1：5 条验收标准 → 全部映射到 Step 2
- B3：6 条验收标准 → 全部映射到 Step 1-5，包括"refresh 不绕过"

**结果**：V8 C2 的 B3 有 5 个 Step（passport select、auth 拦截、注册发信、refreshAuth 检查、测试更新），而 V7 C4 的 B3 只有 4 个 Step（缺少 refreshAuth 检查）。

### 发现 4：noskill 组也有所提升，但假设标注仍是核心差距

V8 C1 noskill 均分 73.7（V7: 71.0，+2.7），分析深度也有提升。但 D5（假设质量）仍为 0 分——所有 noskill 产出都没有使用 `[假设]` 标签。

| 差异点 | C1 noskill | C2 skill |
|--------|-----------|----------|
| 假设标注 | ❌ 隐式 | ✅ [假设] + §2.2 模板 |
| 链路追踪 | 描述性（风险列表） | 结构化（链路追踪表） |
| 完整性自检 | ❌ | ✅ 验收→Step 映射 |
| 配置化方案 | B2 提出env配置 | B2 硬编码（skill 的不足） |

**skill 组在 B2 的配置化方案上不如 noskill**：noskill 提出了 `BODY_LIMIT` 环境变量+Joi 校验+`.env.example`，skill 组只硬编码了 `1mb`。这是一个 skill 组可以改进的点。

所有 7 个维度均无 ≥3 分回退：

## 九、对后续改进的建议

### 9.1 light 模式应保留配置化建议

V8 C2 B2（light）硬编码了 `1mb`，而 noskill 组提出了 `BODY_LIMIT` 环境变量方案。建议在 040-light.md 模板中增加提示：如果限制值是假设的，应建议配置化。

### 9.2 关键链路追踪可增加"配置依赖"链路类型

当前 3 种链路类型（错误处理链、中间件管线、数据流路径）覆盖了大部分场景，但 B2 的 env 配置化建议不在任何链路类型中。可考虑增加"配置依赖"链路类型。

### 9.3 noskill 组的分析深度提升值得关注

V8 C1 noskill 均分比 V7 提升 2.7 分，B2 的 D3=19 与 skill 组持平。这可能是 Composer 2.5 模型自身能力提升所致，也可能是因为测试项目副本与 V7 有细微差异。后续可考虑用相同模型版本做 A/B 测试确认。
