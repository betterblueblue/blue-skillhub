## 五、结论与建议

### 5.1 核心结论

1. **四个 case 全部是"已实现"场景**——所有 5 个 cell 都正确发现了这一点，无 skill 组的表现不亚于有 skill 组。

2. **Skill 的核心增益在 pathfinder**——B6 场景 skill 组平均提升 10.5 分，模板的 13 节结构和认证-鉴权自检机制是关键。

3. **Skill 在"已实现"场景可能引入过度设计**——Composer 有 skill 组在 B1/B2 选了 full 档提出不必要方案，总分反降 21 分。

4. **GLM-5.2 + skill 的组合表现最佳**（均分 88.0），skill 模板帮助 GLM 规范了证据标签和结构，同时 GLM 在判档上更保守（正确选 light）。

5. **Composer 2.5 无 skill 组的表现最稳定**（均分 89.0），自发使用证据标签、行号引用和不确定性标注，工程素养出色。

### 5.2 对 Skill 优化的建议

1. **增加"现状核查优先"规则**：当 Phase 2 发现功能已存在时，应强制走 light 档并输出"零改动确认"，而非继续 full 设计。这可以避免 Composer 有 skill 组的过度设计问题。

2. **pathfinder 的认证-鉴权自检机制值得推广**：第 10 节的字段一致性自检在所有模型上都有效发现了 JWT role select 缺失问题，建议在 impact/impact-pro 中也加入类似的自检机制。

3. **证据标签规范有效**：【已核实:file:line】标签显著提升了可追溯性，建议在无 skill prompt 中也建议使用类似标签（虽然不强制）。

### 5.3 未完成 cell 说明

C2（Opus 4.6 有 skill 组）因模型预算不足未完成。缺少这一组无法形成完整的 2×3 因子对比，特别是无法评估 skill 对 Opus 的增益效果。建议后续补充完成。
