## 三、Ground Truth（评审基准）

> 与第一轮完全一致，五个副本源码相同。

### 3.1 技术栈

| 事实 | 证据 |
|------|------|
| Node.js + Express 4.18 + TypeScript 4.9 | `package.json` |
| Prisma 4.5 + PostgreSQL | `prisma/schema.prisma` datasource |
| JWT（`express-jwt` 8.3 + `jsonwebtoken` 9.0） | `package.json` + `authenticator.ts` |
| bcrypt 密码哈希 | `hashPasswords.ts` |
| Winston 日志 | `package.json` |
| Jest 测试 | `package.json` devDependencies |

### 3.2 数据模型（4 个）

| 模型 | 主键 | 关键字段 | 关系 |
|------|------|---------|------|
| User | `username`（String @id） | email(@unique)、password、bio?、image? | follows/followedBy（自引用 M:N）、authored（1:M Article）、favorites（M:N Article）、comment（1:M Comment） |
| Article | `slug`（String @id） | title(@unique)、description、body、createdAt、updatedAt | author（M:1 User）、favoritedBy（M:N User）、comments（1:M Comment）、tagList（M:N Tag） |
| Comment | `id`（Int @id autoincrement） | body、createdAt、updatedAt | author（M:1 User）、Article（M:1 Article） |
| Tag | `tagName`（String @id） | — | article（M:N Article） |

### 3.3 认证链路

| 事实 | 证据 |
|------|------|
| JWT 签名 HS256 | `authenticator.ts:27` |
| Token + Bearer 两种前缀 | `authenticator.ts:21` |
| payload `{ username, email }` | `createUserToken.ts:15` |
| authenticate（强制）+ optionalAuthenticate（可选） | `authenticator.ts:26-38` |
| JWT_SECRET 从环境变量，缺失报错 | `authenticator.ts:7-9` |
| **无 RBAC** | User 无 role 字段 |

### 3.4 注册流程（C2 基准）

| 事实 | 证据 |
|------|------|
| 注册路由 `POST /api/users` | `routes/api/users.ts:9` |
| 只接收 `{ email, password, username }` | `usersRegister.ts:19` |
| 密码哈希后存入 DB | `usersRegister.ts:22` |
| 注册后立即签发 JWT | `usersRegister.ts:28` |
| **无邮箱验证** | 全流程无 verify 相关代码 |
| **User 无 emailVerified 字段** | `schema.prisma` |

### 3.5 请求体解析（C1 基准）

| 事实 | 证据 |
|------|------|
| `app.use(express.json())` 在 `app.ts:18` | `app.ts:18` |
| **未设 limit**——默认 100kb | 无第二参数 |
| 全局生效 | `app.ts` 只调用一次 |

### 3.6 负向事实表（不存在的能力）

| 事实 | 证据 |
|------|------|
| 无邮件依赖（无 nodemailer 等） | `package.json` |
| 无消息队列 | `package.json` |
| 无缓存层 | `package.json` |
| 无 RBAC | User 无 role |
| 无邮箱验证字段 | `schema.prisma` |

### 3.7 已有方法行为参考（§3.2 验证用）

| 方法名 | 来源文件 | 实际行为 | 异常时 |
|--------|---------|---------|--------|
| `prisma.user.findUnique(` | Prisma Client | 返回 User 或 null | 不抛异常 |
| `prisma.user.findFirst(` | Prisma Client | 返回 User 或 null | 不抛异常 |

## 十、评审结论

### C25（Composer 2.5）

- R1 总分：86（A）→ R2 总分：**86（A）**
- 主要变化：Q2 从 7→**7**（O2 未生效），Q5 从 6→**9**（O1+O3+O4 生效），Q3 从 8→**9**，G8 从 8→**4**（_active-state 缺失）
- 优化生效项：O1 ✅ O2 ❌ O3 ✅ O4 ✅
- 退步项：G8 -4（_active-state.md 缺失，与优化无关，属模型方差）
- 判定：优化**部分有效**——Q5 大幅提升（+3）但被 G8 退步（-4）抵消，总分持平

### S37（Step 3.7 Flash）

- R1 总分：90（A）→ R2 总分：**87（A）**
- 主要变化：Q5 从 7→**8**（O1+O3+O4 生效，但 3 处异常行为错误限制提升幅度），G8 从 8→**4**（_active-state 缺失）
- 优化生效项：O1 ✅（部分）O3 ✅ O4 ✅
- 退步项：G8 -4（_active-state.md 缺失，与优化无关，属模型方差）
- 判定：优化**部分有效**——Q5 提升（+1）但被 G8 退步（-4）超过，总分下降

### 与标杆 G52 的差距变化

| 模型 | R1 质量分 | R2 质量分 | G52 质量分 | R1 差距 | R2 差距 | 变化 |
|------|----------|----------|-----------|---------|---------|------|
| C25 | 46 | 50 | 56 | -10 | **-6** | ↑ 缩小 4 分 |
| S37 | 50 | 51 | 56 | -6 | **-5** | ↑ 缩小 1 分 |

> **质量差距在缩小**，但合规退步掩盖了总分改善。

### 总体结论

**优化在质量维度有效，但被合规维度的模型方差抵消。**

1. **O1+O3+O4（方法验证）完全成功**：两个模型都产出了 §3.2 表，V3 闸门从 N/A 变为 PASS。C25 的 §3.2 表质量甚至超过标杆 G52（异常行为 6/6 准确 vs G52 有 4 处质量问题）。
2. **O2（横切表）对 C25 失败**：模板注释"跳过 = 不完整"未能阻止 C25 跳过横切表。C25 将 §6 改名为"数据迁移方案"绕过了检查。需要更强的约束机制（如将横切表检查加入 `impact_validate.py`）。
3. **_active-state.md 退步是模型方差**：两个模型在 R2 都遗漏了 `_active-state.md`（R1 都有），这与优化无关。PROMPT 和模板中 `_active-state` 的要求未变。
4. **净效果**：C25 质量提升 +4（46→50），合规退步 -4（40→36），总分持平 86。S37 质量提升 +1（50→51），合规退步 -4（40→36），总分下降 87。
5. **下一步建议**：① 将横切表检查加入 `impact_validate.py`（V10 检查）以解决 O2 失败；② 在 PROMPT 中显式提醒 `_active-state.md` 必须产出，减少模型方差。
