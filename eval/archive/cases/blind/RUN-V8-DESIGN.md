> 评测日期：2026-06-25
> 测试模型：Composer 2.5（唯一）
> 评测目标：验证 v3.9 的 4 项改进是否实际生效，且不引入回退

---

## 一、V8 核心目标

### 1.1 要回答的问题

V7 盲测验证了"模糊 prompt 下 skill 增益显著"（+12.1 分），同时暴露了 4 个问题。v3.9 针对这 4 个问题做了改进。V8 要回答：

> **v3.9 的 4 项改进是否在相同的模糊 prompt 下实际生效？是否带来了分析质量的提升？**

### 1.2 与 V7 的区别

| 维度 | V7 | V8 |
|------|-----|-----|
| 模型 | GLM / Composer / Step（3 个） | **仅 Composer 2.5** |
| Cell 数 | 6（3 模型 × noskill/skill） | **2**（1 模型 × noskill/skill） |
| 场景 | B1'/B2'/B3'（同） | **B1'/B2'/B3'（同 V7，直接对比）** |
| Skill 版本 | v3.8 | **v3.9**（4 项改进落地后） |
| 评审重点 | 假设标注能力 | **v3.9 特性存在性 + 有效性 + 无回退** |
| 产出份数 | 18 | **6** |

### 1.3 为什么只用 Composer 2.5

1. V7 已证明三个模型的 skill 增益趋势一致（+11~+12.7），V8 不需要重复验证"是否有增益"
2. V8 的目标是验证 v3.9 改进是否生效，只需要一个模型即可
3. Composer 2.5 是 V7 中表现最强的模型（skill 均分 83.7），用它验证 v3.9 改进最有说服力——如果强模型都能从改进中受益，说明改进不是只对弱模型有效
4. 减少 token 消耗：从 18 份降到 6 份

### 1.4 为什么用相同的 3 个 case

V8 是**回归测试**，不是探索性测试。用相同 case 才能做 V7 vs V8 逐 case 对比：

| Case | V7 C4 skill 分 | V7 遗漏项 | V8 验证重点 |
|------|:---:|------|------|
| B1' 并发登录 | 84 | refreshToken TTL 同步（C2 独有发现） | 关键链路追踪是否帮助发现 |
| B2' 请求体限制 | 83 | Express 默认 100kb、XSS 内存放大 | 关键链路追踪 + 模糊点清单 |
| B3' 邮箱验证 | 84 | passport.ts select、refreshAuth 检查 | 改动完整性自检是否拦截遗漏 |
