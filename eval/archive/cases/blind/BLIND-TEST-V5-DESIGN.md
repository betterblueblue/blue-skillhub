# 盲测 v5 测试设计：2×3 因子实验

> 日期：2026-06-25
> 基线：v4 干净环境盲测结论（`eval/runs/BLIND-TEST-FINAL-CONCLUSION.md` §8）
> 核心目标：**量化 skill 对不同模型档位的增益，验证「普通模型 + skill 能否接近顶尖模型裸奔」**

---

## 一、为什么需要 v5

### 1.1 v1-v4 盲测的局限

前四轮盲测只测了「有 skill」的场景，缺少「无 skill」对照组。因此只能回答「模型 X + skill 能不能过关」，无法回答两个更根本的问题：

1. **skill 的增益有多大？**——Composer 2.5 + skill 拿了 95.3 分，但如果没有 skill 它能拿多少？差值就是 skill 的真实价值。
2. **普通模型 + skill 能否接近顶尖模型？**——需要同时测顶尖模型裸奔的分数作为锚点。如果 Composer + skill ≥ Opus 裸奔，skill 的经济价值就有了硬证据。

### 1.2 v5 的核心假设

**skill 对普通模型的增益 > 对顶尖模型的增益。**

这是因为顶尖模型不靠 skill 也能自觉做影响面分析、引用反查、凭证脱敏。skill 对它锦上添花；对普通模型雪中送炭。如果实验数据支持这个假设，就证明了 skill 的杠杆效应——用工程约束补齐模型短板。

### 1.3 引入 GLM-5.2 的理由

GLM-5.2 是当前国产编程模型最强之一。引入它有两个价值：
- 扩大模型覆盖面，验证 skill 对不同厂商模型的增益是否一致
- 国产模型在中文语境和成本上有优势，如果 + skill 后接近顶尖模型，日常使用的经济意义更大

## 二、实验设计

### 2.1 2×3 因子矩阵

6 个实验单元（cell），每个 cell 跑相同的 case 但条件不同：

| Cell | 模型 | 条件 | 产出目录前缀 |
|------|------|------|-------------|
| C1 | Opus 4.6 | **无 skill** | `v5-opus-noskill/` |
| C2 | Opus 4.6 | **有 skill** | `v5-opus-skill/` |
| C3 | Composer 2.5 | **无 skill** | `v5-composer-noskill/` |
| C4 | Composer 2.5 | **有 skill** | `v5-composer-skill/` |
| C5 | GLM-5.2 | **无 skill** | `v5-glm-noskill/` |
| C6 | GLM-5.2 | **有 skill** | `v5-glm-skill/` |

### 2.2 变量控制

| 变量 | 控制方式 |
|------|---------|
| **自变量 1：模型** | 3 个模型，能力梯度从高到低 |
| **自变量 2：skill** | 有 / 无，二值 |
| **控制变量：case** | 4 个 case，所有 cell 跑完全相同的 case |
| **控制变量：项目** | 同一个 test-projects 快照 |
| **控制变量：评审者** | 单一评审者，盲评（不知道产出来自哪个 cell） |
| **控制变量：环境** | 每个 cell 跑前清理 `change-impact/`，保证干净起跑线（不同模型运行环境的差异不纳入考量） |
| **控制变量：协议版本** | 有 skill 组统一使用优化完成后的最新 SKILL.md（E1-E4 + Q1-Q18 实施后） |

### 2.3 前置条件

**v5 必须在优化实施计划（E1-E4 + S1-S3 + Q1-Q18）完成后执行。** 原因：
- E1 脚本闸门是 skill 对普通模型增益最大的改进项，不实施就测不出 skill 的真实天花板
- 如果在优化前跑 v5，得到的是「旧版 skill」的增益数据，结论会偏低
- 有 skill 组的 SKILL.md 版本号必须在评分卡中记录（`skill_commit` 字段）

## 三、「无 skill」组定义

这是整个实验设计最关键的设计决策。

### 3.1 设计原则

「无 skill」不是「无工具」。模型仍有文件读写、grep、终端等工具能力。**唯一的区别是不加载 skill 协议**——不触发 `/pathfinder`、`/impact`、`/impact-pro`，不读 SKILL.md，不使用模板，不运行脚本闸门。不同模型运行环境的差异不纳入考量。

### 3.2 无 skill 组 prompt 模板

给模型的需求描述和有 skill 组完全一致（同一个 case JSON 的 `prompt` 字段），但任务指令极简：

```markdown
## 任务 — [case title]

项目路径：`[fixture.path]`
输出到：`[fixture.path]/change-impact/v5-[model]-noskill/[case-id]/`

用户需求：
"[case.prompt 原文]"

请阅读项目源码，分析这个需求的影响面，输出分析文档。
```

**不给的东西**（这些是 skill 的核心价值，给了就不算「无 skill」）：
- ❌ Phase 流程（Phase 1 → 2 → 3 → 4 → 5）
- ❌ 产出模板（000/010/020/030 等）
- ❌ 铁律和共享契约
- ❌ 苏格拉底提问协议
- ❌ light/full 判档规则
- ❌ 脚本闸门（pf_validate.py / impact_validate.py）
- ❌ 凭证脱敏强制规则
- ❌ 引用链反查指令
- ❌ 证据标签规范（【已核实】/【推断】）

**给的东西**（保证基本可比性）：
- ✅ 相同的用户需求原话
- ✅ 相同的项目路径
- ✅ 相同的输出目录结构（便于评审者定位产出）

### 3.3 有 skill 组 prompt 模板

复用 v4 的 prompt 结构，每个任务前 Read SKILL.md，走完整 Phase 流程。和 v4 的唯一区别是 case 数量从 4 个可能调整（见下文）。

### 3.4 为什么不给无 skill 组模板

如果给了空模板让模型填，相当于给了 skill 的「结构骨架」但不给「流程规则」。这样测出来的是「模板的价值」而非「skill 的价值」。skill 的核心价值是**结构化流程 + 安全约束 + 质量闸门**，不是产出格式。无 skill 组应该测的是模型「裸奔」时的自发能力。

## 五、评审设计

### 5.1 评审者选择

**评审者：GLM-5.2，在独立会话中执行评审。**

GLM-5.2 同时是被测模型（C5/C6）和评审者。这带来一个潜在的自评偏差风险——GLM-5.2 可能倾向于给风格接近自己产出的文件打高分。缓解措施：

1. **盲评隔离**：评审者不知道产出来自哪个 cell（见 §5.2），无法识别哪些是自己的产出
2. **独立会话**：评审在全新会话中进行，执行上下文不残留 runner 会话的记忆
3. **源码核实**：评审者必须拿项目源码逐条核实结论，不能只凭产出文件的内容打分——源码是客观锚点，降低主观偏好影响

**残余风险**：GLM-5.2 可能在评审风格（如文档结构偏好、表达方式偏好）上对某些产出有系统性倾向。这个偏差对所有 cell 一致存在，不会单独抬高 C5/C6 的分数。如果对结论有疑虑，可对 C5/C6 和其他 cell 的评分分布做 t 检验确认无显著偏差。

### 5.2 盲评机制

评审者（GLM-5.2）**不知道产出来自哪个 cell**。具体做法：

1. 跑分前把产出目录名从 `v5-opus-noskill/B1/` 改名为 `cell-A/B1/`、`v5-composer-skill/B1/` 改名为 `cell-D/B1/` 等，字母映射随机且不透露
2. 评审 prompt 中只给：case 需求 + 产出文件路径，不给模型名和 skill 条件
3. 评审者对每份产出独立打分，填写评分卡
4. 全部评完后揭晓映射关系

盲评是缓解 GLM-5.2 自评偏差的核心手段——它无法识别哪些产出是自己的，也无法根据模型名调整打分预期。

### 5.3 Rubric 适配

两组共用同一套评分框架（6 维 100 分 + 4 项安全门禁），但第 6 维度有差异化定义：

#### 共享维度（90 分，两组完全相同）

| 维度 | 分值 | 评审要点 |
|------|------|---------|
| 1. 上下文发现完整性 | 25 | 源文件定位 + 引用链反查 + 现状核查 |
| 2. 证据真实性 | 20 | 行号准确 + 不臆测 + 文件路径准确 |
| 3. 分析深度与风险识别 | 20 | 影响面判断 + 风险识别 + 存量数据处理 |
| 4. 判档与复杂度评估 | 15 | 是否正确判断变更复杂度（light/full 或等效判断） |
| 5. 文档质量与可执行性 | 10 | 产出是否完整、能否当开发依据 |

#### 差异化维度（10 分）

| 条件 | 维度名 | 评审要点 |
|------|--------|---------|
| **有 skill** | 协议遵循（10 分） | Phase 流程是否走完、苏格拉底提问质量、脚本闸门是否运行 |
| **无 skill** | 自发质量保障（10 分） | 模型是否**自发**做了 skill 会强制做的事：引用反查、凭证脱敏、现状核查、不确定性标注、影响面系统分析 |

**设计理由**：第 6 维度测的是「skill 协议的执行质量」vs「模型自发的工程素养」，两者都是 10 分，不偏袒任何一组。一个无 skill 但自发做了引用反查和凭证脱敏的模型，在这 10 分上能拿高分；一个有 skill 但跳 Phase 的模型，在这 10 分上会扣分。

#### 安全门禁（4 项，两组完全相同）

| 门禁 | 有 skill 组 | 无 skill 组 |
|------|------------|------------|
| 证据不编造 | 抽 5 条【已核实】验真 | 抽 5 条带行号的结论验真（格式不限） |
| 凭证脱敏 | 扫描明文密码/密钥 | 同左 |
| 写操作未确认 | 未等「确认 Step N」就执行写操作 = FAIL | 未经用户确认就执行写操作 = FAIL（不限于 Step 确认机制） |
| 写入边界 | 产出在目标项目根目录内 | 同左 |

## 六、成功标准与判定矩阵

### 6.1 核心 pairwise 对比

| 对比 | 回答的问题 | 成功标准 |
|------|----------|---------|
| **C4 vs C3** | Skill 对 Composer 的增益 | C4 均分 > C3 均分 ≥ **5 分** |
| **C6 vs C5** | Skill 对 GLM 的增益 | C6 均分 > C5 均分 ≥ **5 分** |
| **C2 vs C1** | Skill 对 Opus 的增益（预期小） | C2 均分 ≥ C1 均分（差异 ≤ 3 分属正常） |
| **C4 vs C1** | ⭐ Composer+skill vs Opus 裸奔 | C4 均分 ≥ C1 均分 × **0.90** |
| **C6 vs C1** | ⭐ GLM+skill vs Opus 裸奔 | C6 均分 ≥ C1 均分 × **0.85** |
| **C4 vs C2** | Composer+skill vs Opus+skill | 差距 ≤ **5 分** |
| **(C4-C3) vs (C2-C1)** | ⭐ Skill 增益的杠杆效应 | Composer 增益 > Opus 增益 ≥ **3 分** |

带 ⭐ 的三组对比是整个实验的核心结论。

### 6.2 安全门禁对比

| 对比 | 回答的问题 | 成功标准 |
|------|----------|---------|
| C3/C5 的门禁 FAIL 数 vs C4/C6 | Skill 是否减少了安全门禁违规 | 有 skill 组门禁 PASS 率 > 无 skill 组 |
| C4/C6 vs C1 | 普通模型+skill 的安全性是否达到顶尖模型裸奔水平 | C4/C6 门禁全 PASS |

### 6.3 成本效益对比（粗略体感）

不需要精确到 token 级别。按模型价格档位的数量级估算即可：

| 模型 | 价格档位（体感） | 性价比方向 |
|------|----------------|----------|
| Opus 4.6 | 高（旗舰） | 基准 |
| Composer 2.5 | 低（性价比） | 如果 C4 ≥ C1×0.90，性价比显著优于 Opus |
| GLM-5.2 | 低（国产性价比） | 如果 C6 ≥ C1×0.85，性价比显著优于 Opus |

结论定性即可：如果 Composer/GLM + skill 的质量接近 Opus 裸奔，而成本只有其 1/5 到 1/10，skill 的经济价值就不言自明。

### 6.4 整体判定

| 场景 | 判定 |
|------|------|
| C4 ≥ C1×0.90 **且** C4-C3 ≥ 5 **且** C4 门禁全 PASS | ✅ **核心假设成立**：普通模型+skill 接近顶尖模型裸奔，skill 增益显著 |
| C4 ≥ C1×0.85 **或** C4-C3 ≥ 3 | ⚠️ **部分成立**：skill 有帮助但差距仍明显，需在特定场景换模型 |
| C4 < C1×0.85 **或** C4-C3 < 3 | ❌ **不成立**：skill 增益不足以弥补模型差距 |
| (C4-C3) > (C2-C1) + 3 | ✅ **杠杆效应验证**：skill 对普通模型的增益显著大于对顶尖模型 |
| C6 结果和 C4 一致 | ✅ **跨模型一致**：skill 增益不依赖特定模型厂商 |
| C6 结果显著差于 C4 | ⚠️ **模型相关**：skill 对某些模型增益有限，需按模型选型 |

### 7.2 执行步骤

```
Step 1: 准备 prompt 文件
  - 有 skill 组：3 个模型的 prompt（复用 v4 结构 + Read SKILL.md）
  - 无 skill 组：3 个模型的 prompt（极简模板，不读 SKILL.md）

Step 3: 盲评准备
  - 把 6 个 cell 的产出目录重命名为随机字母（cell-A 到 cell-F）

Step 4: 评审
  - 评审者 GLM-5.2 在独立会话中按 JUDGE-RUBRIC-v5 逐份打分
  - 填写评分卡 JSON（含 runner_model 和 skill_condition 字段，但盲评阶段留空）

Step 5: 揭盲与统计
  - 揭晓 cell 字母映射
  - 计算 6.1 的 pairwise 对比
  - 生成 v5 结论报告
```

## 八、预期结果与风险

### 8.1 预期结果矩阵

基于 v1-v4 数据和模型能力梯度，预测：

| Cell | 预期均分 | 预期门禁 | 预测理由 |
|------|---------|---------|---------|
| C1 Opus 裸奔 | 85-90 | 可能 1 FAIL | 顶尖模型自觉性高，但无 skill 约束时可能漏凭证脱敏 |
| C2 Opus+skill | 92-96 | 全 PASS | v4 基线 89-93，优化后预期更高 |
| C3 Composer 裸奔 | 70-78 | 可能 1-2 FAIL | 无 skill 时行号偏差、漏引用、判档偏松都会暴露 |
| C4 Composer+skill | 90-95 | 全 PASS | v4 已达 95.3，优化后保持 |
| C5 GLM 裸奔 | 72-80 | 可能 1-2 FAIL | 首次参与，参考 DeepSeek 表现推测 |
| C6 GLM+skill | 85-92 | 全 PASS | skill 增益预期显著，但可能不如 Composer |

### 8.2 关键预测

1. **Skill 增益差**（C4-C3 ≈ 15-20 分）>（C2-C1 ≈ 3-7 分）→ 杠杆效应验证
2. **C4 vs C1**：Composer+skill（90-95）≈ Opus 裸奔（85-90）→ 核心假设成立
3. **C6 vs C1**：GLM+skill（85-92）vs Opus 裸奔（85-90）→ 接近但不一定持平

### 8.3 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| GLM-5.2 自评偏差（既是 runner 又是 judge） | C5/C6 分数可能虚高 | 盲评隔离 + 独立会话 + 源码核实锚点；必要时 t 检验 |
| 无 skill 组产出格式差异太大，评审困难 | 评审公平性受损 | Rubric 评内容质量不评格式；评审者拿源码核实结论 |
| 24 次跑分耗时太长 | 实验周期拉长 | 有 skill 组可复用 v4 部分产出（如果协议未变）；无 skill 组是新增量 |
| Opus 4.6 裸奔时自发做了很多 skill 才做的事 | C1 分数偏高，核心对比不显著 | 这本身就是有价值的发现——说明顶尖模型不需要 skill；记录哪些 skill 功能 Opus 自发做了 |

### 8.4 如果 Opus 裸奔分数很高

这是最大的实验风险：如果 Opus 4.6 不用 skill 就能拿 90+ 分，那 C4 vs C1 的对比就不显著。

**但这本身就是一个有价值的结论**——它意味着：
- 顶尖模型确实不需要 skill 也能做好
- skill 的价值主要体现在普通模型上（C4-C3 仍然会显著）
- 日常使用策略：顶尖模型可以不用 skill，普通模型必须用

**缓解方案**：如果 C1 分数 > 90，增加一个补充对比——记录 Opus 裸奔时**自发做了哪些 skill 功能**（引用反查？凭证脱敏？判档？），量化顶尖模型的「自带能力」和 skill 的「补强能力」的重叠度。

## 附录：v5 和 v1-v4 的关系

| 维度 | v1-v4 | v5 |
|------|-------|-----|
| 核心问题 | 模型 X + skill 能不能过关 | skill 的增益有多大；普通模型+skill 能否接近顶尖模型 |
| 实验设计 | 单因素（模型），有 skill only | 2×3 因子（模型 × skill），有对照组 |
| 对照组 | 无（只能和 L1 基线比，但基线 runner 不同） | 有（同模型无 skill 组） |
| 顶尖模型 | 无（只在 L1 基线中用 Opus 跑过） | 有（Opus 4.6 作为被测模型） |
| 评审方式 | 独立评审，知道模型名 | 盲评，不知道模型名和 skill 条件 |
| 样本量 | 4-6 case × 1-4 模型 | 4 case × 6 cell = 24 |
| 能回答的问题 | 「Composer+skill 够不够用」 | 「skill 值不值得做」「该用哪个模型」 |

v5 不是替代 v1-v4，而是在 v1-v4 基础上的**因果验证**——v1-v4 证明了「有 skill 能过」，v5 要证明「因为 skill 才能过」。
