# ImpactRadar 真实使用案例复盘

> 本文是公开版复盘，不包含原始对话全文、私有项目路径或业务敏感代码。案例来源于 [hxd-ggsddu](https://github.com/hxd-ggsddu) 的真实使用反馈。

## 为什么要整理这个案例

`impact` 和 `impact-pro` 最初已经能输出影响分析文档，也有 light/full、Step 确认和验证方案。但真实使用里暴露出一个更难的问题：用户不是一次性给出完整需求，而是在长会话里持续推进一个已有系统的迁移/对齐任务。

这类任务有几个特点：

- 总目标很大，当前 Step 只是其中一小段。
- 用户会隔很久才回复 `确认 Step N`。
- 项目不一定是 Git 仓库，不能依赖 `git diff` 作为唯一审计方式。
- 变更看起来是一两行字段透传，但背后可能是 API 响应、历史兼容、文档和验证等级问题。
- 弱模型或长上下文下，agent 容易把“静态检查过”说成“完整验证过”。

所以这个案例的价值不是“某个字段改对了”，而是让我们看到 Skill 在真实链路里必须更硬的地方。

## 案例抽象

场景可以抽象成：

```text
源系统 A 的某个功能已经存在。
目标系统 B 正在逐步对齐 A。
用户希望 agent 根据进度文档继续实现后续功能。
每一轮只适合落一个小闭环，但总目标还没有结束。
```

真实执行中出现过的典型片段包括：

- agent 先只读发现源系统与目标系统差距。
- agent 给出一个 Step，例如“补齐某个响应字段 fallback”。
- 用户先回复模糊的“确认”，agent 必须要求 `确认 Step N`。
- 执行后需要记录做了什么、为什么这样做、验证到什么等级。
- 后续继续推进时，agent 不能忘记当前总目标和未验证项。

## 暴露的问题

### 1. 长期目标不能只靠聊天记忆

如果没有“长期目标模式”，agent 容易只盯住当前一行代码，忘记它属于迁移/对齐链路的一部分。后果是：

- 后续 Step 没有 backlog。
- 未验证项丢失。
- 新窗口接不上前一轮。

补强结果：

- `impact` / `impact-pro` 增加长期目标模式。
- Context Pack 和 light 摘要中增加总目标、当前 Step、Backlog、阻塞项和运行时未验证项。

### 2. 源系统对齐不能凭命名相似

真实场景里，源系统字段和目标系统字段可能名字相近，但语义、fallback 顺序、默认值、历史兼容并不一致。

补强结果：

- 增加“源系统到目标系统对齐”规则。
- 必须记录可信来源、目标实现、对齐语义、差距证据、本 Step 范围和不确定项。

### 3. API 响应字段需要单独检查

有些改动看起来只是“多透传一个字段”，但它会影响前端、外部消费者、文档、generated client 或历史快照。

补强结果：

- 新增“接口返回检查清单”。
- 向后兼容新增字段可以 light，但删除、重命名、语义变化或消费者不明必须 full 或先补证据。

### 4. 静态验证和运行时验证必须分开

真实使用中，agent 可能读了文件和 grep 过引用，但没有跑构建、单测或接口请求。此时只能说达到 V1，不能写成完整通过。

补强结果：

- 增加 V0-V3 验证等级。
- V1 是静态验证，V2 是构建/测试/lint，V3 是 API/UI/DB/集成真实路径。
- 找不到项目命令时，必须写“V2/V3 不可用/需补证据”，不能套用占位命令。

### 5. 非 Git 项目需要降级保护

真实案例里目标目录不一定是 Git 仓库。没有 Git 时，如果直接写文件，就缺少审计和回滚依据。

补强结果：

- 增加非 Git 降级保护。
- 写操作前必须说明无法使用 Git 审计，给出 before/after、文件 hash、备份路径或用户接受风险。

### 6. 阻塞恢复不能消费旧授权

用户隔很久才回复 `确认 Step N` 时，文件状态可能已经变了。agent 不能直接把旧计划拿来执行。

补强结果：

- 增加阻塞恢复安全闸。
- 恢复后必须复述 pending Step、只读检查当前文件状态、判断风险是否升级，再决定是否需要重新确认。

### 7. 只分析也要给证据

后续 MiniMax M3 复测发现，简单 UI-only 场景下，弱模型可能只输出“已闭环/证据已回填”，但没有展示 profile、Context Pack、判档证据和验证命令来源。

补强结果：

- 增加“只分析 / 不写文件的最小响应契约”。
- 即使不落文档、不写文件，也必须输出上下文摘要、判档证据、风险门禁、验证等级和下一步确认。

## 复测证据

相关公开验证记录：

- `skills/impact/validation-runs/2026-06-08-T04-m3-real-slash-regression.md`
- `skills/impact/validation-runs/2026-06-08-T05-m3-rg3-response-and-recovery.md`
- `skills/impact-pro/validation-runs/2026-06-08-T47-m3-real-slash-node-response-contract.md`
- `skills/impact-pro/validation-runs/2026-06-08-T48-m3-rg3-frontend-monorepo.md`

这些复测证明：

- `impact` 能在 M3 下触发长期目标、阻塞恢复、非 Git 降级、验证等级和最小写操作闭环。
- `impact` 能把响应字段删除判为 full，并且在延迟确认后不直接执行。
- `impact-pro` 能在 Node/Express 响应字段删除场景下避免误判 light。
- `impact-pro` 能在 monorepo 前端文案场景下只读确认后端边界，不扩大实现范围。
- 首轮 React UI-only 复测暴露出“只分析输出过短”问题，修复后复测通过。

## 结论

这个案例把 `impact` / `impact-pro` 从“文档上讲得通”继续推进到“真实长会话里更不容易跑偏”。它的核心价值是暴露了那些单元测试很难覆盖、但 agent 实际使用中很常见的边界：

- 长期目标状态维护
- Step 级确认
- 恢复后重新只读检查
- 静态验证和运行时验证分级
- 非 Git 降级审计
- 只分析也必须展示证据

后续不应继续无边界堆规则，而应按 `docs/impact-regression-protocol.md` 的 RG0-RG3 协议，用真实 agent 复测来判断哪些规则真的必要。
