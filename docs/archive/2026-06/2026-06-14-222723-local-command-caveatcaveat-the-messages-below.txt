❯ L0 L1 L2有什么区别                                                            

  Thought for 4s (ctrl+o to expand)

● L0 / L1 / L2
  是三层由轻到重的复验，区别主要在测什么、谁/什么来测、花多少钱、什么时候跑：

  ┌──────┬────────────────────────────┬────────────────┬─────────┬────────┐
  │ 层级 │           测什么           │     怎么跑     │  成本   │ 触发时 │
  │      │                            │                │         │   机   │
  ├──────┼────────────────────────────┼────────────────┼─────────┼────────┤
  │ L0   │ 文档/契约/引用/schema/fixt │ 脚本           │ 免费（  │ 每次改 │
  │ 静态 │ ure 对不对                 │ skills/<skill> │ 纯本地  │ 动必跑 │
  │ 自洽 │                            │ /tests/run.sh  │ ）      │        │
  ├──────┼────────────────────────────┼────────────────┼─────────┼────────┤
  │ L1   │ subagent 扮用户跑真实      │ eval/run-l1.sh │ 便宜模  │ releas │
  │ 行为 │ case，客观维度 + 安全门禁  │  + 评分卡      │ 型      │ e 前 / │
  │ 契约 │                            │                │         │  定期  │
  ├──────┼────────────────────────────┼────────────────┼─────────┼────────┤
  │ L2   │ 主观质量（苏格拉底式追问、 │ 人工 +         │         │ 里程碑 │
  │ 人审 │ 地图可读性、文档质感）     │ 多模型评委     │ 贵      │  / 红  │
  │ 深度 │                            │                │         │ 线命中 │
  └──────┴────────────────────────────┴────────────────┴─────────┴────────┘

  更直白一点：

  - L0 = 自查格式和规则有没有自相矛盾：铁律写没写、引用文件存不存在、scenario
  是否符合 schema、fixture 锁没锁死。类似编译 + lint，不跑真实业务逻辑。
  - L1 = 让模型真的当用户用一次：给 subagent 一个任务，看 skill
  会不会拒绝不该做的、能不能产出符合 rubric 的交付物、安全门禁 gate_holds 是否全
  true。这是行为层面的回归。
  - L2 = 人/强模型品质量：产出读起来专不专业、有没有漏掉关键盲区、地图能不能让人
  看懂。偏主观，所以贵，只在重要节点或 L1 发现红线问题（比如契约
  PASS→FAIL）时启用。

  文档里的复验循环是：

  改 skill → L0（必跑） → L1（按触发） → 和基线 diff → 红线命中则 L2

  所以 L0 是门槛，L1 是常态把关，L2 是兜底深度审。

  第一步：PreToolUse hook（把"写前确认"从贴纸条变电子锁）

  一句话：Claude Code
  有个机制——在模型调用任何工具（Edit/Write/Bash）真正执行之前，harness
  先跑一段你写的脚本（hook），这段脚本可以拦住这次调用。拦截发生在 harness
  层，不是模型层，所以模型绕不过。

  现在（软规则）：用户说"删 sys_user.remark"。模型跑 impact，读了 SKILL.md
  里的铁律 #1「最高确认法」，于是乖乖做分析、追问、等"确认 Step
  N"。但如果模型今天状态不好（弱模型/上下文压缩后忘了铁律/被用户催），它可能直接
  Edit SysUser.java 把字段删了——skill 文本说不许，但没人物理上拦住这个 Edit
  工具调用，全靠模型自觉。

  模型: 我要 Edit SysUser.java 删 remark
  harness: (没人拦) → 执行了 → 改坏了

  加了 hook（硬机制）：settings.json 里配一段——"凡是 Edit/Write
  目标项目源码，先跑我的 hook 脚本"。hook
  脚本检查：这次写之前，对话里有没有对应的『确认 Step N』？ 没有 → hook 返回
  exit 2，工具根本不执行，模型收到一条"被拦截：需要先确认 Step
  N"的提示，被迫回到分析流程。

  模型: 我要 Edit SysUser.java 删 remark
  hook(在工具执行前自动跑): 检查 → 没有"确认 Step N" → 拦！返回 exit 2
  模型收到: "被拦截，需先确认" → 只能回去做分析、等确认

  类比：软规则 =
  门上贴张「请先确认再进」的纸条（自觉的人遵守，不自觉的直接进）；hook =
  门装电子锁，不刷卡门物理上打不开。你刚修的
  P3D（铁律进压缩区）是在"让纸条更显眼"；hook 是"直接换电子锁"。

  参考 trailofbits/claude-code-config

  ---
  第二步：tree-sitter 依赖图适配器（把"引用发现"从肉眼找变精确索引）

  一句话：tree-sitter 是个代码解析器，能把源码解析成
  AST（精确的结构树），从而算出"这个字段被哪些文件哪些行真实引用"，构建一张真实
  依赖图——是算出来的，不是模型猜的。

  现在（grep + 模型猜）：impact Phase 2 想找"remark 被谁引用"。模型 grep remark
  全仓，拿到 20 个命中文件。然后它得判断：哪些是真引用（resultMap 
  property="remark"）、哪些是注释/字符串/无关变量（别的类也叫
  remark）。弱模型（kimi）可能漏掉 BaseEntity.java（remark
  真正定义处）、或把无关命中当引用。铁律
  C1「不编造」要求模型自觉，但模型靠读片段判断，会错。

  grep remark → 20 个命中 → 模型逐个读片段猜 → 可能漏 BaseEntity、可能误判

  加了 tree-sitter 适配器（真实图）：Phase 2 调用一个 tree-sitter
  工具，它解析全部 .java/.xml，构建符号引用图，直接返回精确清单：

  符号 remark:
    定义: BaseEntity.java:39 (private String remark)
    引用: SysUserMapper.xml (resultMap L26 / select L51 / select L61 / insert
  L160 / update L175 / updateLoginInfo L194)
          index.vue (form.remark L159, L339)
          view.vue (info.remark L120)

  这是 AST 级精确的，不会漏、不会编造行号。模型拿这份真实清单做影响分析，R1 的
  BaseEntity 陷阱、Mapper 6 处引用都是工具给的，不是 grep + 猜的。

  类比：现在 = 用肉眼+放大镜（grep）在一堆文件里找谁提到
  remark，会看漏看错；tree-sitter = 图书馆有一份精确符号目录，查 remark
  所有引用位置一清二楚。

  参考 vk0dev/code-impact-mcp
  (https://github.com/vk0dev/code-impact-mcp)，它就是 tree-sitter 建依赖图 +
  输出 PASS/WARN/BLOCK。

  权衡（为什么是"可选适配器"）：tree-sitter 要装解析器（每语言一个 grammar 或
  tree-sitter-language-pack），是额外依赖。你现在 skill 是"零依赖纯
  markdown，跑哪都行"。所以做成像 db-adapters/ 那样的可选适配器——用户装了
  code-impact-mcp 之类就用真实图，没装就降级回 grep+模型，并在产出里诚实标"V1
  静态/未用图"。不强制、不破坏轻量定位。
