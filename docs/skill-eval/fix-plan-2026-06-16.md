## 修复 1 🔴 P1 — mysql.md 外键查询取错列（发现 E）

### 文件

`skills/impact-pro/db-adapters/mysql.md`

### 问题

`foreign_keys` 查询从 `INFORMATION_SCHEMA.KEY_COLUMN_USAGE` 取了 `UPDATE_RULE` 和 `DELETE_RULE`。

这两列只存在于 `INFORMATION_SCHEMA.REFERENTIAL_CONSTRAINTS`。任何 MySQL 版本执行此 SQL 都会报：
```
ERROR 1054 (42S22): Unknown column 'UPDATE_RULE' in 'field list'
```

### 修复方案

**对齐 impact**（`impact/references/schema-discovery.md:10` 正确版本）：直接删掉 `UPDATE_RULE` 和 `DELETE_RULE` 两列，只保留 KEY_COLUMN_USAGE 真实存在的 4 列。

### 精确修改

**文件：`skills/impact-pro/db-adapters/mysql.md`**

旧文本：
```sql
  foreign_keys: |
    SELECT
      TABLE_NAME,
      COLUMN_NAME,
      REFERENCED_TABLE_NAME,
      REFERENCED_COLUMN_NAME,
      CONSTRAINT_NAME,
      UPDATE_RULE,
      DELETE_RULE
    FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE
    WHERE TABLE_SCHEMA = DATABASE()
      AND (REFERENCED_TABLE_NAME = '目标表' OR TABLE_NAME = '目标表')
      AND REFERENCED_TABLE_NAME IS NOT NULL
```

新文本：
```sql
  foreign_keys: |
    SELECT
      TABLE_NAME,
      COLUMN_NAME,
      REFERENCED_TABLE_NAME,
      REFERENCED_COLUMN_NAME,
      CONSTRAINT_NAME
    FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE
    WHERE TABLE_SCHEMA = DATABASE()
      AND (REFERENCED_TABLE_NAME = '目标表' OR TABLE_NAME = '目标表')
      AND REFERENCED_TABLE_NAME IS NOT NULL
```

### 验收

```bash
# 对任一 MySQL 5.7+ 库执行此 SQL，应返回结果不报错
mysql -u user -p -D database -e "
SELECT TABLE_NAME, COLUMN_NAME, REFERENCED_TABLE_NAME,
       REFERENCED_COLUMN_NAME, CONSTRAINT_NAME
FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE
WHERE TABLE_SCHEMA = DATABASE()
  AND REFERENCED_TABLE_NAME IS NOT NULL
LIMIT 1"
```

## 修复 5 🟡 P2 — generic-sql 方言标注诚实化（发现 D）

### 文件

`skills/impact-pro/db-adapters/generic-sql.md`

### 问题

- `indexes` 查询用 `information_schema.statistics` + `non_unique`/`seq_in_index` → **MySQL 专属**
- `foreign_keys` 查询用 `information_schema.constraint_column_usage` → **PG/SQL 标准，MySQL 无**
- notes 只含糊提"外键/视图/触发器"兼容性，**没提 indexes**

### 修复方案

不改查询（保持兜底面兼容性），只在 notes 如实声明每个查询的方言适用面。

### 精确修改

**文件：`skills/impact-pro/db-adapters/generic-sql.md`**

旧文本：
```yaml
notes:
  limitations:
    - 无任意 SQL 执行能力（execute_sql / query 等）时仅能查询表结构，无法运行数据验证 SQL
    - 外键/视图/触发器依赖具体 DB dialect，可能不兼容
    - 行数统计需 DB-specific SQL
```

新文本：
```yaml
notes:
  limitations:
    - 无任意 SQL 执行能力（execute_sql / query 等）时仅能查询表结构，无法运行数据验证 SQL
    - **通用性诚实声明**：以下查询非真·跨库。在任何单一数据库上至少有一条会失败：
      - `indexes` 基于 INFORMATION_SCHEMA.STATISTICS（列名 non_unique/seq_in_index），是 **MySQL 专属**。PostgreSQL / SQL Server / Oracle / SQLite 均无此视图或列名不同
      - `foreign_keys` 基于 INFORMATION_SCHEMA.CONSTRAINT_COLUMN_USAGE，是 **SQL 标准 / PostgreSQL 专属**。MySQL 无此视图
      - `table_structure` 基于 INFORMATION_SCHEMA.COLUMNS，是真跨库的
      - 因此: MySQL 上 indexes 可用但 foreign_keys 报错；PostgreSQL 上 foreign_keys 可用但 indexes 报错
    - 数据库类型已知时，**必须走对应的 mysql.md / postgresql.md adapter**，不走此兜底
    - 外键/视图/触发器依赖具体 DB dialect，可能不兼容
    - 行数统计需 DB-specific SQL
```
