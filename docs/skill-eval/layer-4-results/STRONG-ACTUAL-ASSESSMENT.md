# 强模型 T01–T06 实际产物评估

> 评估时间：2026-06-16
> 评估对象：强模型（glm5.1 / opus 4.8 等）实际跑测产出的需求/设计/实施/light 文档
> 评估目的：对比 SKILL 模板定义，识别强模型实际跑出的文档有何发挥（超出/不足模板）、质量是否达到行业标准
> 数据来源：
> - 评测报告：`docs/skill-eval/layer-4-results/T0X-*.md`
> - 实际产物：`test-projects/{ruoyi-vue, go-admin, full-stack-fastapi-template}/change-impact/...`
> - 模板定义：`skills/{impact, impact-pro, pathfinder}/templates/*.md`

---

## 评估框架

每场按以下维度打分（10 分制 / 行业标准对照）：

| 维度 | 说明 |
|------|------|
| 模板符合度 | 实际产物与 SKILL 模板定义的章节是否对齐 |
| 超出模板 | 在模板之外给出的增量价值（背景摘要、状态机、回滚方案等） |
| 缺失/弱化 | 模板要求但实际产物未覆盖或敷衍的部分 |
| 行业标准 | vs IEEE 830（需求）/ arc42（设计）/ Agile User Story（实施）的对照 |

行业标准四档：缺（无）/ 弱（提及但不可用）/ 达（达成行业基线）/ 超（超出行业基线）

4. **需求文档** §7 依赖关系比较薄弱（"无外部依赖"）——但 light 场景确实依赖少，可接受

### 质量亮点

6. **实施文档 §3 Step 7** 给出登录校验逻辑的"前/后"完整对比代码（不只是行号）——可执行性极强
7. **实施文档 §5 错误用例** 包含 3 类（admin 拦截 / freezeEndTime=null / freezeEndTime=过去时间）——边界场景覆盖度好

### 缺失/弱化的部分
1. **设计文档 §3.5** 没有"消息队列/事件 Schema 变更"和"监控/告警配置变更"——但本次变更不涉及，可接受
2. **设计文档 §6 数据迁移策略** 给了 DDL + "存量 status 值不变，无需 UPDATE"——未涉及历史快照，但 light 场景无需历史快照，OK
3. **实施文档 §4 回滚方案** 给了每步独立回滚——但未给"组合回滚顺序"（如 Step 1+2+3 同时失败的回滚顺序）——这是 P2 弱化点

5. **需求文档 §7 依赖关系** "无外部依赖"——比 T01 略弱（同样是轻量场景，依赖关系应有"权限数据依赖" `sys_role_menu`）——这是 P3 弱化点

### 质量亮点
- **语义冲突成功捕获**（"account_status vs status"）：报告明确指出"用户原需求'加 account_status 字段'，skill 通过苏格拉底追问发现与现有 status 语义重叠，引导用户合并为复用 status + 枚举扩展，避免多状态字段逻辑混乱"——这是非常专业的领域知识（避免多状态字段逻辑混乱是 20 年经验总结出来的反模式）
- **UserStatus 枚举陷阱识别**：DELETED("2") 已占 code="2"，但对应 del_flag 而非 status，FROZEN("2") 不冲突——这种细节只有阅读源码才能发现，避免了"误判冲突"
- **登录校验位置发现**："status 校验不在 SysLoginService 而在 UserDetailsServiceImpl（第51行）"——这种代码定位精度是高质量技术输出的标志
- **状态机变更章节**：明确画出"停用不可直接转冻结"的状态转换约束，是状态机分析的标准输出
- **9 步分解**：把一个看似简单的"加冻结功能"分解为 9 步（DDL + 枚举 + Entity + Mapper XML + Mapper Java + Service + Controller + 登录校验 + 权限 + 验证）——这种粒度可执行性极强

### 质量亮点
- **诚实指出"已在 Item model 里定义"与实际代码不符**——这是高质量技术输出的核心标志，不盲从用户描述
