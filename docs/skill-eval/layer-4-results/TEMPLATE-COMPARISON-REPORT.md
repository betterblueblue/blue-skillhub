| **版本/修订历史** | IEEE 830 强制 | ❌ 缺 | 10% | 5% | P2 |

**核心结论**：

1. **三 skill 模板是"工程师视角的工程模板"**——安全闸 + 强制确认 + 凭证脱敏 + V0-V3 验证等级**做得好**；**业务/运维/产品视角的完整文档维度**全部缺失。
2. **强模型比弱模型质量高 15-20%**——但**两者的不足大多是同一类**（缺 NFR / 缺 ADR / 缺干系人 / 缺部署）。**说明：模型能力差 15%，模板约束力还差 85%**。
3. **优先改模板，不是改模型**——前 5 项 P0/P1 全部是模板定义层面就缺，模型再强也补不上。

## Part 1：业界基准（4 份范本共同最小集）

来自 [INDUSTRY-TEMPLATES-RESEARCH.md](INDUSTRY-TEMPLATES-RESEARCH.md) 跨范本对比表：

| 维度 | IEEE 830 | arc42 | Google Design Doc | Microsoft SDL |
|------|----------|-------|-------------------|----------------|
| 唯一编号/可追溯 | REQ-N.N | §n.n | doc-as-code | Bug bar / work item |
| 利益相关者表 | 隐式 | **显式 §1** | 隐式 | 隐式 |
| Goals vs Non-Goals | 隐式 | §1.2 质量目标 | **显式 + 强调** | 安全/非安全 |
| 替代方案 + 理由 | 隐式 | §9 决策 | **§4 Alternatives 必填** | Threat model |
| 跨切面关注点 | 隐式 | **§8 Crosscutting** | **显式必含** | privacy/security/IR |
| 反模式自检 | 散落注释 | docs.arc42 FAQ | **原文 6+ 显式禁令** | **"shift-left" 4+ 禁令** |
| 长度上限 | 无硬限 | ~20 页 | **10-20 页 sweet spot** | 阶段交付物制 |
| 修订/版本控制 | 强制 | version.properties | "doc 与现实漂移"警告 | 持续 response 循环 |
| 验证/可追溯矩阵 | §4 verification | §10 质量场景 | review rounds | SAST/DAST/SCA/pen-test |

**4 份范本共同最小集（7 项）**：
1. 唯一编号 + 可追溯矩阵
2. Goals / Non-Goals 分明
3. 替代方案 + trade-off 显式
4. Cross-cutting 必含（安全/隐私/可观测）
5. 反模式自检（"避免 X"清单）
6. 长度/范围约束（反对大部头 + 反对无 trade-off 的伪设计）
7. 修订历史 + 持续维护承诺

## Part 2：模板差距（10 项系统性缺失）

来自 [SKILL-TEMPLATES-INVENTORY.md](SKILL-TEMPLATES-INVENTORY.md) E.2 节，按维度归并：

### P0：行业最低要求

#### 1. 元数据 / 可追溯性（缺 8 子项）
- 缺：版本号、修订历史、生成日期/时间、生成人/审阅人、分发范围
- 缺：需求/设计/实施/验证之间的 traceability matrix
- 缺：模板自身的版本号
- 仅 pathfinder【0】有"生成时间"和"基于 commit"

#### 2. 非功能需求（NFR）— 8 维度全部缺
- 性能（latency / throughput / 资源占用）
- 可用性（uptime / 灾备）
- 可维护性（代码复杂度阈值）
- 可移植性
- 安全性（OWASP）
- 合规（GDPR / PCI / HIPAA）
- 020 line 99 提到"按需展开"但无触发条件
- 缺 SLO / SLA / SLI 定义

### P1：架构文档核心要素

#### 3. Goals / Non-Goals 显式分开
- IEEE 830 隐式、arc42 §1.2 显式、Google Design Doc **显式 + 强调**
- 三 skill 模板全部缺

#### 4. 替代方案 + trade-off
- Google Design Doc §4 **强制必填**
- arc42 §9 决策
- 三 skill 020 决策理由写得简单，缺 trade-off 量化

#### 5. Cross-cutting 关注点
- arc42 §8 + Google Design Doc 显式必含
- 仅 pathfinder【9】风险区域是"半横切面"——但没结构化分类

#### 6. 反模式自检清单
- Google Design Doc 原文 6+ 显式禁令
- Microsoft SDL "shift-left" 4+ 禁令
- pathfinder 已有 8 条反模式防御（做得最好）
- impact / impact-pro 几乎无结构化反模式清单

#### 7. 部署 / 灰度 / Go/No-Go
- Microsoft SDL Release 阶段强制
- 060-preflight 是 yes/no 决定，**无业务指标阈值**（"yes 时延迟 < 200ms 才算 yes"）

### P2：业务/协作/治理

#### 8. 干系人 / RACI
- arc42 §1 显式 stakeholder × expectation
- 三 skill 模板全部缺

#### 9. 长度/范围约束
- arc42 20 页上限、Google 10-20 页 sweet spot
- 三 skill 模板无任何"小变更拆出去"的指导

#### 10. 验证深度
- 测试金字塔（unit / integration / e2e 比例目标）
- 覆盖率阈值（line/branch/path）
- 性能测试 / 压力测试 / 混沌测试
- 契约测试（contract test）
- 回归测试范围
- 测试数据管理

### 强模型共性强项（5 项）
1. **诚实决策**：不被用户描述绑定，按 evidence-based 决策
2. **状态机分析**：T02 独立章节是专业水平
3. **代码定位精度**：行号 + 完整代码片段混合引用
4. **跨语言栈识别**：Go AutoMigrate / Python Alembic / Java MyBatis 演进差异
5. **Mermaid 图精度**：实线/虚线区分已核实/推断

### 强模型共性弱项（5 项）
1. **T03 Go light 缺 6+ 章节**（违反 "light 不跳过安全检查" 原则）
2. **T01/T02 需求文档偏技术语言**（缺业务语言示例）
3. **T03/T04 light 缺错误用例 + 显式回滚命令**
4. **组合回滚顺序缺失**（多步同时失败时怎么回滚没说）
5. **环境回退路径不完整**（3 行 vs 模板要求 4 种典型场景）

### 弱模型共性弱项（3 项）
1. **过度保守**（T07 字段未澄清不写文档）
2. **文档-文件系统不一致**（T10 引用不存在文件）
3. **轻量覆盖**（frontend 未读 / 轻量文档）

## Part 5：改进建议（按优先级）

### P0：模板定义层（影响所有用户/所有 skill）

#### 1. **所有需求/设计/实施/验证模板加「时间戳 + 审阅人 + 版本号」字段**
- **对应业界基线**：IEEE 830 §1.4 修订历史 + 版本号
- **模板改动**：000/010/020/030/040/060 顶部加 5 行 metadata
- **验证方式**：跑测时检查文档是否带"生成时间：date 命令输出" + "审阅人：待定"
- **预计提升**：可追溯性 60% → 95%

#### 2. **010-requirements 加「非功能需求」专节**
- **对应业界基线**：IEEE 830 §3.2 Software system attributes（性能/可靠性/可用性/可观测性/合规）
- **模板改动**：010 现有"业务约束"节后插入"NFR"节，含 5 子项（性能/可用性/安全/可维护性/合规）
- **触发条件**：涉及接口性能/数据敏感/合规场景时强制填写
- **预计提升**：NFR 20% → 70%

#### 3. **010-requirements 加「Goals vs Non-Goals」专节**
- **对应业界基线**：Google Design Doc §2 + arc42 §1.2

## Part 6：强 vs 弱模型发挥差异

### 关键反差 1：T08 弱模型反超

**事实**：T08 弱模型 99/100 vs 强模型 9.5/10（折算 95/100）
**根因**：弱模型**更倾向严格按 SKILL.md 走流程**（7 条触发 full 证据逐条列出 / Step 2 显式高风险标注），强模型倾向"省步骤"
**启示**：skill 的硬性规则对弱模型的**约束力 > 强模型**——弱模型不会"聪明地绕过"

### 关键反差 2：T07 弱模型过度保守

**事实**：T07 弱模型 71/100 vs 强模型 9.0/10
**根因**：弱模型遇到"字段真伪冲突"时选择"暂停等用户"，强模型选择"先产骨架让用户看"
**修复**：templates/040-light.md 增加"未澄清时的最小骨架"段落（建议 9）
