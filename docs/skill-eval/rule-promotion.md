# 经验进入核心规则的条件

这份说明用于判断一条真实使用经验能否进入 Skill 的核心规则、模板、校验脚本、写前拦截、技术栈规则或参考资料。它适用于 IntentAnchor、Pathfinder 和 ImpactRadar。使用记录只负责提供线索；没有足够证据时，先留在迭代清单，不直接修改规则。

## 面向用户的交互

普通用户不需要阅读这份文档，也不需要知道内部编号、候选状态和证据门槛。

只有一次运行暴露出 Skill 自身可能需要改进的具体问题时，Skill 才在收尾内容之后询问：

> 这次发现一个可能值得用于改进 Skill 的问题：<一句话说明问题和后果>。要把它记录下来吗？你回复“记录”或“不用”就行。

普通完成不询问。已有门禁当场拦住并修正的一次性格式问题，也不单独询问。

- 用户回复“记录”：在对话中整理完整的改进记录，不修改 Skill，不写入目标项目。
- 用户回复“不用”：结束，不保存、不继续追问。
- 维护者拿到这份记录后，再按下文完成编号、复现、受理和回归；这些内部步骤不要求用户操作。

## 候选状态

候选编号统一使用 `RC-NNN`，状态按下面顺序更新：

```text
waiting-evidence -> reproduced -> accepted -> implemented -> verified
候选在复核过程中确认不成立 -> rejected
```

- `waiting-evidence`：只有线索，证据或复现不足。
- `reproduced`：当前版本可以复现，已有原始产物或失败测试。
- `accepted`：确认属于 Skill 的通用问题，并确定处理位置和复测范围。
- `implemented`：修改已经完成，等待对应回归。
- `verified`：修改前失败的用例已经转为通过，所需回归没有新增 P0/P1。
- `rejected`：确认是单项目特例、环境问题、重复项或无法证明的问题。保留原因，不删除记录。

## 受理条件

候选进入 `accepted` 前必须同时满足：

1. 有原始证据，例如完整对话、实际产物、diff、命令输出或测试报告。只有模型总结不算证据。
2. 当前版本能够复现；无法直接复现时，必须说明原因，并保留足以核验的历史产物。
3. 根因指向 Skill 的规则、模板、工具或验证缺口，而不是项目特殊约定、环境故障或用户需求本身不清楚。
4. 写清影响范围：哪个 Skill、哪种模式、哪个阶段，以及明确不影响什么。
5. 修改前已经知道怎样验证修复有效，并评估新增约束的误伤风险和使用成本。

最低证据门槛：

| 问题类型 | 最低证据 |
|---|---|
| P0/P1 安全、授权、造假或数据风险 | 1 个可复现的真实失败 |
| P2 稳定性或质量问题 | 至少 3 个同类记录，最好来自两个独立任务 |
| 体验或措辞问题 | 多次出现，并能说明修改后减少什么麻烦 |
| 单项目特殊情况 | 不进入核心规则，留在项目文档 |
| 只有理论风险 | 保持 `waiting-evidence` |

跨模型不是硬性条件。同一模型在独立任务中稳定复现，也可以受理，但要把适用范围和已知边界写清楚。

用户明确提出的产品调整可以作为产品决定直接进入 `accepted`，不必凑满日志数量。记录中必须保留用户原话或可核验来源，并且仍要先定义修改前会失败的测试或检查；不能把产品决定伪装成已经多次发生的失败。

## 选择处理位置

| 问题性质 | 优先处理位置 |
|---|---|
| 字段、章节、状态可以机械核对 | 校验脚本 |
| 必须在写入发生前阻止 | Hook 或宿主工具门禁 |
| 需要理解业务语义 | Skill 规则 + 真实 Agent 评测 |
| 只适用于某个技术栈或场景 | `profiles/` 或 `references/` |
| 只影响可读性或交互体验 | 模板或 README |
| 项目自身特殊约定 | 目标项目的 `AGENTS.md` 或项目文档 |

机器不能稳定判断的问题，不要用字符串匹配伪装成硬门禁。能够放进 reference、profile 或模板的知识，不要继续加重核心 `SKILL.md`。

## 晋级步骤

1. 在 `docs/skill-iteration-backlog.md` 登记候选编号、原始证据、影响范围和状态。
2. 在当前版本复现问题，先得到失败测试、失败 fixture 或可核验的真实失败产物。
3. 判断是否满足受理条件；不满足就保持 `waiting-evidence` 或标记 `rejected`。
4. 确定处理位置、预期效果、明确排除项和需要运行的 RG0-RG3 回归包。
5. 实施最小修改，让修改前的失败用例转为通过。
6. 按 `regression.md` 复测。涉及真实 Agent 行为时，复测只提供任务和原始产物，不提前泄露预期答案或修复方案。
7. 更新候选状态，并链接规则、脚本、测试和复测记录；门禁曾经全绿但产物不合格时，同时更新 `eval/real-projects/escape-ledger.md`。

## 候选记录格式

```markdown
| 编号 | 问题 | 原始证据 | 严重程度 | 影响范围 | 复现 | 处理决定 | 验证 | 状态 |
|---|---|---|---|---|---|---|---|---|
| RC-001 | ... | 路径、命令输出或测试报告 | P0-P3 | skill / 模式 / 阶段 | 未复现或失败用例 | 处理位置和排除项 | 测试与回归包 | waiting-evidence |
```

一条正式规则至少应能追溯到：

```text
使用记录或失败产物 -> RC 候选 -> 修改决定 -> 规则或门禁 -> 回归测试 -> 复测记录
```

使用记录仍默认只留在对话中，不自动写入仓库。只有用户明确要求保存，或评测本身要求归档时，才创建候选记录。
