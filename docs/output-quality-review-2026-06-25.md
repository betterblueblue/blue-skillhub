# 三个 Skill 产出物质量评估

> 日期：2026-06-25
> 评估对象：pathfinder、impact、impact-pro 三个 skill 的所有产出文档模板
> 评估维度：面向人类的可读性、面向 agent 的清晰度、内容的科学性
> 评估方法：逐份阅读模板源文件，按受众分类评估
> 关联文档：[docs/skill-optimization-roadmap-2026-06-25.md](skill-optimization-roadmap-2026-06-25.md)（工程严谨性优化路线图）

---

## 一、产出文档的受众分类

三个 skill 一共产出约 10 类文档。当前的文档体系没有显式区分受众，大部分文档同时面向人类和 agent，导致两头都不够好。

| 文档 | 产出 skill | 主要受众 | 次要受众 | 当前问题 |
|------|-----------|---------|---------|---------|
| `_project-map.md` Executive Summary | pathfinder | **人类** | agent 跳过 | ✅ 唯一显式为人类设计的节 |
| `_project-map.md` 【0】-【13】 | pathfinder | **agent** | 人类参考 | ✅ 结构清晰 |
| `000-context-pack.md` | impact/pro | **agent** | 人类复核 | ⚠️ 和 design 有内容重叠 |
| `010-requirements.md` | impact/pro | **人类** | agent 参考 | ⚠️ 非功能需求举例用了技术语言 |
| `020-design.md` | impact/pro | **双重** | — | ❌ 编号 bug + 三段冗余 |
| `030-implementation.md` | impact/pro | **双重** | — | ⚠️ 混了 agent 行为指令 |
| `040-light.md` | impact/pro | **人类** | agent 执行 | ❌ 混入大量 agent 自检语言 |
| `060-preflight.md` | impact/pro | **人类** | agent 遵守 | ❌ 像 ops runbook，术语密集 |
| `090-execution-record.md` | impact/pro | **agent** | 人类审查 | ⚠️ 嵌入了静态行为规则 |
| `_active-state.md` | impact/pro | **agent** | — | ⚠️ 唯一英文模板，一致性差 |

## 二、面向人类的文档：足够人性化吗？

### 做得好的地方

**1. `010-requirements.md` 的"对的比例/错的比例"引导**

模板直接给了正反例：
- 对的："用户列表支持按手机号搜索"
- 错的："给 SysUserController 的 list 方法加一个 phoneModel 参数"

这种引导比抽象规则有效得多。v4 盲测中 Composer 2.5 和 Step 3.7 Flash 在 B3 的需求文档确实更偏业务语言（优化 8 PASS）。

**2. pathfinder 的 Executive Summary**

唯一显式为人类设计的节。30 秒读懂、Quick Start 5 步、Top 3 风险、Top 3 Gotchas，对刚接手项目的人非常实用。模板还标注了"本节为人类快速认知设计，impact 读取时跳过本节"——这是整个文档体系里唯一一处显式区分受众的地方。

**3. `040-light.md` 的"信息不全时的兜底"**

"不要空等，先输出已确认部分 + 待澄清清单 + 暂定方案"。这避免了模型在信息不全时卡住不出东西，是很好的体验设计。

**4. `020-design.md` 的"替代方案与权衡"**

要求列出多个方案和选择理由，包含"什么情况下会炸"一栏。比只写最终方案更科学，避免了思维惰性。

### 存在的问题

**问题 1：`040-light.md` 名义上是"人类摘要"，实际混入了大量 agent 语言**

`040-light.md` 的定位是"简单改动，一页摘要"，主要读者是人类。但实际内容混入了：

- "定级证据：允许 light 的证据：`[路径/命令/代码]`"——这是 agent 的判档自检，不是给人类看的摘要
- "接口返回检查清单：消费方影响：前端/移动端/外部 API/历史会话/缓存/导出/generated client"——这是 agent 的排查清单
- "验证等级：V0 未验证 / V1 静态验证 / V2 构建或单测 / V3 API/UI/DB/集成验证"——这是 agent 的验证分级

一个非技术 PM 看到这份"摘要"会很困惑。它不像摘要，更像 agent 的自检表。

**改进建议**：把 `040-light.md` 拆成两部分——上半部分是纯人类摘要（变更概述、影响范围、回滚方案、验证结果），下半部分是 agent 自检区（定级证据、验证等级、接口返回检查清单），用分隔线标注"以下为 agent 执行检查项，人类读者可跳过"。

**问题 2：`060-preflight.md` 是最不人性化的文档**

15 行核对表格，每行有"P0/P1 等级""必须证据""当前结果""结论"四列，内容全是 `git status --short --branch`、`_active-state.md`、`确认 Step N` 这类术语。

这个文档的主要读者应该是人类（执行前最后确认），但它的表达方式完全没考虑人类的阅读体验。更像是一份 ops runbook，不是给开发者看的执行前检查。

**改进建议**：把 P0 项（必须满足）和 P1 项（建议满足）分成两个表。P0 表用人类语言描述（"确认代码仓库干净，没有无关改动"而不是"git status --short --branch"），命令放在"验证方式"列。P1 表可以保留当前格式。

**问题 3："凭证脱敏"警告在每份文档头部重复**

`010`、`020`、`030`、`040`、`090` 模板头部都有这段话：

> **凭证脱敏（强制规则）**：任何连接串密码、token、密钥写入本文档前必须脱敏为 `***`，只记键名和来源路径。

对人类读者来说是噪音——这是协议规则，不是文档内容。应该放在 SKILL.md 的强制规则区，不应该出现在每份产出文档的头部。

**改进建议**：从所有模板头部移除凭证脱敏警告，只在 SKILL.md 强制规则区保留。模板头部只保留"生成时间 | 版本 | 生成者"。

**问题 4：`020-design.md` 的"代码风格报告"对人类不友好**

用 `[Java-实体]`/`[DI]`/`[日志]`/`[SQL]` 这种标签引用风格约束（impact-pro 用 `style_axes` 轴名），是给 agent 执行时用的速记符号。但设计文档的主要读者是人类开发者，他们需要的是"这个项目用什么风格写代码"的自然语言描述，不是标签表。

**改进建议**：代码风格报告用自然语言段落描述（"本项目 Entity 类使用 Lombok @Data 注解，不手写 getter/setter；Service 层用 @Transactional 标注事务边界"），标签作为括号附注供 agent 参考。

**问题 5：没有面向"非技术决策者"的文档**

如果产品经理或业务方想了解"这个变更做了什么、影响什么"，最接近的是 `010-requirements.md` 和 `040-light.md`，但前者仍然有"非功能需求：P99 延迟"这类技术指标，后者有 API 契约检查清单。没有一个真正只用业务语言写的"变更摘要"。

full 场景下尤其明显——团队 lead 拿到 4 个文档（000/010/020/030），没有一个地方能 30 秒看完变更全貌。

**改进建议**：在 full 场景中增加一份 `005-change-summary.md`（一页纸变更摘要），只写：变更做了什么、影响哪些系统、风险是什么、需要谁配合、回滚方案。纯业务语言，不出现命令、路径、方法名。

**问题 6：模板里的占位符指引对人类是噪音**

`010` 里"[用业务语言把要做的功能讲明白。读的人应该能一句话转述给同事听，不出现表名、类名、文件路径。]"——这是给 agent 看的填写指引。如果模型没有删掉它们（弱模型很可能不删），人类读到的就是混着方括号指引的半成品。

**改进建议**：模板中的填写指引用 HTML 注释 `<!-- -->` 包裹，人类在渲染后的 Markdown 中看不到，agent 在读取源文件时能看到。

## 三、面向 agent 的文档：足够清晰吗？

### 做得好的地方

**1. `000-context-pack.md` 的结构化设计**

L1/L2/L3 分层、相关性 3/2/1/0 分级、"暂不纳入范围"显式列出、找不到引用写"未找到引用"不写"无影响"——这些规则对 agent 消费来说非常清晰。

**2. `_active-state.md` 的字段设计**

pending_step、last_confirmed_step、v1_only_count、concurrency_lock 等字段覆盖了跨会话恢复的所有关键状态。Step Ledger 表格记录每个 Step 的状态和确认情况，对恢复场景很有用。

**3. `090-execution-record.md` 的高风险清单 PASS/FAIL 表格**

每步都强制过一遍 6 类高风险操作（DROP TABLE / DELETE FROM 无 WHERE / 删旧接口等），对 agent 执行时自检很有效。

### 存在的问题

**问题 1：`_active-state.md` 是唯一用英文写的模板**

所有其他模板都是中文，但这个模板的头部写"Cross-session recovery state for impact / impact-pro"，字段名全是英文（updated_at、pending_step、target_project_root）。一致性差——如果 agent 读其他文档是中文模式，切到这个文档要切语言上下文。

**改进建议**：统一为中文。字段名改为中文（更新时间、待执行步骤、目标项目根目录等），或至少在头部加中文说明。

**问题 2：`000-context-pack.md` 和 `020-design.md` 有内容重叠**

context-pack 的"项目背景摘要"和 design 的"项目背景摘要"/"Context Pack 摘要"几乎是同一份内容的两次拷贝（都列直接修改候选、影响判断候选、暂不纳入范围）。agent 不清楚以哪个为准，人类也不知道为什么要看两遍。

impact-pro 更明显——`020-design.md` 第 2 节直接叫"Context Pack 摘要"，内容就是 context-pack 第 4 节"相关文件和对象"的子集。

**改进建议**：`020-design.md` 不再重复 context-pack 的文件清单，只写"详见 `000-context-pack.md` 第 4 节"，然后直接进入设计概览和变更细则。

**问题 3：`030-implementation.md` 混了人类指令和 agent 执行细节**

"确认提示：执行前必须逐项询问 `确认执行 Step N？请回复：确认 Step N / 跳过 Step N / 其他指令`；模糊确认不得视为写操作确认。"

这是给 agent 的行为指令，但它出现在"实施文档"里。实施文档应该是"做什么"的描述（供人类阅读和 agent 执行），不是"怎么做"的行为指令（应该在 SKILL.md 里）。

同样，"状态更新：询问确认前更新 `_active-state.md` 为 `pending Step N`"也是 agent 行为规则，不应该出现在产出文档模板里。

**改进建议**：从 `030-implementation.md` 模板中移除"确认提示"和"状态更新"字段。这些行为规则在 SKILL.md 的强制规则区和 Phase 5 执行规则中已有定义，不需要在每份产出文档中重复。Step 模板只保留：维度、文件、操作、影响范围、回滚方式、语义约定、验证方式。

**问题 4：`030-implementation.md` 的"语义约定"字段含义模糊**

写的是"已确认定义 / 不涉及 / 未确认"，但没说什么定义。实际含义是"status/enum/常量/错误码/权限名/配置键的语义约定"，但字段名太简略，agent 可能不理解要确认什么。

**改进建议**：改为"语义约定确认（status/enum/常量/错误码/权限名/配置键）：已确认定义 / 不涉及 / 未确认"。

**问题 5：`090-execution-record.md` 嵌入了静态行为规则**

"工具调用约定"一节写"构建验证优先用项目自带的构建工具（mvn test / gradle test）"——这不是执行记录的一部分，是 agent 的行为规则。它嵌在模板里意味着每份执行记录都会重复这段话。

**改进建议**：从 `090-execution-record.md` 模板中移除"工具调用约定"段，移到 SKILL.md 或 references 中。

**问题 6：没有"文档间导航索引"**

agent 读完 `000-context-pack.md` 后，需要知道"下一步读哪个文档"。模板里没有"下一步"指引。虽然 SKILL.md 里有流程，但如果 agent 在长会话中上下文压缩了，只读到产出文件，就不知道流程到哪了。`_active-state.md` 的 `current_phase` 字段部分解决了这个问题，但只对恢复场景有效，不对首次消费有效。

**改进建议**：每份文档的末尾加一行导航："→ 下一步：确认本文档后，进入 `020-design.md`（设计文档）"。或者在每份文档头部加一个"文档流"小节：

```
文档流：000-context-pack → 010-requirements → 020-design → 030-implementation → 060-preflight → 执行
当前位置：010-requirements
```

## 四、内容是否科学？

### 做得好的地方

**1. 信任标签 `【已核实: 证据】` / `【推断: 待验证】`**

区分"证据支撑的事实"和"基于推断的假设"，是工程文档的最佳实践。pathfinder 是三个 skill 中做得最彻底的——每个条目都要求标注可信度。

**2. "找不到写未发现，不编造"原则**

强制模型承认未知，而不是编造填补空白。context-pack 里"找不到引用时写'未找到引用'，不得写成'无影响'"是很好的防骗规则。

**3. V0-V3 验证等级分层**

区分"未验证/静态/构建/运行"四个级别，避免了"已验证"这种模糊表述。

**4. light/full 分档**

简单改动不硬走三文档，复杂变更不偷懒走一页。定级按风险触发，不按文件数量粗暴决定。

### 存在的问题

**问题 1：`020-design.md` 模板有编号 bug（两个 `## 2.`）**

impact 和 impact-pro 的 `020-design.md` 模板都有这个问题：

```
## 2. 当前状态
  ### 数据库
  ### 代码
  ### 配置
## 2. 目标状态       ← 编号重复
  ### 数据库
  ### 代码
  ### 配置
## 3. 变更细则
```

这是模板自身的缺陷，模型填出来后文档结构就是错的。Markdown 渲染器可能自动修正编号，但源文件是错的。

**改进建议**：改为 `## 2. 当前状态` / `## 3. 目标状态` / `## 4. 变更细则`，后续编号顺延。

**问题 2：19 个维度在模板中没有体现**

`020-design.md` 写"[...其他涉及的维度按需展开]"，但不列维度清单。维度定义在 `references/dimensions.md` 里，但模板不交叉引用。模型可能只填了数据库/代码/配置三个维度就认为"按需"够了，漏掉了缓存、消息队列、事务、国际化等维度。

**改进建议**：在 `020-design.md` 的"横切关注点"节列出全部维度清单（作为 checklist），让模型逐项判断"涉及/不涉及"。当前模板的"横切关注点"只列了 6 项（安全/缓存/日志/消息队列/事务/国际化），而 `references/dimensions.md` 有 19 项。

**问题 3：当前状态/目标状态/变更细则三段冗余**

同一个字段变更出现了三次：当前状态列了表结构，目标状态列了变更后的表结构，变更细则又写了"字段变更：ADD/ALTER/DROP"。读者要在三个地方对照才能理解一个字段变更。

**改进建议**：合并为一个"变更明细表"：

| 表名 | 字段名 | 当前定义 | 变更类型 | 目标定义 | 影响说明 |
|------|--------|---------|---------|---------|---------|
| sys_user | phone | 不存在 | ADD | varchar(20) | 新增字段，无存量数据 |

一行展示一个字段的完整生命周期，消除三段冗余。

**问题 4：验证等级分散在多个文档中没有汇总视图**

`030-implementation.md` 每个 Step 有验证等级，`040-light.md` 有整体验证等级，`090-execution-record.md` 每步又有验证等级。但没有一个地方汇总说"本次变更整体达到了 V2 级验证，以下 3 个 Step 只达到 V1"。评审者要自己翻三个文档拼图。

**改进建议**：在 `060-preflight.md` 或 `090-execution-record.md` 的收尾检查中增加一行"整体验证等级汇总"表格：

| Step | 验证等级 | 未达 V2/V3 原因 |
|------|---------|----------------|
| Step 1 | V2 | — |
| Step 2 | V1 | DB 环境缺失 |
| Step 3 | V2 | — |
| **整体** | **V1** | **Step 2 仅静态验证** |

**问题 5：context-pack 的"上下文预算"概念模糊**

模板写"L1 已查看 N 个文件，L2 保留 N 个候选，L3 深入阅读 N 个文件"，但没说预算上限是多少、超过怎么办。"如果超过预算，下一轮收敛问题"——什么预算？谁定的预算？模型填这个字段时大概率是随便写个数。

**改进建议**：要么删除"上下文预算"节（它对 agent 执行没有实际指导意义），要么明确预算规则（如"小仓 ≤20 文件，中仓 ≤50 文件，大仓按模块采样"）。

**问题 6：pathfinder 的 SVG 预览图价值存疑**

模板里每个 Mermaid 图后都跟一段手写的 SVG 代码（约 30-50 行）。模板说"仅用于 Markdown 阅读"，但大多数 Markdown 渲染器（GitHub、VS Code）都能渲染 Mermaid。

307 行模板中有约 120 行是 SVG 代码。模型生成时还要花 token 写 SVG，但实际价值很低——如果 Mermaid 能渲染，SVG 就是冗余的；如果不能渲染，SVG 的维护成本很高（改了 Mermaid 要同步改 SVG）。

**改进建议**：删除模板中的 SVG 预览图段。Mermaid 是图的唯一 canonical source，足够了。如果个别环境不渲染 Mermaid，用户可以装插件或用外部工具渲染。

**问题 7：`010-requirements.md` 的"非功能需求"模板用技术指标举例**

写"性能：如'冻结操作响应时间不超过 200ms'"，"安全：如'冻结操作需要 system:user:freeze 权限'"。后者已经是技术实现（权限标识），不是业务需求。非功能需求应该用业务语言描述，技术指标应该在设计文档里量化。

**改进建议**：举例改为业务语言：
- 性能：如"冻结操作用户不能感觉到卡顿"
- 安全：如"只有管理员能执行冻结操作"
- 可用性：如"冻结期间其他用户操作不受影响"

然后在括号中注明"技术指标见 `020-design.md`"。

## 五、问题汇总与优先级

| # | 类别 | 问题 | 影响 | 严重度 | 改进建议 |
|---|------|------|------|:------:|---------|

## 六、和工程严谨性优化的关系

本文档关注的是"产出物本身对读者是否友好、内容是否科学"，和之前的 [docs/skill-optimization-roadmap-2026-06-25.md](skill-optimization-roadmap-2026-06-25.md) 关注的"协议是否安全、验证是否充分"是两个互补的维度：

| 维度 | 工程严谨性（已有路线图） | 产出物质量（本文档） |
|------|----------------------|-------------------|
| 核心问题 | 协议能不能防住模型犯错 | 产出物对读者好不好用 |
| 典型问题 | 判档错误、方法名编造、facts 不实 | 文档混入 agent 语言、编号 bug、内容冗余 |
| 受影响的人 | 使用 skill 的开发者 | 阅读 skill 产出的所有角色（开发者、PM、reviewer） |
| 验证方式 | 盲测、L1 评测、负向测试 | 模板审查、人类可读性评估 |

两份文档的优化项可以并行推进，互不依赖。工程严谨性优化保证"产出是对的"，产出物质量优化保证"产出是好读的"。
