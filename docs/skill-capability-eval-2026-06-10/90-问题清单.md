## 流程问题（P2 ×2 + P3 ×1）

### P2-001: skill 协议假设交互式用户，subagent 上下文无用户时 Step 确认门禁被绕

- **复现路径**：所有 9 case。subagent 无法获得 `确认 Step N` 回复，统一标"未确认"继续。
- **影响 case 数**：9/9
- **严重等级**：P2
- **实际表现**：subagent 处理得体——在 000/100/200 文档中**显式列出"未确认"段**，Phase 5 显式标"未执行（按 prompt 禁止）"。**不算严重**——但**真实生产场景下用户与 subagent 之间需要明确这种 fallback 协议**。
- **修复建议**：
  - `impact` SKILL.md Phase 5 段加："当 subagent / CI / 自动化环境无法获得 Step 确认时，必须把所有 Step 转为 plan 文档（`200-实施文档.md`），并在 900-执行记录中显式标 'Phase 5 未执行'"
  - 实际生产 subagent 调用方应**主动编排 Step 确认机制**（如每个 Step 单独发回主会话让用户确认）

### P2-002: skill 协议中 Socratic 提问在无人时"必须答"的边界不清

- **复现路径**：R1 / G1 / F1 / F3 等 full case
- **影响 case 数**：4（full case）
- **实际表现**：subagent 在 Phase 3 列了 3-5 轮问题，但**默认假设/建议都列在文档中**（"未确认项 + 默认值"），算合格的 fallback。
- **严重等级**：P2
- **修复建议**：`impact` / `impact-pro` SKILL.md Phase 3 段加："无人值守模式：每轮问题必须同时给出'默认建议'，并明确'未确认不得进入 Phase 5'"

### P3-001: 900-执行记录时间戳由 subagent 自行生成（不是真实 wall-clock）

- **复现路径**：R3 / F3 等
- **影响 case 数**：9/9
- **严重等级**：P3
- **实际表现**：subagent 输出 `[2026-06-10 18:14:00]` 类时间戳，但实际跑分时间可能与子 agent 内部时钟相关。**不影响内容质量**，仅日志准确性。
- **修复建议**：执行记录时间戳由 harness 注入，不让 subagent 自己写

## 工具问题（P2 ×2）

### P2-005: Grep 正则噪音（vue-i18n 类）造成误报

- **复现路径**：R2
- **影响 case 数**：1

## 边界问题（P2 ×2 + P3 ×1）

### P2-007: skill 未识别"F2 dark mode 已完整实现"——prompt 假设 greenfield

- **复现路径**：F2
- **影响 case 数**：1（但意义大）
- **严重等级**：P2
- **实际表现**：subagent 正确发现 `ThemeProvider` + `localStorage("vite-ui-theme")` + `<Appearance />` + E2E **4 段已有实现**——PM 需求实际只是"在 `/settings` 加 Tab"，而不是"建新 dark mode"

## 误判问题（P2 ×1 + P3 ×1）

### P2-010: 原始 mock 报告 P1-001 "EasyExcel 编造"在真实跑分中**未复现**

- **复现路径**：R1 / R4
- **影响 case 数**：0

- **实际表现**：subagent **正确识别 RuoYi 导出机制是 `@Excel` 注解 + `ExcelUtil`**，**没有提到 EasyExcel 也没有编造 EasyExcel 模板类名**。`pom.xml` 中 `com.ruoyi.common.utils.poi.ExcelUtil` 是项目自带工具，subagent 读 `SysUser.java:43-53, 78-85` 看到了 `@Excel` 注解风格。
- **修复建议**：**撤销**原 P1-001 修复。RuoYi 这类项目在 subagent 路径下**已能正确识别**——原始 mock 报告是人审视角的假设性问题。
- **可能的反例**：如果项目实际用 EasyExcel（如 `com.alibaba.excel.EasyExcel` 在 pom 里），subagent 是否仍能正确识别？未在本次 eval 范围。可在下次 eval 加一个 EasyExcel 项目测试。

### P3-004: 原始 mock 报告 P1-003 "i18n 边界"在真实跑分中**未复现**

- **复现路径**：R2
- **影响 case 数**：0
- **严重等级**：P3（**重要修正**）
- **实际表现**：subagent 主动 `Glob **/i18n/**`、`Glob **/lang/**`、读 `package.json` **全部依赖**（71 行）——**确认 RuoYi-Vue 实际上没有 vue-i18n**。文案全是硬编码中文。
- **修复建议**：**撤销**原 P1-003 修复。RuoYi-Vue 不是多语言项目——i18n 维度对它**不适用**。原始 mock 报告 P1-003 是基于"RuoYi 是 i18n 项目"的错误假设。
- **可能的反例**：Vue I18n / Nuxt I18n 项目（如其他 RuoYi-Vue3 改版）可能确实有 i18n。可在下次 eval 加 i18n 项目测试。

## 总结

### 修复优先级

1. **P2 优先（8 项）**：
   - P2-005 Grep 误报模式
   - P2-006 无 DB MCP 降级路径
   - P2-007 prompt 设计补 greenfield check
   - P2-001/002/003/004/009 协议/模板微调

2. **P3 微调（6 项）**：模板描述、时间戳注入、alembic head 必读

3. **撤销（2 项）**：原 P1-001、P1-003 不再是问题

### 真实生产建议

- 两个 skill 在受监督（subagent + 文档 + 执行前检查）场景下**可投入生产**
- subagent 跑分**比人审跑分更稳定**（F2 主动发现已有实现、R3 主动质疑 PM 路径）
- 但**深度仍浅于人审**（F1 未读 alembic down_revision、G1 admin 角色需手动 grep 扩展）
- **建议**：把 subagent 跑分作为 production gate，**人审作为 depth gate**——两层都过才进 release

### 试跑产生的新发现（不只是 mock 假设）

1. **F1 / 新发现**：subagent 在 V3 受限时会**主动写 standalone_test.py** 兜底——这是协议没规定但 subagent 自觉做的行为。**应协议化**为"V3 受限自动进 standalone 模式"。
2. **F1 / 新发现**：subagent 用手写 alembic migration 替代 autogenerate 时**默默执行**没显式记录偏离。**应强制"偏离计划"段必填**。
3. **R3 / 新发现**：trial 提示词不能递归撤销 P0 兜底——元层安全规则。**应写入 SKILL.md**。
4. **R3 / 新发现**：90 天兼容期作为"隐式 P0"——时间维度也算不可逆。**应扩展 P0 兜底规则**。
5. **R3 / 新发现**：subagent 触发 P0 时给出**多层理由**（字面 + 元层 + 时间 + 路径错位）——远超"机械匹配关键词"。**说明 subagent 决策深度足够**。
6. **R3 / 端点计数偏差**：context-pack 标的 24（8 menu + 16 role）实际 v1 是 23（8 menu + 15 role）。归档时需修正。
7. **F1 / 端点 parity 全过**：v1 8 menu + 15 role 端点 v2 全部字节级一致，@PreAuthorize 字符串全等。**subagent 复制 v1 严格保真**。
8. **R3 / Spring Security 鉴权断言未覆盖**：V1V2ParityTest 只写了 2 个模板，剩余 21 个端点 parity 用例需后续补（但模板可复制）。
