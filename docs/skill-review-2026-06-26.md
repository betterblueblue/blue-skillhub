> 评审时间：2026-06-26
> 评审范围：impact / impact-pro / pathfinder / vl-vision
> 评审视角：工程成熟度、设计合理性、可维护性、实际效果

## 一、总体评价

这是一套以「变更影响分析 + 受监督执行」为核心的 agent skill 体系。工程成熟度在 Claude Code skill 项目中属于顶部水平——54+ 验证记录、自动化脚本闸门、L1 回归框架、盲测体系的基础设施比多数开源 agent 项目都扎实。（注：本评审将 eval 体系的存在作为工程质量的正面证据；eval 方法论本身的可靠性——如 V1-V10 盲测设计、控制变量法、runner/judge 分离——也是支撑这一判断的一部分，但本文未对其做专项评审。）

### 工程成熟度排序

```
pathfinder > impact ≈ impact-pro >> vl-vision
```

pathfinder 最干净是因为只读、不写、没有执行阶段的复杂性。impact 系列的复杂性大部分来自 Phase 5 执行安全——但这也正是它的核心价值。

## 三、系统性问题

### 3.1 核心张力：规则密度 vs. 模型遵循率

规则越详细 → 模型在边缘情况下违规概率越高（T53 的 2 个 P1）。更多规则防御更多边缘情况 → context 有限 + 注意力有限 → 规则被"忘掉"。

当前缓解策略：
- 压缩存活区（前 5000 tokens 保留铁律）
- 脚本闸门做事后验证

这是务实的，但是治标不治本。

### 3.2 最大结构性债务：impact / impact-pro 重复

| 重复内容 | 大致比例 |
|----------|----------|
| 安全规则（7 条铁律） | ~95% 相同 |
| Phase 流程（1→2→2.5→3→3.5→4→5） | ~90% 相同 |
| 模板（9 个文件） | 结构完全相同，措辞微调 |
| references（phase-5-execution 等） | ~80% 相同 |
| 自动/确认边界表 | 完全相同 |

每次安全规则修改都要同步两处。`_common/templates/README.md` 已经承认了这个问题但未解决。

### 3.3 中等简单变更的流程开销

> 注：纯前端文案/样式调整 README 明确说不适合用此 skill，不作为论据。这里讨论的是"不太简单但也不复杂"的中间地带。

用户说"把登录失败的错误提示从'密码错误'改成'用户名或密码错误'"——涉及一个 i18n key 或一个 controller 返回值，不涉及 DB/API 契约/权限/状态机。当前流程：

```
Phase 1 意图捕获（输出假设/歧义/规模/标准）
Phase 2 上下文发现（技术栈检测 + 文件扫描 + 现状核查）
Phase 2.5 初步风险预判
Phase 3 苏格拉底式探索（虽然可能 0 题就跳过，但判断本身消耗 context）
Phase 3.5 正式定级
Phase 4 文档输出（light 一页）
Phase 5 执行
```

对于这类变更，Phase 2.5-3.5 的探索和定级步骤增值有限——影响范围在 Phase 2 就能完全确认，不需要苏格拉底追问。问题不是"走了全套流程"，而是"没有快速出口让确认的简单变更跳过探索阶段"。

## 四、优化建议与预期收益

### 优化 1：合并 impact / impact-pro 消除重复

**方案：** 将 impact 改为 impact-pro 的一个 profile（`java-spring-mybatis`），只保留一个 SKILL.md 内核。

**具体做法：**
1. 以 impact-pro SKILL.md 为内核
2. 把 impact 的 Java/RuoYi 专属规则（schema-discovery.md、style-analysis.md 等）收进 `profiles/java-spring-mybatis.md`
3. 只保留一个 skill 名称（`impact`），通过 profile 注入栈规则
4. `_common/templates/` 变为 `templates/`，不再需要同步

**收益：**
- 安全规则修改从改两处 → 改一处，维护成本减半
- 减少约 400 行重复 SKILL.md + 对应重复 references
- 用户不再困惑"什么时候用 impact 什么时候用 impact-pro"
- 评测框架也可以统一（现在 rubric 分两份但内容几乎一样）

**风险：** impact 在 Java/RuoYi 场景下的专精优化可能被稀释。缓解方式：profile Level 3 的 java-spring-mybatis 可以覆盖更多细节。

---

### 优化 2：增加"快速通道"跳过简单变更的冗余流程

**方案：** 在 Phase 1 结束后增加一个"trivial 出口"——任务规模判定为"小"且满足特定条件时，直接跳过 Phase 2.5-3.5 进入执行。

**触发条件（全部满足）：**
- 任务规模：小（≤20 行改动）
- 不涉及 DB/API 契约/权限/状态机/enum/配置键
- 不涉及删除/重命名
- 影响范围可在 Phase 2 完全确认（无未确认项）

**快速通道流程：**
```
Phase 1 → Phase 2（上下文发现 + 现状核查）→ 直接执行（仍需 Step 确认）
```

**收益：**
- 简单变更的对话轮数从 6-8 轮降到 3-4 轮
- 减少 context 消耗约 40%
- 用户体验更流畅，不为改文案走全套流程

**约束：** 安全规则不降级，Step 确认不跳过，只是跳过了中间的探索和定级步骤。

---

### 优化 3：规则分层加载（渐进式披露），缓解 context 压力

**理论依据：** 这对应 Anthropic 在 skill 设计指南中提出的 Progressive Disclosure（渐进式披露）概念——不要一次把所有规则堆给模型，而是按需展开：顶层只放触发条件和核心不变量，细节推迟到模型真正进入那个阶段时再加载。

**当前状态：已有雏形但不彻底**

你的 skill 已经有这个方向的实践：
- `references/` 目录就是延迟加载——Phase 5 执行规则不写在 SKILL.md 正文里，而是 `references/phase-5-execution.md` 按需读取
- "压缩存活区"是另一个实践——把安全铁律浓缩到前 5000 tokens 确保压缩后还在

但问题在于：**SKILL.md 正文本身已经 400+ 行**。模型在 Phase 1 意图捕获阶段就被迫背着 Phase 4 文档格式规则、Phase 5 执行安全闸、认证字段一致性自检这些此刻完全用不上的内容。references 延迟了一部分，但正文里仍然太重——Phase 2.5 现状核查门禁、Phase 3.5 判档决策表、Phase 4 需求文档边界自检这些「进入那个阶段才需要」的规则，全部挤在正文里。

**目标结构：**

```
SKILL.md                          ← 核心层（~150 行），始终在 context 中
├── 强制规则 7 条（安全铁律，不可压缩）
├── 自动/确认边界表
├── 流程骨架（每个 Phase 一句话：做什么 + 出口条件）
├── 行为准则
└── 每个 Phase 的加载指令（见下）

references/
├── phase-1-intent.md             ← Phase 1 进入时加载
├── phase-2-context-discovery.md  ← Phase 2 进入时加载（已有，但需收纳正文内容）
├── phase-2.5-risk-triage.md      ← Phase 2.5 进入时加载（从正文迁出）
├── phase-3-questioning.md        ← Phase 3 进入时加载（已有）
├── phase-3.5-grading.md          ← Phase 3.5 进入时加载（从正文迁出）
├── phase-4-output.md             ← Phase 4 进入时加载（从正文迁出）
├── phase-5-execution.md          ← Phase 5 进入时加载（已有）
└── safety-gates.md               ← 任何写操作前加载（而不是开局就背）
```

**核心层 SKILL.md 正文改写示例：**

当前正文（~60 行描述 Phase 2.5-3.5）变为：

```markdown
## Phase 2.5: 初步风险预判

做什么：基于 Phase 2 发现做初步定级预判（可能 light / 倾向 full / 证据不足）。
出口条件：输出初步风险 + 已确认事实 + 需要澄清项（≤3 个）。

→ **进入前必须 Read `references/phase-2.5-risk-triage.md`**（含现状核查门禁、覆盖范围语义核查完整规则）

## Phase 3: 苏格拉底式探索

做什么：按选中维度分组提问，每轮 ≤3 题，多轮收敛。
出口条件：所有 P0/P1 问题已覆盖，停止条件满足。

→ **进入前必须 Read `references/phase-3-questioning.md`**

## Phase 3.5: 正式定级

做什么：输出判档决策表 + 建议档位，用户确认。
出口条件：用户确认 light 或 full。

→ **进入前必须 Read `references/phase-3.5-grading.md`**（含定级核心条件 6+8 条、证据自洽性检查）
```

**关键设计决策：加载指令必须是流程的一部分，不是备注**

区别在于写法。错误写法：

```markdown
## Phase 3.5
详见 references/phases-detail.md。
```

正确写法：

```markdown
## Phase 3.5
→ **进入前必须 Read `references/phase-3.5-grading.md` 中的定级核心条件**，然后执行以下步骤：
1. 输出判档决策表
2. ...
```

前者是"参考"，模型可能跳过；后者是"指令"，把 Read 动作嵌入流程步骤，模型不执行就无法完成下一步。

**从正文迁出的内容（当前在 SKILL.md 但应移到 references）：**

| 当前位置（SKILL.md 行号范围） | 迁出目标 | 理由 |
|------|------|------|
| Phase 2.5 完整规则（现状核查门禁、覆盖范围语义核查、判定输出格式） | `references/phase-2.5-risk-triage.md` | 只在进入 2.5 时需要 |
| Phase 3.5 判档决策表格式 + 定级证据自洽性 + 退出条件 | `references/phase-3.5-grading.md` | 只在定级时需要 |
| Phase 4 需求文档边界自检 + 设计文档风格报告 + 实施文档预检 + 改动完整性自检 + 产出完整性自检 + 输出验证 | `references/phase-4-output.md` | 只在产出文档时需要 |
| 行号引用格式 + 证据标签规则 | `references/phase-4-output.md` | 同上 |
| 长期目标模式 | `references/phase-1-intent.md` | 触发条件少见，不必常驻 |
| 用户场景覆盖验证 | `references/phase-2-context-discovery.md`（已有，合并） | 只在排除文件时需要 |

**预期收益：**

| 指标 | 当前 | 优化后 |
|------|------|--------|
| SKILL.md 常驻 context | ~8000 tokens | ~3000 tokens |
| 模型在 Phase 1 时背负的无关规则 | Phase 2.5-5 全部细节 | 只有铁律 + 流程骨架 |
| 单阶段规则密度 | 全部规则竞争注意力 | 只有当前阶段规则 |
| 规则遵循率（预期） | T53 暴露 P1 遗漏 | 当前阶段规则完整呈现，遗漏概率降低 |

**风险与缓解：**

| 风险 | 缓解 |
|------|------|
| 模型漏读 reference 导致跳过规则 | 加载指令写成流程步骤（不是备注），不 Read 就无法继续 |
| **弱模型不执行 Read 就直接往下做**（这是核心矛盾——最需要 skill 补偿的弱模型，恰恰最容易跳过 Read 指令。T53 暴露的"规则遗忘"模式与此同源） | ① 脚本闸门做事后兜底——即使规则被跳过，闸门仍能捕获违规产物；② Phase 出口条件设计为「必须引用 reference 中的特定模板/格式才能通过」，间接强制 Read；③ 长期方向：当弱模型遵循率低于阈值时，回退到正文常驻模式（动态分层） |
| 压缩后 reference 加载指令被丢失 | 压缩存活区保留所有加载指令（它们本身很短） |
| 调试时难以一眼看到全部规则 | 保留一份完整版作为人类参考文档（不给模型加载） |
| 脚本闸门检查项散落在多个 reference 中 | `pf_validate.py` / `impact_validate.py` 仍集中校验，不受分层影响 |

**实施顺序建议：**

1. 先在 impact-pro 上试点（它的正文更长，收益更大）
2. 拆分后跑一轮 L1 回归，对比规则遵循率变化
3. 确认无回归后同步到 impact（或合并后只改一份）
4. pathfinder 正文已较精简（~210 行），可后续再做

### 优化 5：vl-vision 定位决策

**两个方向，二选一：**

**方向 A：降级为 utility，不追求 skill 级工程标准**
- 重命名为 `utils/vl-vision`
- 不纳入 skill 评测体系
- 保持当前简洁形态，作为工具箱组件

**方向 B：升级到 skill 级标准**
- 添加 `validation-runs/` 和基础验证（至少模板准确率测试）
- 多 provider 支持（OpenAI、本地 Ollama 等）
- 处理边界情况（大图裁剪、超时重试、token 预算）
- 添加能力探测：先检查当前 LM 是否已有视觉能力，有则不调用

**建议选 A。** 理由：当前主流模型（Claude 4.x、GPT-4o、Gemini）都是多模态的。vl-vision 的使用场景越来越窄，投入升级 ROI 低。作为 utility 保留即可。

---

### 优化 6：pathfinder 的认证-鉴权自检规则泛化

**方案：** 把「认证-鉴权字段一致性自检」从硬编码规则改为可选深挖项。

**做法：**
- 从 Phase 3 `references/phase-3-depth-fill.md` 中的必做步骤改为：
  - 当项目存在认证+鉴权两层链路时（通过 Phase 2 sub-agent 发现），执行字段一致性自检
  - 当项目无明确 RBAC 层时，跳过此检查

**收益：**
- 对无权限系统的项目（纯前端、CLI 工具、数据管线）不浪费 context
- 规则更泛化，减少"某次 bug 的 hotfix 变成通用规则"的感觉

## 六、核心价值与战略定位

### 6.1 impact 系列的两个核心价值

**价值 1：澄清意图 — 把模糊需求变成明确目标（Phase 1 + 3）**

模型的默认行为是：遇到模糊 → 选一个最像训练数据里常见模式的理解 → 直接实现。它不觉得自己不确定——它选了一个理解然后表现得很有信心。用户事后看到结果才发现"不是我要的那个意思"。

impact 的 Phase 1 + Phase 3 把"不确定"从模型内部的隐式猜测，变成了外显的结构化提问。模型不需要自己判断"这里我该不该问"——流程强制它在特定检查点暴露歧义、列出多种理解、追问用户选哪个。

**价值 2：发现后果 — 让模型知道改了会断什么（Phase 2 + 4 + 5）**

每个模型都能改代码。你说"用户表加个 phone 字段"，它立刻动手——改 model、改一个 API、写个 migration。但它不会自己去查：还有 12 处序列化 User 的地方、有缓存层会返回旧结构、有导出报表读这张表、有前端表单校验没覆盖这个字段、有存量数据需要回填策略。

impact 把顺序反过来：先发现后果，再决定要不要动手，动手了再逐步验证。

**两者的关系：** 前者解决"改错方向"，后者解决"改对方向但漏了连带影响"。加在一起，把模型从"拿到指令就跑"变成"先确认理解对不对 → 再确认改了会怎样 → 最后才动手"。重构了 LM 解决问题的认知顺序。

### 6.2 战略定位：认知能力的外置补偿

impact 系列的长期定位不是"给所有模型加流程"，而是**用流程结构弥补弱模型的 meta-reasoning 缺口**：

| 模型能力层 | 配置 | 原理 |
|-----------|------|------|
| 高级模型（Opus 级） | CLAUDE.md（行为准则）就够 | 自带元认知——知道什么时候不确定、什么时候该查引用方、什么时候该追问 |
| 性价比模型（Composer 级） | CLAUDE.md + skill | 缺的不是知识而是判断力——不知道自己不知道。skill 用流程把判断力外置了 |

具体来说：
- 强模型内化了"遇到模糊就追问""改之前查引用方"这些判断，CLAUDE.md 给它准则就行，它自己决定什么时候用
- 弱模型的默认路径永远是"不问就干"。skill 等于给它装了外置的"不确定性检测器"——它不需要有元认知能力来意识到自己不确定，流程会替它做这件事

### 6.3 最终验收标准

```
Composer + CLAUDE.md + skill ≈ Opus + CLAUDE.md（仅行为准则）
```

两种组合在同一批盲测 case 上达到相近的分数，就说明 skill 成功地用流程结构补上了模型能力差。

这个标准可以用现有 eval 框架直接测：跑同一批 case，对比两种组合的分数差。当差距收敛到可接受范围（如 ≤5 分），说明 skill 的外置补偿起效了。

### 6.4 对优化方向的指导

这个战略定位直接影响前述优化方向的优先级判断：

| 优化项 | 战略相关性 |
|--------|-----------|
| 规则分层加载 | **直接服务战略** — 弱模型 context 有限，同时背的规则越少遵循率越高 |
| 快速通道 | **直接服务战略** — 弱模型判断力有限，给它一个明确的"这种情况可以简化"信号比让它自己判断更可靠 |
| 合并 impact/impact-pro | 间接服务 — 统一后弱模型不需要判断"用哪个 skill"，减少一个判断点 |
| profile 覆盖度 | 间接服务 — profile 越准确，弱模型在 Phase 2 的发现质量越高 |

---

## 七、一句话总结

这套 skill 在「让 AI 不乱改生产系统」上做到了同类项目的高水平。它的核心价值是用流程结构为弱模型外置了"澄清意图"和"发现后果"两个判断力，最终目标是让性价比模型+skill 达到高级模型单独工作的同等效果。下一步的重点不是加更多规则，而是**减少重复、降低简单场景的流程开销、让已有规则被更好地遵循**。
