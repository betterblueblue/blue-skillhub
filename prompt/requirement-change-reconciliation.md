# 需求变更对账 Prompt

适用于任务已经进行一段时间、用户中途多次调整要求，当前会话可能同时记着新旧两套需求的情况。把下面的 Prompt 发给仍然保留完整对话上下文的会话，让它先把有效需求理清楚，再继续开发。

```text
这个任务进行过程中，需求已经调整过多次。现在先暂停修改文件，把前后要求对一遍，确认当前到底以哪一版为准。对账完成并得到我确认之前，不要继续编码，不要撤销现有改动，也不要根据你自己的偏好替我做取舍。

请重新查看本次对话、已有需求文档、计划文件、相关 diff 和测试，完成以下工作：

1. 找出用户最初提出的需求和成功标准。能找到原话时保留原话；只能根据上下文整理时要明确说明。
2. 按发生顺序列出用户后来确认的补充、删减和改动。每一条都写清楚它改变了哪项旧要求。
3. 区分用户决定和模型建议。只有用户明确同意的内容才能覆盖旧要求；模型提出但用户没有确认的方案不能算需求变更。
4. 如果用户后来的明确说法与早期要求冲突，以较新的明确决定为准，但必须把被替代的旧要求列出来，不能悄悄删除。
5. 整理出一份“当前有效需求”。它应该能够独立阅读，不需要再翻前面的聊天记录。
6. 对照当前代码、计划和测试，判断哪些内容仍然符合当前需求，哪些已经过时，哪些需要调整。这里只做影响判断，不要实际修改或还原文件。
7. 找出仍然存在的冲突、含糊表述和无法确认的取舍。不要为了得到一份整齐的结论而自行补全。

对账时注意：

- 不要把已经实现的结果反写成用户需求。
- 不要因为某项开发成本高，就擅自把它解释成“可以推迟”。
- 不要把“好的”“继续”“先这样”等缺少明确对象的回复，当成对某项重大变化的确认。
- 已经写进代码但不符合当前需求的内容，只标明影响和建议处理方式，确认前不要删除。
- 找不到原话或时间顺序时写“未确认”，不要猜。

最后按下面的结构输出：

## 最初需求

写出最初目标、成功标准和明确排除项，并标明依据来自用户原话还是现有文档。

## 变更记录

使用表格列出：顺序、用户确认的变化、它替代或补充的旧要求、确认依据。没有用户确认依据的模型建议单独列出，不能混进表格。

## 当前有效需求

整理成一份从零开始也能读懂的编号清单。每项必须明确、可核对，不保留已经失效的旧说法。

## 已失效的要求

列出不再执行的旧要求、失效原因和替代它的新决定。没有则写“无”。

## 对现有工作的影响

按“可以保留、需要调整、可能需要撤销、暂时无法判断”分类列出相关文件、计划和测试，并说明原因。不要在这里实际修改文件。

## 仍需确认的问题

只列会改变实现方向的问题，最多 3 个。每个问题说明不同答案会造成什么实际差别。

## 等待确认

用一句话请我确认是否以“当前有效需求”为后续唯一依据。得到明确确认后才能继续开发；如果我指出有误，先更新对账结果，不要急着改代码。
```
