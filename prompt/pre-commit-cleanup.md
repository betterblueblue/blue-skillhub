# 提交前整理改动 Prompt

适用于任务已经完成、准备提交代码，但工作区里可能混着正式修改、调试残留、用户原有改动和无关文件的情况。把下面的 Prompt 发给当前会话，让它先整理清楚“什么该提交、什么不该动”，再由你决定是否清理和暂存。

```text
这个任务准备提交了。请先整理当前工作区，确认本次到底应该提交哪些改动。在我确认整理结果之前，不要删除或还原文件，不要执行 reset、checkout、clean，不要暂存、提交或推送，也不要顺手修复新发现的问题。

整理前重新查看原始需求、用户后续确认、当前 git status、暂存区、未暂存 diff 和未跟踪文件。能够确认任务起点或基线时，比较基线与当前状态；无法确认时明确写“基线未确认”，不要猜测所有改动都属于本次任务。

请完成以下检查：

1. 逐个查看发生变化的文件，说明每项改动与本次需求有什么关系。不能直接追溯到需求的改动，不要归入正式提交。
2. 分开检查暂存区和未暂存内容。某个文件如果同时包含本次改动和用户原有改动，要明确指出，不能因为文件与任务有关就整文件暂存。
3. 查找本轮调试可能留下的内容，例如临时日志、console.log、print、断点、临时开关、被注释掉的代码、放宽的校验、跳过的测试、临时测试文件和只为排查问题安装的依赖。
4. 检查无关格式化、换行变化、批量重排、自动生成文件、缓存、日志、大文件和意外变化的锁文件。说明它们是否确实由本次需求需要。
5. 检查是否可能包含密钥、Token、密码、.env、真实用户数据、内部地址或其他不应提交的内容。不要在汇报中输出敏感值，只说明文件和风险类型。
6. 核对新文件、删除文件和重命名文件是否都有明确理由。不要把未跟踪文件默认当成垃圾，也不要把删除操作默认当成用户已经授权。
7. 汇总已经运行的验证命令、退出码和结果，并指出代码整理后需要重新运行哪些验证。不要把“之前通过”当成清理后的最终结果。

把每个相关文件归入下面五类之一：

- 本次需求必须提交
- 本轮产生的调试残留
- 用户原有改动或其他任务改动
- 自动生成、缓存或可能含敏感内容
- 归属无法确认

分类必须有 diff、对话原话、需求文档或命令结果作为依据。证据不足时放进“归属无法确认”，不要勉强判断。

最后按下面的结构汇报：

## 本次提交的范围

用几句话重述本次需求、成功标准和明确排除项，作为判断文件是否应该提交的依据。

## 当前工作区

写出项目路径、分支、可确认的任务基线、暂存状态、未暂存状态和未跟踪文件。说明是否存在用户原有改动。

## 改动分类

使用表格列出：文件路径、分类、判断依据、建议动作。建议动作只能写“保留并考虑暂存”“建议清理”“不要动”“需要用户确认”，此时不要实际执行。

## 调试残留和无关变化

逐项列出疑似调试残留、无关格式变化和临时依赖。说明为什么判断它们不属于正式实现，以及清理它们可能影响什么。没有则写“未发现”。

## 敏感内容与生成文件

列出不应提交或需要特别确认的文件，只说明风险，不展示敏感内容。没有则写“未发现”。

## 建议暂存的文件

给出一份精确的文件列表。文件中混有不同归属的改动时，不要建议整文件暂存，说明需要按 diff 片段处理。

## 提交前验证

列出整理完成后需要重新运行的验证命令，以及每条命令要确认什么。

## 提交说明建议

根据实际改动给出一条简洁的提交说明。不要把未完成或未确认的内容写进提交说明。

## 等待确认

明确列出准备清理、保留和暂存的内容，然后停下来等我确认。只有得到明确确认后，才能清理本轮产生的调试残留、重新验证和暂存文件。即使我同意整理，也不代表自动授权提交或推送；提交和推送仍需我单独提出。
```
