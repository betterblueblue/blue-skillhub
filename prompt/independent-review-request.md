# 生成独立验收指令 Prompt

在实现会话完成任务后使用。把下面的 Prompt 发给实现会话，它会生成一段完整的验收指令；将生成的代码块原样粘贴到能访问同一工作区的新会话或其他 agent，即可开始独立验收。

生成结果不需要手工补充“原始需求”或“完成总结”。实现会话负责从对话和工作区提取；找不到的内容必须明确标注，不能自行补全。

```text
请为当前已经完成的任务生成一份可以直接粘贴到新会话或其他 agent 的“独立验收指令”。你现在只负责生成指令，不执行验收，也不要修改任何文件。

生成之前，请重新查看本次对话和当前工作区，收集以下内容：

1. 用户最初提出的需求。能找到原话时尽量保留原话，不要为了贴合当前实现而改写需求。
2. 用户后来确认的补充、范围变化和明确排除项。不要把模型自己的建议写成用户决定。
3. 可以据此判断任务是否完成的成功标准。
4. 项目绝对路径、当前分支、能够确认的任务起点或基线，以及当前未提交改动。基线无法确认时如实说明。
5. 本次实际修改的文件和每个文件的修改目的。有基线时比较基线与当前状态；改动已经提交时检查相关 commit；否则检查当前 diff。不要只凭聊天记忆。
6. 已经运行的验证命令、退出码和关键结果，以及尚未运行或无法运行的验证。
7. 已知风险、仍未确认的内容、用户原有改动和不能覆盖或还原的文件。

如果某项无法从对话、文件或命令结果中确认，写“未找到”或“未确认”，不要猜测。不要向用户索要这些内容，也不要保留任何占位文字。不要包含密钥、Token、密码和其他敏感值。

最终只输出一个 Markdown 代码块，代码块外不要添加解释。代码块中的内容必须是一份可以直接发送给新会话的完整指令，并使用下面的结构：

你现在负责对一个已经完成的开发任务进行独立验收。本轮只做验收，不修改源码、配置或文档。可以运行不会改变业务数据或外部系统的常规检查和测试；可能产生危险副作用的命令不要执行。

## 工作区

写出项目绝对路径、当前分支、能够确认的任务起点或其他可核实基线，以及当前工作区状态。这里必须使用查到的实际值；无法确认的项目写“未确认”。列出与本任务有关的未提交改动，以及明确不属于本任务的用户原有改动。缺少基线时不要猜测改动归属。

## 原始需求

保留用户原话或忠实整理后的需求。将无法找回原文的部分明确标为“根据现有上下文整理”，不能把实现结果反写成需求。

## 后续确认

列出用户后来确认的补充、范围变化、成功标准和明确排除项。没有则写“无”。

## 实现会话的完成说明（待核实）

列出实现会话声称完成的内容、实际 diff 中的文件、已经报告的验证结果，以及它自己承认的限制。这里的内容只是待核实声明，不是验收证据。

## 验收要求

1. 重新读取原始需求和后续确认，逐项列出验收项。
2. 查看当前 git status、实际 diff 和相关文件；改动已经提交时比较相关 commit 或基线与当前状态。不要直接采信上述完成说明。
3. 对修改过的函数、字段、接口、配置和公开行为反查调用方、引用方、注册点、生成物与测试；找不到引用时明确写“未找到”。
4. 检查是否存在漏改、误改、只改一半、无关改动，或者覆盖用户原有改动的情况。
5. 检查测试是否真正覆盖本次需求。能安全判断时，确认测试在旧错误仍存在时会失败；无法验证时如实说明。
6. 重新运行必要的验证命令，记录实际命令、退出码和关键结果。不要引用实现会话的结果代替复验。
7. 测试全部通过也不能直接判定完成；必须回到原始需求逐项核对。

## 输出要求

- 总结论只能是 PASS、FAIL 或证据不足。
- 每项原始需求都要给出结论和具体文件、代码位置或命令证据。
- 单独列出漏改、误改、无关改动、未验证内容和剩余风险；没有也要明确写“未发现”。
- 如果原始需求不足以判断是否完成，不得给出 PASS。缺少任务基线时，说明哪些改动无法确认归属；只有它确实妨碍验收时才判定为证据不足。
- 本轮不要修复发现的问题。先提交验收报告，等待用户决定下一步。
```
