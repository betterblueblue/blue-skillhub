# 无交接恢复现场 Prompt

适用于上一个会话突然中断、上下文已经丢失，而且没有留下 `HANDOFF.md` 的情况。把下面的 Prompt 直接发给新会话，让它先从工作区恢复现场，不要一上来继续改代码。

```text
上一个会话已经中断，也没有留下可靠的交接文档。请先帮我恢复现场。在我确认你的判断之前，不要修改源码、配置或文档，不要安装依赖，不要清理临时文件，也不要还原任何现有改动。

请按下面的顺序检查：

1. 确认当前项目根目录、Git 分支和仓库边界。如果当前目录不是独立仓库，要明确说明。
2. 读取项目里的 AGENTS.md、CLAUDE.md、README、任务文档、计划文件和其他可能记录当前工作的文件。只读与现场恢复有关的内容，不要无限扩大范围。
3. 查看 git status、暂存区、未暂存 diff、未跟踪文件和最近几条提交。不要使用 reset、checkout、clean 等会改变工作区的命令。
4. 阅读发生改动的文件，并检查它们直接相关的调用方、测试和配置。判断哪些改动看起来已经完成，哪些明显只做了一半，哪些可能只是临时调试。
5. 搜索能够说明原任务的证据，例如需求标题、测试名称、TODO、错误信息、计划文档和提交说明。不要仅凭文件名猜任务。
6. 如果工作区里有现成且不会改变业务数据的验证命令，可以说明你建议运行哪一条；先不要运行耗时、会写文件或可能产生外部副作用的命令。

恢复时必须遵守：

- 代码和 diff 只能说明“改过什么”，不一定能证明“用户原本想要什么”。推测原任务时必须标明是推测，并附上依据。
- 不要把所有未提交改动都认定为上一个会话产生的。无法确认归属时写“归属未确认”。
- 不要因为某段代码看起来多余就删除它，也不要替用户整理工作区。
- 找不到证据就写“未确认”或“未找到”，不要补全一个听起来合理的故事。
- 不要输出密钥、Token、密码、用户数据或其他敏感内容。

最后按下面的结构汇报：

## 能确认的现场

写出项目路径、分支、工作区状态、最近相关提交，以及能够从文件和命令确认的事实。每项附上具体路径或命令证据。

## 可能正在进行的任务

用一两句话说明你对原任务的推测，并标注把握程度和依据。如果证据不足，直接写“无法从当前工作区还原原始需求”。

## 改动分类

把相关文件分成：看起来已经完成、仍在进行、疑似临时调试、归属未确认。说明判断依据，不要实际清理。

## 当前卡点

列出能够确认的报错、失败测试、未完成分支和互相矛盾的实现。没有证据时写“尚未找到明确卡点”。

## 不能确认的内容

列出继续工作前必须由我补充或确认的问题，最多 3 个，只问真正影响下一步的问题。

## 建议的下一步

只提出一个最小、可验证的下一步，并说明它能确认什么。不要执行写操作。

汇报后停下来，等我确认你恢复出的任务和下一步。不要自行接着实现。
```
