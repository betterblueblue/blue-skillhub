# Prompt 工具箱

这里收录可以直接发给 AI 编码助手的实用 Prompt。它们不要求安装 Skill，也不依赖某个客户端或插件。

不用先记住所有 Prompt 的名字。找到最接近你当前处境的一项，打开对应文件，复制其中的完整代码块即可。

## 按当前情况选择

| 你现在的情况 | 使用哪个 Prompt | 用完以后 |
|---|---|---|
| 已经有项目想法，但不确定架构和技术路线 | [开工前调研开源项目](open-source-project-research.md) | 检查入选项目和推荐方案，确认后再开始开发 |
| 需求改了几轮，担心 AI 仍在执行已经失效的要求 | [需求变更对账](requirement-change-reconciliation.md) | 目标变了就回到 IntentAnchor；只是改动范围变化就回到 ImpactRadar |
| 同一个问题反复修改，尝试越来越乱 | [卡住时重新梳理](stuck-reassessment.md) | 有新方向就做一次针对性验证；仍无方向再找外援 |
| 当前模型已经没有新思路，准备换更强的模型接手 | [找外援排障](expert-escalation.md) | 把生成的完整求援指令交给新模型 |
| 当前会话还在，但任务需要跨天或换会话继续 | [跨会话交接](session-handoff.md) | 新会话先读取并核对 `HANDOFF.md` |
| 原会话已经丢失，也没有可靠的交接记录 | [无交接恢复现场](workspace-recovery.md) | 用户确认恢复结果后再继续修改 |
| 实现会话说任务已经完成，想换个会话从头检查 | [生成独立验收指令](independent-review-request.md) | 把生成的验收指令交给能访问同一工作区的新会话 |
| 准备提交，但正式修改、调试残留和原有改动混在一起 | [提交前整理改动](pre-commit-cleanup.md) | 用户确认分类结果后，再决定清理、暂存和提交 |

## 开工之前

### 我已经有项目想法，但不确定应该采用什么架构和技术栈

使用 [开工前调研开源项目](open-source-project-research.md)。它会先找出 3–5 个真正相似的 GitHub 项目，核对功能、架构、技术栈、维护状态和许可证，再结合你的目标给出实现方案。你确认方案之前，它不会创建项目或编写代码。

如果你还说不清项目主要给谁用、解决什么问题，先用 [IntentAnchor](../skills/intent-anchor/) 生成 `INTENT.md`。目标已经明确时，直接提供一两段项目想法即可。这个 Prompt 需要交给能够访问 GitHub 的 AI；当前客户端不能联网时，可以接入仓库里的 [网搜 MCP](../mcp/web-search-mcp/)。

## 开发进行中

### 需求改了几轮，我担心 AI 还在同时执行新旧要求

使用 [需求变更对账](requirement-change-reconciliation.md)。它会重新查看原始需求、后续确认和当前实现，分清哪些要求仍然有效、哪些已经失效，以及现有代码需要保留、返工还是删除。

对账完成后，如果连产品目标都变了，回到 IntentAnchor 重新确认意图；如果目标没变，只是已有系统的改动范围变化，回到 ImpactRadar 重新分析影响。

### 同一个问题反复修改，尝试越来越多，却没有更接近根因

先使用 [卡住时重新梳理](stuck-reassessment.md)。它会让当前会话停止继续试错，重新区分已经确认的事实、仍在猜测的原因、已经排除的方向，并只选择一个最能缩小范围的下一步验证。

如果梳理后出现了新的可验证方向，就让当前会话继续。如果仍然没有新线索，再进入找外援，不必一开始就更换模型。

### 当前模型已经绕了多轮，我准备让更强的模型接手

使用 [找外援排障](expert-escalation.md)。它不会继续猜答案，而是把原始问题、复现方法、关键代码、失败尝试、当前证据和仍然未知的内容整理成一份完整指令，让新模型拿到后可以直接开始判断。

这个 Prompt 适合“准备换人”的时候。只是思路混乱、还没有认真清点证据时，先用“卡住时重新梳理”。

## 切换会话

### 当前会话还在，但今天做不完，需要跨天或换会话继续

使用 [跨会话交接](session-handoff.md)。它会在项目根目录生成或更新 `HANDOFF.md`，记录任务目标、已经完成的内容、当前卡点、下一步、验证结果、工作区状态和不能再踩的坑。

新会话的第一句话直接使用文件中提供的“先读 `HANDOFF.md`”指令。新会话仍要核对当前工作区，不能把交接文档当成新的写入授权。

### 原会话已经丢失，而且没有留下可靠的交接文档

使用 [无交接恢复现场](workspace-recovery.md)。它会先只读检查 Git、diff、任务文件和相关代码，尽量还原正在进行的任务、已完成部分、卡点和无法确认的内容。在你确认恢复结果之前，它不会继续修改文件。

如果已有内容完整、仍然可信的 `HANDOFF.md`，先读取并核对它，不需要再走无交接恢复。恢复现场是补救办法，不能保证找回只存在于旧对话里的原始需求。

## 完成以后

### 实现会话说任务已经完成，我想换一个会话从头核实

使用 [生成独立验收指令](independent-review-request.md)。先把它发给实现会话；实现会话会从原始对话和工作区生成一份没有占位符的完整验收指令。再把生成的代码块原样发给能访问同一工作区的新会话或其他 agent。

新会话会按原始需求检查实际 diff、相关调用方和验证结果，不直接采信实现会话的完成说明。这个 Prompt 负责“功能是否真的符合要求”，不负责替你整理提交范围。

### 准备提交，但 diff 里混着正式改动、调试残留和原有文件

使用 [提交前整理改动](pre-commit-cleanup.md)。它会先把文件分成本次必须提交、调试残留、用户原有改动、自动生成内容和归属不明几类，再给出建议；在你确认之前不会擅自删除、还原、暂存或提交。

如果当前 diff 已经清楚，可以先做独立验收；如果连本次究竟要提交什么都说不清，先整理改动。验收产生新的修复后，提交前还要再检查一次工作区。

## 容易选错时看这里

| 两个看起来相近的工具 | 怎么判断 |
|---|---|
| IntentAnchor / 开工前调研开源项目 | 前者确认“到底要做什么”，后者研究“别人怎么做、我们准备怎么实现” |
| 卡住时重新梳理 / 找外援排障 | 前者让当前会话重新判断，后者为更换模型准备完整材料 |
| 跨会话交接 / 无交接恢复现场 | 前者在旧会话还在时主动交接，后者在没有可靠交接材料时恢复现场 |
| 生成独立验收指令 / 提交前整理改动 | 前者检查实现是否符合需求，后者划清哪些改动应该进入提交 |

## 怎么使用

1. 打开与你当前处境对应的 Prompt 文件。
2. 阅读开头的适用条件，确认没有选错。
3. 复制文件中的完整代码块，替换其中明确标出的占位内容，然后发给 AI。
4. Prompt 如果生成了给新会话使用的指令，把生成结果原样交给新会话；不要把原始“生成指令 Prompt”错发给验收或外援会话。
5. 涉及修改、清理、暂存或提交时，先看 AI 的检查结果，再单独确认实际操作。使用 Prompt 本身不代表已经授权写文件。

这些 Prompt 解决的是某个具体时刻的协作问题，不是每次开发都要依次执行的固定流程。完整的场景路线见仓库首页的 [常用完整路线](../README.md#常用完整路线)。
