# 开工前调研开源项目 Prompt

适用于已经有了项目想法，但还没有确定架构、技术栈和实现路线时。它会先研究真正做过类似事情的开源项目，再根据你的目标给出方案；你确认之前，不会创建项目或编写代码。

如果现在连“主要给谁用、解决什么问题”都还说不清，先用 IntentAnchor 梳理意图。已经有 `INTENT.md` 时，直接把文件路径填进下面的 Prompt，不必重新描述一遍。

## 怎么使用

1. 把 `[项目想法或 INTENT.md 路径]` 换成你自己的内容。
2. 将整个代码块发给能够访问 GitHub 的 AI。
3. 先检查它选的项目是否真的可比，再确认或调整实现方案。
4. 只有方案得到你的确认后，才让 AI 进入开发。

```text
我准备做一个项目，当前想法是：

[项目想法或 INTENT.md 路径]

先不要创建项目、安装依赖或编写代码。请先研究 GitHub 上真正解决过类似问题的开源项目，再根据调研结果给我一份适合当前目标的实现方案，等我确认后再动手。

开始前：

1. 如果我提供的是 INTENT.md 路径，先读取它。只把其中“保留”的能力作为本次目标；“推迟”和“放弃”的内容不要重新加回来。
2. 用一句话复述你理解的项目目标。只有缺少的信息会直接影响项目筛选或方案方向时，才向我提问，一次最多问 3 个问题。
3. 如果当前环境不能访问 GitHub，直接说明限制并停止。不要凭记忆编造仓库、活跃度、许可证或代码结构。

调研要求：

1. 选择 3～5 个真正可比的开源项目。核心问题、主要用户或关键工作流程至少有一项与我的目标高度接近，并说明每个项目为什么入选。
2. 不要只按 Star 数或搜索排名选项目。优先考虑相关程度、最近维护情况、文档质量、是否提供可运行版本和许可证；热门但不相关的项目不要列入。
3. 对每个项目至少查看仓库首页、README、主要目录、依赖或构建清单、架构/部署文档，以及能说明核心流程的关键入口。不能只看 README 就断言架构和实现质量。
4. 记录仓库链接、许可证、最近维护情况和你实际查看过的文件或页面。没有找到的内容写“未找到”，不要猜。
5. 明确区分两类内容：
   - 仓库事实：能从代码、配置、文档或提交记录中直接确认。
   - 分析判断：根据这些事实得出的优点、缺点和适用条件。
6. 如果找不到足够相似的项目，如实说明。可以补充相邻领域的参考项目，但必须标明它只能参考什么，不能为了凑数量把不相关项目包装成同类方案。

比较时重点回答：

- 它面向谁，解决什么问题，核心功能是什么。
- 主要架构、技术栈、数据存储和部署方式是什么。
- 哪些实现已经相对成熟，哪些更像演示或早期实验。
- 它最值得借鉴的部分是什么，照搬到我的项目会遇到什么问题。
- 从仓库里能看出哪些维护、扩展、性能、安全或许可证风险。

完成项目比较后，不要把几个项目的功能简单拼在一起。请回到我的目标，给出一份独立的实现方案，包括：

1. 建议的第一版范围：做什么、明确不做什么。
2. 推荐的架构和技术栈，以及选择理由和被放弃的主要备选方案。
3. 核心模块及其关系；涉及数据、接口或权限时，说明关键边界。
4. 哪些设计参考了哪个开源项目，借鉴到什么程度；哪些做法明确不采用，以及原因。
5. 分阶段实施顺序，每一阶段怎样判断已经完成。
6. 当前仍需我决定的问题、主要风险和验证办法。

输出顺序：

## 项目目标理解

写出本次调研使用的目标和范围，标明来自我的描述还是 INTENT.md。

## 入选项目

先列项目、仓库链接、入选理由、许可证和维护情况。

## 对比结果

用一张表比较功能、架构、技术栈、成熟度、值得借鉴的部分和主要问题。表格后补充无法放进表格的重要证据，并注明实际查看过的文件或页面。

## 对当前项目的启发

分别列出“值得借鉴”“不能照搬”“仍缺证据”。

## 推荐实现方案

按第一版范围、架构、技术栈、模块关系、实施顺序和验证方式展开。

## 需要我确认

只列会改变范围、架构或成本的重要决定，并给出你的默认建议和理由。

完成以上内容后停止，不要创建文件、安装依赖或写代码。等我确认方案，或者明确要求修改方案后，再进入下一步。
```

## 和 IntentAnchor 的关系

IntentAnchor 负责把“要做什么”说清楚；这个 Prompt 负责研究“别人怎么做过、我们准备怎么实现”。项目想法还很模糊时，先用 IntentAnchor；目标已经明确时，可以直接从这里开始。
