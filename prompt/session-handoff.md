# 跨会话交接 Prompt

适用于长任务跨天推进、当前会话即将结束或上下文即将压缩时。它不依赖插件，只用项目根目录的一份 `HANDOFF.md` 保存下一次继续工作所需的事实。

## 会话结束前

```text
这个会话即将结束。请为一个完全没有当前对话上下文的新会话编写交接文档，并保存到项目根目录的 HANDOFF.md。

写之前先核对当前工作区、相关文件和本轮实际运行过的验证命令。不要只凭聊天记忆总结，也不要把计划写成已经完成。

HANDOFF.md 至少包含：

1. 我们正在做什么任务：目标、范围、成功标准。
2. 已经完成了什么：关键决策、实际修改的文件、已经通过的验证。
3. 当前状态和卡点：尚未完成、失败原因、不确定项、需要用户决定的内容。
4. 下一步计划：按执行顺序列出具体动作，并写明每一步如何验证。
5. 踩过的坑：已经证伪的方案、失败命令、绝对不要重复的操作，以及原因。
6. 工作区状态：当前分支、未提交改动、用户原有改动、未跟踪文件、不能覆盖或还原的内容。
7. 新会话如何开始：先读哪些文件、先运行哪些只读命令、第一项应该做什么。

记录要求：

- 已完成、进行中、未验证、阻塞必须明确区分。
- 提到代码或配置时写出具体路径；提到验证时写出命令、退出码和关键结果。
- 找不到证据就写“未确认”或“未找到”，不要补全或猜测。
- 不写密钥、Token、密码和其他敏感值。
- 如果 HANDOFF.md 已存在，先读取它，再按当前最新事实更新；删除已经失效的状态，保留仍然有效的背景和警告，不要简单追加一份互相矛盾的新记录。
- HANDOFF.md 只是上下文记录，不代表新的写操作授权，也不能替代用户确认。

写完后重新读取 HANDOFF.md 做一次自检，确认新会话只看这份文件也能继续工作。最后告诉我文件路径，并用不超过 5 条列出最重要的交接内容。
```

## 新会话第一句话

```text
先读HANDOFF.md。读完后先不要修改文件；请用不超过 5 条复述当前任务、已完成内容、卡点、下一步和禁止重复的坑，并核对工作区现状是否仍与交接文档一致。发现不一致时，以当前文件和命令结果为准，先告诉我差异，再继续工作。
```

## 极简版

```text
这个会话要结束了。请把交接文档写到项目根目录 HANDOFF.md，供一个完全没有上下文的新会话使用。写清楚：我们在做什么、已经完成什么、当前卡在哪、下一步是什么、工作区有哪些未提交或不能覆盖的改动、哪些坑绝对不要再踩。先核对文件和命令结果，不要把计划写成已完成；如果 HANDOFF.md 已存在，就按最新事实更新它。写完后重新读取并自检。
```
