# 找外援排障 Prompt

适用于同一个问题已经调试多轮，当前模型仍然没有找到根因，准备把任务交给更强或更擅长该领域的模型时。

它和“卡住时重新梳理”不同：重新梳理仍由当前模型继续排查；找外援则由当前模型整理一份完整求援指令，交给另一个模型独立诊断并继续解决。

把下面的 Prompt 发给当前已经卡住的会话。它会输出一个可以直接粘贴给外援模型的代码块，不需要你手工补充背景。

```text
这个问题已经调试多轮，当前会话仍未解决。现在停止继续修改，请为一个完全没有当前对话上下文、但可以访问同一工作区的外援模型生成一份完整的“排障求援指令”。你现在只负责整理求援材料，不再尝试修复，也不要修改任何文件。

生成之前，重新检查当前工作区、相关代码、git 状态和实际运行过的命令。不要只凭聊天记忆总结。请收集：

1. 原始任务：用户最初要解决什么，成功标准是什么，后来有哪些补充和明确排除项。
2. 当前现象：如何稳定复现，实际命令、退出码、首个错误、关键日志和预期结果分别是什么。不能稳定复现时明确说明。
3. 影响范围：目前确认相关的入口、函数、配置、依赖、调用方和测试。写出具体路径和代码位置；找不到就写“未找到”。
4. 已经尝试的办法：按时间顺序说明每次改了什么、为什么这样改、结果是什么、获得了什么新信息。不要只写“试过但没用”。
5. 已经排除的原因：只列有命令、代码或日志证据支持的结论，并附证据。
6. 仍未确认的判断：把当前怀疑的根因与已确认事实分开，避免把当前模型的猜测传给外援后变成前提。
7. 当前工作区状态：项目绝对路径、分支、相关 diff、未提交改动、临时调试改动、用户原有改动，以及不能覆盖或还原的文件。
8. 环境信息：与问题直接相关的运行时、框架、依赖和平台版本。没有证据支持的版本不要猜。
9. 禁止重复的坑：已经证明无效的命令、修改位置、解决方案和危险操作，并解释为什么不要重试。

只保留定位问题真正需要的日志片段，不要粘贴成百上千行无关输出。不要包含密钥、Token、密码、用户数据和其他敏感内容。某项无法确认时写“未确认”，不要编造，也不要留下需要用户手工填写的占位符。

最终只输出一个 Markdown 代码块，代码块外不要添加解释。代码块必须是一份可以直接发给外援模型的完整指令，并使用下面的结构：

你现在接手一个已经调试多轮仍未解决的问题。你可以访问同一工作区。不要把前一个模型的判断当成事实；先核对工作区和复现结果，再独立判断根因。

## 原始任务

写出用户需求、成功标准、后续补充和明确排除项。尽量保留用户原话；无法找回原话时标明“根据现有上下文整理”。

## 工作区与约束

写出项目绝对路径、当前分支、相关未提交改动、用户原有改动、不能覆盖或还原的内容，以及原任务已经授予的修改范围。不要把这份求援指令当成新的无限制写操作授权。

## 复现方式与实际现象

写出最短稳定复现步骤、完整命令、退出码、首个错误、关键日志位置、实际结果和预期结果。无法稳定复现时说明目前能观察到什么。

## 相关代码与调用关系

列出已经确认相关的文件、函数、配置、依赖、调用方和测试入口，并说明它们为什么相关。未找到的引用要明确写“未找到”。

## 已经尝试过的办法

按顺序列出每次尝试的修改、依据、验证结果和得到的新信息。区分仍留在工作区的改动与已经撤销的尝试。

## 已确认事实

只列有文件、命令或日志证据支持的结论，每条附上证据。

## 尚未确认的判断

列出当前怀疑，但尚未验证的根因和关键问题。不要要求你沿用这些判断。

## 已经排除和不要重复的办法

列出已经证伪的原因、无效方案、重复执行也不会获得新信息的命令，以及可能破坏用户改动或外部数据的危险操作。

## 你需要完成的工作

1. 先读取相关文件并核对 git 状态，确认上述材料与当前工作区一致。发现不一致时，以当前文件和命令结果为准，并指出差异。
2. 在不修改文件的情况下重新复现问题，优先验证最可能区分不同根因的最小实验。不要原样重复已经证伪的尝试。
3. 给出你独立判断的根因及证据。证据不足时继续缩小范围，不要先改代码碰运气。
4. 根因明确后，在原任务授权范围内实施最小修复，保留用户原有改动，不做无关重构。需要扩大范围、改变外部数据或执行危险操作时先询问用户。
5. 添加或更新能够复现问题的测试；如果不适合写自动化测试，说明原因并给出替代验证。
6. 运行必要验证，记录实际命令、退出码和关键结果。测试通过后仍要回到原始成功标准逐项核对。
7. 如果仍无法解决，不要只说“没办法”。写清楚新增证据、剩余可能性，以及下一位专家最值得验证的唯一动作。

## 最终汇报

汇报根因、实际修改、验证结果、没有完成的内容和剩余风险。明确区分“已解决”“部分解决”“仍未解决”，不要在证据不足时宣布完成。
```
