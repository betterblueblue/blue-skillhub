## 背景

T26 已覆盖执行阶段确认门禁压力复测，T27 已补齐 `900-执行记录.md` 模板。但这两项仍不能证明真实生产项目中完整 Phase 5 已经跑通。

本轮将完整执行闭环拆成明确验收项，避免后续把“跑了测试”或“写了模板”误判为生产执行闭环通过。

## Phase 5 闭环通过标准

一次生产级 Phase 5 复验必须同时提供下列证据：

1. 文档确认：light 摘要或 full 三文档已确认；未确认项处理方式明确。
2. 逐项确认：每个写文件、改代码、DDL/DML、配置变更、测试修复操作都有独立确认记录。
3. 影响与回滚：每步记录影响范围、回滚方式、语义约定和验证方式。
4. 执行记录：`900-执行记录.md` 按时间追加，包含命令/文件/SQL/配置键和关键输出。
5. 自动验证：执行后运行项目证据支持的 lint/test/build/SQL/API/UI 验证。
6. 测试失败处理：只自动诊断；任何修复操作再次确认后执行。
7. 收尾状态：已执行、跳过、失败、待用户确认的项都有最终状态和后续动作。

## 推荐复验对象

优先选择低风险但真实的 production-like 样本：

| 候选 | 优点 | 风险 | 建议 |
|------|------|------|------|
| Go RealWorld | Docker Go 测试已跑通，变更面较小 | 仍会写源码；需工作树隔离 | 适合做后端代码 + 测试闭环 |
| Next Learn dashboard | DB/build 前置条件已补齐 | 需 SSL Postgres；容易受 lockfile 警告干扰 | 适合做前端/Server Component 闭环 |
| eShopOnWeb | 生产复杂度高，Docker .NET test 已通过 | 改动成本较高 | 适合后续高置信复验 |

## 下一轮建议

选择 Go RealWorld 做最小真实写操作演练：

- 变更类型：低风险 light 或小 full，例如只改一个 API 错误文案或 serializer 字段展示，不碰 DB。
- 执行方式：在外部验证仓独立工作树/临时分支中执行，避免污染主仓。
- 必须产物：`change-impact/{需求名称}/900-执行记录.md`，记录确认、执行、验证和收尾状态。
- 验证命令：Docker `golang:1.21` + 非 root/临时 DB/`go test -p 1 ./...`，沿用 T24 的已知前置条件。
