### 风险追问

应追问：

1. `overdue` 是人工选择，还是由到期日/定时任务自动计算？
2. `overdue` 是否允许在 create/edit 表单中直接选择，还是只读状态？
3. 统计卡片是否需要新增 overdue 汇总，还是只影响列表/详情？
4. 历史 `pending` 发票是否需要按日期批量迁移为 `overdue`？
5. `overdue` 的视觉样式、排序和筛选规则是什么？

### 判档

应判定 full。

理由：

- 涉及状态枚举和表单校验。
- 涉及 Server Action 写入和缓存刷新。
- 涉及数据查询/聚合和 UI 展示。
- 可能涉及历史数据迁移或派生状态规则。
- 虽然 schema 中 `status` 是 `VARCHAR(255)`，但 TS union/Zod enum 是实际契约约束，不能按纯 UI light 处理。

### 命令验证

- `pnpm install` 已执行成功，安装 `next`、`react`、`next-auth`、`postgres`、`zod` 等依赖。
- `pnpm run build` 已复跑。
- 结果：Next.js 编译和 TypeScript 阶段通过；预渲染 `/dashboard` 时访问 Postgres 失败，报 `ECONNREFUSED`，最终 build 退出码 1。
- 关键证据：失败来自 `fetchCardData()` 等 Server Component 数据读取路径，不是 profile 文件发现失败。
- 项目 `package.json` 没有 `test`、`lint` script，profile 必须标注未确认，不得输出为已验证命令。

这个结果说明 Next.js profile 必须把 build 视为可能触发服务端数据访问的验证命令：若无 DB/环境变量/外部服务，必须标注运行时前置条件，而不是简单写“build 通过/失败”。
