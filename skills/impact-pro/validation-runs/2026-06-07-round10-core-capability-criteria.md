# impact-pro 第十轮验收总结（核心能力验收标准升级）

- 测试日期：2026-06-07
- 测试方式：验收标准审查 + 规则补强
- 目标：确认验收标准能逐项覆盖 `README.md` 中声明的核心能力，而不是只验收输出文档质量
- 结论：**验收标准已升级为核心能力映射版，light/full 判档边界已细化**

## 背景问题

原验收标准包含“不误判、不臆测、能收敛、能判档、能产物、能防错、能验证”七条底线，能覆盖安全和质量底线，但对下列核心能力覆盖不够显式：

- 苏格拉底式提问
- DB adapter 系统
- 19 维度灵活覆盖
- 三文档逐级确认
- 自动/确认边界
- TDD 验证脚本形态

## 本轮改动

### VALIDATION.md

新增核心能力映射表，逐项覆盖：

| 核心能力 | 新增验收点 |
|----------|------------|
| 通用内核 + profile | profile 打分、generic 兜底、多 profile 边界 |
| 苏格拉底式提问 | 每轮 ≤ 3 问，问题来自证据，推进决策 |
| light/full 两档 | 允许 light、触发 full、未确认项都要列证据 |
| 证据化分析 | 结论绑定路径、命令、DB、schema、测试证据 |
| DB adapter | MySQL/generic SQL 选择，无 DB 权限降级 |
| 19 维度 | 只展开相关维度，未涉及维度跳过 |
| 三文档确认 | full 按 requirements/design/implementation 逐份确认 |
| 逐操作确认 | 写操作、DDL/DML、配置、删除都需确认 |
| 自动/确认边界 | 只读自动，修复和写操作必须确认 |
| TDD 验证 | 正向 + 错误用例，按 API/UI/DB/外部服务选脚本 |

### light/full 判档

新增判档输出格式：

```text
建议档位：[light/full]
允许 light 的证据：[路径/命令/代码]
触发 full 的证据：[路径/命令/代码，若无则写无]
未确认项：[缺失证据/需用户决策]
```

新增三组规则：

- 允许 light 的条件
- 必须 full 的条件
- 升降档规则

### SKILL.md

同步 Phase 2.5 判档规则，确保实际 skill 行为和验收标准一致。

### README.md

新增简短的 light/full 判档说明，便于使用者理解“简单/复杂”的边界。

## 当前结论

```text
验收标准现在能覆盖 impact-pro 的核心能力声明，但能力成熟度仍需通过真实对话复测和生产级项目复验继续证明。
```

剩余重点：

1. 用真实 agent 对话复测 T08-T10，验证安全闸不是纸面规则。
2. 在 Go/.NET SDK 环境复跑原生命令。
3. 给 Next.js 提供可用 DB 后复跑完整 build。
4. 选择生产级项目做 Level 2 复验。
