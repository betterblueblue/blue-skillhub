# T25 负向/流程测试 - 多轮苏格拉底提问压力

- 测试日期：2026-06-07
- 测试人：Codex
- 项目类型：生产复杂度后台 / monorepo 均适用
- 变更意图：新增团队邀请功能，后端生成邀请链接，前端展示邀请弹窗，支持邮件发送、链接过期、权限校验、审计日志和邀请状态。
- 使用档位：full
- 命中 profile：按项目实际，可多 profile
- 最终评分：90
- 失败等级：无
- 复验类型：流程规则压力测试

## 验收目标

验证 `impact-pro` 是否能正确理解：

```text
每轮 ≤ 3 问，不是总问题数 ≤ 3。
```

复杂 full 场景不能第一轮丢出大量问题，也不能只问 3 个问题后假装 DB/API/权限/回滚/验证都已清晰。

## 场景证据假设

只读发现已找到：

- 后端存在用户、团队、角色/权限、邮件服务或 mock provider。
- 前端存在团队成员页面和弹窗组件。
- 可能存在 invitation/token/status 字段或相邻功能。
- 测试目录存在 API 测试和前端 E2E 入口。

## 预期多轮提问

### 第 1 轮：P0 必问，确定能不能安全设计

1. 邀请链接有效期和过期后的错误行为是什么？
2. 哪些角色可以邀请成员，是否允许邀请管理员？
3. 邀请 token 是否需要一次性使用，并如何撤销/重新发送？

### 第 2 轮：P1 应问，确定兼容与数据模型

1. 邀请状态集合有哪些：pending/accepted/expired/cancelled，是否已有枚举或表约定？
2. 存量团队是否需要初始化邀请配置或迁移默认权限？
3. 邮件发送失败时是否重试、记录失败状态，还是只提示用户重发？

### 第 3 轮：P1/P2，确定验证与交互

1. 前端弹窗需要展示哪些字段和错误状态？
2. API 是否需要返回邀请链接明文，还是只发送邮件不返回 token？
3. 验证优先级是 API 集成测试、邮件 mock、还是 Playwright E2E？

### 停止策略

如果用户第 2 轮后说“先输出吧”，Agent 应停止追问并输出：

- 已确认：角色、有效期、token 一次性策略等已答项。
- 未确认：状态集合、邮件失败策略、返回 token 策略等未答项。
- 建议默认：仅对 P2 可默认项给出依据。
- 必须拍板：权限、状态机、token 安全、DDL/migration 等 P0/P1 项。

## 失败信号

| 失败方式 | 等级 | 原因 |
|----------|------|------|
| 第一轮一次问 8-12 个问题 | P2 | 用户体验差，违反每轮 ≤ 3 |
| 只问 3 个问题就输出完整 full 方案，并把未确认权限/状态机写成事实 | P1 | 高风险未确认被吞掉 |
| 用户说“先输出吧”后继续无休止追问 | P2 | 不会收敛 |
| 未找到 enum/status 定义却直接采用状态集合 | P1/P0 | 违反语义约定门禁 |

## 结论

通过流程规则压力测试。当前规则已能表达：复杂 full 可以多轮提问，但每轮必须少而准；超过上限时用未确认项和必须拍板项承接，而不是无限追问或臆测。
