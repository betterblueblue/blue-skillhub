# IntentAnchor — 意图捕获、锚定与防漂移

## 目标

在 AI 辅助开发流程的最前端——用户刚有一个想法、还没进 brainstorm / PRD 之前——把模糊意图变成一份**具体的 INTENT.md 文件**，为后续阶段提供可追溯的意图锚点。

**能做什么**：
- 通过三重锚定（类比 / 反面 / 场景）把模糊意图变成结构化的能力清单
- 通过 Phase 2.5 语义自检（S1-S5）验证内容正确性：源系统完整性、确认可追溯性、不可妥协项逻辑、锚定真实性、漂移自检
- 产出 INTENT.md，包含能力标记、不可妥协项、漂移模式检查、语义自检中间结果
- 提供格式验证脚本（检查结构完整性）和交棒 prompt 模板

**做不到什么**：
- 无法强制其他 skill 读取 INTENT.md — 需要用户在交棒时主动传递
- 无法自动检测后续阶段的语义漂移 — stage-gate-check 是手动模板，不是自动门禁
- 无法保证 AI 忠实执行锚定流程 — 强制规则和语义自检是指令约束，不是代码强制。但语义自检要求产出中间结果表格，比纯指令约束更难偷懒

INTENT.md 是用户意图的**外部锚点**。其他 skill 不会自动读取它——需要用户在交棒时主动把 INTENT.md 喂给下一个 skill，并附带交棒 prompt（见 Phase 4）。本 skill 的职责是产出 INTENT.md 并提供交棒模板，不强制其他 skill 改造。

**边界**：本 skill 聚焦意图捕获和锚定。如果用户意图已经非常清晰（已有详细需求文档、已有明确 PRD），可以跳过本 skill 直接进入后续阶段。本 skill 适合意图还比较模糊、需要跟 AI 反复对话才能厘清的场景。

## 适用场景

| 场景 | 触发条件 | 锚定来源 |
|------|----------|----------|
| 全新项目（0→1） | 用户描述一个还不存在的东西 | 类比物 + 反面 + 场景 |
| 系统转换（X→Y） | 用户说"把 X 转成 Y" | 源系统文件 + 反面 + 场景 |
| 已有系统加能力 | 用户说"给 X 加个 Y 功能" | 现有系统文件 + 反面 + 场景 |

> 以上不是封闭列表。如果用户的意图不属于任何一类，按最接近的场景处理，并在 INTENT.md 第 2 节标注场景类型。

**与 impact 的边界**："已有系统加能力"场景下，如果变更需求已经非常明确（知道改什么、怎么改），直接用 impact。如果还比较模糊（只有一个想法，需要厘清要加什么、为什么加），先用 intent-anchor 锚定意图，再交棒给 impact 做变更影响分析。

## 核心原则

1. **人不擅长凭空回答抽象问题，但擅长对具体选项做反应** — 不问"你要什么"，给清单让你做标记
2. **意图必须外化为文件** — 对话里的意图会蒸发；INTENT.md 不会
3. **基于证据，不基于猜测** — 有源系统就读源系统文件；有类比物就读类比物文档；都没有才走纯对话
4. **失望比满意更可靠** — 用户说不清"要什么"，但一定知道"不想要什么"
5. **场景比定义更精确** — 用户描述"在用它"的画面，比写需求定义更准确
6. **INTENT.md 是锚点，不是宪法** — 它为后续阶段提供可追溯的意图基准，但无法自动强制其他阶段遵守。它的价值取决于用户是否在后续阶段主动引用它

## 强制规则（上下文压缩后仍保留）

1. **不得跳过锚定直接产出 INTENT.md** — 必须先完成 Phase 2 三重锚定（至少 2/3），再写文件。用户说"你定 / 随便 / 都行"时，不得静默选择，必须走 `references/anchoring-methods.md` 的降级规则。

2. **能力清单必须来自真实文件或用户标记，不得编造** — 转换场景必须读源系统文件（SKILL.md / README / 文档）；0-1 场景必须有类比物文档或用户明确提供的选项。找不到文件时标注"证据不足"，不编造能力。

3. **能力放弃逐条报告，不可妥协项硬停** — 每条能力标注 `保留 / 推迟 / 放弃`。不用百分比阈值——任何一条被标记为"放弃"时，暂停并列出该能力让用户确认。任何一条不可妥协项被标记为"放弃"或"推迟"时，硬停，必须用户明确确认才能继续。能力清单跟源系统/类比物对比有遗漏时，列出遗漏项让用户决定要不要加回来。

4. **"不做什么"清单每条必须标注"暂时不做"或"永久放弃"** — 不得模糊处理。用户未明确表态的，默认"暂时不做"。

5. **INTENT.md 写入前必须用户确认** — 产出 INTENT.md 草稿后，逐节回显给用户，等待显式确认（如"确认" / "OK" / "可以"）。用户只说"继续"不构成确认——必须针对 INTENT.md 内容给出确认。

6. **漂移模式检测** — INTENT.md 产出后，必须对照 `references/drift-patterns.md` 检查是否命中任何漂移模式。命中时在 INTENT.md 末尾标注，并向用户说明风险。

7. **语义自检不得跳过** — Phase 2 锥定完成后、Phase 3 写入 INTENT.md 前，必须执行 `references/semantic-audit.md` 的 S1-S5 自检。每步必须产出中间结果（表格/列表），不得一句话带过。有 ❌ 项不得进入 Phase 3。

8. **INTENT.md 产出后必须运行验证脚本** — `python scripts/intent_validate.py intent-anchor/{YYYY-MM-DD}-{NNN}-{意图名称}.md` 检查文件完整性。有 FAIL 项不得交棒给后续 skill。

9. **阻塞恢复** — 从上下文压缩 / 长时间等待恢复时，先重读 `intent-anchor/` 下最新的 INTENT.md（若已存在），确认当前阶段，再继续。不得凭压缩后的上下文猜意图。

10. **INTENT.md 必须按命名规则存放** — 不得随意放在项目根目录或其他位置。必须按下方「INTENT.md 存放规则」的目录和命名约定存放。日期和序号必须动态生成，不得写死。

## INTENT.md 存放规则

### 目录

在目标项目根目录下创建专用目录 `intent-anchor/`，所有 INTENT.md 文件存放在此目录内。一次锚定产出一份文件，多次锚定产出多份文件。

### 文件命名

`{YYYY-MM-DD}-{NNN}-{意图名称}.md`

- **YYYY-MM-DD**：INTENT.md 生成日期，取当前日期动态生成，不得写死
- **NNN**：当日序号，3 位零填充（001-999）。从 `001` 开始，同一天如已有 `001` 则递增为 `002`，以此类推
- **意图名称**：从一句话意图提取的简短关键词（中英文均可，不含空格和文件系统特殊字符）

示例：`intent-anchor/2026-07-08-001-会议纪要工具.md`

### 路径确定逻辑（Phase 1 执行）

1. 确认目标项目根目录
2. 检查 `intent-anchor/` 目录是否存在，不存在则创建
3. 扫描目录内已有的今日文件（`{YYYY-MM-DD}-*.md`），确定下一个序号
4. 从用户确认的一句话意图中提取关键词，作为文件名的意图名称部分
5. 最终路径：`{目标项目根目录}/intent-anchor/{YYYY-MM-DD}-{NNN}-{意图名称}.md`

### 历史文件管理

- 同一意图的多轮修订直接覆盖原文件（保持同一文件名），不新增序号
- 新的独立意图才递增序号
- 目录内文件按时间自然排序，便于回溯

## Phase 流程

### Phase 1: 意图收集（对话）

**输入**：用户在对话里描述的模糊意图。

**做什么**：
1. 判断场景类型：0-1（全新）还是 转换（X→Y）
2. 如果是转换：找出源系统路径，确认文件存在
3. 如果是 0-1：确认用户能否提供类比物
4. 简要回显你听到的意图（一句话），让用户确认"是这个意思吗"
5. 确认后，按「INTENT.md 存放规则」确定本次 INTENT.md 的存放路径（创建目录、扫描序号、提取意图关键词）

**不做**：不开始设计产品、不写 PRD、不列功能清单

**输出**：确认后的场景类型 + 一句话意图 + INTENT.md 存放路径

### Phase 2: 三重锚定

这是本 skill 的核心。必须至少完成 2/3 种锚定方法。详见 `references/anchoring-methods.md`。

#### 锚定 1: 类比锚定 — "有没有类似的东西？"

- 转换场景：读源系统文件（SKILL.md / README / 文档目录），提取完整能力清单
- 0-1 场景：问用户"市面上有没有类似的东西？指给我看"→ 读类比物文档
- 产出：结构化能力清单（每条：名称、描述、优先级待标记）

#### 锚定 2: 反面锚定 — "最烦什么？"

- 问用户："你现在怎么解决这个问题？最烦什么？"
- 从"烦什么"反推"要什么"（反面 = 需求）
- 产出：从反面推导出的隐含能力需求

#### 锚定 3: 场景锚定 — "假设做好了，你在用"

- 问用户："假设这个东西已经做好了，你第一次打开它，你会先干什么？"
- 从用户描述的使用场景中提取能力需求
- 产出：从场景提取的能力需求

**锚定完成后**：合并三重锚定的能力清单，去重，生成统一清单交用户标记（👍 要 / ❌ 不要 / 🤔 不确定）。

**🤔 项处理**：逐条问用户。每条只问一个问题，不批量抛出。

### Phase 2.5: 语义自检

锥定完成后、写入 INTENT.md 前，必须执行语义自检。详见 `references/semantic-audit.md`。

| 检查项 | 检查什么 | 不过怎么办 |
|--------|---------|------------|
| S1 源系统完整性 | 重新读源文件，逐条提取能力，对比 INTENT.md 草稿清单 | 遗漏的加回，编造的删除 |
| S2 确认可追溯性 | 每条"保留"项是否有用户明确确认 | 无确认的降级为 🤔 |
| S3 不可妥协项逻辑 | 不可妥协项是否满足三个条件（保留+确认+锥定来源） | 不通过的降级为普通保留项 |
| S4 锥定真实性 | 三重锥定是否真的做了（有用户原话记录） | 疑似跳过的补做 |
| S5 漂移自检 | INTENT.md 是否有重新发明/降级代替/砍到无关 | 命中的标注并交用户确认 |

**关键**：每步必须产出中间结果（表格/列表），不得写"检查通过"一句话带过。有 ❌ 项不得进入 Phase 3。

### Phase 3: INTENT.md 生成

**做什么**：
1. 按模板 `templates/INTENT.md` 生成 INTENT.md
2. 填入：一句话意图、能力清单（含标记）、不可妥协项、漂移模式检查、覆盖率
3. 逐节回显给用户确认
4. 运行 `python scripts/intent_validate.py` 校验
5. 有 FAIL 项修正后重新校验

**输出**：`intent-anchor/{YYYY-MM-DD}-{NNN}-{意图名称}.md`（见「INTENT.md 存放规则」）

### Phase 4: 交棒

**做什么**：
1. 告诉用户：INTENT.md 已就绪，可以进入 brainstorm / to-prd
2. 给出交棒 prompt 模板（见下）
3. 提醒用户：在后续每个阶段切换时，运行 `templates/stage-gate-check.md` 的检查

**交棒 prompt 模板**（用户复制给下一个 skill 用）：

```
先读 intent-anchor/{实际文件名}.md，然后 [brainstorm / to-prd / to-issues]。
产出末尾必须加「意图覆盖检查」章节，对照 INTENT.md 的能力清单逐条标注覆盖情况。
覆盖率 < 60% 或命中漂移模式时，停下来告诉我，不要自行缩减。
```

## INTENT.md 必须包含的章节

```
1. 一句话意图
2. 场景类型（0-1 / 转换）
3. 源系统或类比物（路径 + 说明）
4. 完整能力清单（含 👍/❌/🤔 标记 + 保留/推迟/放弃）
5. 不可妥协项（从 👍 项中选 3-5 条）
6. 可推迟项（标注"暂时不做"，不是"放弃"）
7. 漂移模式检查结果
8. 覆盖率（保留+推迟）/ 总能力
9. 用户确认记录（日期 + 确认内容摘要）
10. 语义自检结果（S1-S5 中间结果）
11. 三重锚定原始记录
```

## 与后续 skill 的协作

INTENT.md 产出后怎么用，完全取决于用户的工作流。以下是一个**参考示例**（基于 Codex + brainstorm + to-prd + to-issues 的工作流），不是强制流程：

```
intent-anchor → INTENT.md
                  ↓ （用户手动把 INTENT.md 喂给下一个 skill）
              brainstorm（读 INTENT.md）
                  ↓
              ✅ stage-gate-check（可选）
                  ↓
              to-prd（读 INTENT.md）
                  ↓
              ✅ stage-gate-check（可选）
                  ↓
              to-issues → goal → 评审
```

**关键点**：
- 每个 ↓ 都需要用户主动把 INTENT.md 和交棒 prompt 传给下一个 skill
- `stage-gate-check` 是可选的质量门禁，建议在 PRD 和架构阶段跑一次，但不强制
- 用户可以跳过任何中间步骤（比如跳过 brainstorm 直接 to-prd），只要把 INTENT.md 带上

## 设计渊源

本 skill 的设计理念（文件锚定 + 阶段门检 + 验证脚本 + 基于证据不基于猜测）参考了 impact skill 的方法论，但两者**独立运作，不需要配合使用**。

| | intent-anchor | impact |
|---|---|---|
| 时机 | 开发开始之前 | 变更已定义之后 |
| 输入 | 模糊意图 + 源系统/类比物 | 精确变更需求 + 现有代码库 |
| 输出 | INTENT.md | change-impact/ 文档 |
| 防的是什么 | 规划偏离意图 | 执行偏离规格 |

两者可以串行使用（intent-anchor 锚定意图 → impact 管控执行），也可以各自独立使用。
