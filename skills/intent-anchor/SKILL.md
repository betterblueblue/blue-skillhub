# IntentAnchor

## 目标

在 brainstorm、PRD 和开发之前，把仍然模糊的产品意图整理成一份 `INTENT.md`。文件应记录用户想解决的问题、能力取舍、不可妥协项、尚未解决的冲突，以及每项决定来自哪里。

本 skill 只负责澄清和记录意图：

- 不设计产品方案，不写 PRD，不拆任务，不修改业务代码。
- 不能证明内容一定符合用户真实想法；语义复核只留下可供用户检查的证据。
- 不能强制后续 skill 自动读取 `INTENT.md`；交接仍需用户主动传递文件路径。
- `intent_validate.py` 只检查结构、字段关系和确认记录，不验证事实真假。

如果用户已经给出明确 PRD 或可执行规格，直接进入后续流程。已有系统的变更如果已经清楚“改什么”，使用 impact；只有目标和取舍仍模糊时才先使用 intent-anchor。

## 场景分流

| 场景 | 主要证据 | 能力盘点范围 |
|---|---|---|
| 全新项目（0→1） | 用户问题、现有做法、使用场景；类比物可选 | 只提取与用户意图有关的候选能力，不追求复制类比物 |
| 系统转换（X→Y） | 源系统文件、用户希望保留或改变的体验 | 完整盘点源系统能力，再由用户决定保留、推迟或放弃 |
| 已有系统新增能力 | 现有系统相关文件、目标用户和使用场景 | 只盘点相关模块，不扩展到整个系统 |

无法归类时，先用一句话说明你的理解并请用户确认场景。不要默默套用最接近的流程。

## 决策契约

能力清单使用稳定 ID（`C01`、`C02`……），只保留一列“决策”。能力 ID 一旦展示给用户，含义就不得改变或复用；新增能力使用新 ID，合并能力时保留最早展示的 ID。详细规则见 `references/anchoring-methods.md`。

- `保留`：属于目标产品或功能。
- `推迟`：当前不做，后续可以重新讨论。
- `放弃`：明确不再追求。
- `待确认`：模型提出了候选项，但用户尚未决定。最终交接前不得保留 `待确认`。

每项决策还必须记录“决策来源”：

| 决策来源 | 含义 | 允许的决策 | 能否成为不可妥协项 |
|---|---|---|---|
| 用户明确确认 | 用户看过具体能力后明确选择 | 保留 / 推迟 / 放弃 | 可以 |
| 用户授权模型决定 | 用户明确说“你定”，模型说明选择和理由 | 保留 / 推迟 | 不可以 |
| 模型建议（未确认） | 模型推导的候选项，用户尚未确认 | 待确认 | 不可以 |

必须遵守以下规则：

1. 用户描述痛点或场景只是证据，不等于确认某个解决方案。
2. 用户说“你定 / 随便 / 都行”是授权模型选择，不是用户本人赞同该选择。不得把它写成“用户明确确认”。
3. `放弃`必须由用户明确确认；模型受托决定时只能选择`保留`或`推迟`。
4. 不可妥协项允许为零。每一项都必须引用一个“保留”能力 ID，并有用户明确确认。
5. 不同来源互相冲突时，一律标记为`待确认`并请用户裁决。场景描述不会自动覆盖用户先前的明确选择。
6. 可以分批展示能力，每批 5-10 项。只对冲突、高影响项和仍不清楚的项逐条追问。
7. 只有问题在用户作答前明确约定“未选视为推迟”，未勾选项才能记为`推迟 / 用户明确确认`，并保留问题约定和用户选择作为证据。普通的未提及或未勾选不算确认，也不能据此记为`放弃`。
8. 用户原话必须逐字摘录。不得为了通过校验补写、改写或拼接用户没有说过的确认用语。

## 锚定完成标准

类比、反面、场景是三种取证方法，不是必须凑够数量的打卡项。详细操作见 `references/anchoring-methods.md`。

一种方法只有同时留下以下内容才算完成：

1. 原始证据：文件路径、原文摘录或用户原话。
2. 从证据提取的候选能力。
3. 用户对候选能力的决定，或明确记录尚未确认。

按场景选择方法：

- 系统转换必须完成源系统盘点，并用反面或场景确认用户为什么要转换。
- 0→1 不要求类比物。通常使用反面和场景；用户已经提供等价证据时，不重复提问。
- 0→1 首批候选默认只列 5-10 项用户可感知能力。状态机、按键缓冲、资源加载器等实现细节留给 PRD 或开发阶段；只有用户明确把技术、许可或交付方式作为约束时才记录。
- 已有系统新增能力只读取相关模块，再用场景确认目标行为。

如果证据仍不足，继续追问或保留`待确认`，不要为了进入下一阶段声称锚定已经完成。

## 强制规则

1. **先确认一句话意图**：未确认前不列完整功能清单。
2. **先取证再推导**：有文件就引用文件；没有文件时保留用户原话。推导必须标明是推导。
3. **先确认再写文件**：在对话中展示完整草稿，得到针对全文的明确确认后，才创建目录和写入文件。“继续”不算全文确认。
4. **语义复核留证据**：写入前执行 `references/semantic-audit.md` 的 S1-S7；它是自查记录，不是正确性证明。
5. **结构校验必须通过**：写入后运行 `intent_validate.py`。FAIL 只表示当前文件不满足结构契约，不代表内容一定错误；PASS 也不代表语义正确。
6. **恢复时确认活动文件**：上下文压缩或长时间等待后，优先读取本会话已记录的活动文件路径。路径不明或目录中有多份文件时，询问用户，不得自动选择“最新一份”。

## 文件存放

最终文件放在目标项目根目录：

```text
intent-anchor/{YYYY-MM-DD}-{NNN}-{意图名称}.md
```

- 日期使用生成当天的实际日期。
- `NNN` 是当日三位序号。
- 同一意图的修订覆盖原文件；独立意图使用新序号。
- Phase 1 只能计算候选路径，不创建目录。全文确认后才创建 `intent-anchor/`。

## 工作流程

### Phase 1：确认问题

1. 确认目标项目根目录。
2. 判断是 0→1、系统转换还是已有系统新增能力。
3. 用一句话回显你理解的意图，请用户确认。
4. 计算候选文件路径，但不创建目录、不写文件。

输出：确认后的场景、问题和候选路径。

### Phase 2：收集证据并形成决策

1. 按场景选择类比、反面和场景方法。
2. 保留原始证据并提取带稳定 ID 的候选能力。
3. 合并重复项；冲突项保持`待确认`。不得改变已经展示给用户的能力 ID 含义。
4. 分批请用户决定`保留 / 推迟 / 放弃`。
5. 用户授权模型决定时，记录为`用户授权模型决定`，并说明理由。
6. 询问是否存在不可妥协项；允许回答“没有”。
7. **识别设计标准**：检查项目目录中是否存在设计/原型素材（`prototype/`、`*.fig`、设计稿导出、可点 HTML 等）。如果存在，向用户确认是否作为 UI 验收基线，并记录到第 12 节。如果不存在，记录用户确认“没有设计素材”。不得跳过此步骤——即使用户没有主动提供设计素材，也要检查项目目录。
8. **标记术语**：逐项审查能力表中的“能力”和“描述”列，识别需要向普通人解释才能看懂的行业黑话。为每个术语给出人话翻译和界面文案，记录到第 13 节。如果能力描述中确实没有术语，记录“无术语需要翻译”。

输出：没有未解释冲突的能力清单和决策来源、设计标准（第 12 节）、术语表（第 13 节）。

### Phase 2.5：语义复核

按 `references/semantic-audit.md` 生成 S1-S7 记录：

| 项目 | 要回答的问题 |
|---|---|
| S1 证据覆盖 | 候选能力能否找到文件或用户原话；转换场景是否漏掉源能力 |
| S2 决策来源 | 每项决定究竟是用户确认、用户授权模型决定，还是仍待确认 |
| S3 不可妥协项 | 是否引用保留能力，并且由用户明确确认 |
| S4 锚定充分性与冲突 | 所用方法是否留下证据；冲突是否交给用户裁决 |
| S5 漂移复核 | 是否新增、替换、遗漏或重新引入了未经确认的能力 |
| S6 设计标准 | 项目中的设计/原型素材是否被识别为 UI 验收基线；用户是否确认 |
| S7 术语标记 | 能力描述中的行业黑话是否被标记并翻译为界面文案 |

任何`待确认`、未解决冲突或来源不明的能力都不得进入最终交接。

### Phase 3：确认并写入

1. 完整读取 `templates/INTENT.md`，以其中的章节和表格为固定骨架生成草稿；不要先自由起草再依赖校验器重写。
2. 在回复中展示完整草稿。第 9 节在用户确认前明确写“待用户全文确认”，不得预填确认原话。
3. 逐节或分块展示草稿，最后明确询问用户是否确认整份内容。
4. 得到全文确认后，把用户回复逐字写入第 9 节，再创建目录并写入候选路径。
5. 运行：

   ```bash
   python "{intent-anchor skill 目录}/scripts/intent_validate.py" "{目标项目根目录}/intent-anchor/{实际文件名}.md"
   ```

6. 修复结构问题后重新运行。若修复改变了能力、证据、决策或用户原话，之前的全文确认立即失效，必须展示更新后的全文并重新确认。
7. 校验器不识别用户原话时，不得改写原话凑关键词；应保留原话并说明校验限制，必要时请用户重新明确确认。

输出：通过结构校验的 `INTENT.md`。

### Phase 4：交接

给用户以下可直接传给下一会话的 Prompt，并替换真实路径：

```text
先读 intent-anchor/{实际文件名}.md，再完成接下来的 [brainstorm / PRD / 任务拆分 / 影响分析]。

只把“保留”的能力作为目标范围；“推迟”和“放弃”的能力不能悄悄加回来。产出末尾增加「意图核对」：
1. 逐项列出本阶段如何处理每个“保留”能力：已体现 / 本阶段不适用（说明原因）/ 遗漏。
2. 单独核对所有不可妥协项。
3. 列出本阶段新增但 INTENT.md 中没有的能力，并等待我确认。

如果 INTENT.md 第 12 节包含设计标准（原型/设计稿等），必须遵守：
- PRD 在 Implementation Decisions 中列出对照的设计文件路径。
- 涉及界面的工单，Acceptance criteria 必须包含“对照 {设计文件} 结构一致”。
- Done 定义：功能能走通 + UI 对照设计标准通过，缺一不可。

如果 INTENT.md 第 13 节包含术语表，必须遵守：
- 界面和用户提示文案使用术语表中的“人话翻译”，不得直接使用原始术语。
- 原始术语只允许出现在开发文档和内部沟通中。

发现遗漏、冲突或未经确认的新增项时停下来告诉我，不要自行改写原意。
```

`templates/stage-gate-check.md` 是可选的人工检查表，不是自动门禁。

## 改进记录提示

只有本次运行暴露出 Skill 自身可能需要改进的具体问题时，才在交接内容之后询问。适合询问的情况包括：结构校验通过但意图仍明显失真、决策来源被错误记录、同类问题重复发生，或者流程让用户反复卡住。普通完成不询问；已有校验当场拦住并修正的一次性格式问题也不询问。

面向用户只说：

> 这次发现一个可能值得用于改进 Skill 的问题：<一句话说明问题和后果>。要把它记录下来吗？你回复“记录”或“不用”就行。

不要向用户展示内部编号、状态名、维护文档路径或证据门槛。

- 用户回复“记录”：在对话中整理一份完整记录，包含日期、Skill、任务、问题、原始证据、实际后果、校验表现和最终结果。不要修改 Skill，也不要把记录写进目标项目。
- 用户回复“不用”：不保存、不继续追问。
- 其他回复不视为同意记录。

## INTENT.md 必需章节

1. 一句话意图
2. 场景类型
3. 证据来源
4. 能力与决策
5. 不可妥协项
6. 推迟与放弃
7. 漂移复核
8. 决策统计与未决问题
9. 用户确认记录
10. 语义复核记录
11. 锚定原始记录
12. 设计标准
13. 术语表

## 与 impact 的边界

| | intent-anchor | impact |
|---|---|---|
| 使用时机 | 目标和取舍仍模糊 | 变更需求已经明确 |
| 输入 | 用户问题、场景、源系统或可选类比物 | 明确需求和现有代码库 |
| 输出 | `INTENT.md` | `change-impact/` 文档和实施记录 |
| 主要作用 | 防止后续误解用户意图 | 防止实施遗漏影响范围 |

两者可以串行使用，也可以独立使用。
