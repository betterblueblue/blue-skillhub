- 阶段：{brainstorm / PRD / 架构 / 任务拆分 / 开发完成}

## 1. 文件与范围

- 使用的是当前活动 INTENT.md：是 / 否
- 产出明确引用该文件：是 / 否
- 检查对象只包括第 4 节中决策为“保留”的能力：是 / 否
- 推迟和放弃项没有被默认算入当前范围：是 / 否

任一项为“否”时先修正检查范围，不继续计算结果。

## 2. 保留能力逐项核对

<!-- 状态只使用：已体现 / 本阶段不适用 / 遗漏。选择不适用时必须说明原因。 -->

| 能力 ID | 保留能力 | 状态 | 证据或原因 |
|---|---|---|---|
| C01 | {能力} | 已体现 | {产出中的位置} |
| C02 | {能力} | 本阶段不适用 | {为什么应在后续阶段处理} |

判定规则：

- 存在`遗漏`：停止进入下一阶段，补充产出或请用户重新决定。
- `本阶段不适用`缺少理由：视为遗漏。
- 不使用“达到 60% 即通过”之类的阈值。

## 3. 不可妥协项

## 6. 阶段专项检查

### brainstorm 或 PRD

- 新提出的能力是否都标明为“已在 INTENT.md”或“待用户确认”？
- “不做什么”是否分别对应推迟或放弃，而不是含糊写成“不追求”？

### 架构

- 架构是否支持所有本阶段相关的保留能力？
- 技术方案是否把某个能力换成了不同目标？

### 任务拆分

- 每个任务是否能追溯到保留能力、必要工程工作或用户确认的新增项？
- 是否存在没有任务承接的保留能力？

### 开发完成

- 每个本阶段保留能力是否有实现或验证证据？
- 是否有能力被静默降级、遗漏或重新加入？
