# intent-anchor

> 在 AI 辅助开发的最早阶段——用户刚有一个想法、还没进 brainstorm / PRD 之前——捕获和锚定用户意图，产出一份结构化的 INTENT.md 作为后续阶段的意图锚点。

## 为什么会有这个 skill

这个 skill 诞生于一次失败复盘。

用户用 Codex + GPT-5.5 走了一条看起来很规范的工作流：聊需求 → brainstorm → to-prd → to-issues → goal 驱动开发。每个环节的执行质量都很高——PRD 自洽、架构合理、issue 粒度合适、代码质量过关。

但最后做出来的东西跟用户想要的不是一个东西。

原因不是执行质量，而是**意图在阶段传递中逐级蒸发了**：

```
用户口述意图 → AI 提取关键词（丢了大部分）→ brainstorm 基于关键词探讨（已经偏了）
→ PRD 基于偏了的探讨写产品定义（更偏）→ issue 基于偏了的 PRD 拆任务（越来越偏）
→ goal 基于偏了的 issue 写代码（偏到天边）
```

每个环节都在优化"当前产出的质量"，没有人优化"是否忠于原始意图"。因为原始意图只存在于对话里，到了第三个环节它已经被压缩或遗忘了。

**根因**：意图只存在于对话里（软约束），没有外化为文件（硬约束）。AI 会选择性遵守软约束，但必须遵守硬约束。

## 它解决什么问题

| 问题 | 怎么解决 |
|------|---------|
| 意图在对话中蒸发 | 把意图外化为 INTENT.md 文件 |
| AI 重新发明产品而非保留源系统能力 | 三重锚定（类比/反面/场景），能力清单来自真实文件 |
| 能力被静默砍掉 | 放弃项逐条报告，不可妥协项硬停 |
| AI 说"检查过了"但没真查 | 语义自检 S1-S5，每步必须产出中间结果表格 |
| AI over-promise | 明确写"能做什么/做不到什么" |
| 后续阶段不读 INTENT.md | 诚实承认做不到自动强制，提供交棒 prompt 模板让用户手动传递 |

## 核心方法论：三重锚定

人不擅长凭空回答抽象问题，但擅长对具体选项做反应。给你看 5 个选项，你立刻知道哪 3 个不要。让你凭空说你要什么，你说不出来。

所以不问"你要什么"，而是问三个问题：

| 锚定方法 | 问什么 | 从哪提取能力 |
|---------|--------|------------|
| 类比锚定 | "有没有类似的东西？指给我看" | 读源系统/类比物文件，提取能力清单 |
| 反面锚定 | "你现在怎么解决？最烦什么？" | 从痛点反推能力需求 |
| 场景锚定 | "假设做好了，你第一次打开它会先干什么？" | 从使用画面提取能力需求 |

至少完成 2/3 种。合并去重后交用户标记（👍 要 / ❌ 不要 / 🤔 不确定）。

## 工作流程

```
Phase 1: 意图收集（对话）
  判断场景类型，确认一句话意图
         ↓
Phase 2: 三重锚定
  类比锚定 + 反面锚定 + 场景锚定
  合并能力清单，交用户标记
         ↓
Phase 2.5: 语义自检
  S1 源系统完整性（重读文件，对比清单有没有遗漏）
  S2 确认可追溯性（每条"保留"项有没有用户确认）
  S3 不可妥协项逻辑（三条件检查）
  S4 锚定真实性（三重锚定真的做了吗）
  S5 漂移自检（有没有重新发明/降级代替/砍到无关）
         ↓
Phase 3: INTENT.md 生成
  按模板生成，逐节回显用户确认
  运行验证脚本（9 项检查）
         ↓
Phase 4: 交棒
  提供交棒 prompt 模板，用户手动传递给下一个 skill
```

## 诚实的边界

**能做什么**：
- 通过三重锚定把模糊意图变成结构化的能力清单
- 通过语义自检验证内容正确性
- 提供格式验证脚本和交棒模板

**做不到什么**：
- 无法强制其他 skill 读取 INTENT.md — 需要用户手动传递
- 无法自动检测后续阶段的语义漂移 — stage-gate-check 是手动模板
- 无法保证 AI 忠实执行 — 强制规则是指令约束，但语义自检要求产出中间结果，比纯指令更难偷懒

## 设计渊源

设计理念（文件锚定 + 阶段门检 + 验证脚本 + 基于证据不基于猜测）参考了 [impact](https://github.com/betterblueblue/blue-skillhub/tree/master/skills/impact) skill 的方法论，但两者独立运作。

| | intent-anchor | impact |
|---|---|---|
| 时机 | 开发开始之前 | 变更已定义之后 |
| 输入 | 模糊意图 + 源系统/类比物 | 精确变更需求 + 现有代码库 |
| 输出 | INTENT.md | change-impact/ 文档 |
| 防的是什么 | 规划偏离意图 | 执行偏离规格 |

## 自我验证

这个 skill 用自己的方法论给自己跑了一遍意图锚定，产出了 `INTENT.md`。验证脚本 9 项全部 PASS。过程中暴露并修复了 2 个验证脚本 bug 和 1 个设计缺陷（60% 覆盖率阈值不合理，改为逐条报告）。

这本身就是方法论的一次实战检验。
