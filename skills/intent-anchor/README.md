# IntentAnchor

> 在 brainstorm 和 PRD 之前，把还没说清楚的产品意图整理成一份可检查、可交接的 `INTENT.md`。

## 为什么需要它

AI 辅助开发最麻烦的问题之一，是早期想法在多次转述后慢慢变了。需求讨论、PRD、任务拆分和开发单独看都可能很完整，最后却在认真完成另一个目标。

IntentAnchor 先把用户原话、候选能力和明确取舍写进文件。后续会话可以重新读取这些记录，而不是依赖已经被压缩的聊天上下文。

它不能保证 AI 一定理解正确，也不能强制其他 Skill 自动遵守 `INTENT.md`。它提供的是一份更容易复核的外部记录，以及一套避免把模型推导冒充用户决定的规则。

## 什么时候使用

适合：

- 只有一个新产品想法，还没有 PRD。
- 想把现有系统转换成另一种形态，但哪些能力要保留还没决定。
- 想给已有系统增加能力，但目标用户、使用方式或取舍仍然模糊。
- 希望先把意图说清楚，再交给 brainstorm、PRD 或 ImpactRadar。

不适合：

- 普通文件格式转换或代码语法迁移。
- 已经有明确 PRD、验收标准或可执行规格的开发任务。
- 只需要查看陌生项目结构；这种情况使用 Pathfinder。

## 三种锚定方法

| 方法 | 主要问题 | 作用 |
|---|---|---|
| 类比 | 有没有接近的系统或产品？具体参考哪部分？ | 在系统转换中盘点源能力；在 0→1 中帮助用户比较偏好 |
| 反面 | 现在怎么解决？最不满意什么？ | 从痛点发现候选能力 |
| 场景 | 假设已经做好，第一次使用时会怎么做？ | 从实际动作发现候选能力和冲突 |

三种方法按信息缺口选用，不要求机械完成固定数量。系统转换必须完整盘点源系统；0→1 可以完全没有类比物。每种方法只有留下原始证据、候选能力和用户反馈才算完成。

## 决策不会混在一起

每项能力使用一个稳定 ID，并记录两个不同概念：

| 字段 | 合法值 |
|---|---|
| 决策 | 保留 / 推迟 / 放弃 / 待确认 |
| 决策来源 | 用户明确确认 / 用户授权模型决定 / 模型建议（未确认） |

这一区分很重要。用户说“你定”代表允许模型代为选择，不代表用户本人赞同该选择。模型受托时只能选择保留或推迟，不能永久放弃能力，也不能自行指定不可妥协项。

不可妥协项没有最低数量。用户可以明确回答“没有”；如果存在，每一项都必须引用一个保留能力，并留下用户针对该项的明确确认。

## 工作流程

```text
Phase 1：确认问题
  判断场景，用一句话回显意图，只计算候选路径，不写文件
        ↓
Phase 2：收集证据并形成决策
  按场景选择类比、反面和场景方法，分批确认能力
        ↓
Phase 2.5：语义复核
  S1 证据覆盖
  S2 决策来源
  S3 不可妥协项
  S4 锚定充分性与冲突
  S5 漂移复核
        ↓
Phase 3：确认并写入
  先在对话中展示完整草稿，用户确认全文后才写文件
  运行 9 项结构与交叉引用检查
        ↓
Phase 4：交接
  提供带真实文件路径的 Prompt，由用户传给下一会话
```

语义复核会留下证据和中间判断，方便用户检查；它不是正确性证明。`intent_validate.py` 只检查章节、决策关系、引用、统计和确认记录，PASS 不代表内容一定符合用户真实想法。

## 后续阶段怎么核对

交接检查不再使用统一完成百分比。后续产出需要：

- 逐项说明每个保留能力是“已体现”“本阶段不适用”还是“遗漏”。
- 单独核对所有不可妥协项。
- 排除推迟和放弃项，除非用户重新确认。
- 报告 INTENT.md 中没有的新增能力。

`templates/stage-gate-check.md` 提供人工检查表，但不会自动阻止后续 Skill。

## 文件结构

```text
intent-anchor/
├── SKILL.md
├── INTENT.md                         ← 早期设计过程的历史记录
├── references/
│   ├── anchoring-methods.md          ← 三种锚定方法和决策规则
│   ├── drift-patterns.md             ← 7 种通用漂移模式
│   └── semantic-audit.md             ← S1-S5 语义复核规则
├── templates/
│   ├── INTENT.md                     ← 当前输出模板
│   └── stage-gate-check.md           ← 可选人工检查表
├── scripts/
│   └── intent_validate.py            ← 9 项结构与交叉引用检查
└── tests/
    ├── fixtures/valid-intent.md       ← 当前契约的有效样本
    └── test_intent_validate.py        ← 行为回归测试
```

根目录下的 [INTENT.md](INTENT.md) 记录了 Skill 设计初期的一次自我验证，沿用当时的格式，不是当前模板的标准样本，也不保证通过当前校验器。当前契约以 `templates/INTENT.md`、校验器和测试 fixture 为准。

## 能力边界

IntentAnchor 能够：

- 把用户原话、文件证据、模型推导和最终决定分开记录。
- 在系统转换中检查源能力是否遗漏。
- 暴露冲突、未确认项和后续阶段的范围变化。
- 生成可供新会话读取的意图文件。

IntentAnchor 做不到：

- 自动判断模型的语义推导是否正确。
- 自动证明用户原话没有被遗漏或伪造。
- 强制其他 Skill 读取 INTENT.md。
- 自动判断 PRD、架构或代码是否偏离；阶段检查仍需人工或另一个会话执行。

## 与 ImpactRadar 的关系

| | IntentAnchor | ImpactRadar |
|---|---|---|
| 使用时机 | 目标和取舍仍然模糊 | 变更需求已经明确 |
| 输入 | 用户问题、场景、源系统或可选类比物 | 明确需求和现有代码库 |
| 输出 | `INTENT.md` | `change-impact/` 文档和实施记录 |
| 主要作用 | 防止后续误解用户意图 | 防止实施遗漏影响范围 |

两者可以串行使用，也可以独立使用。
