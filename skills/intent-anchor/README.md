# IntentAnchor

> 在头脑风暴和 PRD 之前，把模糊想法整理成一份可检查、可交接的 `INTENT.md`。

## 为什么需要它

很多项目最后做偏，并不是因为代码写得差，而是因为最初的想法在一次次转述中变了样。

IntentAnchor 来自一次真实复盘。用户用 Codex + GPT-5.5 依次讨论需求、生成 PRD、拆分任务并完成开发。每个阶段单独看都没有明显问题，但最终产品并不是用户原本想做的东西。

问题出在信息传递：

```
用户描述想法 → AI 提炼关键词 → 根据关键词讨论方案
→ 根据讨论写 PRD → 根据 PRD 拆任务 → 根据任务写代码
```

如果最初的理解有一点偏差，后面的文档和代码可能都很完整，却在认真解决另一个问题。IntentAnchor 会先把原始意图写进文件，让后续阶段随时可以回看和核对。

## 它解决什么问题

| 问题 | 怎么解决 |
|------|---------|
| 关键想法只留在对话里 | 写入 `INTENT.md`，方便后续阶段回看 |
| AI 擅自补充或重做产品定义 | 从类比、现状痛点和实际使用场景中提取能力 |
| 原有能力在规划中被悄悄删掉 | 所有能力都要明确选择保留、推迟或放弃 |
| AI 声称检查过，却没有留下过程 | S1-S5 语义自检必须给出中间结果 |
| 后续阶段不知道该读什么 | 提供交接 Prompt，由用户把 `INTENT.md` 交给下一个会话 |

## 核心方法论：三重锚定

人往往很难直接回答“你到底想要什么”，却很容易判断一个具体方案哪些部分有用、哪些部分不对。

IntentAnchor 因此从三个角度提问：

| 锚定方法 | 问什么 | 从哪提取能力 |
|---------|--------|------------|
| 类比锚定 | "有没有类似的东西？指给我看" | 读源系统/类比物文件，提取能力清单 |
| 反面锚定 | "你现在怎么解决？最烦什么？" | 从痛点反推能力需求 |
| 场景锚定 | "假设做好了，你第一次打开它会先干什么？" | 从使用画面提取能力需求 |

至少完成其中两个角度。合并并去重后，再请用户逐项选择“保留”“推迟”或“放弃”。

## 工作流程

```
Phase 1：意图收集（对话）
  判断场景类型，确认一句话意图
         ↓
Phase 2：三重锚定
  类比锚定 + 反面锚定 + 场景锚定
  合并能力清单，交用户标记
         ↓
Phase 2.5：语义自检
  S1 源系统完整性（重读文件，对比清单有没有遗漏）
  S2 确认可追溯性（每条"保留"项有没有用户确认）
  S3 不可妥协项逻辑（三条件检查）
  S4 锚定真实性（三重锚定真的做了吗）
  S5 偏离检查（有没有重新定义产品、用低配方案替代目标或删掉相关能力）
         ↓
Phase 3：生成 INTENT.md
  按模板生成，逐节回显用户确认
  运行验证脚本（9 项检查）
         ↓
Phase 4：交接
  提供交接 Prompt，用户手动传递给下一个 Skill
```

## 文件结构

```
intent-anchor/
├── SKILL.md                          ← 核心定义（场景识别 + 4 个 Phase + 强制规则）
├── INTENT.md                         ← 本 Skill 自己生成的意图文件
├── references/
│   ├── anchoring-methods.md          ← 三重锚定详细规则
│   ├── drift-patterns.md             ← 9 种偏离模式检查清单
│   └── semantic-audit.md             ← 语义自检 S1-S5 详细步骤
├── templates/
│   ├── INTENT.md                     ← 意图锚定输出模板（11 个章节）
│   └── stage-gate-check.md           ← 阶段检查模板（可选）
└── scripts/
    └── intent_validate.py            ← INTENT.md 完整性校验（9 项检查）
```

## 能力边界

**能做什么**：

- 通过三重锚定把模糊意图变成结构化的能力清单
- 通过语义自检验证内容正确性
- 提供格式验证脚本和交棒模板

**做不到什么**：

- 不能强制其他 Skill 自动读取 `INTENT.md`，仍需用户主动传递
- 不能自动判断后续 PRD 或代码是否偏离意图，`stage-gate-check.md` 只是人工检查模板
- 不能保证所有模型都严格执行流程，因此仍需保留对话记录和中间结果供人复核

## 设计渊源

IntentAnchor 参考了 [ImpactRadar](https://github.com/betterblueblue/blue-skillhub/tree/master/skills/impact) 中“把关键信息写进文件、用脚本检查格式、结论需要证据”的做法，但两者可以独立使用。

| | IntentAnchor | ImpactRadar |
|---|---|---|
| 时机 | 开发开始之前 | 变更已定义之后 |
| 输入 | 模糊意图 + 源系统或类比物 | 明确的变更需求 + 现有代码库 |
| 输出 | `INTENT.md` | `change-impact/` 文档 |
| 主要防止 | 规划偏离原始意图 | 实际改动偏离已确认要求 |

## 验证

这个 Skill 曾用自己的流程生成 [INTENT.md](INTENT.md)，并通过 `intent_validate.py` 的 9 项检查。校验脚本还有独立的行为测试，CI 会验证非法决策值、能力名称不一致和确认记录缺失等情况。

验证脚本只检查能够明确判断的格式和交叉引用，内容是否真正符合用户想法，仍需用户确认。
