# 三重锚定详细规则

> 本文件包含 Phase 2 三种锚定方法的详细操作规则。SKILL.md 正文只保留概要。

## 核心理念

**人不擅长凭空回答抽象问题，但擅长对具体选项做反应。**

三种锚定方法对应三种不同的"给选项"方式：

| 方法 | 给什么选项 | 触发什么反应 | 产出什么 |
|------|-----------|-------------|---------|
| 类比锚定 | 已有系统的能力清单 | "这个要 / 不要" | 结构化能力清单 |
| 反面锚定 | 现状中的痛点 | "我最烦这个" | 隐含能力需求 |
| 场景锚定 | 使用画面 | "我在做这个" | 场景提取的能力需求 |

至少完成 2/3 种。如果三种都完成了，合并去重后交用户标记。

---

## 锚定 1: 类比锚定

### 适用场景

- **转换项目**：源系统就是类比物，直接读文件
- **0-1 项目**：用户能指出一个类似的产品 / skill / 文档

### 转换项目操作

1. 确认源系统路径（如 `/path/to/source-system/`）
2. 读取以下文件（按优先级）：
   - `SKILL.md` 或 `README.md`（总览）
   - `references/commands/*.md` 或等效目录（逐个能力）
   - 任何 `VALIDATION.md` / `CHANGELOG.md`（边界和约束）
3. 提取能力清单，每条包含：
   ```
   | # | 能力名称 | 描述 | 来源文件 | 优先级（待标记）|
   |---|---------|------|----------|----------------|
   | 1 | kickoff | 初始化辅导档案 | commands/kickoff.md | 🤔 |
   ```
4. 交用户做标记（👍 要 / ❌ 不要 / 🤔 不确定）

### 0-1 项目操作

1. 问用户："市面上有没有一个东西，跟你想要的比较接近？不用一模一样，接近就行。"
2. 如果用户给了类比物：
   - 如果有文档/网站可读 → 读文档，提取能力清单
   - 如果只有名字 → 用 Web 搜索找文档，提取能力清单
   - 如果用户说"没有" → 跳过此锚定，依赖反面 + 场景
3. 能力清单格式同上

### 用户标记规则

- 👍 要：保留这个能力
- ❌ 不要：明确放弃（必须标注"暂时不做"或"永久放弃"）
- 🤔 不确定：逐条追问，每条只问一个问题

### 🤔 项追问模板

```
关于「{能力名称}」：
{能力描述}

你想在新项目里保留这个吗？
- 如果不保留：是暂时不做（以后可能加），还是确定不要？
```

**不批量追问**。一次只问一条。用户回答后立即记录，再问下一条。

### 类比锚定的常见失败

| 失败模式 | 症状 | 防御 |
|----------|------|------|
| 只读总览不读细节 | 能力清单只有 3-4 条（实际有 24 条） | 必须读 commands/ 目录下每个文件 |
| AI 自己缩减清单 | "我觉得这几条不重要就省略了" | 禁止。全部列出，让用户标记 |
| 编造不存在的能力 | 清单里出现源系统没有的东西 | 每条必须标注来源文件 |

## 锚定 2: 反面锚定

### 适用场景

所有项目都适用。这是最容易触发用户真实需求的锚定。

### 操作

1. 问用户："你现在怎么解决这个问题的？"
2. 听完后追问："这个过程里，什么东西让你最烦？"
3. 从每个"烦"反推"要"：

```
用户："我最烦 ChatGPT 让我打字练习面试，它不让我开口说"
反推能力：
  - 语音输入（因为烦打字）
  - 语音对话（因为烦"不让我开口"）

用户："最烦的是每次练完关掉，下次打开从头来，什么记录都不留"
反推能力：
  - 跨会话状态（因为烦"从头来"）
  - 练习历史（因为烦"不留记录"）
```

4. 把反推的能力加入清单，标注 `[反面推导]`
5. 交用户确认："从你最烦的点，我推导出这些能力需求，对吗？"

### 反面推导规则

- 一个"烦"可能推导出多个能力
- 推导出的能力必须回显给用户确认，不自行加入最终清单
- 用户说"对"→ 加入清单（👍）
- 用户说"不完全对"→ 追问哪里不对
- 用户说"不是这个意思"→ 重新理解，重新推导

### 反面锚定的常见失败

| 失败模式 | 症状 | 防御 |
|----------|------|------|
| 推导过度 | 用户说"烦打字"，AI 推导出"需要 ASR 引擎 + TTS + VAD" | 只推导产品能力，不推导技术实现 |
| 忽略用户的原话 | 用户说"烦打字"，AI 记成"需要语音" | 必须记录用户原话，推导基于原话 |
| 不回显就加入 | 推导完直接写进清单 | 必须回显确认 |

## 锚定 3: 场景锚定

### 适用场景

所有项目都适用。这是最能发现隐含需求的锚定。

### 操作

1. 问用户："假设这个东西已经做好了，你第一次打开它，你会先干什么？"
2. **闭嘴，让用户说。不要追问，不要引导。** 用户说完后再分析。
3. 从用户描述的场景中提取能力需求：

```
用户："我打开页面，它问我今天想练什么。我说'模拟面试'。
它出了一道题，我开口回答了 3 分钟。它给了反馈，我能听到它说哪里好哪里不好。
我看了一个分数，点了下一题。做完 5 道题，它给我一个总结。"

提取能力：
- 命令系统（"它问我今天想练什么" → 多个入口）
- 语音输入（"我说'模拟面试'"）
- 语音输出（"我能听到它说"）
- 评分体系（"我看了一个分数"）
- 多题连续（"点了下一题"）
- 模拟面试（"做完 5 道题"）
- 复盘总结（"它给我一个总结"）
```

4. 把提取的能力加入清单，标注 `[场景提取]`
5. 回显给用户："从你描述的场景里，我提取了这些能力，有没有漏的或多加的？"

### 场景提取规则

- 从动作提取能力，不从情绪提取（"我很开心"不提取能力）
- 从具体名词提取，不从形容词提取（"好看的 UI"不提取，"一个分数"提取"评分体系"）
- 如果用户描述的场景跟前面锚定的结果矛盾，以场景为准（场景是用户最真实的表达）
- 用户说漏的场景动作，不要追问——用户没提到的能力可能不是必要的

### 场景锚定的常见失败

| 失败模式 | 症状 | 防御 |
|----------|------|------|
| 引导用户 | "你是不是想要一个分数显示？" | 只问"你会先干什么"，不问"你是不是想要 X" |
| 提取过度 | 用户说"我看了一个分数"，提取出"多维度评分 + 校准引擎 + 趋势追踪" | 只提取用户明确提到的，不推断细节 |
| 忽略矛盾 | 场景里有"评分"，前面标记"❌ 不要评分" | 标注矛盾，问用户哪个为准 |

## 三重锚定合并规则

1. 合并三种锚定产出的能力清单
2. 去重（同一能力从不同锚定来源出现，合并为一条，标注所有来源）
3. 如果同一能力在不同锚定中标记冲突（👍 vs ❌），标注冲突，交用户裁决
4. 生成统一清单，格式：

```
| # | 能力 | 来源 | 标记 | 说明 |
|---|------|------|------|------|
| 1 | 语音输入 | 类比+反面+场景 | 👍 | 三重确认 |
| 2 | 5维评分 | 类比+场景 | 🤔 | 类比说要、场景提到分数，但用户未确认是5维 |
| 3 | 简历优化 | 类比 | ❌ 暂时不做 | 用户明确说先不做 |
```

5. 交用户做最终标记确认

## 用户偷懒时的降级规则

> 与 impact skill 的"澄清被拒绝"规则对齐。

### 用户说"你定 / 随便 / 都行"

1. 不得静默选择
2. 选**最保守的理解**（保留能力而非删除）
3. 一句话回显："按默认理解保留这个能力，依据是 {来源}；如果你不要，现在说一声"
4. 记入 INTENT.md 的确认记录："【用户委托默认: 日期 选项X 依据{来源}】"

### 用户说"不知道 / 不确定"

1. 不得强行选一个
2. 列出每个选项的代价/风险（表格形式）
3. 标出"最安全的默认"（保留 > 推迟 > 放弃）
4. 用户仍无法决定时："建议先保留，后续在 brainstorm 阶段再决定"
5. 标记为 🤔 推迟，不标记为 ❌

### 用户说"图省事 / 你看着办"

1. 同"你定"处理
2. 额外提醒："现在省的事，后面可能要返工。建议至少把 👍/❌ 标完再走。"
