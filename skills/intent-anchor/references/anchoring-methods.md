# 锚定方法

本文件说明如何使用类比、反面和场景三种方法收集证据。它们是按需选择的工具，不是必须凑够数量的步骤。

## 共同要求

每种方法只有同时留下以下内容才算完成：

1. 原始证据：文件路径、原文摘录或用户原话。
2. 从证据提取的候选能力。
3. 用户对候选能力的决定，或明确标为`待确认`。

用户描述的问题、痛点和使用画面只能证明“这里可能有需求”，不能直接证明某个解决方案已经获得确认。

## 按场景选择方法

| 场景 | 必须使用 | 视情况使用 |
|---|---|---|
| 系统转换 | 类比锚定（源系统完整盘点）+ 反面或场景 | 另一种补充方法 |
| 全新项目（0→1） | 反面和场景提供的信息；已有等价信息时不重复提问 | 类比锚定，可完全跳过 |
| 已有系统新增能力 | 读取相关模块 + 场景锚定 | 反面、外部类比物 |

选择方法的依据是信息缺口。不能因为“已经问过两个问题”就宣布锚定完成。

## 类比锚定

### 系统转换

源系统是迁移基线，需要完整盘点：

1. 确认源系统路径。
2. 先读总览，再读取承载具体能力的文件、命令目录、配置和边界说明。
3. 为每项能力分配稳定 ID，并记录来源位置和原文摘录。
4. 不替用户删减“不重要”的能力；全部列出后由用户决定。

候选表：

```text
| ID | 候选能力 | 证据位置 | 原文摘录 |
|---|---|---|---|
| C01 | 初始化档案 | commands/kickoff.md:L23 | “初始化用户档案……” |
```

如果文件很多，先给出文件清单和预计盘点范围，再分批处理。不得只读 README 就声称完整。

### 全新项目

类比物只是帮助表达偏好的参考，不是必须复制的需求基线：

1. 问：“有没有接近的产品或做法？你想参考它的哪一部分？”
2. 用户给了文档或链接时，只提取与当前问题有关的部分。
3. 用户只给名称时，可以使用 WebSearch / WebFetch 查找公开资料；工具不可用时，请用户提供链接或文字，并把未核实内容标成“待核实”。
4. 用户说没有类比物时，正常跳过，继续使用反面和场景方法。

禁止把类比物的完整功能列表默认变成用户需求。

## 反面锚定

反面锚定用于理解用户为什么不满意现状：

1. 问：“你现在怎么解决这个问题？”
2. 问：“这个过程中最不满意的是什么？”
3. 记录用户原话。
4. 把痛点转换成一个或多个候选能力，并明确标为模型推导。
5. 向用户展示候选能力，让用户决定。

示例：

```text
用户原话：“每次重新打开都要从头开始。”

模型推导：
- C03 跨会话保存进度
- C04 历史记录

这两项都是候选解决方案，不等于用户已经确认。请用户分别选择保留、推迟或放弃。
```

只推导产品能力，不提前指定 ASR、数据库、状态机等技术实现。

## 场景锚定

场景锚定用于理解用户实际会做什么：

1. 问：“假设它已经做好了，你第一次使用时会怎么做？”
2. 让用户先完整描述，不在叙述过程中塞入候选答案。
3. 从动作和具体对象中提取候选能力。
4. 展示提取结果，请用户检查遗漏和误解。

不要从形容词直接发明功能。例如“页面好看”不能自动推导出设计系统；“我看到一个分数”只能推导出“显示结果或评分”这一候选，不足以推导多维评分、趋势分析等细节。

如果场景与此前决定冲突：

1. 保留两边的原始证据。
2. 把相关能力标记为`待确认`。
3. 清楚说明冲突，请用户决定哪个表达代表当前意图。

场景不会自动覆盖用户先前的明确决定。

## 合并与决策

1. 合并三种方法产生的候选能力。
2. 同一能力出现多次时合并为一个 ID，并保留全部证据来源。
3. 不同来源互相冲突时保持`待确认`，不自行选择。
4. 按 5-10 项一组展示给用户；冲突、高影响或解释不清的能力再逐条追问。
5. 最终能力表只使用：`保留 / 推迟 / 放弃 / 待确认`。

决策示例：

```text
| ID | 能力 | 证据来源 | 决策 | 决策来源 |
|---|---|---|---|---|
| C01 | 语音输入 | 用户场景原话 | 保留 | 用户明确确认 |
| C02 | 趋势分析 | 类比物 | 待确认 | 模型建议（未确认） |
| C03 | 简历优化 | 用户明确说当前不做 | 推迟 | 用户明确确认 |
```

最终交接前不得存在`待确认`。

## 用户授权模型决定

用户说“你定 / 随便 / 都行 / 你看着办”时：

1. 明确说明你将代为选择，而不是把选择记成用户观点。
2. 基于已有证据选择`保留`或`推迟`，并给出简短理由。
3. 决策来源写`用户授权模型决定`。
4. 不得选择`放弃`，也不得把该能力列为不可妥协项。
5. 用户之后明确选择时，更新为`用户明确确认`并记录原话。

用户说“不知道”但没有授权模型决定时，保持`待确认`。不要把“最安全默认”伪装成最终决定。

## 不可妥协项

能力清单确认后，问：“这些保留项里，有没有任何一项是后续绝对不能丢的？没有也可以。”

- 数量可以是零或任意合理数量，不设最低值。
- 每项必须引用一个`保留`能力 ID。
- 必须由用户明确确认；用户授权模型决定不够。
- 用户拒绝或推迟对应能力时，立即从不可妥协项移除，并重新确认最终列表。
