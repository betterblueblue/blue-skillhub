# 漂移模式清单

> 本文件列出意图偏离的常见模式。Phase 3 生成 INTENT.md 后必须逐条检查，后续阶段切换时也需复查。

## 通用漂移模式（所有项目）

### D1: 重新发明漂移

**症状**：AI 发明了新概念代替源系统/类比物的已有概念。

**检测**：当前产出的核心概念（评分体系、交互模式、反馈格式）是否能在源系统/类比物里找到对应？如果找不到，是"新发明"还是"用户明确要求的新增"？

**案例**：
- PRD 发明"四档状态标签"代替源系统的"多维度评分" → 命中
- PRD 发明"说一版→拆解→重说"代替 skill 的"practice + mock" → 命中
- PRD 新增"语音输入"功能（skill 里没有）→ 不命中（用户明确要求的新增）

**处理**：命中时在 INTENT.md 标注，向用户说明"这个概念在源系统里不存在，是 AI 新发明的，确认要保留吗？"

---

### D2: MVP 砍到无关漂移

**症状**：MVP 砍掉的能力跟保留的能力没有共享基础，变成了两个不同的产品。

**检测**：MVP 保留的能力集合是否是完整产品的一个真子集？还是完全不同的产品？

**案例**：
- 完整产品：多模块全生命周期系统（例如 20+ 个命令）
- MVP 保留："说一版→拆解→重说"（1 个循环）
- MVP 砍掉：其余 23 个命令 + 评分 + 故事库 + 模拟面试
- 保留的跟砍掉的共享的只有"面试"这个主题 → 命中

**处理**：命中时在 INTENT.md 标注覆盖率，向用户说明"MVP 和完整产品共享的基础只有 X，建议保留至少 Y 个核心能力作为共享基础"。

---

### D3: 降级代替漂移

**症状**：用更弱的版本"代替"源系统能力，而不是"简化"或"推迟"。

**检测**：被替换的能力，其替换物是否在源系统里存在？替换物是否比原版弱？如果是"更弱的新东西"而非"原版的子集"，就是漂移。

**案例**：
- 多维度评分（1-5 分 × 多个维度）→ 四档标签（4 个文字状态）
  - 标签不是评分的简化版，是完全不同的东西 → 命中
- 多维度评分 → 只做部分维度的评分
  - 是原版的子集 → 不命中（简化，不是代替）
- 8 级练习阶梯 → 只做第 1 级
  - 是原版的子集 → 不命中（推迟，不是代替）

**处理**：命中时向用户说明"X 被 Y 代替了，Y 不是 X 的简化版而是新发明。建议改为'只做 X 的子集'或'推迟 X'"。

---

### D4: 单循环锁定漂移

**症状**：架构只支持一条固定流程，无法扩展到多命令/多模式。

**检测**：后端是否有命令调度器？状态机是否只描述一条路径？

**案例**：
- ws.ts 1700 行全是单循环，没有 command router → 命中
- 后端有 command router，每个命令独立 handler → 不命中

**处理**：命中时在 INTENT.md 的不可妥协项里标注"必须支持多命令"。

---

### D5: 沉默输出漂移

**症状**：AI 只通过屏幕文字反馈，不通过语音输出。用户说"语音交互"，AI 理解成"语音输入+文字输出"。

**检测**：架构是否允许 AI 语音输出在全程传输？还是只在特定阶段？

**案例**：
- 架构限制"只有开场阶段允许 assistant audio" → 命中
- 架构允许全程 response.create + audio.delta → 不命中

**处理**：命中时在 INTENT.md 的不可妥协项里标注"AI 必须能开口说话"。

---

### D6: "不追求"清单吃掉核心能力

**症状**：PRD 的"不追求/不做"清单里包含了源系统的 P0 能力。

**检测**："不追求"清单里的每一条，是否对应 INTENT.md 里标记为 👍 的能力？

**案例**：
- PRD 写"不追求多轮压力面" → 对应 skill 的 mock（P0）→ 命中
- PRD 写"不追求完整简历系统" → 对应 skill 的 resume（P2）→ 不命中（确实可推迟）

**处理**：命中时向用户说明"'不追求 X' 对应的是你标记为 👍 的能力 Y，确定要放弃吗？"

---

### D7: 上下文蒸发漂移

**症状**：后续阶段的产出没有引用 INTENT.md，说明意图已经从上下文里蒸发了。

**检测**：当前产出是否包含对 INTENT.md 的引用？是否包含覆盖率检查章节？

**案例**：
- PRD 全文没有出现 "INTENT.md" → 命中
- PRD 有"意图覆盖检查"章节但覆盖率为 0% → 命中
- PRD 有覆盖率表且 ≥ 60% → 不命中

**处理**：命中时停下来，要求重跑该阶段并在 prompt 里加入"先读 INTENT.md"。

---

## 转换项目专属漂移

### D8: 源系统未被引用

**症状**：PRD / 架构 / issue 全文没有出现源系统的名字或文件路径。

**检测**：全文搜索源系统名称和路径。

**处理**：命中时停下来。源系统文件必须作为 PRD 的输入被引用。

---

## 0-1 项目专属漂移

### D9: 类比物蒸发

**症状**：用户在锚定阶段给了类比物，但 PRD 阶段完全没有参考类比物的能力。

**检测**：PRD 的能力清单是否跟类比物能力清单有交集？

**处理**：命中时向用户确认"你在锚定时提到了 X，但 PRD 里没有 X 的任何能力，是有意偏离还是遗忘了？"

---

## 漂移检测时机

| 时机 | 检查哪些模式 | 怎么检查 |
|------|-------------|---------|
| INTENT.md 生成后 | D1, D2, D3 | 对照源系统/类比物检查 |
| brainstorm 完成后 | D1, D7 | 产出是否引用 INTENT.md？是否新发明概念？ |
| PRD 完成后 | D1, D2, D3, D6, D7, D8/D9 | 全量检查 |
| 架构完成后 | D4, D5, D7 | 架构是否支持多命令？语音输出？ |
| issue 拆分后 | D7 | 每个 issue 是否可追溯到 INTENT.md？ |
| 开发完成后 | D4, D5, D7 | 最终回归 |
