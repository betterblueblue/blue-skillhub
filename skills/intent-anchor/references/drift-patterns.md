# 通用漂移模式

本文件只保留适用于不同产品和技术栈的通用模式。语音、命令调度器等特定能力不属于通用规则；只有它们已经出现在当前 `INTENT.md` 中时，才按普通能力检查。

检查结果只使用三个值：`未命中 / 命中 / 不适用`。`命中`表示发现风险，不得用对勾或叉号代替，避免含义相反。

## D1：未确认新增

**症状**：能力清单或后续产出出现了没有证据、没有决策来源的新能力。

**检查**：每个新增概念能否对应能力 ID、用户原话、源文件或明确标注的模型建议？

**处理**：标为`待确认`并交用户决定。不得因为模型认为“应该需要”就直接保留。

## D2：目标替换

**症状**：原本确认的目标被换成另一个看似接近、实际解决不同问题的目标。

**检查**：新表述是否仍然满足用户确认的能力和场景，还是只保留了相似主题？

**处理**：并列展示原目标和替代目标，请用户裁决。不能由场景、类比物或模型偏好自动覆盖原决定。

## D3：能力降级

**症状**：保留能力被更弱、但并非其明确子集的方案代替。

**检查**：当前方案是原能力的一部分，还是另一个能力？缩减是否由用户确认并记录为推迟？

**处理**：如果是明确子集，列出保留和推迟的部分；如果是不同能力，回到 D2 处理。

## D4：保留项遗漏

**症状**：后续产出没有处理某个标为`保留`的能力。

**检查**：逐项记录为`已体现 / 本阶段不适用（说明原因）/ 遗漏`。推迟和放弃项不计入检查对象。

**处理**：存在遗漏时停止交接，补充产出或由用户重新决定该能力。

## D5：推迟或放弃项被重新加入

**症状**：后续方案悄悄加入了用户决定推迟或放弃的能力。

**检查**：新增内容是否对应 INTENT.md 中的推迟或放弃项？

**处理**：说明重新加入的原因并等待用户确认；未确认前不得进入目标范围。

## D6：决策来源失真

**症状**：模型推导、用户授权模型决定或模糊反馈被改写成“用户明确确认”。

**检查**：决策来源和用户原话是否一致？用户是否真的看过并决定了具体能力？

**处理**：恢复真实来源。没有明确确认的能力改为`用户授权模型决定`或`待确认`；不可妥协项必须移除，直到用户重新确认。

## D7：交接信息丢失

**症状**：后续会话或产出没有读取当前 INTENT.md，也没有逐项核对保留能力。

**检查**：是否引用了正确的活动文件路径？是否包含“意图核对”？如果目录里有多份 INTENT.md，是否明确使用了哪一份？

**处理**：停下来读取正确文件，再重新核对。不得自动选择目录中最新的文件。

## 检查时机

| 时机 | 重点检查 |
|---|---|
| INTENT.md 草稿完成后 | D1、D2、D3、D6 |
| brainstorm 或 PRD 完成后 | D1-D7 |
| 架构或任务拆分完成后 | D3-D7 |
| 开发完成后 | D3-D7 |

检查模式只负责暴露风险。任何处理如果改变了能力、决策或不可妥协项，都必须回到用户确认，不能由检查过程自行改写意图。
