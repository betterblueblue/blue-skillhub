# 语义复核规则

本文件定义写入 `INTENT.md` 前必须留下的 S1-S5 复核记录。复核的作用是把证据、推导和决定摊开供用户检查，不是证明内容一定正确。

`intent_validate.py` 只能检查这些记录是否具备规定结构，不能判断引用是否真实、推导是否合理，也不能代替用户确认。

## S1：证据覆盖

目的：确认每项候选能力都有可追溯证据，并按场景检查遗漏。

### 系统转换

1. 重新读取源系统能力文件，不依赖记忆。
2. 列出读取范围、原文摘录和位置。
3. 将源能力与候选清单按 ID 对照。
4. 漏掉的能力补回并标为`待确认`；找不到来源的新增能力标明其用户原话或删除。

系统转换只有完成源能力清单对照后才能通过 S1。

### 全新项目

类比物不是完整性基线。检查对象是用户的问题、痛点、场景原话以及用户明确选择参考的类比部分。没有类比物属于正常情况，不得因此编造或阻塞。

### 已有系统新增能力

只检查相关模块和目标行为，不要求把整个系统的既有能力都放进 INTENT.md。

记录格式：

```text
### S1 证据覆盖

| 能力 ID | 证据类型 | 证据位置或用户原话 | 复核结果 |
|---|---|---|---|
| C01 | 用户场景 | “我第一次打开会先……” | 有依据 |
| C02 | 源系统文件 | commands/export.md:L18 | 有依据 |

场景专项结论：{转换场景是否完成源能力对照；0→1 是否存在无来源候选；已有系统检查了哪些相关模块}
```

## S2：决策来源

目的：防止模型把自己的推导或默认选择写成用户确认。

逐项记录：

```text
### S2 决策来源

| 能力 ID | 决策 | 决策来源 | 用户原话或授权 | 复核结果 |
|---|---|---|---|---|
| C01 | 保留 | 用户明确确认 | “这个要保留” | 通过 |
| C02 | 推迟 | 用户授权模型决定 | “你根据现有信息决定” | 通过，不算用户本人确认 |
| C03 | 待确认 | 模型建议（未确认） | 无 | 阻塞 |
```

规则：

- 痛点和场景原话只能作为能力证据，不能直接作为解决方案的确认。
- `用户授权模型决定`必须保留授权原话，不得写成`用户明确确认`。
- `放弃`必须有用户针对该能力的明确原话。
- 任何`待确认`都必须解决后才能写入最终文件并交接。

## S3：不可妥协项

目的：确认不可妥协项确实来自用户，而不是模型替用户排序。

有不可妥协项时：

```text
### S3 不可妥协项

| 能力 ID | 决策为保留 | 用户明确确认 | 用户原话 | 复核结果 |
|---|---|---|---|---|
| C01 | 是 | 是 | “这个绝对不能丢” | 通过 |
```

没有不可妥协项时，写明：

```text
### S3 不可妥协项

无不可妥协项。用户明确确认：“没有必须绝对保留的能力”。
```

数量不设下限。只要缺少“保留能力 ID”或“用户明确确认”，就不能列为不可妥协项。

## S4：锚定充分性与冲突

目的：确认所使用的方法留下了足够证据，并且没有把冲突静默覆盖。

```text
### S4 锚定充分性与冲突

| 方法 | 原始证据 | 提取的能力 ID | 用户是否看过候选 | 结论 |
|---|---|---|---|---|
| 反面 | “现在最烦……” | C01、C02 | 是 | 信息充分 |
| 场景 | “第一次打开……” | C01、C03 | 是 | C03 曾冲突，已由用户裁决 |

未解决冲突：无
```

不能只写“完成了类比、反面、场景”。一种方法没有原始证据、候选能力或用户反馈时，就不算完成。

## S5：漂移复核

目的：确认第 7 节中的通用漂移模式已经逐项处理。

```text
### S5 漂移复核

| 模式 | 第 7 节状态 | 证据或处理结果 |
|---|---|---|
| D1 未确认新增 | 未命中 | 所有能力均有证据和决策来源 |
| D2 目标替换 | 命中 | C04 已恢复为待确认并交用户裁决 |
```

任何`命中`都必须写出处理结果。处理结果如果改变了能力或决策，需要重新获得用户确认。

## 执行要求

1. S1-S5 分别保留记录，不合并成一句“全部通过”。
2. 每项至少要有证据表或规定的“无不可妥协项”说明。
3. 复核记录写入 INTENT.md 第 10 节。
4. 复核发现未决问题时回到 Phase 2，不得直接写入最终文件。
5. 最终判断仍由用户作出；模型不得把自查结果描述为客观证明。
