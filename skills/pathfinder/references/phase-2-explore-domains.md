# Phase 2: 并行专探

> 本文件含 Phase 2 完整规则。5 路 explore 子 agent 分域扫描,替代原串行广度扫描。
> 核心变化:把单 agent 串行 Glob/Grep/Read 改为 5 路 explore 并行,每路有独立上下文预算。

## 原则:按域分路并行，各路预算独立

- 每路 agent 负责一个**正交域**,互不重叠。
- 每路有自己的上下文预算,不和其他路抢。
- 主 agent 合并结果,统一标可信度。

## 5 路并行模型

| Agent | 职责 | 核心产出 |
|-------|------|----------|
| **A1 架构** | 模块边界、分层、目录结构、模块间依赖方向 | 模块列表 + 依赖方向 |
| **A2 数据** | schema / model / ORM 映射 / migration 文件 | 实体列表 + 关系 |
| **A3 入口/API** | HTTP 路由、CLI 入口、MQ consumer、定时任务 | 入口列表 |
| **A4 权限** | authn 方式、authz 机制、中间件/守卫位置 | 权限模型摘要 |
| **A5 运维** | 构建命令、运行命令、测试命令、部署配置、CI | 命令列表 |

## 为什么按域分而不是按模块分

5 个域(架构/数据/API/权限/运维)互不重叠——任何项目都有这五个维度。按模块分路(如"user 模块一个 agent、order 模块一个 agent")会导致每个 agent 需要跨域扫描(既要看路由又要看数据又要看权限),子 agent 的输入上下文重复、产出难以合并。按域分路保证每路有独立上下文预算、产出不重叠、主 agent 合并无冲突。

## 每个子 agent 的设计

每个 explore agent 遵守:

1. **只读**:只用 Read/Grep/Glob/Bash(git),不写任何文件
2. **输入**:主 agent 传入目标项目路径 + 关注重点 + `facts/` 脚本输出摘要(文件数、扩展名 top 5、栈类型、top_level_dirs、hotspots、recent_commit_modules)
3. **输出格式**:在对话中输出结构化报告,包含:
   - 发现的模块/实体/路由/权限等(含文件路径证据)
   - 推断项(标【推断】,注明依据)
   - 未确认项(声明「本域未覆盖」的部分)
   - `hypotheses_challenged`:至少尝试找 1 条「目录名假说 vs facts 矛盾」。找不到则写「未发现矛盾」——不强制编造
4. **可信度规则**:所有实线依赖/关系必须有证据,无证据标推断

## 子 agent 与脚本的关系

- `pf_scan.py` 和 `pf_git.py` 的产出在 Phase 1.5 已运行完毕
- 主 agent 将脚本产出的摘要(文件数、扩展名 top 5、top_level_dirs、hotspots、栈类型)作为输入传给每个子 agent
- 子 agent 不再重复 Glob 文件计数——直接读脚本产出
- 子 agent 的核心工作是**解读和深挖**,不是搬运事实

## Phase 2.5: 假说反证合并

取消独立的 Phase 2.5。假说反证并入每个子 agent 的输出(`hypotheses_challenged` 字段)。主 agent 在 Phase 3 汇总后写入地图。

理由:独立的 Phase 2.5 需要再多一轮 agent 调度;并入子 agent 不增加轮次,且反证所需的上下文在专探时已经拿到了。

## 降级策略

降级触发条件(按优先级检查):

1. **Task 工具不在可用工具列表** → 完全降级为串行(5 路 → 1 路,等同于 v1 Phase 2)
2. **Task 可用但启动子 agent 返回空/错误**(3 次重试后仍失败) → 该路结果丢弃,其余路正常合并
3. **任意子 agent 超过 5 分钟未返回** → 该路超时丢弃,其余路正常合并
4. **3 路及以上超时/失败** → 完全降级为串行

降级后:
- 在概览头部或【13】未覆盖项中注明:`并行模式: N 路成功 / K 路降级 / 串行回退`
- 丢弃的子 agent 域标记为「未深入」,由 Phase 3 主 agent 串行补扫

## 栈探测

栈探测规则不变,仍然读清单文件识别技术栈。**完整映射表见 `stack-detection.md`。** 子 agent A5(运维)负责从清单文件中提取构建/运行/测试命令。

## 相关性分级(沿用 impact 约定)

广度扫描时给模块/文件标相关性,辅助 Phase 3 决定深挖谁:

| 分值 | 含义 |
|------|------|
| 3 | 核心模块,Phase 3 必深挖(尤其落在聚焦区的) |
| 2 | 重要支撑模块,预算允许则深挖 |
| 1 | 背景 / 工具 / 配置,广度提及即可 |
| 0 | 噪音(依赖、构建产物、生成物),排除 |

## Phase 2 输出

并行专探结束后,主 agent 在对话里输出合并骨架草案(不写文件):

```text
技术栈: [已核实/推断]
模块地图: [N 个模块,各一行职责]
关键入口: [列表]
聚焦区候选深挖: [按关注重点 + 相关性 3 选出的模块]
假说挑战: [汇总各子 agent 的 hypotheses_challenged]
并行模式: [5 路成功 / N 路降级 / 串行回退]
```

确认骨架方向后进入 Phase 3 深挖。
