# Phase 1: 体量测量 + 预算分档

> 本文件含 Phase 1 完整规则。SKILL.md 正文只保留概要。
> 目的:陌生项目大小未知,先统计项目大小再定"读多深",避免一上来就扫全仓把上下文撑爆。

## Step 1.1: 体量测量(只读、便宜)

用快命令量三个数,不读文件内容:

**前置:Git 仓库归属检查**

先确认当前目录是独立 Git 仓库,不是另一个仓库的子目录:

```bash
# bash — 用 .git 目录检测,避免 msys2 路径格式不兼容
test -d "$(pwd)/.git" && echo "独立仓库" || echo "子目录或非Git"
```
```powershell
# PowerShell
if (Test-Path (Join-Path (Get-Location) '.git')) { "独立仓库" } else { "子目录或非Git" }
```

> **注意**:原 `git rev-parse --show-toplevel` vs `$(pwd)` 字符串比较在 Git Bash (msys2) 上恒失败,因为前者返回 Windows 格式路径(`E:/...`),后者返回 Unix 格式(`/e/...`)。改用 `.git` 目录存在性检测后再用 `git -C "$(pwd)" rev-parse --show-toplevel` 获取根路径(概览头部用),不依赖路径字符串比较。

- 返回"独立仓库" → 正常记录 HEAD(`git -C "$(pwd)" rev-parse --short HEAD`)。
- 返回"子目录或非Git" → 概览头部 `基于 commit` 写"非独立 Git 仓库(HEAD 来自父仓库)，以扫描时间为准"，不记录父仓库的 HEAD。
- `.git` 目录不存在(非 Git) → 写"非 Git，以扫描时间为准"。

| 指标 | bash 示例 | PowerShell 示例 |
|------|-----------|------------------|
| 跟踪文件数 | `git ls-files \| wc -l` | `(git ls-files \| Measure-Object).Count` |
| 非 Git 文件数 | `find . -type f -not -path './.git/*' \| wc -l` | `(Get-ChildItem -Recurse -File).Count` |
| 顶层目录数 | `ls -d */ \| wc -l` | `(Get-ChildItem -Directory).Count` |
| 主要模块数 | 数 src 下子模块 / monorepo 下 packages、apps 子项目数 | 同左 |

**忽略噪音**:`node_modules`、`vendor`、`dist`、`build`、`.git`、`target`、`__pycache__`、锁文件不计入"代码体量"。优先用 `git ls-files`(天然排除 gitignore 内容)。

### 大规模仓库降级（保留）处理

**`git ls-files` 超时**:仓库超过 10 万文件时 `git ls-files` 可能耗时 > 5 秒。若命令超过 5 秒未返回,终止并降级为 `find . -type f -not -path './.git/*' | wc -l`,设 10 秒超时。降级时标注「文件数来自 find(含 gitignore 内容),偏高」。

**sparse-checkout 感知**:若 `.git/info/sparse-checkout` 存在,说明仓库使用稀疏检出。此时 `git ls-files` **偏少**(只统计 sparse 范围内的文件),而 `find` **偏多**(统计物理存在的所有文件,可能包含 sparse 外的内容)。这种情况下在概览头部标注「sparse-checkout 仓库,文件计数仅覆盖检出子集」。

## Step 1.2: 分档

按跟踪文件数为主、模块数为辅定级:

| 档位 | 跟踪文件数(参考) | 含义 |
|------|------------------|------|
| **小仓** | < 200 | 单服务 / 单包,可较全面深挖 |
| **中仓** | 200 – 2000 | 多模块,深挖按关注重点取舍 |
| **大仓** | 2000 – 8000 | 大型单体 / 中型 monorepo,深挖严格限聚焦区 |
| **超大仓** | > 8000 | monorepo / 巨型系统,只做 L1 广度 + 聚焦区,其余全进未覆盖项 |

> 文件数只是参考,模块数 / 语言数 / monorepo 结构可上调一档。定级理由回显给用户。

## Step 1.3: 上下文预算(按档位)

| 档位 | 广度扫描(Phase 2) | 深挖文件(Phase 3) | 单片段行数 |
|------|---------------------|----------------------|-----------|
| 小仓 | 目录树 + 全部入口 | ≤ 20 个文件 | ≤ 80 |
| 中仓 | 目录树 + 模块级入口 | ≤ 25 个文件(聚焦区优先) | ≤ 80 |
| 大仓 | 目录树到 2 层 + 聚焦区入口 | ≤ 25 个文件(限聚焦区) | ≤ 80 |
| 超大仓 | 仅顶层 + 聚焦子项目 | ≤ 20 个文件(仅聚焦区) | ≤ 80 |

- 深挖文件预算是**总预算**,不是每节预算。优先花在关注重点指向的模块。
- 超预算时**不继续扩大读取**:把剩余模块写入【13】未覆盖项清单,等用户扩展。

## Step 1.4: 回显档位

Phase 1 结束必须回显:

```text
体量: 跟踪文件 ~N 个 / 顶层目录 M 个 / 主要模块 K 个
定级: [小/中/大/超大]仓(理由: ...)
预算: 深挖上限 ~X 个文件;聚焦区 = [用户关注重点 / 无→均匀]
```

## 超大仓特别处理

超大仓不硬扫。明确告知用户:

> "这是个超大仓(~N 文件)。本轮我只做顶层广度 + 你关心的 [聚焦区],其余模块会列进地图的『没挖深的部分』,你之后可以说『再挖 X』我再补。"

不得假装已通读全仓;覆盖度声明必须如实列出大量"未深入"。

## 维护注意

- 文件数阈值是**易变**参考值,可按实际经验调整,但分档→预算的映射关系不变。
- 改预算上限必须同步 `phase-2-explore-domains.md` / `phase-3-depth-fill.md` 的引用。
