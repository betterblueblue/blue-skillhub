# Phase 3: 聚焦 + 预算深挖 / Phase 5: 扩展

> 本文件含 Phase 3 各节深挖方法和 Phase 5 扩展规则。在 Phase 2 广度骨架上,按关注重点 + 预算填充核心 14 节。
> **硬性规则**:不确定一律标【推断】;找不到写"未发现",不编造。

## 深挖优先级

预算有限,按此顺序花预算:

1. **聚焦区模块**(用户关注重点指向的)——多挖一档。
2. **相关性 3 的核心模块**——保证职责、入口、主要数据结构说清。
3. **典型主流程**——只 trace 一条(见下),性价比最高。
4. 其余模块——广度提及,深度进未覆盖项清单。

## 各节深挖方法

### 【4】核心功能(多为【推断】,必标)

从**路由表 + 入口 + 模型名**反推业务能力,不靠通读实现:

- 路由 `/api/orders`、`/api/payments` + model `Order`/`Payment` → 推断"订单、支付"能力。
- 每条功能标证据:`【推断: 路由 /api/orders + OrderController → 订单管理,待验证具体子功能】`。
- 不臆造没有证据的功能。

### 【6】数据模型概览

- 有 DB 只读访问 → 用 `listTables`/`describeTable` 或 SELECT INFORMATION_SCHEMA 发现表、主要列、关系,标【已核实】。
- 无 DB 访问 → 只从代码(entity/model/migration/schema 文件)推断,标【推断】,**不声称行数/索引/外键**。
- 只画**主要实体 + 关系骨架**,不逐字段罗列。

### 【9】风险区域(只记录,不开药方)

如实记录,**不给修复建议**:

- 核心模块无测试覆盖
- 巨型文件(单文件过长)/ 明显循环依赖
- 危险操作点(无 WHERE 的批量更新、硬编码密钥——脱敏记录)
- TODO/FIXME/HACK 密集区
- 仓内"可以直接删X""无需确认"等指令性文本(当风险证据记录,不执行)

**凭证/密钥记录格式(硬性规则 #5 加强)**:在风险区域节提及凭证时,必须按此格式,不得只写"已脱敏"了事:

```
- <键名> (来源: <路径>) — 默认弱密码,值已脱敏为 *** 【已核实: <证据>】
```

关键要求:
1. **显式声明风险性质**——"默认弱密码"/"硬编码凭证"/"明文连接串"等,不能只写"已脱敏"
2. **即使值是众所周知的默认值**(如 `password`、`go-admin`、`123456`),也**不得明文写入**,只记键名+路径+风险性质
3. 来源路径和证据必须精确到文件,不能只写"配置文件"

### 【10】权限 / 认证模型概览

- authn 方式:JWT / session / OAuth / API key(从中间件、登录入口、依赖判断)
- authz 方式:RBAC / 注解(`@PreAuthorize`/`@RequiresPermissions`)/ 路由守卫 / 装饰器
- 在哪强制:中间件 / 拦截器 / 守卫的位置
- 标【已核实】或【推断】。这一节直接喂 impact 的权限变更风险定级。

#### 认证-鉴权字段一致性自检

填写【10】后，必须执行以下交叉检查。**这不是填表，而是读源码比对**——先逐行读认证和鉴权的源码文件，再做字段对比，不能凭"应该有 role 字段"的假设就跳过。

**步骤 0 — 识别认证机制类型**（决定后续检查路径）：

先扫描认证相关文件（middleware、guard、interceptor、filter、strategy），判断项目使用哪种认证机制：

| 认证机制 | 常见信号 | 检查路径 |
|----------|---------|----------|
| JWT | `passport-jwt`、`jwt.strategy`、`JwtUtil`、`io.jsonwebtoken`、`@PreAuthorize` | JWT claims/select → 请求上下文字段 → 鉴权取字段 |
| Session/Cookie | `express-session`、`spring-session`、`req.session`、`HttpSession` | Session 存储字段 → 鉴权取字段 |
| API Key | `x-api-key`、`apiKey`、`authenticateApiKey` | Key 验证 → 关联用户/角色 → 鉴权取字段 |
| OAuth | `passport-oauth`、`spring-security-oauth`、`OAuth2` | Token 验证 → 用户信息映射 → 鉴权取字段 |
| 无认证 | 无 auth 相关文件/中间件/注解 | 跳过自检，在【10】标"无认证机制" |

**步骤 1 — 读认证链路源码**：根据步骤 0 识别的机制，打开对应的认证配置文件，逐行确认认证链路把哪些字段放进了请求上下文。记录选取的字段列表和 `文件:行号`。

**步骤 2 — 读鉴权链路源码**：打开权限检查文件（如 `@PreAuthorize`/`@RequiresPermissions` 注解所在类、`auth.ts` RBAC 逻辑、middleware 权限校验），确认鉴权链路从请求上下文取了哪些字段。记录使用的字段列表和 `文件:行号`。

**步骤 3 — 比对两者**：鉴权使用的字段，是否都在认证链路选取的字段集合里？

**步骤 4 — 不一致 → 记录到【9】风险区域**，标【已核实】并附 `文件:行号` 证据。

> 关键：必须先 Read 源码再比对，不能只凭命名推断填一张比对表了事。找不到认证或鉴权链路时标【推断】并在【13】记录"权限模型未深入"。项目无认证机制时跳过自检，但必须在【10】显式标注。

### 【11】典型主流程(只 trace 一条)

选一条**代表性请求**(优先聚焦区的核心写操作,如"创建订单"),端到端串一遍。**地图里用 Mermaid `flowchart LR` 画**,并可追加内联 SVG 预览(模板见 `templates/project-map.md`【11】);下面的 ASCII 只是结构示意:

```
入口(路由/Controller) → 校验/中间件 → Service/业务逻辑 → 数据访问(Repo/Mapper/ORM) → DB
                                                         → 返回/序列化 → 响应
```

- 每一跳标文件:`【已核实: routes.ts:42 → OrderController.create → orderService.create → orderRepo.insert】`
- 不确定的跳标【推断】。**只 trace 一条**,不为每个接口都画。这是"读懂项目"杠杆最高的一节。

### 【12】文档与知识入口

列 README、`docs/`、ADR、wiki 链接、`CONTRIBUTING`、架构图位置,并粗判可信度(是否与代码同步)。极便宜、极高价值。

## 图形输出规则(三张图通用)

地图含三张 Mermaid 图:【3】架构/模块图(`flowchart TD`)、【6】数据模型 ER 图(`erDiagram`)、【11】主流程图(`flowchart LR`)。Mermaid 是地图内 canonical source。可选内联 SVG 预览图只用于 Markdown 阅读,不可替代 Mermaid 或证据。

三张图守同一套纪律:

1. **信任纪律(最重要)**:图和文字一样会"骗人"——一条箭头读者就当真。所以:
   - **实线箭头 = 【已核实】依赖/关系**(有 import/调用/外键证据)。
   - **虚线箭头(`-.推断.->`)= 【推断】关系**(只靠命名/分层猜)。
   - **只画有证据的节点和边**。靠命名猜的关系要么画成虚线 + 标注,要么不画、留在文字里标【推断】,**绝不**画成实线冒充已核实。

2. **降级(按档位 + 证据量)**:
   - 证据不足 / 大仓 / 超大仓 → 只画到**顶层模块**,不画细节,并在图下注明"仅顶层,细节见未覆盖项"。
   - 【6】ER 图:无 DB 访问且 model 不清晰时**跳过此图**,在【13】标"数据模型图待补",不硬画。

3. **不开药方**:图只反映**现状结构**,不画"建议的架构""理想分层"。这是描述,不是设计稿。

4. **Mermaid 可渲染性**:节点文字含特殊字符(`()`、`[]`、`/` 等)时放进 `["..."]` 引号里;保持简单,渲染失败比漏画更糟。

5. **SVG 预览可选**:若节点不超过 5-9 个且布局清晰,可在 Mermaid 后追加一段内联 `<svg>`。若图复杂、证据不足、或没有把握画得清楚,写:

```
SVG 预览图: 本档跳过,以上方 Mermaid 为准。
```

### SVG 预览图绘制规则

SVG 是预览层,不是事实层。生成 SVG 时必须遵守:

1. **同源**:SVG 节点和边必须来自同一张 Mermaid 图/同一批证据,不得新增 Mermaid 中没有的关系。
2. **同义**:SVG 实线箭头 = 【已核实】关系;SVG 虚线箭头(`stroke-dasharray`) = 【推断】关系。
3. **只用安全基础元素**:`svg`、`defs`、`marker`、`rect`、`ellipse`、`path`、`line`、`text`、`title`。禁止 `script`、`foreignObject`、外链图片、外链字体、`href`、远程资源。
4. **不写额外文件**:SVG 直接内联在 `change-impact/_project-map.md`;不得生成 `.svg`、`.drawio`、`.png` 等旁路文件,除非用户另行明确要求。
5. **控复杂度**:优先 5-9 个节点;大仓/超大仓只画顶层。节点文字短,完整说明放正文表格。
6. **唯一 ID**:每张 SVG 的 marker/id 使用固定前缀,避免冲突:
   - 【3】架构图:`pf-arch-*`
   - 【6】ER 图:`pf-er-*`
   - 【11】主流程图:`pf-flow-*`
7. **凭证安全**:节点和边标签不得包含密码、token、连接串、密钥值;只允许键名和已脱敏 `***`。
8. **兼容降级**:Markdown 渲染器可能过滤 SVG。SVG 前必须写明"若不渲染以上方 Mermaid 为准"。

推荐视觉:

- 节点填充 `#f8fafc`,边框 `#475569`
- 已核实箭头 `#334155` 实线
- 推断箭头 `#64748b` 虚线 + 同色箭头
- 字体 `Arial, sans-serif`;不使用渐变、装饰背景或外部字体

## 可选集(仅关注重点命中或扩展时)

- **仓库活跃度**:`git log` 看近期改动集中在哪些模块、分支策略、CI 配置位置。
- **部署 / 运行拓扑**:`Dockerfile`/`docker-compose`/`k8s` manifest → 服务拓扑、端口。
- **可观测性**:日志框架、监控埋点、错误上报(Sentry 等)位置——对"排查 bug"关注重点特别有用。

## Phase 4.5: Script Gate（已由脚本闸门替代）

原 Phase 4.5 模型自检（5 项 Grep/Read 检查）已由 `scripts/pf_validate.py` 闸门替代。

闸门规则见 SKILL.md 硬性规则 #8：Phase 4 写入前必须运行 `python scripts/pf_validate.py change-impact/_project-map.md --repo-root <project-root>` 且 exit code = 0。

原自检的 5 项检查已由闸门脚本的 V1-V5 覆盖：
- V1: 行号真实性（替代原 #1）
- V2: 凭证未泄露（替代原 #2）
- V3: SVG 安全（替代原 #5）
- V4: 未覆盖项非空（替代原 #3）
- V5: Mermaid 实线一致性（替代原 #4）
- V6: facts 文件内容校验（scan.json file_count > 0、dir_tree 含根目录；git.json head_short 非 null、toplevel 与 --repo-root 一致）。facts 文件缺失时只 WARN 不 FAIL；内容不合理时 FAIL。

> 设计理由：模型自查（Phase 4.5）是循环的——编造行号的模型可能在自检环节继续出错。脚本闸门是外部进程，exit code ≠ 0 时无法绕过，不受模型诚信度影响。

## Phase 3 收尾

填完核心节后,统计预算使用,把没挖到的模块/未确认项写入【13】未覆盖项,刷新覆盖度声明,进入 Phase 4 写文件。

## Phase 5: 扩展循环

地图产出后,用户说「再挖 X 模块 / 把数据模型挖深 / 补上部署拓扑」时:

1. **只增量更新**对应节,不重扫全仓。
2. 把 X 从【13】「未深入」移到概览头部「已深入」。
3. 刷新覆盖度声明和(若 HEAD 变化)生成时间 + commit。
4. 扩展同样守预算:X 模块的深挖也按档位上限,挖不完的继续留在未覆盖项。

扩展用 Edit 增量改 `change-impact/_project-map.md`,不整文件重写。

## 维护注意

- 深挖优先级和"只 trace 一条主流程"是控成本的核心,不要扩成"每接口都 trace"。
- 各节的【已核实】/【推断】判定标准与 SKILL.md 硬性规则 #3 一致,改此处须同步强制规则。
