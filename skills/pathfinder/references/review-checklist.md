# Pathfinder 地图质量 Review Checklist

> 适用文件：`change-impact/_project-map.md`
>
> 地图的两个核心作用：
> 1. 让人快速看懂「这项目是干啥的、核心功能是什么」
> 2. 给 agent/impact 提供上下文，在现有架构 + 现有风格轨道上可控迭代
>
> 评级：每项标 **必过** / **建议** / **加分**；人类层和 Agent 层各有总评（达标 / 基本达标 / 不达标）。
> 机械门禁由 `pf_validate.py` 自动执行，人工评审用本文件的 §一 + §二。

---

## 一、人类用户 KPI（「30 秒 + 5 分钟」测试）

测试方法：只读 Executive Summary + 扫一眼【1】【4】【8】【9】，不问 AI 补充。

### 1.1 三十秒测试（Executive Summary）

| # | 检查项 | 级别 | 通过标准 |
|---|--------|------|----------|
| H1 | 一句话说清项目 | 必过 | 能回答：做什么、用什么栈、给谁用；有【已核实】证据 |
| H2 | Quick Start 可执行 | 必过 | ≥3 步是真实命令/路径，不是「配置数据库」这类空话 |
| H3 | 5 个必读文件 | 必过 | 每文件有「为什么重要」；含入口 + 路由/架构 + 数据或 auth 至少各 1 个 |
| H4 | Top 3 风险 | 必过 | 项目特有风险 ≥1 条（不只「默认密码/弱 JWT」样板风险） |
| H5 | Top 3 Gotcha | 建议 | 环境/前后端分离/dev-prod 差异/隐藏依赖等「README 容易漏」的点 |
| H6 | 导航不误导 | 必过 | 导航行不触发 V4 误匹配（已由 V4 修复保障）；链接到正确章节 |

**人类 30 秒总评**：H1–H4 全过 → 达标；缺 H4 或 H2 占位 → 不达标。

### 1.2 五分钟测试（核心认知）

| # | 检查项 | 级别 | 通过标准 |
|---|--------|------|----------|
| H7 | 核心功能可枚举 | 必过 | 【4】≥3 条，每条有路由/模块/文件证据或明确【推断】 |
| H8 | 架构一层图能看懂 | 必过 | 【3】表 + Mermaid：入口 → 业务 → 数据 三层关系清楚 |
| H9 | 怎么跑起来 | 必过 | 【8】构建/运行/测试/迁移有关键命令；找不到写「未发现」 |
| H10 | 数据长什么样 | 必过 | 【6】主要实体 + 关系；标明 DB 是否只读推断 |
| H11 | 权限怎么工作（有 auth 的项目） | 建议 | 【10】authn/authz 各一句人话 + 在哪强制 |
| H12 | 诚实边界 | 必过 | 【13】≥1 条未深入；概览头部「未深入」与【13】一致 |

**人类 5 分钟总评**：H7–H10、H12 全过 → 达标；【4】全是【推断】且无证据 → 基本达标。

## 二、Agent / impact KPI（「少盲扫 + 风格可控」测试）

测试方法：模拟 impact Phase 2——只读地图 + `_style-rules.md`（若有），不扫全仓，能否定位变更入口并保持风格。

### 2.1 L1 导航（impact Phase 2 预读）

| # | 检查项 | 级别 | 通过标准 |
|---|--------|------|----------|
| A1 | 模块边界可切片 | 必过 | 【3】覆盖本次变更可能涉及的顶层模块；相关性 3 的有职责描述 |
| A2 | 入口可定位 | 必过 | 【5】HTTP/CLI/定时任务入口表完整；API 前缀明确 |
| A3 | 构建测试可引用 | 必过 | 【8】命令可直接填进 context-pack，非占位 |
| A4 | 权限变更有起点 | 有 auth 则必过 | 【10】机制类型 + 强制点 + 认证-鉴权字段交叉检查（有读源码比对，不是填表） |
| A5 | 风险不进实施建议 | 必过 | 【9】只记录不开药方；凭证已脱敏 |
| A6 | 可信度可消费 | 必过 | impact 可把【推断】整段当未确认；HEAD/commit 在【0】可做过期检测 |
| A7 | 主流程可 follow | 建议 | 【11】一条端到端链路，每跳有 file:line 或【推断】 |

**Agent 导航总评**：A1–A3、A5–A6 全过，且有 auth 时 A4 过 → 达标；缺【5】或【8】→ impact 仍要全仓扫 → 不达标。

### 2.2 风格可控（impact Phase 2/3 风格链）

| # | 检查项 | 级别 | 通过标准 |
|---|--------|------|----------|
| A8 | 【14】存在且非空壳 | 必过 | V7 通过；观察表有实质内容（≥2 条非标题行）。机械门禁只查"非空壳"，"≥5 轴有内容"是人审标准不是门禁 |
| A9 | 【14】只描述不规定 | 必过 | 无「必须/禁止/应该统一为」；是「Controller 返回 X」而非「应返回 X」 |
| A10 | 采样来源声明 | 必过 | 写清样本文件、覆盖模块、未覆盖模块（impact 据此降级【14】权重） |
| A11 | 【14】与【13】一致 | 建议 | 未采样模块在【13】或【14】未覆盖里对齐，不 silent 遗漏 |
| A12 | 与 `_style-rules.md` 分工清晰 | 建议 | 地图不写规范性规则；`_style-rules.md` 存在时 impact 优先级更高（地图不冲突） |

**Agent 风格总评**：A8–A10 全过 → 可控迭代基础具备；只有 V7 标题无表内容 → 空壳，不算达标。

### 2.3 impact 接手模拟（一条变更走通）

用下面 3 问做桌面推演（换真实需求词即可）：

| 推演问题 | 地图应能回答 |
|----------|-------------|
| S1：「改用户手机号字段」从哪进？ | 【3】【5】【6】指向 user 相关路由/model |
| S2：「改登录鉴权」会碰哪些文件？ | 【10】【11】+ 【9】已知 defect |
| S3：「新接口返回格式」跟现有一致吗？ | 【14】api_response 轴 + 可选 `_style-rules.md` |

三条都能从地图指到具体路径/风格 → Agent KPI 达标；只能答 S1 → 基本达标；都要重新 grep 全仓 → 不达标。

## 三、机械门禁（Script Gate，自动化必过）

| 检查 | 不过则 |
|------|--------|
| V1 行号真实 | 不可信，impact 不应引用 |
| V2 凭证脱敏 | 红线，禁止归档 |
| V3 SVG 安全 | SVG 已从模板移除，残留即 FAIL |
| V4 【13】非空 | 假装全覆盖（已修复：只匹配 `##` 标题行，不误匹配导航行） |
| V5 Mermaid 一致性 | 实线箭头源节点须在正文出现 |
| V6 facts 与磁盘一致 | 体量/目录造假 |
| V7 【14】存在且有内容 | 风格链断裂（已增强：空壳标题也 FAIL） |

```bash
python skills/pathfinder/scripts/pf_validate.py \
  change-impact/_project-map.md --repo-root <project-root>
# 期望：SUMMARY: 7 passed, 0 failed, 0 warnings
```

> **注意**：V7 只查「节存在 + 非空壳」，不查【14】内容质量 → A8–A10 必须人工或 L2 评审。

## 四、红线条款（任一即「不可交付」）

- 把【推断】写成【已核实】
- 凭证/密码明文进地图（含风险节）
- 父仓库 HEAD 冒充子项目 commit
- 缺【13】却暗示「已全项目覆盖」
- 【14】写「必须/禁止」当规范（越界成 RuleBlade/impact 的事）
- Script Gate exit ≠ 0 仍交付

## 五、快速打分卡（Review 时用）

| 维度 | 权重 | 问一句 |
|------|------|--------|
| 人类 30s | 25% | 同事能否一句话说清项目并知道怎么跑？ |
| 人类 5min | 15% | 能否列出 3 个核心功能和主要风险？ |
| Agent 导航 | 30% | impact 能否跳过全仓扫、直接切到变更模块？ |
| Agent 风格 | 20% | 【14】+ 采样声明能否约束「新代码像旧代码」？ |
| 机械门禁 | 10% | 7/7 PASS？ |

| 总分 | 结论 |
|------|------|
| ≥90% 且无红线 | 可交付，可作 impact L1 |
| 75–89% | 可交付但标注缺口（【13】写清待补） |
| <75% 或触红线 | 返工 |

---

## 六、建议用法

1. **Pathfinder 写完** → 先跑 Script Gate → 再按 §一 + §二 人工 10 分钟 review
2. **impact 第一次读地图** → 用 §2.3 推演验证 L1 是否够用；不够就「再挖 X」
3. **跑几次 impact 后** → 把确认的 style 规则沉淀到 `_style-rules.md`，地图【14】只作观察补充
