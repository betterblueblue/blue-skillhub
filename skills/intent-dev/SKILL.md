# Intent-Dev

## 目标

读取 `INTENT.md`、`PRD` 和工单文件，按依赖顺序逐个开发工单。每个工单用 TDD 循环开发：先写测试看到红灯，再写代码看到绿灯，最后重构。绿灯就是工单级验证通过的证据。全部工单开发完成后，交接给 intent-verify 做端到端验收。

本 skill 只负责开发和工单级验证：

- 不做需求澄清（那是 intent-anchor 的事）。
- 不写 PRD（那是 intent-prd 的事）。
- 不拆工单（那是 intent-issues 的事）。
- 不做端到端验收（那是 intent-verify 的事）。
- 不代替用户开发代码（用户写代码，skill 管流程和验证纪律）。

## 前置条件

1. 必须存在通过 `intent_validate.py` 校验的 `INTENT.md`。
2. 必须存在通过 `prd_validate.py` 校验的 `PRD`。
3. 必须存在通过 `issues_validate.py` 校验的工单文件。
4. 三者缺一不可。

## 验证等级

每条 Then 的验证结果使用以下等级：

| 等级 | 含义 | 证据要求 | 能否标 done |
|---|---|---|---|
| V0 | 未验证 | 无 | 不能 |
| V1 | 代码审查 | 代码位置和审查结论 | 不能单独标 done，必须有 V2+ |
| V2 | 实际运行验证通过 | 真实命令 + 输出摘要 | 可以 |

**铁律：标 V2 但没有真实命令输出 → 视为 V1 冒充，不允许标 done。**

## TDD 规则

按工单类型适用不同规则：

### 新功能工单：推荐 TDD（Red-Green-Refactor）

1. **Red**：根据工单验收标准的每条 Then，先写测试。运行测试，确认测试失败（因为功能还没写）。记录红灯输出。
2. **Green**：写代码让测试通过。运行测试，确认全绿。记录绿灯输出。
3. **Refactor**：重构代码，保持测试仍然全绿。运行测试确认。

绿灯输出就是 V2 证据。

### 修 bug 工单：必须先写复现测试

1. **Red**：先写一个能复现 bug 的测试。运行，确认测试失败。记录红灯输出。
2. **Green**：修代码。运行，确认测试通过。记录绿灯输出。

修 bug 不走 TDD（先修后补测试）→ 不允许标 done。

### 纯样式/配置工单：不要求 TDD

直接写代码，实际运行看结果。运行输出就是 V2 证据。

## 验证命令来源

**不预设任何技术栈。** 验证手段不限于构建+测试命令——任何能实际运行并产生真实输出的方式都算。

按以下优先级确定验证手段：

### 第一优先级：项目自带的构建/测试命令

1. **找配置文件**：在目标项目根目录查找，如 `package.json`、`pom.xml`、`build.gradle`、`go.mod`、`pyproject.toml`、`Cargo.toml`、`Makefile`、`CMakeLists.txt` 等。
2. **从文件内容提取命令**：找到配置文件后，读取其内容提取实际命令。例如：
   - `package.json` → 读 `scripts` 字段
   - `Makefile` → 读 target 列表
   - `go.mod` → `go test ./...`
   - `pom.xml` → 检查 surefire 插件配置
3. **实际运行**，捕获输出。

### 第二优先级：直接运行项目入口

找不到标准构建/测试配置时，尝试直接运行项目本身来验证行为：

- **脚本类项目**：直接运行入口文件，如 `python src/main.py`、`node index.js`，看输出是否符合预期。
- **编译型项目**：先编译再运行，如 `javac Main.java && java Main`、`go run main.go`。
- **服务类项目**：启动服务后验证，如 `python app.py` 启动后用 `curl` 测接口。

### 第三优先级：写最小验证脚本

项目没有入口、没有测试、也没法直接运行时，针对工单的 Then 写一个最小验证脚本：

- 验证函数输出 → 写几行调用代码，运行看结果。
- 验证文件生成 → 运行代码后检查文件是否存在且内容正确。
- 验证配置生效 → 加载配置，打印关键字段确认。

**这种验证脚本不是交付物，不写入项目，只用于获得 V2 证据。**

### 都不行 → V1

以上三种手段都无法实现时，降级为 V1（代码审查）。此时工单不能标 done，必须提示用户：

> 本工单无法自动验证（原因：{具体原因}）。请用户提供验证方式，或确认接受 V1 标 done。

用户确认接受 V1 标 done 时，在工单状态中记录"用户确认 V1 标 done"。

**禁止行为：** 在没有实际运行任何命令的情况下，凭空写命令名称和编造输出。

## 工作流程

### Phase 1：前置检查与项目探索

1. 确认 INTENT.md、PRD 和工单文件路径。
2. 读取三者全文。
3. 运行 `intent_validate.py`、`prd_validate.py` 和 `issues_validate.py` 确认通过。任一 FAIL 则停止。
4. **探索目标项目根目录**，按"验证命令来源"节的步骤查找构建/测试配置文件，从文件内容中提取实际命令。找不到配置文件则问用户。
5. 确定构建命令和测试命令。用户确认没有测试环境则标"V2 不可用，工单无法标 done"。
6. 从输入文件路径推导链路目录，DEV-RECORD 写入同一目录下的 `dev-record.md`。不创建目录、不写文件。
7. 如果 dev-record 已存在（跨会话恢复），读取现有记录，复述当前进度。

输出：确认后的文件路径、工单清单、构建/测试命令和当前进度。

### Phase 2：工单开发（TDD 循环）

**验证依据：工单文件里该工单的验收标准（Given/When/Then）。** 只验当前工单的 Then，不验整条用户路径。

对每个工单，按依赖顺序执行：

1. 用户声明开始某个工单。
2. 如果工单有 `Blocked by` 依赖，确认被依赖的工单已验证通过。未通过的不允许开始。
3. **按工单类型走 TDD 循环**：
   - **新功能工单**：
     - Red：根据工单的每条 Then 写测试，运行，确认失败，记录红灯输出。
     - Green：写代码，运行测试，确认全绿，记录绿灯输出。
     - Refactor：重构（如有），运行测试确认仍然全绿。
   - **修 bug 工单**：
     - Red：先写复现 bug 的测试，运行，确认失败，记录红灯输出。
     - Green：修代码，运行测试，确认全绿，记录绿灯输出。
   - **纯样式/配置工单**：
     - 直接写代码，实际运行看结果，记录输出。
4. **验证判定**：
   - 实际运行通过 → V2，附命令和输出摘要。
   - 实际运行失败 → 未通过，附失败原因和命令输出。
   - 无法运行任何验证 → V1（代码审查），附代码位置和审查结论。
5. 每条 Then 必须有明确的验证等级和证据。V2 必须附真实命令输出，不得仅凭"应该能通过"标注。
6. 所有 Then 达到 V2 以上 → 工单标 done，可进下一个工单。
7. 有未通过项或仅有 V1 → 要求修复或补测试环境，不得跳到下一个工单。
8. 每个工单验证完成后，追加写入 dev-record.md。

输出：更新后的 dev-record.md。

### Phase 3：交接

1. 确认所有工单都标 done。
2. 如果任一工单未 done，停止，不允许交接。
3. 运行 `dev_validate.py` 确认 dev-record 结构通过。
4. 给用户以下提示：

```text
开发阶段完成。intent-chain/{链路目录}/dev-record.md 记录了每个工单的 TDD 过程和验证结果。

下一步：使用 intent-verify 做端到端验收。提供以下文件：
- intent-chain/{链路目录}/intent.md
- intent-chain/{链路目录}/prd.md
- intent-chain/{链路目录}/issues.md
- intent-chain/{链路目录}/dev-record.md
```

## 强制规则

1. **三个前置文件必须存在且通过校验**：不通过则不进入开发。
2. **验证必须有真实输出**：可以是构建/测试命令、直接运行入口、或最小验证脚本，但不能只有代码审查。
3. **每个工单完成后必须运行验证**：不得只凭代码审查标 done。
4. **V2 必须附真实命令输出**：标 V2 但无输出 → 视为 V1 冒充，P0。
5. **每条 Then 必须有验证等级和证据**：V0/V1/V2，不得留空。
6. **修 bug 工单必须先写复现测试**：先修后补测试 → 不允许标 done。
7. **有未通过项的工单不得进下一个工单**：修复后重新验证。
8. **所有工单 done 后必须交接给 intent-verify**：不自己端到端验收。
9. **先确认再写文件**：在对话中展示验证结果草稿，得到确认后才写入。
10. **结构校验必须通过**：写入后运行 `dev_validate.py`。

## dev-record.md 必需章节

1. 开发概述（产品名称、前置文件路径、工单总数、构建命令、测试命令）
2. 每个工单的开发记录
   - 工单标题
   - 工单类型（新功能 / 修 bug / 纯样式配置）
   - TDD 过程
     - Red：测试内容 + 红灯输出摘要（纯样式配置工单写"不适用"）
     - Green：代码内容摘要 + 绿灯输出摘要
     - Refactor：重构内容（无则写"无"）
   - 验证结果（每条 Then 的等级、状态和证据）
   - 工单状态（done / 未通过）
3. 开发总结（工单总数、done 数、未通过数）

## 文件存放

最终文件放在目标项目根目录：

```text
intent-chain/{链路目录}/dev-record.md
```

- 链路目录由 intent-anchor 创建，DEV-RECORD 写入同一目录。
- 同一产品的修订覆盖原文件。

## 能力边界

Intent-Dev 能够：

- 探索目标项目，发现真实的构建和测试命令。
- 按 TDD 循环管理每个工单的开发过程。
- 实际运行验证命令，捕获真实输出。
- 根据命令输出判定每条 Then 的验证等级。
- 通过校验器检查 dev-record 的结构完整性和验证等级合规性。

Intent-Dev 做不到：

- 代替用户写代码和测试。
- 做端到端验收（那是 intent-verify 的事）。
- 强制用户修复未通过项（只阻止进下一个工单）。
- 验证不在工单验收标准中的内容。
