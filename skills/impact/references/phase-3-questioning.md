# Phase 3 & 3.5: 探索与定级详细规则

> 本文件包含 Phase 3（苏格拉底式探索）和 Phase 3.5（正式定级）的完整规则。SKILL.md 正文只保留核心结构与关键定级条件，详细规则见此。
> impact 与 impact-pro 通用；栈专属追问（Next.js Server Action、Nuxt API 等）由 profile 提供。

## 多轮收敛协议

- **light**：通常 0-1 轮，只问阻塞执行的关键问题；能从现有风格、代码约定或用户原话确认的，不追问。
- **full**：通常 1-3 轮，每轮最多 3 题；按风险从 DB/API/权限/兼容/回滚/验证逐层收敛。
- **高风险 full**：最多 5 轮；超过 5 轮仍不清晰时，停止追问，输出"已确认 / 未确认 / 建议默认 / 必须用户拍板"。

**每轮 <= 3 题，不是总问题数 <= 3**。每轮必须基于真实上下文，不泛泛而谈，并收敛一个决策层级。

问题不是越多越好。Agent 不追求一次问完所有问题，而是按风险分层逐轮收敛；用户可以要求先输出当前版本，但文档必须标注未确认项。涉及写操作、DDL/DML、API 契约、权限、状态机的未确认项，不得被默认值悄悄吞掉。

## 问题优先级

| 等级 | 定义 | 处理方式 |
|------|------|----------|
| P0 必问 | 不问就可能做错、破坏数据/API/权限/状态机或无法回滚 | 本轮优先问；未确认不得进入执行 |
| P1 应问 | 影响兼容性、设计质量、验证方案或用户体验 | full 中优先问；未确认必须入文档 |
| P2 可默认 | 可按项目现有风格、代码约定或常见默认给建议 | 说明依据，不阻塞文档输出 |
| P3 可延后 | 不影响需求/设计，可在实施前确认 | 放入实施前确认清单 |

## 维度分组（技术栈规则可覆盖）

> 维度定义见 `references/dimensions.md`。

| 组 | 维度 | 优先级 |
|----|------|--------|
| A | 数据库、代码、接口/契约 | 必问 |
| B | 配置、凭证/密钥、安全/权限 | 高 |
| C | 缓存、存储/文件、消息队列/事件 | 中 |
| D | 其余维度 | 低（按需） |

栈专属维度（Next.js 的 RSC、Go 的 goroutine、Python 的 async 语义等）由 profile 注入。

## 质量底线追问

当变更涉及功能的质量要求空白/模糊时追问：

> "关于 [功能]，有具体质量要求吗？比如导出格式、显示样式、交互细节，还是与现有同类功能保持一致即可？"

不追问就按现有同类功能风格实现，并在文档标注"按现有约定"。

## 风险靶向追问（按发现触发）

按 Phase 2 反向引用检查的命中情况，按风险类别触发对应追问：

- **DROP/RENAME**：发现 N 个文件引用，删除/重命名后会报错，是否需过渡期/别名？API 返回此字段，现有消费者是否中断？
- **多表外键**：[表A] 被 [B/C/D] 外键引用，是否级联？
- **存量数据**：该表 N 行，现有数据如何处理？是否需迁移脚本？
- **视图/存储过程**：N 个引用此表，是否同步更新？
- **业务逻辑**：Service `[X]` 用此字段做 [逻辑]，是否受影响？
- **缓存**：N 个缓存 Key 存此数据，失效策略？
- **消息队列**：变更会发布到 [Topic]，消费者是否受影响？
- **版本兼容**：N 个版本消费者，是否需同时更新？

## 停止条件

关键风险已覆盖；用户说「够了/输出吧」；超 5 轮未收敛则主动提议输出已确认信息，并把剩余问题分级写入未确认项。

## 维护注意

- 本文件的"高风险 full 5 轮上限"是软上限，证据不足时即使超过 5 轮也应在文档标注"未确认/建议默认/必须用户拍板"的分级清单。
- 改本文件任何追问模板必须同步检查模板 `templates/010-requirements.md` 是否有引用。

# Phase 3.5: 正式定级 + 确认

依据 Phase 2 发现和 Phase 3 澄清结果，由 Agent 基于证据先行建议档位，用户复核确认。定级不是按文件数量决定，而是按风险触发条件决定。

## 正式定级输出格式

```text
建议档位：[light/full]
允许 light 的证据：[路径/命令/schema/代码]
触发 full 的证据：[路径/命令/schema/代码，如发现则必须升 full；确认无触发项才写无]
未确认项：[缺失证据/需用户决策]
行为准则检查：[本任务规模对应的必检规则是否满足]
```

## 定级证据自洽性（强制）

定级输出的「触发 full 的证据」一行必须与 Phase 2/3 的分析内容一致，不得与分析自相矛盾：

- 分析节（背景分析 / 影响范围 / 存量数据 / 引用检查等）中**已经识别**的任何上述"必须 full"条件，必须在定级的「触发 full 的证据」里**逐条如实列出**，不得写"无"。
- **反例（禁止）**：影响范围节写了"admin 种子 status='2' 若 2=disabled 需回填为 1"（= 存量数据回填，命中"必须 full"），但定级「触发 full 的证据」却写"无"并建议 light —— 这是定级结论与自身分析证据矛盾，是不允许的。
- 出现矛盾时二选一，不得并存：① 升 full；② 回头修正分析（删掉或否定那条触发 full 的证据）。定级证据是结论的依据，不是事后粉饰。
- **未确认项 ≠ "无"**：某条件是否触发 full 尚未确认时，列入「未确认项」并按"证据不足 + 高风险区域"规则默认 full 或暂停补证据；不得借"未确认"在「触发 full 的证据」写"无"、再判 light 掩盖。

## 允许 light 的条件

同时满足以下条件，才可建议 light：

- 影响面局限：通常只涉及单页面、单组件、单 handler 文案、单测试断言、局部样式、局部配置展示
- 无结构变更：不改 DB schema、migration、实体字段、DTO 结构、OpenAPI/GraphQL 契约、generated client
- 无核心逻辑变更：不改权限、认证、支付、状态机、事务、缓存失效、消息队列、外部服务副作用
- 无破坏兼容：不删除字段/接口/路由，不重命名公开契约，不改变 HTTP status 或错误结构
- 验证完成简单：可通过局部 lint/test/E2E 或手动 UI 检查验证
- 证据充分：已找到相关文件和测试/命令；未确认项不会影响安全性和兼容性

典型 light：UI placeholder、按钮文案、toast 文案、footer 文案、局部样式/图标、API 自然语言 message、前端本地状态展示、文档或日志文案。

## 兼容性新增 API 响应字段

- 仅新增向后兼容的 API 响应字段，且旧消费者可忽略新字段、字段语义和类型有证据、无需 generated client/OpenAPI/SDK 同步时，可以建议 light。
- 只要出现删除/重命名字段、字段类型或语义变化、HTTP status 或错误结构变化、generated client/OpenAPI/SDK 需要同步、外部消费者不明、历史数据/缓存/持久化快照或前后端必须同步修改，必须建议 full 或先暂停补证据。
- 不得一边把消费者、generated client、OpenAPI/SDK 或持久化快照写成"未确认/不明"，一边正式建议 light；除非用户补证据或仓库内证据证明这些项不存在。
- light 涉及 API 返回字段时，必须在 light 摘要中填写"接口返回检查清单"。

## 必须 full 的条件

出现以下任何条件，默认建议 full：

- DB schema、migration、索引、唯一约束、外键、存量数据或回填
- API/DTO/OpenAPI/GraphQL 契约变更，或 generated client 需要再生成
- 权限、认证、支付、订单、状态机、审计、风控等高风险业务逻辑
- 跨模块、跨服务、前后端联动
- 缓存、消息队列、异步任务、文件存储、邮件/短信/第三方 API 等外部副作用
- 删除、重命名、DROP、批量替换、破坏兼容、迁移旧接口
- 引用检查发现 API、DB、权限、状态机、生成物、测试或外部消费者必须同步修改
- 证据不足但涉及 DB/API/权限/状态机等高风险区域

## 定级责任与对话

> "这次变更看起来是 **[light / full]**（理由：…）。light 产一页影响摘要后进入执行前检查；full 产三文档。确认走哪档？"

- Agent 负责初判和正式建议，必须给证据。
- 用户负责复核确认、补充业务决策或要求升档。
- 用户可以要求简化 full 的文档形式，但不能把已触发 full 的高风险变更降为跳过安全检查的 light。
- 用户可以要求 light 升级为 full，以获得三文档和更完整验证。

**退出条件**：用户随时说「直接改 / 别写文档 / 简化」，可以简化文档形式，但不能跳过写操作确认、破坏性变更影响发现、执行前检查和验证方案。

## 只分析 / 不写文件的最小响应契约

当用户要求"只分析""只定级""不要写文件""不落文档"时，仍必须输出最小证据结构，不能只给一句结论或"已完成"自我声明。

最小响应必须包含：

1. **上下文摘要**：直接相关文件/对象、引用检查结果、排除项。
2. **定级结论**：light/full，列出允许 light 的证据、触发 full 的证据、未确认项。
3. **风险与安全闸**：是否涉及 API/DB/权限/状态机/写操作/非 Git 回退方案/阻塞恢复。
4. **验证等级**：当前最高 V0/V1/V2/V3，已运行和未运行的验证及原因。
5. **下一步确认**：如果要执行，明确需要用户回复的 `确认 Step N`；如果本轮只读，写明"本轮未执行写操作"。

不得用"已完成四项任务""证据已回填""已完成"替代上述内容。

## 验证等级

所有执行结论必须标注验证等级：

| 等级 | 含义 |
|------|------|
| V0 | 未验证 |
| V1 | 静态验证：`rg`/读取代码/片段对照/文档检查通过 |
| V2 | 构建、编译、lint、类型检查或单元测试通过 |
| V3 | API/UI/DB/集成/E2E 或真实运行路径验证通过 |

无法运行验证命令时，不得写成"已完整验证"；必须写明当前最高等级、失败/缺失的命令、环境限制和运行时未验证项。

只读分析本身也可以达到 V1：如果已经读取文件、`rg` 搜索或做了片段对照，分析结论的验证等级应写 V1；尚未执行写操作时，实施状态可另写"未执行"。不要把"未写入/未运行时验证"误写成 V0。

## 破坏性请求保护

当用户要求「直接删」「全部替换」「不用分析」「删旧接口」「DROP/RENAME 表字段」「批量重构」时：

1. 不执行写操作。
2. 先只读搜索引用、消费者、schema、测试和回滚线索。
3. 回显破坏面、缺失证据和未确认项。
4. 至少追问兼容期、回滚、消费者、迁移策略。
5. 用户确认后仍按 Phase 5 逐项确认执行。
