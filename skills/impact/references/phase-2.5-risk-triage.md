# Phase 2.5: 初步风险预判详细规则

> 本文件包含 Phase 2.5 的完整规则。SKILL.md 正文只保留概要。

## 初步风险预判

依据 Phase 2 发现，只做初步风险预判，不最终定档：

- **可能 light**：局部文案、局部样式、单 handler message、单字段且无兼容/存量/外键/API 风险。
- **倾向 full**：DB schema、存量数据、外键/索引、API 契约、权限、状态机、缓存、MQ、跨模块、删除/重命名、破坏兼容。
- **证据不足**：涉及高风险区域但证据不完整，先补证据或在 Phase 3 追问。

初步预判输出：

```text
初步风险：[可能 light / 倾向 full / 证据不足]
已确认事实：[路径/命令/代码]
需要澄清：[最多 3 个问题]
```

如果已经发现 full 触发条件，必须标记"倾向 full"，并在 Phase 3 针对触发原因提问。用户可以要求简化输出形式，但不能把已触发 full 的高风险变更降成跳过证据、确认和验证的 light。

## 覆盖范围语义核查（定级前必做）

用户表述中出现"每次/所有/全部/任何/一律/每个"等全量词时，必须核实现有实现（如有）是否真的全覆盖——**不要因为"功能已存在"就判 light**。"已存在"不等于"已全覆盖"。

1. **提取全量词，确认用户要求的覆盖范围**：如"每次接口请求都要记录"= 所有接口（含登录、验证码等无注解接口）。
2. **打开现有实现代码，确认实际覆盖范围**：如 AOP 切面/中间件 `@annotation(xxx)` 或路由级 hook 只覆盖部分请求，未覆盖的不记录。
3. **两者有缺口 → 标记"倾向 full"**：覆盖范围缺口意味着需要补充实现（扩大覆盖面或新增逻辑），不是"零改动确认"。

典型模式：用户说"每次接口请求都记录耗时"，但现有日志中间件/切面只覆盖部分路由或注解接口 → 存在覆盖范围缺口 → 应定 full 而非 light。这个核查在 agent 内部静默执行。

## 现状核查门禁（Phase 2 出口强制检查）

Phase 2 上下文发现完成后、进入 Phase 3 前，必须对用户需求做现状核查判定。判定结果决定后续流程走向：

1. **已完整实现** → 强制 light 档，输出"零改动确认"摘要（附证据），跳过 Phase 3 苏格拉底探索和 full 文档流程。用户确认后直接结束。
   - 判定标准：用户需求中的每一项功能点都能在代码中找到对应实现，且有证据（`【已核实: file:line】`）。
   - 摘要中必须逐条列出：用户需求功能点 → 代码实现位置 → 证据。

2. **部分实现** → 继续 Phase 3，但必须在 Phase 3 中针对"缺口"部分提问，**不得将缺口直接标为 Out of Scope**。缺口必须列入"待确认项"，由用户决定是否纳入本次范围。

3. **未实现** → 正常走 Phase 2.5 → Phase 3 → Phase 3.5 流程。

现状核查输出格式：

```text
现状核查：[已完整实现 / 部分实现 / 未实现]
核查依据：[用户需求功能点 → 代码实现位置，逐条对照]
缺口：[部分实现时列出缺失项；已完整实现写"无"]
后续流程：[跳过 Phase 3 走 light / 继续 Phase 3 / 正常流程]
```

> **禁止**：发现功能已完整实现后仍走 full 档提出新增方案。这是过度设计，违反"简单优先"原则。已在 V5 盲测中验证：有 skill 组在功能已实现时仍选 full 档，导致总分反降 21 分。
