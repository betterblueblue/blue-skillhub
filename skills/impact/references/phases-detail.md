# Phase 3 & 3.5 详细规则

> 本文件包含 Phase 3（苏格拉底式探索）和 Phase 3.5（定级）的完整流程规则。SKILL.md 正文只保留核心结构，详细规则见此。

## Phase 3: 不确定项分类（提问前必做）

Phase 2.5 产出的不确定项，在进入苏格拉底式提问之前，必须先逐一分类。这一步解决的核心问题是：**用户可能不熟悉项目，无法回答技术细节问题。**

### 分类标准

| 类型 | 判断标准 | 示例 |
|------|----------|------|
| **代码可推断** | 答案可以从现有代码/配置/schema/迁移/测试中直接读出 | 现有路由用什么权限名、schema 有没有某字段、中间件读哪个 param、现有路由是否有 swagger 注释 |
| **业务需决策** | 答案不在代码中，是产品/业务/用户体验层面的选择 | 是否加唯一约束、是否需要短信验证码、错误消息文案、是否需要审计日志 |

### 处理方式

**代码可推断项**：
1. Agent 在 Phase 2/3 只读分析中自行查证
2. 标注 `【代码推断: file:line】` 并写出推断结论
3. **不列入待确认问题，不问用户**
4. 在 context-pack §7「已确认事实」中记录

**业务需决策项**：
1. 在苏格拉底式提问中问用户
2. 但必须附上"基于代码现状的默认建议"，让用户确认或纠正，而非从零拍板
3. 格式：`[问题] — 默认建议：[基于代码的推断]，理由：[代码证据]`
4. 用户未回答时，可标 `[假设]` 但必须标"待确认"

### 边界判定原则

当不确定一项是"代码可推断"还是"业务需决策"时，按以下原则判定：

- **如果代码中有先例/约定** → 代码可推断（对齐现有模式）
- **如果代码中无先例，但常见默认做法明确** → 代码可推断（用默认值 + 标注依据）
- **如果代码中无先例，且多种方案各有取舍** → 业务需决策
- **如果涉及业务规则、产品流程、用户体验** → 业务需决策

### 反例（禁止）

- ❌ 把"现有路由用什么权限名"列为待确认问题 — 这能从 `auth('manageUsers')` 读出来
- ❌ 把"路径 param 用 :id 还是 :userId"列为待确认问题 — 这能从 `req.params.userId` 读出来
- ❌ 把"是否补 Swagger 注释"列为待确认问题 — 这能从现有路由是否有注释块读出来
- ❌ 把"phone 是否唯一"直接列为待确认问题而不附默认建议 — 应该附上"代码现状：schema 无 @unique 先例，默认不加"

### 与「不替用户拍板」原则的关系

核心原则 #3 说"不替用户拍板"——指的是**业务决策**不替用户拍板。代码事实不是决策，是观察。把代码能回答的问题推给用户，不是"尊重用户决策权"，而是把本应由 Agent 承担的查证工作甩给了用户。这在用户熟悉项目时只是多余，在用户不熟悉项目时（Pathfinder 场景）直接导致流程卡住。

> **Pathfinder 场景适配**：Pathfinder 的用户画像是"刚接手项目、不熟悉代码"。如果 Agent 把"鉴权范围是什么""param 名怎么定"这类问题推给用户，用户无法回答。Step 3.0 确保这类问题由 Agent 通过代码查证消化，只在真正的业务岔路口才停下来。

## Phase 3: 多轮收敛协议

- **light**：通常 0-1 轮，只问阻塞执行的关键问题；能从现有风格、代码约定或用户原话确认的，不追问。
- **full**：通常 1-3 轮，每轮最多 3 题；按风险从 DB/API/权限/兼容/回滚/验证逐层收敛。
- **高风险 full**：最多 5 轮；超过 5 轮仍不清晰时，停止追问，输出"已确认 / 未确认 / 建议默认 / 必须用户拍板"。

**每轮 <= 3 题，不是总问题数 <= 3**。每轮必须基于真实上下文，不泛泛而谈，并收敛一个决策层级。

**多轮触发条件**：第 1 轮未覆盖所有 P0/P1 级问题（含用户原话模糊点 + 链路追踪发现的副作用风险点）时，必须进入第 2 轮，不得用"每轮 3 题"作为停止理由。

问题不是越多越好。Agent 不追求一次问完所有问题，而是按风险分层逐轮收敛；用户可以要求先输出当前版本，但文档必须标注未确认项。涉及写操作、DDL/DML、API 契约、权限、状态机的未确认项，不得被默认值悄悄吞掉。

## Phase 3: 问题优先级

| 等级 | 定义 | 处理方式 |
|------|------|----------|
| P0 必问 | 不问就可能做错、破坏数据/API/权限/状态机或无法回滚 | 本轮优先问；未确认不得进入执行 |
| P1 应问 | 影响兼容性、设计质量、验证方案或用户体验 | full 中优先问；light 可写入未确认项 |
| P2 可默认 | 可按项目现有风格、代码约定或常见默认给建议 | 说明依据，不阻塞文档输出 |
| P3 可延后 | 不影响需求/设计，可在实施前确认 | 放入实施前确认清单 |

## Phase 3: 维度分组（技术栈规则可覆盖）

| 组 | 维度 | 优先级 |
|----|------|--------|
| A | 数据库、代码、接口/契约 | 必问 |
| B | 配置、凭证/密钥、安全/权限 | 高 |
| C | 缓存、存储/文件、消息队列/事件 | 中 |
| D | 其余维度 | 低（按需） |

## Phase 3: 质量底线追问

变更涉及功能的质量要求空白/模糊时追问：

> "关于 [功能]，有具体质量要求吗？比如格式、样式、交互细节，还是与现有同类功能保持一致即可？"

## Phase 3: 风险靶向追问（按发现触发）

- **DROP/RENAME**：发现 N 个文件引用，删除/重命名后会报错，是否需过渡期/别名？
- **删除对外契约字段**：被删字段出现在 API response / SDK / 公共类型中，"彻底删除"（breaking change）还是"保留兼容桩（空数组/null）"（向后兼容）？这是业务岔路，不能自行决定保留兼容桩——必须交用户确认。
- **多表外键**：[表A] 被外键引用，是否级联？
- **存量数据**：该表 N 行，现有数据如何处理？是否需迁移脚本？
- **API 破坏**：发现 [path] 返回此字段，现有消费者是否中断？
- **缓存**：N 个缓存 Key 存此数据，失效策略？
- **消息队列**：变更会发布到 [Topic]，消费者是否受影响？
- **版本兼容**：N 个版本消费者，是否需同时更新？
- **证据不足**：未找到合法值/状态枚举/schema/test，是否提供约定或允许我继续查证？
- **DB 无权限**：当前只能从代码/迁移推断 schema，是否提供连接或接受未确认标注？

## Phase 3: 风格分歧检测（触发式）

当实施计划中的代码风格与已加载的风格来源不一致时，触发风格分歧追问。这不是每轮必问的维度，而是**检测到分歧才触发**。

### 触发条件（任一命中即触发）

- 实施计划中的新增/修改代码使用了与 `_project-map.md`【14】观察到的现状不同的风格模式
- `_style-rules.md` 中的强制规则在实施计划中未被遵循
- profile `style_axes` 提示的通用模式与项目实际代码不一致（运行时从代码确认发现偏差）

### 不触发的情况

- 没有 `_style-rules.md` 也没有【14】时，**不主动做风格分歧检测**（退回现有行为，只靠 profile `style_axes` + 运行时从代码确认）
- `_style-rules.md` 中的规则已在实施计划中遵循时，不追问

### 追问格式

> "现有 [文件/模块] 统一使用 [风格模式]（证据：`file:line`）。你这次新增的 [代码] 没有对齐这个模式。这是故意的，还是应该统一？"

### 用户回答后的处理

**用户回答"应该统一"时**：
1. 在实施计划中修正代码风格以对齐
2. 如果该规则不在 `_style-rules.md` 中 → 在 Phase 5 追加一个 Step 提议写入 `_style-rules.md`（需 `确认 Step N`），格式见 `templates/_style-rules.md` 的"积累记录"表

**用户回答"故意的"时**：
- 记录为已确认的设计决策，不追问

### 渐进积累的价值

风格规则不需要提前写全。第一次跑 impact 时可能没有 `_style-rules.md`，但在实际变更中发现的每一条风格分歧，经用户确认后都会追加写入 `_style-rules.md`。同一个项目多跑几次，规则越来越完整，后续变更自动遵循，不用重复问。

## Phase 3: 问题格式（强制规则 #13）

Phase 3 的每个问题必须是**带依据的选择题**，包含三要素：

1. **岔路选项 A / B**（各一句话，不能用"等"或"之类"模糊带过）
2. **为什么代码判不了**（引 `file:line` 说明两种走向在代码里都成立/都缺失）
3. **默认建议 + 选错的后果**（基于代码现状给出推荐选项，说明选错会发生什么）

### 合格示例

> 给文章加置顶功能时，置顶作用于哪个范围？
> - A. 全局置顶——所有 feed 的第一条
> - B. 按作者置顶——每个作者的 feed 内置顶
> 代码判不了：`src/routes/articles/routes.ts:42` 的 feed 查询没有 WHERE 过滤置顶字段，两种实现都不与现有代码冲突。
> 默认建议 A（全局置顶），理由：`package.json` 依赖里有 `feed-ranking` 中间件（`src/middleware/feed-ranking.ts:8`），全局排序改动更小。选 B 的后果：需要在每个 author 查询里加子查询，性能风险更高。

### 不合格示例

- ❌ "你想怎么做？" — 无依据的开放式问题
- ❌ "置顶功能要全局还是按作者？" — 缺选项描述、缺代码依据、缺默认建议
- ❌ "需要加索引吗？" — 代码可推断（看 schema 有没有 index）

### 执法方式

- **脚本层（V21）**：000-context-pack §7 已确认事实的每一条必须带来源标签——`【用户确认】`/`【代码推断: file:line】`/`【用户委托默认: …】`三者之一，缺标签即 FAIL。脚本验不了标签真假，但强制标注来源让编造变得显式可审计。
- **eval 层（歧义陷阱 case）**：补强 A 的提问精确率指标查问题相关性——是否命中 must_ask、是否误问 must_not_ask、是否假提问。

## Phase 3: 澄清被拒绝时的降级规则（强制规则 #12）

用户回"你定 / 随便 / 别问了 / 都行"时，Agent 不得静默选择，必须走四步：

### 步骤

1. **选代码现状支持的默认理解**（不是模型偏好）。从现有代码、schema、配置中找到最接近现状的选项作为默认。
2. **一句话回显**：`"按默认理解 X 执行，依据 <file:line>；如果你要的是 Y，现在说一声"`。回显必须包含依据和纠正机会，不能只写"按 X 执行"。
3. **写进 000-context-pack §7 已确认事实**，格式：`【用户委托默认: <YYYY-MM-DD> 选项X 依据<file:line>】`。错了能一眼追溯到假设，而不是散在代码里。
4. **高风险岔路不可委托**：岔路涉及 DB/API 契约/权限/enum/删除时，"你定"不构成授权，仍须用户显式选择——与强制规则 #2 高风险拦截对齐。此时回复：`"这个选择涉及 <DB/权限/契约>，我不能替你定。需要你选 A 或 B。"`

### 边界判定

| 用户回复 | 处理方式 |
|---------|---------|
| "你定" / "随便" / "都行" | 走四步降级（非高风险岔路） |
| "你定"（高风险岔路） | 拒绝委托，要求显式选择 |
| "按 X 来" | 接受为显式选择，记 `【用户确认】` |
| "不知道" / "不确定" / "你看着分析" | 走三步辅助决策（见下） |
| 沉默 / 不回答 | 不得假设，标为未确认项，继续只读分析 |

### 用户"不知道"时的辅助决策（三步）

用户说"我也不知道 / 不确定 / 你看着分析"时，和"你定"不同——用户不是不想花精力，而是缺乏判断依据。Agent 不能直接套用"你定"的降级流程，必须先帮用户建立判断基础。

1. **列出选项代价矩阵**：用表格形式展示每个选项的代价、风险和可逆性。

   | 选项 | 代价 | 风险 | 可逆性 |
   |------|------|------|--------|
   | A: ... | ... | ... | 可逆 / 不可逆 |
   | B: ... | ... | ... | 可逆 / 不可逆 |

2. **标出最安全的默认**：在选项中找出"可逆、不破坏现有数据、向后兼容"的那个，明确说"如果你还是拿不准，我建议选 A，因为……"。这一步和"你定"的降级区别在于：不是直接执行，而是给用户最后一根救命稻草，用户可以说"那就 A 吧"完成显式确认。

3. **用户仍无法决定时的处理**：
   - **非高风险岔路**：建议"先只做分析不执行变更，等你想清楚再做"或"分阶段：先做可逆步骤（比如加字段但不删旧字段），观察一段时间后再决定是否删旧字段"。
   - **高风险岔路**（DB/API 契约/权限/enum/删除）：不适用"最安全的默认"。必须回复 `"这个选择涉及 <DB/权限/契约>，我无法替你判断业务优先级。建议先不做变更，等你和团队确认后再来。"` 不能为了推进任务而猜一个。

## Phase 3: 停止条件

关键 P0/P1 风险已覆盖；用户说「够了/输出吧」；超 5 轮未收敛则主动提议输出已确认信息，并把剩余问题分级写入未确认项。

## Phase 3: 模糊点覆盖率自检（Phase 3 输出前必做）

当用户需求中存在模糊表述（如未指定具体数值、策略、范围、拦截点），agent 必须对每个模糊点做出处理。处理方式按问题优先级决定：

- **P0 模糊点**（不问就可能做错、破坏数据/API/权限）→ **必须在 Phase 3 提问**，不得假设
- **P1 模糊点**（影响兼容性/设计质量）→ **应该在 Phase 3 提问**；用户未回答时可标 `[假设]`，但必须同时标"待确认"
- **P2 模糊点**（可按项目现有风格/常见默认推断）→ 可直接标 `[假设]`，说明依据

提交确认前执行以下自检：

1. **回顾用户原话**：逐句扫描用户原始需求，标记每个模糊点（如"加个限制"未说限制多少、"不验证也能用"未说在哪拦截）
2. **回顾链路追踪**：扫描 000-context-pack.md §5 关键链路追踪表的"发现的二级影响"列，标记需要用户决策的副作用风险点（如"可能导致 XX 被滥用"、"不同环境可能需要不同值"）。这些不是用户原话中的模糊点，而是分析过程中发现的风险，同样需要确认或假设。
3. **对照处理清单**：模糊点 + 副作用风险点是否都有处理（已提问确认 / `[假设]` / 列入未确认项）？
4. **未覆盖的项**：补提问、补假设、或补未确认项——不能遗漏
5. **P0/P1 检查**：是否有 P0 或 P1 级问题（含原话模糊点和链路副作用）被直接假设而未提问？如果有，必须补提问，不得用假设替代

> 同一需求不同实现下假设数量可能差异很大，遗漏模糊点会导致分析偏差。此自检确保模糊点处理完整，且 P0 级模糊点不被假设带过。
>
> agent 容易在第 1 轮问满 3 题就停止，即使链路追踪已发现额外的副作用风险点（如"重发验证邮件可被无限调用"）也不再追问。步骤 2 确保这些发现回到澄清环节。

## Phase 3.5: 允许 light 的条件

同时满足以下条件，才可建议 light：

- 影响面局限：通常只涉及单页面、单组件、单 handler 文案、单测试断言、局部样式、局部配置展示
- 无结构变更：不改 DB schema、migration、实体字段、DTO 结构、OpenAPI/GraphQL 契约、generated client。**注意：仅调整读路径 SQL 的 WHERE 过滤或 SELECT 投影（如列表不返回 body 字段），不构成结构变更——这不改 schema、不改写路径、不改 DTO 定义，属于 light**
- 无核心逻辑变更：不改权限、认证、支付、状态机、事务、缓存失效、消息队列、外部服务副作用
- 无破坏兼容：不删除字段/接口/路由，不重命名公开契约，不改变 HTTP status 或错误结构
- 验证通过简单：可通过局部 lint/test/E2E 或手动 UI 检查验证
- 证据充分：已找到相关文件和测试/命令；未确认项不会影响安全性和兼容性

典型 light：UI placeholder、按钮文案、toast 文案、footer 文案、局部样式/图标、API 自然语言 message、前端本地状态展示、文档或日志文案、**读路径 SQL 过滤/投影调整（不改 schema、不改写路径、不改 DTO 结构）**。

## Phase 3.5: 兼容性新增响应字段

- 仅新增向后兼容的 API 响应字段，且旧消费者可忽略新字段、字段语义和类型有证据、无需 generated client/OpenAPI/SDK 同步时，可以建议 light。
- 只要出现删除/重命名字段、字段类型或语义变化、HTTP status 或错误结构变化、generated client/OpenAPI/SDK 需要同步、外部消费者不明、历史数据/缓存/持久化快照或前后端必须同步修改，必须建议 full 或先暂停补证据。
- 不得一边把消费者、generated client、OpenAPI/SDK 或持久化快照写成"未确认/不明"，一边正式建议 light；除非用户补证据或仓内证据证明这些项不存在。
- light 涉及 API 返回字段时，必须在 light 摘要中填写"接口返回检查清单"。

## Phase 3.5: 必须 full 的条件

出现任一条件，默认建议 full：

- DB schema、migration、索引、唯一约束、外键、存量数据或回填
- API/DTO/OpenAPI/GraphQL 契约变更，或 generated client 需要再生成
- 权限、认证、支付、订单、状态机、审计、风控等高风险业务逻辑
- 跨模块、跨服务、前后端联动、monorepo 多技术栈规则共同实施
- 缓存、消息队列、异步任务、文件存储、邮件/短信/第三方 API 等外部副作用
- 删除、重命名、DROP、批量替换、破坏兼容、迁移旧接口
- 引用检查发现 API、DB、权限、状态机、生成类型、测试或外部消费者必须同步修改
- 证据不足但涉及 DB/API/权限/状态机等高风险区域

典型 full：新增表字段并在接口返回；新增状态值并影响状态流转；前端新增入口调用后端 API；修改登录、支付、订单、邀请链接、邮件发送；Next.js Server Action 或 Nuxt server API 涉及写入、缓存刷新或外部依赖。

## Phase 3.5: 判档决策表（定级时必须输出）

定级时必须附判档决策表（SKILL.md Phase 3.5 已要求），把推理过程写出来让人审查：

| 用户原话关键词 | 现有实现覆盖范围 | 缺口 | 判档依据 |
|--------------|---------------|------|---------|
| "[关键词]" | [基于代码核查的覆盖范围] | [量化缺口，无写"无"] | light/full |

规则：
- 每行对应用户原话中的一个关键需求描述
- "现有实现覆盖范围"必须基于代码核查填写，不能写"全部覆盖"了事
- "缺口"无缺口写"无"，有缺口必须量化
- 如果"缺口"非空但判 light，表后必须附理由

### "已实现"场景的判档

当现状核查结果为"已完整实现"时，判档决策表应如下填写：

| 用户原话关键词 | 现有实现覆盖范围 | 缺口 | 判档依据 |
|--------------|---------------|------|---------|
| "[关键词]" | [基于代码核查的覆盖范围，附 `【已核实: file:line】`] | 无 | light（零改动确认） |

当现状核查结果为"部分实现"时，缺口行不得留空，且判档依据中必须写明缺口的处理方式（纳入本次 / 待确认）。

## Phase 3.5: 定级证据自洽性(强制)

定级输出的「触发 full 的证据」一行必须与 Phase 2/3 的分析内容一致,不得与分析自相矛盾:

- 分析节(背景分析 / 影响范围 / 存量数据 / 引用检查等)中**已经识别**的任何上述"必须 full"条件,必须在定级的「触发 full 的证据」里**逐条如实列出**,不得写"无"。
- **反例(禁止)**:影响范围节写了"admin 种子 status='2' 若 2=disabled 需回填为 1"(= 存量数据回填,命中"必须 full"),但定级「触发 full 的证据」却写"无"并建议 light —— 这是定级结论与自身分析证据矛盾,是不允许的。
- 出现矛盾时二选一,不得并存:① 升 full;② 回头修正分析(删掉或否定那条触发 full 的证据)。定级证据是结论的依据,不是事后粉饰。
- **未确认项 ≠ "无"**:某条件是否触发 full 尚未确认时,列入「未确认项」并按"证据不足 + 高风险区域"规则(见下文升降档)默认 full 或暂停补证据;不得借"未确认"在「触发 full 的证据」写"无"、再判 light 掩盖。

## Phase 3.5: 升降档规则

- 用户可以要求从 full 简化输出，但不能跳过：① `_active-state.md` 创建（恢复基础设施）；② 执行前检查；③ 破坏性变更发现；④ 写操作确认；⑤ 验证方案（详见硬规则 #10）
- 用户可以要求 light 升级为 full，以获得三文档和更完整验证
- 当证据不足且风险区域高，必须先按 full 或"暂停并补证据"处理，不能用 light 掩盖未知
- UI-only 变更如果发现 generated client、服务端写入、权限或外部副作用，应立即升级 full

定级责任：

- Agent 负责初判和正式建议，必须给证据
- 用户负责复核确认、补充业务决策或要求升档
- 用户可以要求简化 full 的文档形式，但不能把已触发 full 的高风险变更降为跳过安全检查的 light。简化模式的安全底线见硬规则 #10

## Phase 3.5: 只分析 / 不写文件的最小响应契约

用户要求"只分析""只定级""不要写文件""不落文档"时，仍必须输出最小证据结构，不能只给一句结论或"已通过"自我声明。

最小响应必须包含：

1. **技术栈和 profile**：探测证据、加载的 profile、generic 备选方案或多 profile 边界。
2. **上下文摘要**：直接相关文件/对象、引用检查结果、排除项。
3. **定级结论**：light/full，列出允许 light 的证据、触发 full 的证据、未确认项。
4. **风险与检查点**：是否涉及 API/DB/权限/状态机/写操作/非 Git 降级/阻塞恢复。
5. **验证等级**：当前最高 V0/V1/V2/V3，验证命令来自哪些项目证据，已运行和未运行的验证及原因。
6. **下一步确认**：若要执行，明确需要用户回复的 `确认 Step N`；若本轮只读，写明"本轮未执行写操作"。

不得用"已完成四项任务""证据已回填""已通过"替代上述内容。

## Phase 3.5: 验证等级

所有执行结论必须标注验证等级：

| 等级 | 含义 |
|------|------|
| V0 | 未验证 |
| V1 | 静态验证：`rg`/读取代码/片段对照/文档检查通过 |
| V2 | 构建、编译、lint、类型检查或单元测试通过 |
| V3 | API/UI/DB/集成/E2E 或真实运行路径验证通过 |

无法运行验证命令时，不得写成"已完整验证"；必须写明当前最高等级、失败/缺失的命令、环境限制和运行时未验证项。

只读分析本身也可以达到 V1：如果已经读取文件、`rg` 搜索或做了片段对照，分析结论的验证等级应写 V1；尚未执行写操作时，实施状态可另写"未执行"。不要把"未写入/未运行时验证"误写成 V0。
