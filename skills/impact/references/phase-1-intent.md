# Phase 1: 意图捕获详细规则

> 本文件包含 Phase 1 的长期目标模式判定规则和快速通道判定规则。SKILL.md 正文只保留概要。

## 长期目标模式

用户表达"继续""按进度文档往下做""逐步补齐""迁移/对齐/重构/长期实现"等意图，或 Agent 发现当前变更只是总目标中的一个 Step，且后续明显还有 backlog 时，进入长期目标模式。

典型场景包括：

- 迁移类：旧系统迁新系统、框架/ORM/前端版本迁移、跨语言实现迁移。
- 对齐类：让 A 项目的行为逐步对齐 B 项目。
- 重构类：分多轮拆核心模块、抽服务层、改权限模型、改状态机。
- 大功能接入类：在现有系统里接支付、审计、消息通知、权限框架等。
- 债务清理类：持续补测试、补接口契约、补历史数据兼容、消除 N+1。
- 多 Step 变更链：当前 Step 只是总目标的一小块，后面已有明确 backlog。

长期目标模式不代表每一步都 full。每个 Step 仍按证据独立判定 light/full，但必须维护：总目标、当前 Step、已完成 Step、待确认 Step、backlog、阻塞项、验证等级和运行时未验证项。

## 用户意图→设计假设映射

用户口语化需求中隐含的设计决策，Agent 应主动推断并标注为 `[假设]`，而不是默认保守方案。常见模式：

| 用户原话模式 | 隐含意图 | 常见错误推断 |
|-------------|---------|------------|
| "先存 X，想好了再 Y" | 创建时默认 X 状态，后续可主动转为 Y（需新增转换端点） | 默认 Y，只支持手动指定 X |
| "能不能不返回 Z" | 列表/摘要场景去掉 Z，详情仍需 Z | 全局去掉 Z |
| "每次都…太慢了" | 优化高频路径，不是全局重构 | 推翻现有架构 |
| "能不能加个 X 功能" | 新增能力，不改现有行为 | 改现有行为来兼容 |

推断后必须在 `010-requirements.md` 的假设清单中列出，并说明依据。推断错误成本可控（用户会纠正），保守错误（漏掉隐含意图）成本更高（交付不符合预期）。

> **歧义 ≠ 假设**：上表说的是从用户口语中推断隐含意图（"先存X想好了再Y" → 默认X状态），这是合理的代码推断。但如果用户的需求存在多种理解（如分析范围可以是某一层、跨层链路、全部相关数据结构），这属于歧义——agent 不得自行选择一种理解继续，必须停下来问用户。

## 快速通道（trivial 出口）

Phase 1 结束后预判（规模 + 不涉及高风险），Phase 2 完成后最终确认（影响范围 + 现状核查）。如果**全部满足**以下条件，可以跳过 Phase 2.5-3.5，直接从 Phase 2 进入 Phase 4 light 输出：

### 触发条件（全部满足）

- 任务规模：小（≤20 行改动）
- 不涉及 DB schema/migration/字段/索引/约束
- 不涉及 API 契约变更（新增/删除/重命名字段、改变 HTTP status 或错误结构）
- 不涉及权限/认证/状态机/enum/错误码/配置键
- 不涉及删除/重命名/DROP
- Phase 2 上下文发现中影响范围已完全确认（无未确认项——**agent 在 Phase 1 识别的歧义也属于未确认项，不得自行消解后再走快速通道**）
- 现状核查结果为"未实现"（不是"已完整实现"——那走零改动确认 light；"部分实现"不走快速通道，因为有缺口需要 Phase 3 澄清）

### 快速通道流程

```
Phase 1 意图捕获 → Phase 2 上下文发现（确认影响范围）→ Phase 4 light 文档输出 → Phase 5 执行
```

快速通道判定不是流程终点。判定成立后不得只回复"可以走快速通道"就停下；除非用户明确要求暂停，否则必须继续产出 Phase 4 light 文档、运行 `impact_validate.py`，再进入 Phase 5 执行前检查。

### 快速通道仍需执行

- Phase 2 完整上下文发现（技术栈检测 + 文件扫描 + 现状核查）
- Phase 4 light 文档输出（040-light.md）
- Phase 5 执行前检查（060-preflight.md）+ Step 确认
- 输出验证脚本（impact_validate.py）

### 快速通道跳过

- Phase 2.5 初步风险预判
- Phase 3 苏格拉底式探索
- Phase 3.5 正式定级（直接判 light）

### 安全保障

快速通道不降级安全规则：
- 仍需 `确认 Step N`
- 仍需执行前检查
- 仍需验证方案
- 如果 Phase 2 发现任何 full 触发条件，立即退出快速通道，走正常流程

> 快速通道的设计目标：让"改个错误提示""调个文案"这类简单变更不需要走 7 个 Phase。对弱模型尤其有效——给它一个明确的"这种情况可以简化"信号，比让它自己判断更可靠。
