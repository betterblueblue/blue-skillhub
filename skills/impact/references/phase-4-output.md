# Phase 4: 文档输出详细规则

> 本文件包含 Phase 4 的完整规则。SKILL.md 正文只保留目录结构概要。
> 进入 Phase 4 前必须 Read 本文件。

## 文档产出规则

- **context-pack** → `templates/000-context-pack.md`，light 和 full 模式均必产出。写入前必须获得用户确认（批量执行模式下自行假设并标注 `[假设]`）。context-pack 必须包含"关键链路追踪"节，对错误处理链、中间件管线、数据流路径、配置依赖做追踪式分析，识别二级影响——避免流程化检查漏掉错误处理链和中间件管线里的副作用
- **light** → `templates/040-light.md`，一页输出，确认后进入 Phase 5 执行前检查。light 模式必须完成"关键链路深度检查"（见模板），不跳过错误处理链、中间件管线和配置依赖的兼容性扫描。**light 模式不做方法级验证（§3.2 API 方法验证表仅 full 模式产出）**——light 的方法级风险由"关键链路深度检查"兜底：如果变更邻域调用了已有方法，关键链路检查必须覆盖该调用的异常行为
- **full** → `templates/010-requirements.md` → `020-design.md` → `030-implementation.md`，**每份确认后再出下一份**。提交 010 确认时，如果"模糊点处理清单"中有 `[假设]` 条目，必须单独列出请用户逐条确认/修正/拒绝
- **active-state** → `templates/_active-state.md`，在本需求目录第一次写入文档时创建；之后每次文档状态、pending Step、执行结果、验证等级或阻塞项变化都更新。该文件只能写在当前需求目录内，不能替代任何确认。

## 需求文档内容边界（010-requirements.md，生成后自检）

`010-requirements.md` 只写**业务需求**——做什么、为什么、怎么算做完。技术实现细节属于 `020-design.md` 和 `030-implementation.md` 的范畴。

**需求文档中禁止出现**：表名（如 `User`、`sys_user`）、类名/方法名（如 `userService.createUser`）、文件路径（如 `src/services/user.service.ts`）、代码片段、数据库字段定义（如 `String? @unique`、`varchar(100)`）、schema 变更方案、API 契约细节、依赖注入方式、ORM 用法等技术实现细节。

**需求文档中应该出现**：业务场景描述、功能需求清单、非功能需求（性能/安全/可用性/合规的业务指标）、业务约束、验收标准（业务视角）。

> 现状核查的结论可以写在背景里，但只写事实结论（如"密码已用 BCrypt 加密"），不贴代码、不引方法名、不写文件路径。

**生成后自检**（提交用户确认前，静默执行）：扫描 `010-requirements.md` 是否混入技术细节。发现表名/类名/方法名/文件路径/代码片段/字段类型定义 → 移到 `020-design.md`，需求文档只保留对应的业务描述。

## 设计文档的「代码风格报告」

每个风格项基于技术栈规则中的 `style_axes` 提取，附**完整、未截断**的参考代码片段。设计文档中的代码风格报告用自然语言段落描述，标签作为括号附注。标签保留供 agent 实施时参考，但主体是自然语言。

## 分析依据与待确认问题

每份文档必须包含：

- 已确认事实：路径、命令、DB 查询或测试证据
- 代码推断项：Phase 3 Step 3.0 分类为「代码可推断」的项，标 `【代码推断: file:line】`，已自行查证
- 未确认项：缺失文件、无权限、命令不可用、业务决策空白
- 不采用的推断：说明为什么不把猜测写入方案

涉及前端-only 变更时，不生成数据库实施步骤；涉及 DB 但无 DB 权限时，不声称已确认行数、索引、外键。

## 设计原则约束（写进设计文档）

- **简单优先**：不添加用户未要求的功能，不做推测性设计
- **精准修改**：只改必须改的文件，不"改进"相邻代码
- **质量底线**：最小改动 ≠ 最低质量；变更范围内功能须达项目同等质量标准
- **语义约定**：涉及 status/enum/常量/错误码/权限/配置键时，必须引用原定义；找不到则列入未确认项

## 实施文档代码引用预检（自动执行，提交确认前必做）

`030-implementation.md` 生成后、提交用户确认前，agent 内部静默执行以下检查。不需要用户额外确认；发现问题直接修正实施文档后再提交。

### 检查 1：API 方法名存在性验证

实施文档中引用的所有「已有代码库方法名」必须经 grep 验证存在，并将验证结果写入实施文档 §3.2「API 方法验证」表。模型可能凭训练数据中的常见命名模式编造方法名，此检查用于拦截。

1. 从实施文档 §3 执行步骤中提取所有形如 `对象.方法名(` 的调用
2. 排除标注为「新增」的方法（本次变更新定义的方法不在检查范围）
3. 对每个方法名执行 grep 搜索，确认在代码库中存在定义
4. 找到不存在的方法名 → 搜索正确方法名并修正实施文档
5. 无法确定正确方法名 → 在实施文档 §3.2 表中标注「待确认」，不得直接写死
6. **验证结果必须写入实施文档 §3.2「API 方法验证」表**，标注 `【已核实: path:line】`

### 检查 2：被调方法异常行为确认

实施文档中，如果调用了已有方法并将其返回值用于条件判断（null 检查、空值检查等），必须确认该方法的实际行为——它可能在异常情况下抛异常而不是返回 null。**验证结果必须写入实施文档 §3.2「API 方法验证」表的「异常行为」列。**

1. 打开方法定义源码，检查是否有 throw 语句
2. 方法在异常情况下抛异常而不是返回 null → 修正实施代码，使用 try-catch 替代 null 检查
3. 方法确实返回 null → 在 §3.2 表中记录"返回 null 不抛异常"

常见模式参考（必须打开源码确认，以下仅为提示）：

**命名约定模式**：
- `getXxxUser()`/`getLoginUser()` → 通常抛异常不返回 null（业务约定：调用方假定用户已登录）
- `findXxx()`/`selectXxxById()` → 通常返回 null/undefined（查询不到时）
- `matchXxx()`/`compareXxx()` → 通常返回 boolean（不抛异常）
- `hashPassword()`/`encryptXxx()` → 通常返回字符串（不抛异常，除非参数非法）

**Prisma ORM 模式**（逐一查 Prisma 文档或源码确认）：
- `prisma.xxx.findUnique()` → 找不到返回 null，不抛异常
- `prisma.xxx.findFirst()` → 找不到返回 null，不抛异常
- `prisma.xxx.findMany()` → 找不到返回空数组 []，不抛异常
- `prisma.xxx.create()` → 唯一约束冲突抛 P2002；不抛 P2025（P2025 是 update/delete 找不到记录）
- `prisma.xxx.update()` → 记录不存在抛 P2025
- `prisma.xxx.delete()` → 记录不存在抛 P2025
- `prisma.xxx.upsert()` → 不会抛 P2002/P2025（内部处理了存在性）

> 典型错误：把 `create()` 的异常码写成 P2025（实际是 P2002），或把 `hashPassword()` 标为"抛异常"（实际返回哈希字符串）。这些错误源于从记忆中推断而非读源码。

### 检查 3：判档表事实一致性

`030-implementation.md` 中的判档决策表（§10 或等效位置）引用的事实声明，必须与 `000-context-pack.md` §7「已确认事实」中的对应条目一致，不得出现自相矛盾。

1. 提取判档表中每行的「现有实现覆盖范围」和「缺口」列中的事实声明
2. 对每条事实声明，在 context-pack §7 中查找对应条目
3. 发现矛盾（如 §7 写"updateUserById 默认 keys 不含 password"，判档表写"updateUserById 默认 select 含 password"）→ 以 §7 为准修正判档表
4. §7 中无对应条目但判档表引用了新事实 → 补充到 §7，或标注「待核实」

> 典型场景：Agent 在 context-pack 中已精确区分了 getUserById（默认含 password）和 updateUserById（默认不含 password），但在写 030 判档表时从记忆中拷贝了错误版本。此检查确保两份文档的事实声明一致。

> 以上三项检查在 agent 内部执行，**验证结果必须写入实施文档 §3.2「API 方法验证」表**。问题修正后实施文档才提交用户确认；未修正的问题必须标注「待确认」，不得隐瞒。跳过 §3.2 表 = 提交不完整，不得提交确认。

## 改动完整性自检（实施文档提交确认前必做）

`030-implementation.md` 生成后、提交用户确认前，agent 内部静默执行：对照 `010-requirements.md` 的每条验收标准，逐条确认是否有对应的实施 Step。如果存在验收标准没有对应 Step 覆盖，必须补充 Step 或在"未确认项"中说明原因。此自检防止弱模型在 skill 流程中过早收尾——只产出 1 个 Step 而遗漏错误处理、测试更新等关键改动。

## 产出完整性自检（文档输出后、提交确认前执行）

如果本次定级为 full，检查以下文件是否都已产出：000/010/020/030。如果缺少任一文件：
1. 不得提交用户确认
2. 在已有产出的文件头部标注"产出不完整"
3. 输出提示："本次分析产出不完整，缺少 [文件列表]。可能原因：执行模型产出能力有限。建议：换更强模型重跑，或人工补齐缺失文档。"

## 行号引用格式与证据标签（强制）

文档中引用源码行号时使用以下格式：
- `` `path/to/file.ext:42` `` — 单行引用（反引号包裹）
- `` `path/to/file.ext:42-88` `` — 行范围引用

**证据标签（强制）**：所有结论性陈述必须附带证据标签 `【已核实: path/to/file.ext:line】`。仅描述性内容（如目录结构概述）可不附标签。未核实的陈述必须标注 `【待核实】`，不得以事实口吻陈述。

脚本闸门会随机抽查 3 条行号引用，验证文件和行号是否存在。

## 输出验证（Phase 4 文档输出后、提交用户确认前必须运行 — 不可跳过）

> **⛔ 此步骤为 Phase 4 的最后一步，跳过 = Phase 4 未完成。** 未运行验证脚本不得向用户提交确认。验证脚本会检查文件完整性、§3.2 方法验证表、§6 横切关注点表等必产出项。有 FAIL 项不得提交确认。
>
> **_active-state.md 的「最近验证」节必须填入实际运行的脚本命令和结果，不得写 N/A 或「未执行」。** 只读分析阶段也必须运行验证脚本——脚本只读检查文件内容，不修改项目代码。

运行 `python skills/impact/scripts/impact_validate.py <需求目录> --mode <light|full> --repo-root <项目根目录>` 完成输出验证。

| 检查项 | 编号 | 失败动作 |
|--------|------|---------|
| 文件完整性 + _active-state.md | V1 | FAIL（核心文件），WARN（_active-state.md） |
| 需求文档边界 | V2 | WARN，提示移到 020 |
| 方法名验证标记 | V3 | FAIL/WARN，缺 §3.2 表则阻止提交 |
| 判档决策表 | V4 | WARN，提示补表 |
| 凭证脱敏 | V5 | WARN，提示可能的凭证泄露 |
| 行号抽查 | V6 | WARN，偏差报 WARN |
| 判档合理性 | V7 | FAIL/WARN（见下） |
| 风格规则校验 | V8 | WARN，提示无法自动执行的规则 |
| 判档表事实一致性 | V9 | WARN，提示矛盾或未确认事实 |
| 横切关注点表 | V10 | FAIL（缺 §6），WARN（行数不足） |

V7 判档合理性检查包含三个子检查：
- **全量词覆盖门禁（FAIL）**：用户原话出现全量词（每次/所有/全部/任何/一律/每个）时，产出必须包含覆盖范围分析（覆盖范围/缺口）。缺失则 FAIL，阻止提交。
- **过度判档检测（WARN）**：Full 模式但实现步骤 ≤2 且必须修改文件 ≤3 且无全量词 → WARN，提示可能应为 Light。
- **判档不足检测（WARN）**：Light 模式但必须修改文件 >5 → WARN，提示可能应为 Full。

V9 判档表事实一致性检查包含两个子检查：
- **矛盾检测（WARN）**：判档决策表中引用的事实声明（含/不含、存在/不存在等）与 `000-context-pack.md` §7 已确认事实矛盾时报 WARN。典型场景：agent 在 context-pack 中正确区分了 `getUserById`（默认含 password）和 `updateUserById`（默认不含 password），但在写判档表时从记忆中拷贝了错误版本。
- **未确认事实检测（WARN）**：判档表引用了 §7 中没有对应条目的事实声明时报 WARN，提示补充到 §7 或标注「待核实」。

- 任何 FAIL 项：必须修复后重新运行，不得提交用户确认
- 任何 WARN 项：在提交确认时向用户说明，由用户决定是否接受
