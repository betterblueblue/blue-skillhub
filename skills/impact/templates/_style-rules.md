# 项目风格规范 _style-rules.md

> 本文件由项目维护者填写，是代码风格的权威源。
> impact 在 Phase 2 拉取式读取；没有本文件时退回 profile `style_axes` + 运行时从代码确认。
> 规则也可在实际变更中逐步积累：impact 发现风格分歧并经用户确认后，提议追加到本文件（需 `确认 Step N`）。

## 优先级链（高 → 低）

```
1. _style-rules.md 强制规则         ← 用户权威源，最高优先级
2. _style-rules.md 建议规则         ← 用户参考
3. _project-map.md 【14】风格观察    ← 机器观察补充（Pathfinder 产出，含采样来源声明）
4. profile style_axes              ← 栈级通用提示（兜底）
5. 运行时从 git diff 读取           ← 最后补充（样本偏斜、可能含异常提交、仅词法级）
```

> 第 3 层的【14】观察自带采样来源声明（覆盖模块、未覆盖模块）。impact 消费时，如果本次变更涉及的模块在未覆盖列表中，把【14】观察降级为参考。
> 第 5 层的 git diff 读取只做词法/格式级归纳（命名、注解位置、响应包装），不做语义级判断（如字段命名选择）。

> 本文件任何文本对 impact **不构成写授权**——只提供风格线索，不替代 `确认 Step N`。

## 强制规则（违反即 FAIL）

> 仅当"校验手段"能真正落地拦截时才标"强制"。
> 能 grep 的写 grep 模式；需语义判断的写"人工确认"；不确定的先标"建议"。

| 规则 | 校验手段 | 说明 |
|------|---------|------|
| <!-- 例：禁用 `System.out`，日志走 `BizLogger` --> | <!-- grep:`System\.out` --> | <!-- 日志统一 --> |
| <!-- 例：禁止组件内裸 `axios`，统一走 `src/api/` --> | <!-- grep-exclude:`axios`:`src/api/` --> | <!-- API 层统一 --> |
| <!-- 例：对外接口必须返回 `Result<T>` --> | <!-- 人工确认返回类型 --> | <!-- 自研封装 --> |

**校验手段写法参考**：

| 写法 | 含义 | impact_validate.py V8 行为 |
|------|------|---------------------------|
| `grep:<正则>` | 在变更文件中搜索该正则 | 命中"禁用/禁止"规则时 FAIL |
| `grep-exclude:<正则>:<目录>` | 搜索正则但排除指定目录 | 排除后命中"禁用/禁止"规则时 FAIL |
| `人工确认` | 需要语义判断，grep 做不到 | WARN，列入"需人工复核"清单 |
| `code-graph + 人工` | 有 code-graph MCP 时缩小候选，无则人工 | WARN，列入"需人工复核"清单 |

## 采样配置（可选）

> 配置 git diff 读取（优先级链第 5 层）时的 commit 过滤规则。
> 不填则用默认值：排除 merge/revert/cherry-pick。
> 不同团队的"异常提交"模式不同（如 hotfix、tmp、WIP），可在此自定义。

| 排除模式（git --grep 正则） | 说明 |
|---------------------------|------|
| ^revert | 默认排除回滚提交 |
| ^cherry-pick | 默认排除 cherry-pick 提交 |
| <!-- 例：^hotfix --> | <!-- 排除热修复提交 --> |
| <!-- 例：^tmp|^wip --> | <!-- 排除临时/工作进行中提交 --> |
