# ImpactRadar — 系统变更影响分析 Skill

## 这个 Skill 是干什么的

帮开发者在做任何系统变更前，通过苏格拉底式提问暴露盲区，最终产出三份完整版文档并协助执行。

不是替开发者做分析，而是帮开发者把"我想改一下数据库"这种模糊意图，变成清晰、可执行、可验证的变更方案。

## 核心能力

- **苏格拉底式提问** — 基于实际 schema 和代码发现上下文，针对性提问，不泛泛而谈
- **19个维度灵活覆盖** — 数据库、代码、配置、接口、缓存、消息队列、基础设施、前端、文档、监控、测试、日志、凭证、网络、存储、安全、版本兼容性、事务、依赖包
- **三文档逐级确认** — 需求文档 → 设计文档 → 实施文档，每份文档输出后等待确认再输出下一份
- **设计原则约束** — 简单优先、精准修改、质量底线三条原则内嵌于设计环节，最小改动不等于最低质量
- **统一输出目录** — 所有文档写入 `change-impact/{需求名称}/` 目录（000-需求文档.md / 100-设计文档.md / 200-实施文档.md / 300-测试脚本 / 900-执行记录.md）
- **自动测试验证** — 风格合规检查 + 单元测试自动跑，E2E 生成脚本手动触发；验证步骤必须包含正向用例 + 错误用例（TDD 思路）
- **测试失败 fix 循环** — 测试失败先诊断，Bug 自动修，设计问题停等用户

## 什么时候该用这个 Skill

不是所有变更都需要走这个流程。用错了反而费劲。

**适合用的场景：**

| 场景 | 原因 |
|------|------|
| 涉及 3 个以上维度的变更 | DB + 代码 + 前端 + 配置全套，维度越多漏掉的概率越高 |
| 需要多人协作的变更 | 文档是团队沟通的依据，干活的人需要知道"为什么改" |
| 涉及数据库 schema 变更 | 影响面广，需要系统性地想清楚再动手 |
| 需要合规/审计追溯的变更 | 文档和执行记录是追溯的凭证 |
| 重构核心模块 | 改动大、风险高，需要完整的设计文档指导实施 |
| 对接外部 API / 消息队列 | 涉及消费者协调，接口变更需要文档化 |

**不适合用的场景：**

| 场景 | 原因 |
|------|------|
| 单一字段的简单加字段 | 直接让 agent 干，5 分钟的事不要写 30 分钟文档 |
| 纯前端样式/文案调整 | 不涉及后端逻辑，维度少，文档价值低 |
| 个人项目快速原型 | 速度优先，文档是负担 |
| 明显的小 bug 修复 | 目标明确，不需要需求澄清 |

**判断标准**：如果说不清楚"这个变更影响多少人、涉及多少系统"，就先想清楚再动手。如果自己都清楚，直接让 agent 干。

## 触发方式

在 Claude Code 终端输入 `/impact` 即可激活。

## 典型对话流程

```
你：我想在 user 表加一个 phone 字段
↓
Skill 推断涉及维度（数据库+代码+接口），请确认
↓
Skill 发现 schema 上下文和代码引用
↓
Skill 提问：存量数据怎么处理？API 消费者会不会受影响？
↓
你：回答问题
↓
Skill 继续提问（或收敛）
↓
你：够了，输出文档
↓
Skill 写入需求文档 → change-impact/用户个性签名/000-需求文档.md
↓
你：确认
↓
Skill 写入设计文档 → change-impact/用户个性签名/100-设计文档.md
↓
你：确认
↓
Skill 写入实施文档 + E2E脚本 → change-impact/用户个性签名/200-实施文档.md + 300-测试脚本/
↓
你：确认，开始执行
↓
Skill：执行操作 1/5，确认吗？
↓
你：yes
↓
Skill：执行 → 风格合规检查 + 单元测试 + 正向用例 → 全部通过 → 写入 900-执行记录.md
↓
你：yes
↓
（错误用例验证脚本生成）
↓
Skill：执行操作 2/5，确认吗？
↓
...（每步操作后自动跑 A+B 测试）
↓
E2E 测试脚本已生成 → change-impact/用户个性签名/300-测试脚本/
```

## 真实测试用例

### ComplianceGuard — 自定义合规检查规则

**场景**：给 ComplianceGuard 加一个自定义添加合规检查规则的功能。

**Phase 2 发现**：Skill 探索出了完整的 ComplianceGuard 结构：
- 4个 `.drl` 规则文件（采购合同/项目审批/人事任命/通用公文）
- `DroolsEngine.java` 的 `loadRuleFromString` 方法
- Schema 类（ProcurementContractSchema 等，16个抽取字段）
- 双通道融合引擎（RuleEngineChannel + AgentChannel）
- API 端点（`/api/review` 系列）
- **注**：代码风格分析在此次测试中未实际执行，策略在后续 grill-me 讨论中完善

### 真实测试用例 2：RuoYi-Vue — 用户个性签名字段

**场景**：给 RuoYi-Vue 用户管理模块添加个性签名字段。

**第一次测试（评审 1/6 通过）**：
- E2E 测试脚本缺失
- 参考代码大量截断（"@Autowired priv..."），不可用
- 风格约束未嵌入实施步骤
- 维度漏掉安全/权限

**第二次测试（改进后）**：
- ✅ 主动发现了缓存同步问题（tokenService.setLoginUser 缓存刷新）
- ✅ 文档写入了 `change-impact/添加用户个性签名/` 目录
- ✅ 维度推断后主动问"有没有漏掉的维度"
- ✅ 5步实施步骤结构清晰
- ⚠️ 待验证：文档内容（格式、风格约束、E2E脚本、错误用例）需对照 REVIEW.md 逐项打分

**验证结果**：skill 框架在起作用，苏格拉底提问质量提升，Phase 2 发现能力增强。文档内容完整性待下一次测试验证。

## 迭代记录

### v1.0（初始版本）

第一个版本的定位是"帮用户想清楚再动手"，最终输出一个结构化 prompt 给下一个 AI agent 做影响分析。

**设计问题：**
- Skill 本身发现了所有风险和上下文，却把分析结论外包给没有上下文的下一个 agent
- 没有文档产出，只有 prompt 输出
- 没有执行能力

### v2.0（本次重构）

**主要改进：**

1. **去掉 prompt 输出，改出完整文档** — 需求文档、设计文档、实施文档，不再外包分析
2. **插入决策点** — 每个文档之间用户确认，不连续输出
3. **支持执行** — 用户确认后可以逐操作执行，每步确认一次
4. **19个维度覆盖** — 初始版只关注数据库，重构版覆盖所有相关维度
5. **维度灵活选择** — 不是每次变更都要覆盖全部19个，Skill 自动推断 + 用户确认
6. **代码风格分析** — Phase 2 自动分析项目代码风格，产出风格报告并嵌入设计文档，指导实施阶段不偏离项目规范
7. **测试验证框架** — 风格合规检查 + 单元测试自动跑，E2E 生成脚本手动触发；测试失败进入 fix 循环，自动修 Bug 或停等用户
8. **统一输出目录** — 所有文档写入 `change-impact/{需求名称}/` 目录，执行记录持续追加

### v2.1（TDD 验证思路）

**改进：**
- 验证步骤必须包含正向用例（功能正常）+ 错误用例（异常边界）
- 错误用例类型：边界值、空值NULL、格式校验、XSS注入安全风险、幂等性、缓存一致性
- E2E 测试脚本必须覆盖正向用例 + 错误用例
- 借鉴 TDD 思路：正向用例回答"功能做对了没有"，错误用例回答"异常情况下有没有崩溃"

**为什么需要错误用例**：正向用例验证功能正常工作，错误用例暴露的是边界条件和安全风险——这些是测试中最容易被忽略、线上最容易出问题的部分。

### v2.2（设计原则约束）

**改进：**
- Phase 2 内嵌最小改动原则：上下文地图只记录涉及文件，不主动扩大范围，不额外提问
- Phase 3 新增质量底线追问原则：用户描述质量要求模糊时（如导出格式、校验规则），主动追问
- 设计文档模板新增"设计原则约束"章节：简单优先 / 精准修改 / 质量底线三条原则

**核心逻辑**：最小改动 = 范围最小 ≠ 质量最低。Agent 不能以"要最小改动"为由做出低于项目现有标准的实现。

**确认的设计决策：**

| 决策点 | 最终选择 |
|--------|---------|
| 文档深度 | 完整版（接近 PRD 格式） |
| 文档数量 | 三份（需求+设计+实施） |
| 确认节奏 | 逐文档确认（不是一次性输出） |
| 维度选择 | Skill 自动判断 + 用户调整 |
| 执行模式 | 逐操作确认（不是自动执行） |

### 关键设计原则

- **不做假设** — 用工具发现真实上下文，不靠猜测
- **不替决策** — 问清楚风险，但最终决策权在用户
- **不退让执行** — 每步操作必须用户明确确认，不自动执行
- **按需覆盖** — 简单变更不需要全部19个维度

## 三个文档的关系

```
需求文档 ← 回答"为什么改"
    ↓
设计文档 ← 回答"怎么改"（schema/代码/配置方案）
    ↓
实施文档 ← 回答"怎么执行"（步骤/回滚/验证）
    ↓
执行阶段 ← 实际操作（每步确认）
```

需求文档是变更的"为什么"，设计文档是"怎么改"，实施文档是"怎么做"。三者缺一不可，顺序不能颠倒。

## 代码风格分析

Phase 2 在发现上下文时同步分析项目代码风格，产出结构化风格报告，嵌入设计文档供实施阶段参考。

### 两级策略

**基础层（始终执行）**：从最近 20 条 git commits diff 中自动提取风格特征。Git diff 本身包含命名、格式、注解位置的上下文，token 消耗极低（约 5K-15K tokens）。提取内容覆盖：命名规范、Lombok 使用、依赖注入方式、事务注解位置、异常处理风格、日志规范、响应格式、分层模式、DTO 使用情况。

**深入层（按维度触发）**：只对本次变更涉及的维度做深入分析，每个维度不超过 3 个文件。例如：变更涉及数据库则深入分析 Entity 注解风格，涉及接口则分析 Controller 响应格式。

**Git 不可用时**：降级为扫描目标模块下 6 个代表性文件（Service/Controller/Entity/Config 各 1-2 个），提取相同风格特征。

**Token 保护**：基础层上限 20 条 git diff 或 6 个文件，深入层每个维度上限 3 个文件，超出时提示用户。

### 风格报告结构

```markdown
### 基础风格（Git Diff 提取）
| 维度 | 发现 |
|------|------|
| 类命名 | PascalCase |
| 依赖注入 | Constructor injection |

### 深入分析（按需维度）
[本次变更涉及的维度细节]

### 风格约束
[实施阶段必须遵循的要点]
```

### 设计原则约束

设计文档包含"设计原则约束"章节，明确三大原则：
- 简单优先：不添加用户未要求的功能
- 精准修改：只改必须改的文件
- **质量底线**：最小改动不等于最低质量，变更范围内的功能必须达到项目同等质量标准

### 设计文档嵌入位置

风格报告作为独立章节插入设计文档，排在"变更细则"之后、"数据迁移策略"之前。原因：代码风格是"怎么改"的重要约束，实施阶段直接参考，不需要跨文档翻找。
