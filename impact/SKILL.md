# ImpactRadar — 苏格拉底式需求澄清与变更实施 Skill

## 目标

帮助开发者把模糊的变更意图变成精确的结构化需求，通过三个完整版文档（需求文档→设计文档→实施文档）逐级确认，最终协助执行。每步操作需用户明确确认后再执行。

## 核心原则

1. **先想清楚，再动手** — 通过靶向提问暴露盲区，不做推测性变更
2. **逐文档确认** — 每份文档输出后等待用户确认，再输出下一份
3. **逐操作确认** — 执行阶段每个操作前都询问用户，不自动执行
4. **维度按需覆盖** — 19个维度按变更类型灵活选择，不强制全覆盖

## 流程总览

```
Phase 1: 意图捕获
    ↓
Phase 2: 上下文发现（DBHub/DB MCP + 代码搜索）
    ↓
Phase 3: 苏格拉底式探索（选中的维度逐个提问）
    ↓
[决策点] 用户确认：是否进入文档输出
    ↓
Phase 4: 需求文档 → [确认] → 设计文档 → [确认] → 实施文档
    ↓
Phase 5: 执行（逐操作确认）
```

## Phase 1: 意图捕获

询问用户：

> "你想做什么变更？尽可能自然描述——涉及哪些表/字段/模块，或者只是一个模糊的想法都可以。"

等待用户回复后再进入 Phase 2。

## Phase 2: 上下文发现

根据用户意图，静默使用工具发现相关上下文。

### 前置检查：DBHub / DB MCP 可用性

先尝试调用 `search_objects` 搜索 "table"：
- **调用成功**：继续后续发现步骤
- **调用失败**：告知用户可降级为纯代码搜索模式，但缺少 schema 精准度，询问是否继续

### 数据库 Schema 发现

使用 DBHub 的 `search_objects` 和 `execute_sql`，或 DB MCP 的对应工具：

- 搜索表：`search_objects({ pattern: "表名关键词" })`
- 表结构：`SELECT COLUMN_NAME, DATA_TYPE, COLUMN_TYPE, IS_NULLABLE, COLUMN_DEFAULT, COLUMN_COMMENT FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_NAME = '目标表' AND TABLE_SCHEMA = DATABASE()`
- 外键引用：`SELECT TABLE_NAME, COLUMN_NAME, REFERENCED_TABLE_NAME, REFERENCED_COLUMN_NAME FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE WHERE REFERENCED_TABLE_SCHEMA = DATABASE() AND (REFERENCED_TABLE_NAME = '目标表' OR TABLE_NAME = '目标表')`
- 视图引用：`SELECT TABLE_NAME, VIEW_DEFINITION FROM INFORMATION_SCHEMA.VIEWS WHERE TABLE_SCHEMA = DATABASE() AND VIEW_DEFINITION LIKE '%目标表%'`
- 触发器引用：`SELECT TRIGGER_NAME, EVENT_OBJECT_TABLE, ACTION_STATEMENT FROM INFORMATION_SCHEMA.TRIGGERS WHERE EVENT_OBJECT_TABLE = '目标表' OR ACTION_STATEMENT LIKE '%目标表%'`
- 行数估算：`SELECT TABLE_ROWS FROM INFORMATION_SCHEMA.TABLES WHERE TABLE_NAME = '目标表' AND TABLE_SCHEMA = DATABASE()`

### 代码引用发现

使用 `Grep` 搜索：

1. MyBatis XML：`Grep` 搜索表名在 `**/*.xml`
2. JPA/Entity：`Grep` 搜索表名在 `**/*.java`
3. Service/Repository/Mapper：`Grep` 搜索在 `**/*Service*.java`、`**/*Repository*.java`、`**/*Mapper*.java`
4. Controller：`Grep` 搜索在 `**/*Controller*.java`
5. 配置文件：`Grep` 搜索 `**/application*.yml`、`**/application*.properties`
6. 对关键文件使用 `Read` 了解使用方式

### 代码风格分析

分两级进行，确保风格分析全面但不导致 token 爆炸。

#### 基础层：Git Diff 风格提取（优先）

检测是否使用 Git：
- **Git 可用**：执行 `git log --oneline -20 -- <module_path>` 获取最近 20 条 commits，对每条 commit 执行 `git show <hash>` 获取 diff。从 diff 中提取以下风格特征：
  - 类命名规范（PascalCase / 下划线）
  - 方法命名规范（camelCase）
  - Lombok 使用情况（@Data / @Getter / @Setter）
  - 依赖注入方式（field injection @Autowired vs constructor injection `private final XxxService`）
  - 事务注解位置（@Transactional 在 Service 还是 Controller）
  - 异常处理风格（自定义异常 vs 标准 RuntimeException）
  - 日志框架和占位符风格（Slf4j `{}` vs Log4j string concat）
  - API 响应格式（统一 Response 包装 vs 直接返回）
  - 分层模式（Controller → Service → Repository）
  - DTO/VO 使用情况

- **Git 不可用**：降级为文件扫描策略。扫描目标模块下代表性文件（Service/Controller/Entity/Config 各 1-2 个），从中提取上述风格特征。同时扫描 `.editorconfig`、`checkstyle`、`pmd` 等风格配置文件。

**Token 保护**：基础层扫描上限为 20 条 git diff 或 6 个文件，超出时提示用户。

#### 深入层：按维度按需分析

只对本次变更涉及的维度做深入分析，每个维度分析不超过 3 个文件：

| 维度 | 分析范围 | 提取内容 |
|------|---------|---------|
| 代码 | Service + Controller 各 1-2 个文件 | 分层模式、异常处理、参数校验、DI 方式 |
| 数据库 | Entity 类 + Repository 接口 | JPA 注解风格、字段映射、查询方法命名 |
| 接口/契约 | Controller 类 | 响应格式、路径规范、参数接收方式 |
| 配置 | application.yml + 配置类 | 配置项命名、读取方式 |
| 日志 | Service 层文件 | 日志级别选择、占位符风格 |
| 事务 | Service 类 | @Transactional 位置和参数 |
| 安全/权限 | Controller + Service | 鉴权注解位置 |
| 测试/用例 | 现有测试文件 | 测试类/方法命名、断言风格 |
| 包结构 | 模块根目录 | 包层级深度、目录组织方式 |
| 空值处理 | Service 层 | null 检查风格、Optional 使用频率 |
| 常量定义 | 枚举或常量类 | 枚举 vs 常量类 vs 接口常量 |
| 时间处理 | Entity + Service | LocalDateTime vs Date、时区处理 |

分析结果输出为结构化风格报告（见设计文档模板）。

## Phase 3: 苏格拉底式探索

基于选中的维度分组提问，每次 2-3 个问题。

### 维度分组

将 19 个维度分为 4 组，减少提问轮数：

| 组 | 包含维度 | 优先级 |
|----|---------|--------|
| A | 数据库、代码、接口/契约 | 必问 |
| B | 配置、凭证/密钥、安全/权限 | 高 |
| C | 缓存、存储/文件、消息队列/事件 | 中 |
| D | 基础设施、前端、网络/路由、文档/注释、监控/告警、测试/用例、日志/埋点、版本兼容性、事务/一致性、依赖包/SDK | 低（按需问） |

### 提问原则

- 基于实际发现的上下文提问，不泛泛而谈
- 一次不超过 3 个问题
- 用户回答后，根据回答内容决定下一轮方向
- 高影响维度的隐藏风险优先问清楚

### 常见风险问题模板

**DROP / RENAME 类**：
- "发现 [N] 个代码文件引用此字段，删除/重命名后这些代码会报错。是否需要过渡期或别名？"
- "API端点 [path] 返回此字段，现有消费者是否会中断？"

**涉及多表外键**：
- "发现 [表A] 被 [表B, C, D] 通过外键引用。变更是否需要级联到这些表？"

**有存量数据**：
- "该表有 [N] 行数据，变更后现有数据如何处理？是否需要迁移脚本？"

**有视图/存储过程**：
- "发现 [N] 个视图/存储过程引用此表。变更后是否需要同步更新？"

**有业务逻辑**：
- "Service层 [X] 使用此字段做 [业务逻辑]。变更是否影响此逻辑？"

**有缓存**：
- "发现 [N] 个缓存 Key 存储此数据。变更后缓存失效策略是什么？"

**有消息队列**：
- "发现此数据变更会发布到 [Topic]。消费者是否会受影响？"

**版本兼容性**：
- "此 API 有 [N] 个版本消费者。变更是否需要同时更新？"

### 停止条件

- 用户回答已覆盖所有选中维度的关键风险
- 用户主动说"够了"、"输出吧"、"整理一下"
- 超过 5 轮仍未收敛时，主动提议输出当前已确认的信息

### 决策点

苏格拉底探索完成后，询问：

> "需求已经比较清晰了。现在输出需求文档？确认后写入 change-impact/{需求名称}/000-需求文档.md，然后继续生成设计文档和实施文档。"

## Phase 4: 文档输出

三个文档按顺序逐个写入 `change-impact/{需求名称}/` 目录，每份文档需用户确认后再输出下一份。

**目录结构：**

```
change-impact/
└── {需求名称}/
    ├── 000-需求文档.md
    ├── 100-设计文档.md
    ├── 200-实施文档.md
    ├── 300-测试脚本/
    │   └── playwright-xxx.spec.js      # E2E 测试脚本
    └── 900-执行记录.md                  # 每步操作的时间戳记录
```

**文件名合规化**：需求名称转为文件夹名时，空格→下划线，特殊字符去除，最大长度 50 字符。

## Phase 5: 执行与验证

### 测试与验证框架

每步操作执行完成后，进入测试验证阶段。分两类测试：

**A. 风格合规检查（自动执行）**

执行 `grep` / `Bash` 命令验证代码风格符合设计文档约束：

```
- [ ] 新增类使用 Constructor injection（grep "@Autowired private" 应返回空）
- [ ] 日志使用 Slf4j + {} 占位符（grep "logger.*\+ " 应返回空）
- [ ] 异常使用现有异常体系（不新增 CustomXxxException 类）
- [ ] Entity 注解在字段上（不在 getter 上）
- [ ] 分层符合规范（Controller 只调 Service，不直接调 Repository）
```

每项检查附具体命令和预期结果。全部通过后继续下一步。

**B. 单元测试（自动执行）**

执行 `mvn test` 或 `gradle test`，确保新增代码不破坏现有测试。

**C. 正向用例 + 错误用例（自动执行）**

验证步骤必须同时包含正向用例（功能正常工作）和错误用例（异常边界处理）。错误用例参考 TDD 思路，测试以下几类：

| 错误用例类型 | 验证内容 |
|------------|---------|
| 边界值 | 字段最大长度+1、超出范围的值 |
| 空值/默认值 | NULL 提交、必填项缺失 |
| 格式校验 | 非法格式（手机号格式/邮箱格式/特殊字符） |
| 安全 | XSS 注入、SQL 注入风险 |
| 幂等性 | 重复提交是否安全 |
| 缓存一致性 | DB 成功但缓存失败时的处理 |

正向用例 + 错误用例构成完整的测试覆盖。

**C. E2E 测试（必须生成脚本，手动触发）**

在 `change-impact/{需求名称}/300-测试脚本/` 目录下**必须**生成 Playwright 测试脚本（不是可选项）。脚本包含：
- 启动被测服务的命令（如 `mvn spring-boot:run` 或 npm script）
- Playwright 测试用例：**正向用例 + 错误用例都要覆盖**

### 测试失败处理

测试失败时，Skill 自动进入 fix 循环：

```
测试失败
    ↓
Skill 分析失败原因
    ↓
是 Bug → Skill 自动修复 → 重新跑测试
是设计问题 → 停止，输出分析结果 → 等用户决定
    ↓
全部通过 → 继续下一步骤
```

Skill 根据失败信息判断类型：
- **编译错误 / 测试断言失败** → Bug，自动修复
- **测试超时 / 服务启动失败 / 端口占用** → 环境/基础设施问题，停止等待用户处理
- **风格检查失败** → 自动修复（按设计文档 §4 的约束修正代码）

## 行为准则

- **绝不自己做影响分析** — 任务是帮用户澄清需求，不是替用户做决策
- **始终基于实际 schema 和代码提问** — 用工具发现真实上下文，不靠猜测
- **输出语言跟随用户** — 用户用中文就用中文，用户用英文就用英文
- **每个文档之间等待用户确认** — 不连续输出多份文档
- **每个操作之前等待用户确认** — 不自动执行任何操作
- **维度按需选择** — 不强制覆盖全部 19 个维度
- **测试失败先诊断** — 不盲目回滚，先分析是 Bug 还是设计问题，Bug 自动修，设计问题停等用户
